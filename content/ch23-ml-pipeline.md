# ML 프로젝트 파이프라인

머신러닝 논문을 읽으면, 모든 것이 깔끔하게 정리되어 있다. 데이터가 표로 정리되어 있고, 모델이 논리적으로 선택되었고, 결과가 예쁜 그래프로 제시된다. 마치 처음부터 끝까지 순탄하게 진행된 것처럼 보인다.

현실은 전혀 다르다. 데이터에 결측치가 가득하고, 어떤 모델을 써야 할지 모르겠고, 첫 번째 모델은 전혀 작동하지 않는다. 코드를 고치고, 데이터를 다시 정리하고, 하이퍼파라미터를 수십 번 바꿔가며, 수 주간의 시행착오 끝에 "논문에 쓸 만한" 결과가 나온다. 이 지저분한 과정을 체계적으로 관리하는 것이 ML 프로젝트 파이프라인이다.

이 장에서는 ML 프로젝트의 전체 흐름을 단계별로 설명한다. 각 단계에서 무엇을 해야 하는지, 어떤 실수를 피해야 하는지, 어떤 도구를 쓸 수 있는지를 구체적으로 다룬다. 특히 공학 연구에서 자주 발생하는 문제, 즉 소규모 데이터, 물리적 제약, 재현성에 초점을 맞춘다.

## 전체 파이프라인 개관

ML 프로젝트는 대략 여덟 단계로 구성된다. 각 단계는 순서대로 진행되지만, 실제로는 이전 단계로 돌아가는 반복(iteration)이 빈번하다. 모델 결과가 안 좋으면 전처리를 다시 하고, 전처리를 바꿔도 안 되면 데이터를 추가 수집하고, 데이터를 추가해도 안 되면 문제 정의 자체를 수정한다.

## 데이터 수집

데이터 없이는 ML을 할 수 없다. 이것은 자명하다. 그러나 "어떤 데이터를, 얼마나, 어떻게 모을 것인가"는 자명하지 않다. 데이터 수집은 ML 프로젝트에서 가장 먼저 부딪히는 현실적 문제이며, 많은 프로젝트가 이 단계에서 좌초한다.

### 데이터 소스별 특징

**직접 실험.** 가장 신뢰할 수 있는 데이터 소스다. 실험 조건을 직접 통제하고, 측정 방법을 알고 있으므로 데이터의 품질을 보장할 수 있다. 다만 비용과 시간이 문제다. 콘크리트 시편 하나의 28일 강도를 측정하려면, 시편 제작 → 28일 양생 → 시험까지 최소 한 달이 걸린다. 시편 100개를 만들려면 재료비, 인건비, 시험비가 상당하다. 현실적으로 실험 데이터만으로 수천 개의 데이터를 확보하기는 어렵다.

**시뮬레이션 생성.** 유한요소 해석(FEA), 전산유체역학(CFD), 분자 동역학(MD) 등으로 가상 실험 데이터를 생성한다. 실험보다 훨씬 빠르고 저렴하게 대량의 데이터를 만들 수 있다. 입력 변수를 체계적으로 변화시키면서(Design of Experiments) 시뮬레이션을 돌리면, 입력-출력 관계를 효율적으로 탐색할 수 있다.

단, 시뮬레이션 데이터는 시뮬레이션 모델의 정확도에 종속된다. 시뮬레이션이 현실을 완벽하게 반영하지 못하면, 그 위에 학습된 ML 모델도 현실과 괴리가 생긴다. 가능하면 일부 실험 데이터로 시뮬레이션 모델을 검증한 후에 대량의 시뮬레이션 데이터를 생성하는 것이 바람직하다.

**공개 데이터셋.** 연구 커뮤니티에서 공유하는 데이터셋이다. UCI Machine Learning Repository, Kaggle, 논문의 supplementary data, GitHub 저장소 등에서 찾을 수 있다. 무료이고 즉시 사용 가능하며, 다른 연구와 비교가 용이하다는 장점이 있다. 예를 들어, UCI의 "Concrete Compressive Strength" 데이터셋은 콘크리트 ML 연구에서 가장 많이 사용되는 벤치마크 데이터다.

그렇지만 공개 데이터셋이 자기 연구 문제와 정확히 맞지 않을 수 있다. 변수가 다르거나, 측정 조건이 다르거나, 데이터의 범위가 제한적일 수 있다. 공개 데이터셋만으로 연구를 하면 "새로운 기여"가 제한될 수 있으므로, 자기만의 데이터를 추가하는 것이 좋다.

**문헌 데이터 수집.** 기존 논문에 보고된 실험 결과를 표에서 수동으로 수집하는 방법이다. 공학 분야, 특히 재료/구조 분야에서 매우 흔하다. 수십 편의 논문에서 데이터를 모아 수백~수천 개의 데이터셋을 구축할 수 있다.

이 방법의 장점은 비용이 거의 들지 않는다는 것이고, 단점은 데이터의 일관성을 보장하기 어렵다는 것이다. 논문마다 실험 조건, 측정 방법, 보고 형식이 다르다. 같은 물성이라도 논문 A에서는 ASTM 기준, 논문 B에서는 KS 기준으로 측정했을 수 있다. 이런 차이를 주의 깊게 처리해야 한다.

### 데이터가 없을 때의 대안

데이터가 전혀 없는 상황도 있다. 이때의 대안은 다음과 같다.

**데이터 증강(Data Augmentation)**: 기존 데이터에 변환(회전, 반전, 잡음 추가 등)을 적용하여 데이터를 늘린다. 이미지 데이터에서 특히 효과적이다.

**전이 학습(Transfer Learning)**: 관련 분야의 대규모 데이터에서 사전 학습된 모델을 자기 데이터에 미세 조정한다. 적은 데이터로도 높은 성능을 달성할 수 있다.

**능동 학습(Active Learning)**: 모델이 가장 불확실한 데이터 포인트를 선택하여 추가 실험/시뮬레이션을 수행한다. 최소한의 데이터로 최대 성능을 달성하는 전략이다. Gaussian Process와 결합하면 강력하다.

**합성 데이터 생성**: GAN(Generative Adversarial Network)이나 VAE(Variational Autoencoder)로 인공 데이터를 생성한다. 다만 합성 데이터의 품질을 검증하는 것이 어렵고, 물리적으로 비현실적인 데이터가 생성될 수 있으므로 주의가 필요하다.

## EDA와 전처리

EDA(Exploratory Data Analysis, 탐색적 데이터 분석)와 전처리는 ML 프로젝트에서 가장 과소평가되지만 가장 중요한 단계다. 이 단계를 대충 하면 아무리 좋은 모델을 써도 성능이 안 나온다. "전처리가 성능의 80%를 좌우한다"는 말은 과장이 아니다.

### EDA: 데이터를 눈으로 보라

데이터를 모델에 넣기 전에, 반드시 데이터를 "눈으로" 봐야 한다. 통계 요약(평균, 표준편차, 최솟값, 최댓값)만으로는 부족하다. 시각화가 필수적이다.

**히스토그램**: 각 변수의 분포를 확인한다. 정규분포를 따르는가? 한쪽으로 치우쳐 있는가(skewed)? 이상한 봉우리(peak)가 있는가? 분포가 심하게 치우쳐 있으면 로그 변환이나 Box-Cox 변환을 고려한다.

**산점도(Scatter Plot)**: 입력 변수와 출력 변수의 관계를 시각적으로 확인한다. 선형 관계인가? 비선형인가? 관계가 없는 변수가 있는가? 이 단계에서 "이 문제에 ML을 적용할 수 있겠다"라는 직관이 생긴다.

**상관관계 행렬(Correlation Matrix)**: 변수 간 상관관계를 한눈에 보여준다. 두 입력 변수 간 상관관계가 매우 높으면(>0.9), 다중공선성(multicollinearity) 문제가 발생할 수 있다. 하나를 제거하거나 PCA로 차원을 축소하는 것을 고려한다.

**박스 플롯(Box Plot)**: 이상치를 시각적으로 식별한다. 상자 밖의 점들이 이상치 후보다. 각 이상치가 물리적으로 가능한 값인지 도메인 지식으로 판단한다.

### 결측치 처리

실제 데이터에는 빈 칸이 있기 마련이다. 센서가 고장나서, 실험이 중단되어서, 기록이 누락되어서. 이유는 다양하다. 결측치를 어떻게 처리하느냐가 모델 성능에 영향을 미친다.

**삭제(Deletion)**: 결측치가 있는 행을 삭제한다. 결측 비율이 낮고(<5%), 데이터가 충분할 때 적합하다. 간단하나, 데이터 손실이 발생한다.

**대체(Imputation)**: 결측치를 추정값으로 채운다. 평균/중앙값 대체(가장 간단), 그룹별 평균 대체(범주 변수별), K-NN 기반 대체(유사한 데이터의 값 사용), 회귀 기반 대체(다른 변수로 예측) 등의 방법이 있다.

결측치가 랜덤으로 발생했는지(MCAR), 다른 관측 변수와 관련이 있는지(MAR), 결측값 자체와 관련이 있는지(MNAR)에 따라 적절한 방법이 다르다. 공학 데이터에서는 센서 고장(MCAR에 가까움)이나 측정 범위 초과(MNAR)가 흔하다.

### 정규화 (Scaling)

입력 변수의 스케일(단위, 범위)이 다르면, 일부 알고리즘의 성능이 떨어진다. 예를 들어, "강도"가 20-60 MPa 범위이고 "길이"가 0.001-0.01 m 범위이면, 스케일이 수천 배 차이난다. 이때 경사하강법 기반 알고리즘(신경망 등)은 큰 스케일의 변수에 의해 학습이 지배되어, 작은 스케일 변수의 영향을 제대로 학습하지 못한다.

**MinMaxScaler**: 모든 값을 0과 1 사이로 변환한다. `x_scaled = (x - x_min) / (x_max - x_min)`. 분포의 형태를 유지하나, 이상치에 민감하다.

**StandardScaler**: 평균을 0, 표준편차를 1로 변환한다. `x_scaled = (x - mean) / std`. 이상치에 상대적으로 덜 민감하다. 대부분의 경우 StandardScaler가 무난한 선택이다.

**선택 기준**: 변수가 특정 범위에 있어야 하는 경우(예: 이미지 픽셀값, 확률) MinMaxScaler가 적합하다. 그 외에는 StandardScaler를 기본으로 사용한다. 트리 기반 모델(RF, XGBoost)은 스케일에 둔감하므로 정규화가 필수는 아니지만, 적용해도 해가 되지는 않는다.

### 특성 엔지니어링 (Feature Engineering)

특성 엔지니어링은 기존 변수에서 새로운 변수를 만드는 과정이다. 이것이 ML에서 공학자가 가장 빛나는 영역이다. AI 전공자는 알고리즘은 잘 알지만, 공학 문제의 물리적 관계를 모른다. 공학 전공자는 도메인 지식을 활용하여 의미 있는 특성을 설계할 수 있다.

예시: 콘크리트 강도 예측에서, "물"과 "시멘트"를 각각 입력으로 넣는 것보다 "물-시멘트비(W/C)"를 계산하여 입력으로 넣는 것이 훨씬 효과적이다. W/C는 콘크리트 강도를 결정하는 가장 중요한 인자로 알려져 있다. 이 도메인 지식을 모르는 AI 전공자는 물과 시멘트를 별도 변수로 넣을 것이고, 모델이 스스로 W/C의 관계를 발견해야 한다. 데이터가 충분하면 발견할 수도 있지만, 데이터가 부족하면 발견하지 못한다.

물리적으로 의미 있는 특성의 예:
- 비율: 물-시멘트비, 종횡비(aspect ratio), 세장비
- 무차원수: 레이놀즈 수, 프루드 수
- 파생 물리량: 응력/변형률, 에너지 밀도, 곡률
- 통계적 특성: 이동 평균, 표준편차(시계열에서)

## 모델 학습

데이터 준비가 끝나면 드디어 모델을 학습시킨다. 그러나 모델 학습은 "model.fit(X, y) 한 줄"로 끝나지 않는다. 어떤 모델을 쓸지, 하이퍼파라미터를 어떻게 설정할지, 어떤 전략으로 학습할지를 결정해야 한다.

### 하이퍼파라미터 튜닝

하이퍼파라미터는 모델의 구조와 학습 방식을 결정하는 설정값이다. 데이터에서 학습되는 것(가중치, 편향)이 아니라, 사람이 미리 정해주는 것이다. Random Forest의 트리 수(n_estimators), 신경망의 층 수와 뉴런 수, 학습률(learning_rate) 등이 하이퍼파라미터다.

하이퍼파라미터 튜닝 방법에는 세 가지가 있다.

**Grid Search**: 하이퍼파라미터의 가능한 조합을 모두 시도한다. 예: `n_estimators = [100, 200, 500]`, `max_depth = [5, 10, 20]`이면 9가지 조합을 모두 시도하고, 가장 좋은 것을 선택한다. 간단하나, 하이퍼파라미터가 많으면 조합 수가 폭발적으로 증가한다.

**Random Search**: 하이퍼파라미터의 범위에서 랜덤으로 샘플링하여 시도한다. Grid Search보다 효율적이다. 같은 계산 비용으로 더 넓은 범위를 탐색할 수 있다. 실무에서는 Random Search가 Grid Search보다 좋은 결과를 내는 경우가 많다.

**Bayesian Optimization**: 이전 시도의 결과를 바탕으로, 다음에 시도할 하이퍼파라미터를 지능적으로 선택한다. Gaussian Process를 대리 모델로 사용하여 "어떤 하이퍼파라미터 조합이 가장 좋을 것 같은가"를 예측한다. Random Search보다 효율적이며, 특히 하이퍼파라미터가 많고 학습 시간이 길 때 유용하다. Optuna, Hyperopt 등의 라이브러리가 이를 지원한다.

### 교차 검증 전략

하이퍼파라미터 튜닝에서 교차 검증은 필수적이다. 하나의 데이터 분할에서 성능이 좋은 하이퍼파라미터가, 다른 분할에서는 나쁠 수 있다. K-fold 교차 검증으로 여러 분할에서의 평균 성능을 기준으로 하이퍼파라미터를 선택해야 한다.

공학 데이터에서 주의할 점이 있다. 시계열 데이터는 일반적인 K-fold 교차 검증을 사용하면 안 된다. 시계열에서는 시간 순서가 중요하므로, 미래 데이터로 학습하고 과거 데이터로 검증하는 "데이터 누수"가 발생할 수 있다. 시계열 데이터에는 TimeSeriesSplit(시간 순서를 유지하는 교차 검증)을 사용해야 한다.

또 하나, 같은 시편에서 여러 번 측정한 데이터가 있다면, 같은 시편의 데이터가 학습 세트와 검증 세트에 동시에 들어가지 않도록 GroupKFold를 사용해야 한다. 같은 시편의 데이터는 서로 매우 유사하므로, 학습-검증에 같이 들어가면 성능이 과대 평가된다.

## 평가

모델을 학습시켰으면 성능을 평가해야 한다. "이 모델이 좋은 모델인가?"를 판단하는 것이다. 이때 메트릭(평가 지표) 하나만 보면 안 된다. 여러 메트릭을 종합적으로 보고, 물리적 타당성까지 확인해야 한다.

### 공학 문제에 적합한 메트릭

**R² (결정계수)**: 0에서 1 사이의 값으로, 모델이 데이터 분산의 몇 퍼센트를 설명하는지를 나타낸다. R² = 0.9면 데이터 변동의 90%를 모델이 설명한다는 뜻이다. 직관적이고 해석이 쉬워서 가장 많이 사용되지만, 함정이 있다. 데이터 범위가 넓으면 R²가 자동으로 높아진다. 좁은 범위의 세밀한 예측에서는 R²가 낮게 나올 수 있다.

**RMSE (Root Mean Squared Error)**: 예측 오차의 제곱 평균의 제곱근. 출력 변수와 같은 단위를 가지므로 물리적 해석이 가능하다. "평균적으로 ±5 MPa 오차로 예측한다"처럼 말할 수 있다. 큰 오차에 가중치를 준다.

**MAE (Mean Absolute Error)**: 예측 오차의 절대값 평균. RMSE보다 이상치에 덜 민감하다. "평균적으로 ±3 MPa 오차로 예측한다"처럼 해석한다.

**MAPE (Mean Absolute Percentage Error)**: 오차를 퍼센트로 나타낸다. 다른 스케일의 데이터 간 비교가 가능하다. 반면 실제값이 0에 가까우면 MAPE가 폭발적으로 커지는 문제가 있다.

**물리적 타당성**: 메트릭에 포함되지 않지만 가장 중요한 평가 기준이다. 모델이 물리 법칙에 위배되는 예측을 하지 않는가? 외삽(extrapolation) 영역에서 합리적인 예측을 하는가? 입력 변수의 변화에 대한 출력의 변화 방향이 물리적으로 올바른가?

### 메트릭 하나만 보지 않기

R² = 0.95라고 해서 모델이 좋다고 단정하면 안 된다. R²는 높은데 RMSE가 공학적으로 허용 불가능한 수준일 수 있다. 예를 들어, 콘크리트 강도 예측에서 R² = 0.95이지만 RMSE = 8 MPa라면, 설계에서 이 오차가 허용되는지를 확인해야 한다.

예측값 vs 실측값 산점도를 반드시 그려야 한다. 대각선(y = x)에 점들이 모여 있어야 좋은 모델이다. 특정 영역에서 점들이 대각선에서 벗어나면, 그 영역에서 모델이 잘 못하고 있다는 뜻이다. 이 시각적 확인이 숫자 하나보다 훨씬 많은 정보를 준다.

잔차(residual = 실측값 - 예측값) 분석도 중요하다. 잔차가 0을 중심으로 균일하게 분포하면 좋은 모델이다. 잔차에 패턴이 보이면(예: 큰 값에서 잔차가 커지는 경향) 모델이 그 패턴을 포착하지 못하고 있다는 의미이며, 모델 개선의 여지가 있다.

## 결과 해석

ML 모델이 좋은 성능을 보여도, "왜 이 예측이 나왔는가?"를 설명하지 못하면 공학 논문으로서 불완전하다. 리뷰어는 반드시 물어볼 것이다. "모델의 물리적 해석은 무엇인가?", "어떤 변수가 가장 중요한가?", "입력 변수와 출력의 관계가 기존 공학 지식과 일치하는가?"

### Feature Importance (변수 중요도)

트리 기반 모델(RF, XGBoost)은 학습 과정에서 각 변수의 중요도를 자동으로 계산한다. 변수 중요도는 "각 변수가 모델의 예측에 얼마나 기여하는가"를 수치로 나타낸 것이다.

변수 중요도를 막대 그래프로 시각화하면, 어떤 변수가 예측에 가장 큰 영향을 미치는지 한눈에 보인다. 이것을 도메인 지식과 비교하면 된다. 콘크리트 강도 예측에서 물-시멘트비가 가장 중요한 변수로 나왔다면, 이것은 기존 공학 지식과 일치하므로 모델의 신뢰성이 높아진다. 반면, 물리적으로 관련이 없어야 하는 변수(예: 시편 번호)가 높은 중요도를 보이면, 데이터에 문제가 있을 가능성이 높다.

### SHAP (SHapley Additive exPlanations)

SHAP은 게임 이론의 Shapley value를 ML 모델 해석에 적용한 방법이다. Feature Importance가 "전체적으로 어떤 변수가 중요한가"를 보여준다면, SHAP은 **"개별 예측에서 각 변수가 어떤 방향으로 얼마나 기여했는가"**를 보여준다.

SHAP summary plot은 모든 데이터 포인트에 대한 각 변수의 SHAP 값을 보여준다. 이를 통해 "변수 X가 높을 때 출력이 증가하는가 감소하는가?"를 시각적으로 확인할 수 있다. 이것은 Partial Dependence Plot과 유사한 정보를 주지만, 변수 간 상호작용도 반영한다는 장점이 있다.

### Partial Dependence Plot (PDP)

PDP는 특정 변수가 변할 때 모델의 예측이 어떻게 변하는지를 보여주는 그래프다. 다른 변수를 고정하고 한 변수만 변화시켰을 때의 평균적인 영향을 나타낸다.

PDP를 그려보면 모델이 학습한 입력-출력 관계의 형태를 시각적으로 확인할 수 있다. 물-시멘트비가 증가하면 강도가 감소하는 패턴이 PDP에 나타나면, 이것은 공학적으로 합리적이다. 반대로 물-시멘트비가 증가하면 강도가 증가하는 패턴이 나타나면, 모델이나 데이터에 문제가 있다는 신호다.

## 재현성

재현성(Reproducibility)은 "다른 사람이 같은 코드, 같은 데이터로 실행했을 때 같은 결과를 얻을 수 있는가"이다. 과학 연구의 기본 원칙이지만, ML 연구에서 재현성을 확보하는 것은 생각보다 어렵다.

ML의 재현성이 어려운 이유가 있다. 랜덤 초기화(random initialization), 데이터 분할(random split), 하드웨어 차이(GPU 연산의 비결정성), 라이브러리 버전 차이 등이 결과에 영향을 미친다. 같은 코드를 돌려도 랜덤 시드가 다르면 결과가 달라진다.

## 흔한 실수

ML 프로젝트에서 반복적으로 나타나는 실수들이 있다. 이 실수들은 경험 많은 연구자도 가끔 저지르고, 초보자는 거의 반드시 저지른다. 미리 알고 있으면 피할 수 있다.

### 데이터 누수 (Data Leakage)

데이터 누수는 ML에서 가장 위험한 실수다. 테스트 데이터의 정보가 어떤 경로로든 학습 과정에 흘러들어가는 것이다. 데이터 누수가 있으면 성능이 비정상적으로 높게 나오고, 실제 적용에서는 성능이 급격히 떨어진다.

**흔한 데이터 누수 유형:**

첫째, 전처리 누수. 전체 데이터에 대해 정규화(StandardScaler)를 적용한 후 데이터를 분할하면, 테스트 데이터의 통계(평균, 표준편차)가 학습 과정에 사용된 셈이다. 올바른 방법은 학습 데이터에서만 스케일러를 학습(fit)하고, 테스트 데이터에는 변환(transform)만 적용하는 것이다.

둘째, 미래 정보 사용. 시계열 데이터에서 미래 시점의 데이터를 입력으로 사용하는 것이다. 예: "내일의 온도"를 입력으로 넣어 "오늘의 에너지 소비량"을 예측. 모델 안에서는 잘 작동하나, 실제로는 "내일의 온도"를 미리 알 수 없으므로 쓸모없는 모델이다.

셋째, 타깃 누수. 출력(y)에서 파생된 변수를 입력(X)에 넣는 것이다. 예: "파괴 여부"를 예측하는 모델에 "파괴 시 하중"을 입력으로 넣으면, 당연히 정확도가 높다. 그런데 파괴 시 하중은 파괴가 일어나야 알 수 있는 값이므로, 사전 예측에는 사용할 수 없다.

**데이터 누수 진단법:** 성능이 비정상적으로 높으면(R² > 0.99 등) 일단 의심한다. 입력 변수 중 출력과 직접적으로 관련된 변수가 있는지 확인한다. 전처리 파이프라인에서 데이터 분할 전후의 순서를 점검한다.

### 과적합

과적합은 이전 장에서도 다뤘지만, 재강조가 필요하다. 공학 데이터는 보통 소규모(수십~수백 개)이므로, 과적합 위험이 특히 높다. 100개의 데이터에 1,000개의 파라미터를 가진 신경망을 적용하면, 모델은 100개를 완벽하게 외울 수 있지만 새로운 데이터에는 무용지물이다.

**과적합 방지 원칙:** 모델의 파라미터 수가 데이터 수보다 지나치게 많지 않게 한다. 교차 검증으로 모델을 평가한다. 정규화(L1/L2, Dropout)를 적용한다. Early Stopping을 사용한다. 간단한 모델부터 시작하여 필요한 만큼만 복잡도를 높인다.

### 불균형 데이터

분류 문제에서 클래스 간 비율이 크게 다른 경우다. 예: 결함 탐지에서 정상 시편 950개, 결함 시편 50개. 이 데이터를 그대로 학습시키면, 모델은 "전부 정상"이라고 예측해도 정확도 95%를 달성한다. 그렇지만 이것은 쓸모없는 모델이다.

**대처법:**
- 오버샘플링(Oversampling): 소수 클래스의 데이터를 복제하거나 합성(SMOTE)하여 균형을 맞춘다.
- 언더샘플링(Undersampling): 다수 클래스의 데이터를 줄여 균형을 맞춘다.
- 클래스 가중치(Class Weight): 소수 클래스의 오분류에 더 큰 페널티를 부여한다.
- 평가 메트릭 변경: 정확도(accuracy) 대신 F1-score, Precision, Recall, AUC-ROC를 사용한다.

### 검증 없는 테스트 성능 보고

테스트 세트에서의 성능만 보고하고, 교차 검증 결과를 제시하지 않는 것은 불완전한 평가다. 하나의 데이터 분할에서 좋은 성능이 나왔어도, 다른 분할에서는 나쁠 수 있다. 교차 검증의 평균과 표준편차를 함께 보고하면, 모델 성능의 안정성을 보여줄 수 있다.

예를 들어, "R² = 0.92 (5-fold CV: 0.89 ± 0.03)"이라고 보고하면, 테스트 성능뿐 아니라 모델의 일반화 능력과 변동성도 함께 보여주는 것이다. 리뷰어는 이런 보고를 높이 평가한다.

## 정리

ML 프로젝트는 화려한 모델이 아니라 견고한 파이프라인이 핵심이다. 데이터 수집과 전처리에 충분한 시간을 투자하고, 간단한 모델부터 시작하고, 교차 검증으로 모델을 평가하고, 결과를 물리적으로 해석하고, 재현성을 확보하는 것. 이 기본에 충실하면 대부분의 공학 ML 프로젝트에서 의미 있는 결과를 얻을 수 있다.

데이터 누수, 과적합, 불균형 데이터 같은 흔한 실수를 미리 알고 있으면 시행착오를 크게 줄일 수 있다. 그리고 무엇보다, ML은 연구의 도구이지 목적이 아니라는 것을 잊지 말아야 한다. 공학 문제를 이해하는 것이 먼저고, 적절한 도구를 선택하는 것이 그 다음이다.

## 공학 데이터를 위한 특성 공학

앞서 특성 엔지니어링의 기본 개념을 다뤘지만, 여기서는 한 단계 더 들어간다. 일반 ML 교과서에서 다루는 특성 공학은 대부분 텍스트나 범주형 데이터 기반이고, 공학 연구에서 실제로 효과적인 특성은 잘 소개하지 않는다. 센서 신호, 시뮬레이션 결과, 물리적 측정값을 다루는 공학 연구자에게 필요한 특성은 다른 범주에 속한다.

<div class="card-grid">
<div class="card">
<h4>통계적 특성</h4>
<p>신호나 측정 데이터의 기본적인 분포 정보를 요약하는 특성이다. 평균, 표준편차, 첨도(kurtosis), 왜도(skewness), RMS(Root Mean Square)가 대표적이다. 진동 신호에서 RMS는 전체 에너지 수준을, 첨도는 충격성 이벤트의 존재를 포착한다. 계산이 간단하고, 물리적 해석이 직관적이라는 장점이 있다.</p>
</div>
<div class="card">
<h4>주파수 특성</h4>
<p>FFT(Fast Fourier Transform)로 추출한 주요 주파수 성분, 파워 스펙트럼 밀도(PSD), 대역별 에너지 비율 등이 해당한다. 회전 기계의 고장 진단에서 특정 주파수의 변화가 결함 유형을 직접 가리키는 경우가 많다. 진동, 음향, 전기 신호를 다루는 연구에서 거의 필수적인 특성 범주다.</p>
</div>
<div class="card">
<h4>시간 영역 특성</h4>
<p>피크 값(peak value), 크레스트 팩터(crest factor, 피크/RMS 비율), 영교차율(zero-crossing rate) 등이 포함된다. 크레스트 팩터는 신호에 순간적인 충격이 존재하는지를 보여주고, 영교차율은 신호의 주파수 특성을 간접적으로 나타낸다. FFT 없이도 시간 영역에서 바로 계산할 수 있어서 실시간 모니터링에 적합하다.</p>
</div>
<div class="card">
<h4>물리 기반 특성</h4>
<p>무차원수(레이놀즈 수, 누셀트 수 등), 에너지 비율(운동에너지 대비 변형에너지), 감쇠 계수, 응력비(stress ratio) 등 물리 법칙에서 유도된 특성이다. 이런 특성은 데이터에서 자동으로 학습되기 어려운 물리적 관계를 모델에 직접 주입하는 효과가 있다. 도메인 지식이 있는 공학 연구자만이 설계할 수 있는 특성이며, 이것이 공학 ML 연구의 차별화 포인트가 된다.</p>
</div>
</div>

어떤 특성을 넣어야 할지 모르겠다면, 이 4가지 범주에서 각 2~3개씩 추출해서 시작하는 것을 추천한다. 총 8~12개의 특성으로 베이스라인 모델을 만들고, 특성 중요도를 확인한 후에 효과적인 특성을 추가하거나 불필요한 특성을 제거하는 방식으로 반복하면 된다. 처음부터 수십 개의 특성을 넣으면 차원의 저주와 과적합의 위험이 커지므로, 적은 수에서 시작하는 것이 안전하다.

## AutoML vs 수동 튜닝

<div class="highlight-box info">
<span class="highlight-box-icon">ℹ️</span>
<div class="highlight-box-content">
<p><strong>AutoML의 장점과 한계</strong></p>
<p>AutoML 도구(Optuna, Auto-sklearn, FLAML 등)는 하이퍼파라미터 탐색과 모델 선택을 자동화한다. 사람의 편향 없이 넓은 공간을 빠르게 탐색하고, 수동으로는 시도하지 않았을 조합을 발견할 수 있다는 것이 가장 큰 장점이다. 반면 한계도 분명하다. AutoML은 물리적 의미를 고려하지 않는다. 데이터만 보고 최적의 모델을 찾으므로, 공학적으로 해석 불가능한 블랙박스 모델이 선택될 수 있다. 또한 특성 공학이나 데이터 전처리 전략은 자동화하기 어렵기 때문에, AutoML이 해결해주는 범위는 파이프라인 전체의 일부에 불과하다.</p>
</div>
</div>

현실적인 접근은 이렇다. AutoML로 빠르게 좋은 모델을 찾고, 왜 그 모델이 좋은지를 분석하는 것이 이상적인 순서다. Optuna로 최적 하이퍼파라미터를 탐색한 다음, 그 결과를 SHAP이나 PDP로 해석하여 물리적 타당성을 검증한다. AutoML이 선택한 모델이 도메인 지식과 일치하면 그대로 쓰고, 일치하지 않으면 왜 그런지를 파악하여 모델을 수정하거나 특성을 재설계한다.

주의할 점이 하나 있다. AutoML 결과를 논문에 보고할 때, "AutoML이 이 모델을 선택했다"로 끝내면 안 된다. 리뷰어는 "왜 이 모델이 최적인가?"를 물을 것이다. AutoML은 탐색 도구이지, 설명 도구가 아니다. 선택된 모델에 대한 공학적 해석과 물리적 타당성 분석은 연구자의 몫이다.

## 실험 추적: MLflow와 Weights & Biases

ML 연구는 본질적으로 수많은 실험의 반복이다. "이 하이퍼파라미터로 학습", "저 데이터셋으로 학습", "다른 모델 구조로 학습"을 수십, 수백 번 반복한다. 이 모든 실험을 손으로 기록하면 일주일 만에 무엇이 무엇이었는지 잊어버린다. **실험 추적 도구**는 이 문제를 해결한다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>실험 추적 없이는 재현할 수 없다</strong></p>
    <p>"내가 한 달 전에 가장 좋은 결과를 얻었던 실험의 정확한 설정이 무엇이었는가?" 이 질문에 즉시 답할 수 없다면, 연구는 재현 불가능한 상태다. 6개월 후 논문 리비전에서 그 결과를 다시 만들어야 할 때 큰 문제가 된다. 모든 실험을 자동으로 기록하는 도구가 필요한 이유다.</p>
  </div>
</div>

**MLflow.** 가장 인기 있는 오픈소스 실험 추적 도구다. 로컬 또는 서버에 설치하여 사용한다. 핵심 기능: 첫째, **실험 로그**(하이퍼파라미터, 메트릭, 결과). 둘째, **모델 저장과 버전 관리**. 셋째, **모델 배포**(REST API, Docker). 넷째, **웹 UI로 실험 비교**.

```python
import mlflow
import mlflow.pytorch

mlflow.set_experiment("concrete_strength_prediction")

with mlflow.start_run():
    # 하이퍼파라미터 기록
    mlflow.log_param("lr", 0.001)
    mlflow.log_param("batch_size", 32)
    mlflow.log_param("model", "ResNet18")

    # 모델 학습
    model = train_model(...)

    # 메트릭 기록
    mlflow.log_metric("train_loss", 0.05)
    mlflow.log_metric("val_loss", 0.08)
    mlflow.log_metric("test_r2", 0.93)

    # 모델 저장
    mlflow.pytorch.log_model(model, "model")

    # 시각화 자동 저장
    mlflow.log_figure(fig, "training_curve.png")
```

이 코드 한 번 작성하면, 모든 실험이 자동으로 기록된다. 웹 UI(`mlflow ui`)에서 모든 실험을 비교할 수 있다.

**Weights & Biases (wandb).** 클라우드 기반 실험 추적 도구로, 학계와 산업계에서 가장 인기 있다. 무료 학생/연구자 플랜을 제공한다. MLflow보다 시각화가 화려하고 협업 기능이 강력하다.

```python
import wandb

wandb.init(project="concrete_strength", config={
    "lr": 0.001,
    "batch_size": 32,
    "epochs": 100,
})

# 학습 루프
for epoch in range(100):
    train_loss = train_one_epoch(...)
    val_loss = validate(...)
    wandb.log({"train_loss": train_loss, "val_loss": val_loss, "epoch": epoch})

# 모델 저장
wandb.save("model.pth")
```

wandb의 강점은 다음과 같다. 첫째, **자동 GPU/시스템 모니터링**. GPU 사용률, 메모리, 온도가 자동으로 추적된다. 둘째, **하이퍼파라미터 스윕**(sweep) 기능. 그리드/베이지안 탐색을 자동으로 수행. 셋째, **공동 작업**. 팀 멤버와 실험 결과를 공유. 넷째, **리포트 생성**. 실험을 정리한 인터랙티브 리포트를 만들 수 있다.

**다른 옵션들.**

| 도구 | 특징 |
|------|------|
| MLflow | 오픈소스, 로컬, 자체 호스팅 |
| Weights & Biases | 클라우드, 학생 무료, 시각화 우수 |
| Neptune.ai | 클라우드, 작은 팀 무료, 메타데이터 강력 |
| Comet ML | 클라우드, 학술용 무료, 협업 |
| TensorBoard | 무료, TF/PyTorch 지원, 가벼움 |
| ClearML | 오픈소스+클라우드, 파이프라인 통합 |

**언제 어느 것을 쓸까.** 첫째, **혼자 작업하고 인터넷이 제한적**: MLflow 로컬. 둘째, **혼자 또는 작은 팀, 학생**: Weights & Biases 무료 플랜. 셋째, **간단한 추적만 필요**: TensorBoard. 넷째, **대학원 프로젝트의 표준**: wandb (가장 흔한 선택). 박사 과정에서 한 가지 도구를 선택하여 모든 실험에 사용한다. 도중에 바꾸면 이전 실험 데이터가 사라진다.

**실험 추적 도구를 사용하는 5가지 원칙.** 첫째, **첫 실험부터 사용한다**. "나중에 정리하면 되지"가 가장 큰 함정. 첫 실험부터 사용하면 습관이 된다. 둘째, **모든 하이퍼파라미터를 기록한다**. "이건 안 중요하니 안 기록해도 되지"라고 생각한 것이 나중에 결정적일 수 있다. 셋째, **실험에 의미 있는 이름을 붙인다**. "experiment_1, experiment_2"가 아니라 "resnet18_lr1e-3_aug_strong"처럼 6개월 후에도 알아볼 수 있는 이름. 넷째, **태그를 활용한다**. "baseline", "ablation", "final" 같은 태그로 실험을 분류한다. 다섯째, **정기적으로 정리한다**. 매주 실패한 실험은 삭제하거나 아카이브하여, 의미 있는 실험만 남긴다.

## MLOps 기초: 학술 코드를 넘어서

MLOps(Machine Learning Operations)는 ML 모델을 학술 환경에서 산업/제품 환경으로 배포하고 운영하는 분야다. 박사 과정 학생에게 MLOps의 모든 것이 필요하지는 않지만, 기본 개념을 알아두면 졸업 후 진로(특히 산업체)에 큰 도움이 된다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>학술 ML과 산업 ML의 차이</strong></p>
    <p>학술 ML은 "한 번 실행하여 좋은 결과를 얻고 논문에 발표하는 것"이 목표다. 산업 ML은 "매일 수만 건의 실시간 예측을 안정적으로 제공하고, 모델이 점점 나빠지면 자동으로 재학습하는 것"이 목표다. 이 차이를 이해하면 연구가 산업에 어떻게 적용될 수 있는지가 명확해진다. 그리고 연구가 사실은 산업 적용까지 한 단계 더 가야 한다는 것을 인식하게 된다.</p>
  </div>
</div>

**MLOps의 5가지 구성 요소.**

첫째, **데이터 파이프라인**(Data Pipeline). 원본 데이터에서 학습 데이터까지의 전처리를 자동화. 매번 수동으로 처리하지 않고, 코드로 정의된 파이프라인이 자동 실행된다. 도구: Apache Airflow, Prefect, Dagster.

둘째, **모델 학습 파이프라인**(Training Pipeline). 데이터 수집 → 전처리 → 학습 → 평가 → 모델 저장의 전 과정을 자동화. 도구: Kubeflow, ZenML, Metaflow.

셋째, **모델 레지스트리**(Model Registry). 학습된 모델을 버전 관리하고 메타데이터(성능, 학습 데이터, 하이퍼파라미터)와 함께 저장. 도구: MLflow Model Registry, DVC, Weights & Biases Models.

넷째, **모델 서빙**(Model Serving). 학습된 모델을 REST API 또는 gRPC로 제공. 사용자의 요청에 실시간으로 응답. 도구: TorchServe, TensorFlow Serving, FastAPI + 모델, BentoML, Seldon Core.

다섯째, **모니터링과 알림**(Monitoring & Alerting). 배포된 모델의 성능, 지연 시간, 입력 데이터 분포를 지속적으로 모니터링. 이상이 감지되면 알림. 도구: Prometheus + Grafana, Evidently AI, Arize, WhyLabs.

**학생을 위한 MLOps 단계별 학습.**

첫 단계: **재현 가능한 학습 코드**. 학술 코드를 환경 설정, 데이터 로드, 학습, 평가의 명확한 모듈로 분리. requirements.txt 또는 environment.yml 작성. ch16, ch20 참조.

두 번째 단계: **실험 추적**. MLflow 또는 wandb로 모든 실험을 자동 기록. (앞 섹션 참조).

세 번째 단계: **모델 저장과 로드**. 학습된 모델을 표준 형식(PyTorch `.pth`, ONNX, SavedModel)으로 저장. 다른 컴퓨터에서 로드하여 예측 가능.

네 번째 단계: **간단한 API 서빙**. FastAPI로 모델을 REST API로 노출. 로컬 또는 학교 서버에 배포.

```python
from fastapi import FastAPI
import torch

app = FastAPI()
model = torch.load("model.pth")
model.eval()

@app.post("/predict")
def predict(data: dict):
    x = torch.tensor(data["input"])
    with torch.no_grad():
        y = model(x)
    return {"prediction": y.tolist()}
```

다섯 번째 단계: **Docker로 패키징**. 모델과 환경을 Docker 컨테이너로 패키징하여 어디서든 같은 방식으로 실행 가능하게.

여섯 번째 단계 이후는 산업 환경의 영역이다. 박사 졸업 후 산업체나 스타트업에 들어가면 그때 본격적으로 학습한다.

**박사 과정에서 MLOps의 가치.** 첫째, **모델을 산업체 파트너에게 데모할 때**. "이 모델은 작동합니다"보다 "이 API에 데이터를 보내면 즉시 결과를 받을 수 있습니다"가 훨씬 강력하다. 둘째, **박사 논문의 영향력 확대**. 많은 사용자가 모델을 사용해 보고 인용한다. Hugging Face Hub, GitHub Releases에 모델을 공개한다. 셋째, **산업체 취업 시 큰 가산점**. 박사 졸업생 중 학술 코드만 짤 줄 아는 사람과 MLOps도 아는 사람의 차이는 매우 크다.

## 데이터 드리프트와 컨셉 드리프트

배포된 모델은 시간이 지남에 따라 성능이 떨어진다. 학술 환경에서는 한 번 학습한 모델로 끝이지만, 실제 환경에서는 데이터가 변한다. 이것을 **데이터 드리프트**(data drift)와 **컨셉 드리프트**(concept drift)라고 한다.

**두 종류의 드리프트.**

**데이터 드리프트(Data Drift)**: 입력 데이터의 분포가 시간에 따라 변하는 것. 예를 들어, 자율주행 모델이 여름에 학습되었는데 겨울에 사용되면, 눈 덮인 도로 이미지의 분포가 학습 데이터와 다르다. 모델은 새로운 분포에 익숙하지 않아 성능이 떨어진다.

**컨셉 드리프트(Concept Drift)**: 입력과 출력의 관계가 변하는 것. 예를 들어, 코로나 이전과 이후로 사람들의 구매 패턴이 완전히 바뀌었다. 같은 입력에도 출력이 다르게 나와야 한다. 모델은 이전 패턴을 학습했으므로 새 패턴에 맞지 않는다.

**공학 분야의 드리프트 예시.** 첫째, **센서 노화**: 같은 측정에도 시간이 지나면서 센서의 출력이 변한다(데이터 드리프트). 둘째, **장비 마모**: 새 장비와 마모된 장비의 측정 패턴이 다르다. 셋째, **환경 변화**: 계절, 온도, 습도의 영향. 넷째, **공정 변경**: 제조 공정이 개선되거나 변경되면서 데이터 분포가 바뀜. 다섯째, **재료 변경**: 같은 제품이라도 공급자가 바뀌면 재료 특성이 다르다.

**드리프트 탐지 방법.**

첫째, **통계적 검정**: 학습 데이터와 새 데이터의 분포가 다른지 통계적으로 검정. Kolmogorov-Smirnov test, Population Stability Index(PSI), Wasserstein distance.

둘째, **모델 성능 모니터링**: 새 데이터에 대한 모델의 성능(정확도, 오차)을 지속적으로 측정. 성능이 떨어지면 드리프트의 신호. 라벨이 있어야 가능하다.

셋째, **예측 분포 모니터링**: 라벨이 없을 때, 모델의 예측 분포를 모니터링. 예측이 특정 값으로 치우치거나 분포가 변하면 드리프트의 신호.

넷째, **드리프트 탐지 라이브러리**: Evidently AI, Alibi Detect, NannyML 같은 도구를 사용.

**드리프트 대응 전략.**

전략 1: **재학습(Retraining)**. 새 데이터로 모델을 다시 학습. 가장 단순하지만 효과적. 정기적으로(매월, 매분기) 또는 드리프트 탐지 시 자동으로.

전략 2: **점진적 학습(Incremental Learning)**. 기존 모델을 새 데이터에 미세조정. 처음부터 학습하는 것보다 빠름.

전략 3: **앙상블**. 여러 시점에 학습한 모델들의 앙상블. 시간 변화에 강건.

전략 4: **온라인 학습(Online Learning)**. 새 데이터가 들어올 때마다 모델을 즉시 업데이트. 매우 동적인 환경에 적합.

**박사 연구에서 드리프트의 의미.** 박사 연구가 특정 시점/조건의 데이터로 학습되었다는 것을 인지한다. 모델이 다른 시점/조건에서도 작동할지 검증하는 것이 일반화 가능성의 일부다. 논문에 "Limitations" 섹션에서 다음을 언급한다. "Our model is trained on data from [period]. Performance may degrade if applied to substantially different conditions, such as [examples]." 이런 인지가 연구의 신뢰성을 높인다.

## 학술 ML과 산업 ML의 차이

박사 과정 동안 개발한 ML 모델은 학술적 가치를 위해 만들어졌다. 같은 모델이 산업에 적용되려면 많은 추가 작업이 필요하다. 이 차이를 이해하면 연구의 위치와 한계를 명확히 알 수 있다.

**5가지 핵심 차이.**

| 측면 | 학술 ML | 산업 ML |
|------|---------|---------|
| 목표 | 새로운 발견, 논문 출판 | 비즈니스 가치, 사용자 만족 |
| 데이터 | 깨끗한 벤치마크 데이터 | 더럽고 변하는 실제 데이터 |
| 평가 지표 | 정확도, F1 등 단일 지표 | 정확도 + 지연 시간 + 비용 + 사용성 |
| 시간 척도 | 한 번의 실험 | 24/7 운영, 지속적 업데이트 |
| 우선순위 | 정확도 최대화 | 안정성 + 신뢰성 + 유지보수성 |

**학술 ML의 함정.** 학술 ML 연구자가 산업에 가면 다음을 흔히 경험한다. 첫째, **데이터의 더러움**에 충격받는다. 학술 데이터셋은 이미 정제되어 있지만, 실제 데이터는 누락, 오류, 이상치, 일관성 없음으로 가득하다. 90% 시간을 데이터 정리에 쓴다. 둘째, **정확도가 1순위가 아님**에 충격받는다. "내 모델은 95% 정확도인데 왜 안 채택되는가?" 답은 추론 시간이 너무 길거나, 메모리가 너무 크거나, 유지보수가 어렵기 때문이다. 셋째, **재학습의 부담**. 학술 환경에서는 한 번 학습하면 끝이지만, 실제 환경에서는 매주 또는 매월 재학습이 필요하다.

**산업 ML 관점이 학술 연구에도 도움된다.** 학술 논문이 산업에 적용 가능성을 가지려면 다음을 의식한다. 첫째, **단일 정확도가 아닌 다차원 평가**. 논문에 정확도뿐만 아니라 추론 시간, 모델 크기, 메모리 사용을 보고. 둘째, **강건성 평가**. 모델이 노이즈, 누락, 이상치에 얼마나 강한지 보고. 셋째, **계산 효율성**. 학습 시간, 추론 시간, 에너지 소비를 보고. 넷째, **재현성**. 코드와 데이터를 공개하여 다른 사람이 결과를 재현할 수 있게.

**Bridging the Gap: 할 수 있는 것.** 박사 과정 중 산업 ML 관점을 가지면 연구가 더 실용적이고 인용도 잘 받는다. 첫째, **벤치마크 외 데이터에서 평가**. 모델을 1-2개의 다른(더 어려운) 데이터셋에서도 평가. 둘째, **실제 사용 시나리오 시뮬레이션**. 모델을 실시간 환경 또는 제한된 리소스 환경에서 시뮬레이션. 셋째, **산학 협력**. 연구를 산업체 파트너와 협력하여 실제 데이터에 적용해 본다(ch30 참조). 넷째, **공개와 데모**. 모델을 Hugging Face Hub, Streamlit, Gradio로 데모 가능하게 만든다. 다른 사람이 직접 시도할 수 있으면 연구가 훨씬 가시화된다.

## 체계적 오류 분석 — 실패 사례에서 배우는 법

모델이 90% 정확도를 달성했다고 하자. 이제 끝일까? 아니다. 90%는 출발점이다. 나머지 10%의 실패를 이해하는 것이 연구를 평범한 것에서 뛰어난 것으로 바꾼다. ML 연구에서 가장 저평가된 기술이 **체계적 오류 분석(Systematic Error Analysis)**이다. 단순히 "정확도가 몇 %"라고 보고하는 것과, "어떤 경우에 실패하고 왜 실패하는지"를 설명하는 것은 논문의 품질을 완전히 다른 차원으로 올린다.

<div class="highlight-box highlight-important">

**왜 오류 분석이 중요한가.** 리뷰어의 관점에서 "95% 정확도" 숫자는 많은 논문에서 볼 수 있다. 하지만 "이 모델이 X 조건에서만 실패하는데 그 원인이 Y이고, 우리는 이렇게 대응했다"는 서술은 드물다. 후자가 리뷰어에게 "이 저자는 모델을 진짜로 이해하고 있다"는 신호를 보낸다. Ablation Study(ch21)가 방법의 어떤 요소가 기여하는지를 보여준다면, 오류 분석은 방법이 어디서 한계에 부딪히는지를 보여준다.

</div>

**오류 분석의 5단계 워크플로.**

**1단계: 오류 샘플 수집.** 테스트 셋에서 모델이 틀린 예측을 한 모든 샘플을 별도로 저장한다. 분류 문제라면 오답 샘플, 회귀 문제라면 오차가 큰 샘플(상위 5-10%). 이 오류 샘플들이 분석의 원재료다.

```python
# 분류 예시
wrong_mask = (y_pred != y_true)
error_samples = {
    'input': X_test[wrong_mask],
    'true_label': y_true[wrong_mask],
    'pred_label': y_pred[wrong_mask],
    'confidence': probs[wrong_mask].max(axis=1),
    'index': np.where(wrong_mask)[0]
}
```

신뢰도(confidence)도 함께 저장하는 것이 중요하다. "자신 있게 틀린 예측"과 "머뭇거리며 틀린 예측"은 매우 다른 오류다.

**2단계: 무작위 50-100개 샘플의 육안 검사.** 통계만 보지 말고 실제 샘플을 직접 본다. 이미지라면 이미지를 열어 본다. 센서 데이터라면 시계열을 그래프로 그린다. 텍스트라면 문장을 읽는다. 이 "**인간이 데이터를 직접 보는**" 단계가 자동화된 분석이 놓치는 통찰을 준다.

육안 검사 중 다음 질문을 한다.
- 본인이라면 이 샘플을 올바르게 분류할 수 있었을까?
- 정답 라벨이 실제로 맞는가? (라벨 오류 가능성)
- 모델의 예측이 "합리적으로 틀린" 것인가, "터무니없이 틀린" 것인가?
- 이 샘플이 학습 데이터와 얼마나 다른가?

이 단계에서 발견되는 것의 30-50%는 실제로 **데이터 라벨이 틀린 경우**다. 이것을 알면 본인 모델의 진짜 성능이 보고된 수치보다 높다는 것이 드러난다.

**3단계: 오류를 카테고리로 분류.** 육안 검사 후 오류 샘플들을 카테고리별로 그룹화한다. 보통 4-8개의 카테고리가 나온다. 예를 들어 이미지 분류에서:

| 오류 카테고리 | 비율 | 특징 |
|--------------|------|------|
| 라벨 오류 | 15% | 데이터셋 자체가 잘못 라벨됨 |
| 배경 혼동 | 30% | 객체와 유사한 배경에 속음 |
| 작은 객체 | 20% | 객체 크기가 입력 해상도의 10% 미만 |
| 가려짐 | 15% | 객체의 50% 이상이 가려짐 |
| 드문 클래스 | 12% | 학습 데이터에 10개 미만 |
| 기타 | 8% | 명확한 패턴 없음 |

이 표 하나가 논문의 Discussion 섹션에서 엄청난 가치를 가진다. 리뷰어는 이런 표를 좋아한다.

**4단계: 카테고리별 대응 전략 수립.** 각 오류 카테고리에는 다른 대응이 필요하다.

- **라벨 오류**: 데이터셋을 수정하거나, "uncleaned" 버전과 "cleaned" 버전을 모두 보고.
- **배경 혼동**: 데이터 증강에 배경 변형 추가, Attention 메커니즘 도입.
- **작은 객체**: 입력 해상도 증가, 멀티스케일 학습.
- **가려짐**: 가려짐 시뮬레이션 증강(Cutout, Random Erasing).
- **드문 클래스**: Class weight 조정, Oversampling, Focal Loss (ch21의 불균형 데이터 참조).

모든 카테고리를 다 해결할 필요는 없다. 가장 큰 비중을 차지하는 것 1-2개에 집중하여 개선 실험을 한다. 이 개선이 논문의 한 섹션이 된다.

**5단계: 개선 후 재평가.** 대응을 적용한 후, 같은 오류 분석을 다시 수행한다. 어느 카테고리가 얼마나 줄었는가? 새로운 카테고리가 생겼는가? 이 비교가 개선의 효과를 카테고리별로 명확히 보여준다. 단순히 "정확도가 90%에서 92%로 올랐다"보다 훨씬 강력한 논증이다.

**오류 분석을 위한 실용 도구.**

- **Confusion Matrix**: 분류 문제의 기본. 어떤 클래스가 어떤 클래스로 혼동되는지 행렬로 본다. seaborn의 `heatmap`으로 시각화.
- **Error Slicing**: 테스트 데이터를 여러 부분집합으로 나누고 각각의 정확도를 측정. 예: 이미지 밝기별, 객체 크기별, 시간대별. 전체 평균은 괜찮은데 특정 슬라이스에서 처참한 경우가 많다.
- **Residual Plot (회귀)**: 예측값과 실제값의 차이(잔차)를 그래프로. 잔차가 특정 입력 범위에서 체계적으로 크다면, 그 범위에서 모델이 편향을 가지는 것이다.
- **Calibration Plot**: 모델이 "80% 확신"이라고 한 예측이 실제로 80% 정답률인가? 그렇지 않으면 모델이 과신(over-confident)하는 것이다. 신뢰도 기반 의사결정이 필요한 응용에서 중요.
- **Attention Visualization (딥러닝)**: Transformer나 CNN의 주목 영역을 시각화. ch21의 Grad-CAM과 연결.

**오류 분석 예산 배정의 원칙.** 연구 시간을 이렇게 배분하는 것이 이상적이다. 첫 번째 결과 얻기 40%, 오류 분석 20%, 개선 실험 30%, 논문 쓰기 10% (수치는 대략적). 많은 학생이 "첫 결과 얻기"에 70-80%를 쓰고 오류 분석에 거의 시간을 쓰지 않는다. 그러면 논문의 깊이가 얕아진다. 오류 분석에 의도적으로 시간을 배정하는 것이 필요하다.

**분야별 특수성.** 오류 분석의 원칙은 보편적이지만 분야에 따라 구체적 기법은 다르다. 시계열/신호 데이터(ch37)에서는 오류가 큰 시점의 원본 신호를 봐야 한다. 시뮬레이션-ML 혼합 연구에서는 물리적으로 말이 안 되는 예측이 있는지 따로 체크한다. 이미지 분류에서는 어떤 부분에 주목했는지 Grad-CAM으로 확인한다. 본인 분야의 도메인 지식을 오류 분석에 활용하는 것이 핵심이다.

**오류 분석을 논문에 녹이는 법.** Discussion 섹션에 "Error Analysis" 또는 "Failure Modes"라는 하위 섹션을 만든다. 다음 요소를 포함한다.

1. 전체 오류의 카테고리 분포 (표 또는 파이 차트)
2. 각 카테고리의 대표 사례 (이미지, 그래프, 또는 예시)
3. 각 카테고리의 원인에 대한 해석
4. 본인 방법의 한계와 향후 연구 방향
5. (가능하면) 어떤 카테고리는 개선했고 어떤 것은 못 했는지

이 섹션이 있는 논문과 없는 논문은 리뷰 점수가 크게 다르다. 리뷰어는 한계를 정직하게 논의하는 저자를 신뢰한다.

**오류 분석의 함정: "너무 완벽한 오류 분석"의 위험.** 간혹 오류 분석이 너무 치밀해서 모델의 한계가 너무 많이 드러난 경우, 저자가 이를 두려워하여 논문이 약해 보이지 않을까 고민한다. 이것은 잘못된 걱정이다. 리뷰어는 한계가 **없는** 논문을 신뢰하지 않는다. 모든 방법에는 한계가 있고, 저자가 이를 정직하게 인정하는 것이 오히려 신뢰를 높인다. "방법은 모든 경우에 잘 작동한다"는 서술이 있는 논문은 리뷰어가 즉시 의심한다.

> 오류 분석은 모델과 진지하게 마주하는 시간이다. 90%의 성공보다 10%의 실패가 더 많은 것을 가르친다. 그 10%를 무시하고 논문을 쓰면 연구가 표면적이 된다. 오류 분석에 며칠을 투자하는 것이 논문의 깊이를 결정한다. 이 기술은 박사 과정 초기에 익히는 것이 늦을수록 손해다.

## 데이터 분할의 함정 — 무작위 분할이 해가 될 때

ML 프로젝트에서 가장 기본적이면서도 가장 자주 잘못되는 단계가 **데이터 분할**이다. 대부분의 튜토리얼은 `train_test_split(X, y, test_size=0.2)`로 시작하고 끝낸다. 무작위 80/20 분할이 모든 경우에 통한다고 가정한다. 그러나 공학 데이터의 상당수는 이 단순한 가정이 성립하지 않는다. 잘못된 분할은 모델이 실제로는 작동하지 않는데 **테스트 세트에서 높은 성능**을 보이는 데이터 누출(data leakage)을 만든다. 이것은 논문 리젝의 가장 흔한 원인 중 하나다.

<div class="highlight-box highlight-warning">

**"무작위 분할이면 안전하다"는 치명적 오해.** 90% 정확도를 보고했는데 리뷰어가 "무작위 분할 때문에 누출이 있었던 것 아닌가"라고 지적하면, 전체 실험을 다시 해야 한다. 최악의 경우 논문이 철회된다. 데이터 분할은 "단순한 1줄의 코드"가 아니라 **본인 연구의 가장 중요한 설계 결정 중 하나**로 대해야 한다.

</div>

**데이터 누출(Data Leakage)의 6가지 종류.**

**1. 시간적 누출 (Temporal Leakage).**
시계열 데이터에서 미래 데이터가 학습에 사용되는 경우. 2020년 데이터로 학습한 모델로 2019년을 "예측"하면 의미 없다. 무작위 분할은 이것을 알지 못하고 섞는다.

**예시**: 주가 예측, 날씨 예측, 센서 이상 탐지.

**2. 그룹 누출 (Group Leakage).**
같은 "개체"의 여러 샘플이 train과 test에 분산된다. 한 환자의 MRI 여러 장을 무작위로 분할하면, 같은 환자의 이미지가 양쪽에 들어간다. 모델이 "이 환자"를 기억할 수 있다.

**예시**: 의료 이미지, 사람 재식별, 반복 측정 데이터.

**3. 공간적 누출 (Spatial Leakage).**
공간적으로 인접한 샘플이 train과 test에 분산. 한 도시의 인접 픽셀을 무작위로 나누면, 주변 상황이 거의 같은 두 픽셀이 양쪽에 들어간다.

**예시**: 원격 탐사, 지질 데이터, 의료 영상의 인접 슬라이스.

**4. 중복 누출 (Duplicate Leakage).**
데이터에 중복 또는 거의 동일한 샘플이 있는데 무작위 분할로 같은 것이 양쪽에 들어간다.

**예시**: 웹에서 크롤링한 이미지, 뉴스 기사 모음.

**5. 특징 누출 (Feature Leakage).**
목표 변수와 너무 직접 연관된 특징이 입력에 포함된다. "A 환자가 B 질병에 걸릴까"를 예측하는데 입력에 "B 질병 치료제 복용 여부"가 있으면, 모델이 100% 정확도를 낸다.

**6. 전처리 누출 (Preprocessing Leakage).**
데이터를 분할하기 **전**에 정규화나 평균 제거를 수행한다. 이때 테스트 데이터의 통계가 학습에 스며든다. **올바른 순서**: 분할을 먼저, 학습 세트의 통계로 양쪽을 정규화.

**잘못된 분할의 실제 사례.**

**사례 1: 의료 영상 분류.**
한 연구팀이 "폐 X-ray로 폐렴을 95% 정확도로 진단"을 발표. 그러나 데이터셋은 100명 환자의 5000장 이미지였고, 무작위 분할로 같은 환자의 이미지가 양쪽에 들어갔다. 환자별 분할로 재실험하니 70%로 떨어졌다.

**사례 2: 주식 예측.**
학생이 5년 데이터를 무작위 80/20으로 분할하여 98% 정확도. 시간순 분할로 재실험하니 52%(랜덤 수준)로 떨어졌다.

**사례 3: 원격 탐사 지표 분류.**
위성 이미지를 픽셀 단위로 무작위 분할하여 93% 정확도. 인접 지역으로 분할하니 75%로 떨어졌다.

**사례 4: 재료 물성 예측.**
같은 조성의 여러 샘플을 무작위 분할. 94% 정확도. 조성별 분할로 재실험하니 68%.

**올바른 분할 전략 — 데이터 종류별.**

**전략 1: 시간적 분할.**
```python
# 잘못됨
train, test = train_test_split(data, test_size=0.2, random_state=42)

# 올바름
data_sorted = data.sort_values('timestamp')
split_idx = int(len(data_sorted) * 0.8)
train = data_sorted[:split_idx]
test = data_sorted[split_idx:]
```

교차검증에는 `TimeSeriesSplit`을 사용.
```python
from sklearn.model_selection import TimeSeriesSplit

tscv = TimeSeriesSplit(n_splits=5)
for train_idx, test_idx in tscv.split(X):
    X_train, X_test = X[train_idx], X[test_idx]
    y_train, y_test = y[train_idx], y[test_idx]
```

**전략 2: 그룹별 분할.**
```python
from sklearn.model_selection import GroupKFold

gkf = GroupKFold(n_splits=5)
for train_idx, test_idx in gkf.split(X, y, groups=patient_ids):
    # 같은 환자의 데이터는 한 쪽에만
    ...
```

**전략 3: 공간적 분할.**
```python
# 위도 기준 분할
north_region = data[data['lat'] > 37.5]  # 학습
south_region = data[data['lat'] <= 37.5]  # 테스트
```

원격 탐사, 지질 조사에 필수. Block Cross-Validation이 더 엄격한 방법.

**전략 4: 계층적 분할 (Stratified Split).**
불균형 데이터에서 클래스 분포를 유지. 다른 전략과 결합 사용.

```python
from sklearn.model_selection import StratifiedKFold

skf = StratifiedKFold(n_splits=5, shuffle=True)
for train_idx, test_idx in skf.split(X, y):
    ...
```

**전략 5: 중복 제거 후 분할.**
```python
import hashlib
data['hash'] = data['content'].apply(lambda x: hashlib.md5(x.encode()).hexdigest())
unique_data = data.drop_duplicates(subset='hash')
train, test = train_test_split(unique_data, test_size=0.2)
```

이미지나 텍스트는 정확 중복뿐 아니라 near-duplicate도 문제. Perceptual hash, MinHash 등 fuzzy 매칭이 필요.

**전략 6: 도메인별 분할 (Domain-Based Split).**
데이터를 수집한 조건(센서, 실험실, 환경)별로 분할.

```python
# 실험실 A에서 학습, 실험실 B에서 테스트
train = data[data['lab'] == 'A']
test = data[data['lab'] == 'B']
```

이 분할이 가장 엄격하고 실용적이다. 모델이 한 환경에서 다른 환경으로 얼마나 잘 전이되는지 확인.

**여러 분할 전략의 결합.**

때로는 여러 전략을 동시에 적용해야 한다. 예: 의료 데이터에서 시간과 환자 모두 중요한 경우.
- 먼저 시간 순으로 분할 (2020년까지 학습, 2021년 테스트)
- 동시에 환자별 분리 (같은 환자가 양쪽에 없도록)
- 두 조건을 모두 만족하는 샘플만 사용

**교차검증의 고급 형태.**

**Nested Cross-Validation**: 외부 루프는 모델 평가, 내부 루프는 하이퍼파라미터 튜닝. 가장 엄격한 평가.

```python
# 외부 5-fold
outer_cv = KFold(n_splits=5)
for train_idx, test_idx in outer_cv.split(X):
    X_train, X_test = X[train_idx], X[test_idx]
    
    # 내부 3-fold (하이퍼파라미터 튜닝)
    inner_cv = KFold(n_splits=3)
    best_model = find_best_model(X_train, y_train, inner_cv)
    
    # 외부 테스트
    score = best_model.score(X_test, y_test)
```

**Leave-One-Out 변형**: 데이터 수가 적을 때(N < 100). Leave-One-Group-Out, Leave-One-Subject-Out 등.

**Repeated Cross-Validation**: 같은 분할을 여러 시드로 반복. 분산 추정용.

**분할의 검증 — "분할이 올바른가"를 확인하는 법.**

**검증 1: Train/Test 통계 비교.**
두 세트의 기본 통계(평균, 분산, 클래스 분포)가 비슷해야 한다.

**검증 2: 베이스라인 모델의 성능.**
단순 모델(Logistic Regression, Random Forest)로 먼저 돌려본다. 비현실적으로 높으면(99%) 누출 가능성이 높다. **"너무 좋은 결과는 의심하라"**.

**검증 3: 반대 방향 예측.**
"train과 test를 구분할 수 있는가"를 확인. 두 세트를 합쳐 "train이냐 test냐"를 예측하는 classifier를 학습. 50% 근처(랜덤)면 분할이 잘 되었다. 90%면 두 세트가 시스템적으로 다르다는 뜻.

**검증 4: 시간 순서 확인 (시계열).**
```python
assert test['timestamp'].min() > train['timestamp'].max(), "Temporal leakage!"
```

**검증 5: 그룹 ID 확인.**
```python
assert len(set(train['patient_id']) & set(test['patient_id'])) == 0
```

이런 assertion을 코드에 넣어 두면 실수가 즉시 드러난다.

**논문에서의 분할 보고.**

**필수 정보**:
- 분할 전략 (temporal, group, spatial 등)
- 선택 이유
- Train/val/test 크기와 비율
- 교차검증 fold 수
- 랜덤 시드

**예시 서술**:
> "We used a temporal split: data from 2018-2022 for training, 2023 for validation, and 2024 for testing. This prevents future information from leaking into training. Additionally, we ensured that the same measurement session was not split across sets, addressing group leakage."

**분할 전 체크리스트.**

<div class="highlight-box highlight-info">

**데이터 분할 전 체크리스트**

- [ ] 데이터에 시간 순서가 있는가? → Temporal split
- [ ] 데이터에 개체 ID가 있는가? → Group split
- [ ] 데이터가 공간적으로 분포하는가? → Spatial split
- [ ] 데이터에 중복 가능성이 있는가? → Duplicate removal
- [ ] 데이터가 여러 도메인에서 수집되었는가? → Domain split
- [ ] 클래스 불균형이 있는가? → Stratified split 추가
- [ ] 전처리를 분할 전에 하고 있지 않은가?
- [ ] 특징 중에 목표와 너무 연관된 것이 있는가?
- [ ] Train/test 분포가 비슷한가?
- [ ] 베이스라인 성능이 비현실적이지 않은가?

</div>

**리뷰어의 흔한 지적.**

1. "How did you split the data? Random split seems inappropriate for this problem."
2. "Is there temporal/group leakage in your setup?"
3. "Your results seem too good. Can you verify with a more rigorous split?"
4. "Please provide an ablation study with different splitting strategies."
5. "How do you ensure generalization to unseen [patients/locations/time periods]?"

이런 질문들에 대한 답을 미리 준비해두면 리젝 위험이 크게 줄어든다.

**Kaggle과 학계의 차이.**

Kaggle에서는 무작위 분할이 흔하지만, 학술 논문에서는 더 엄격한 분할이 기대된다. Kaggle 경험만으로 학술 논문을 쓰면 "이것은 Kaggle 스타일이지 과학적 평가가 아니다"라는 지적을 받을 수 있다.

> 데이터 분할은 "1줄의 코드"가 아니라 연구의 과학적 유효성을 결정하는 설계다. 데이터에 적합한 분할을 선택하고, 누출이 없는지 검증하고, 논문에 상세히 보고한다. 이것을 소홀히 한 수백 편의 논문이 "재현 실패"로 비판받고 있다. 논문이 그 중 하나가 되지 않도록, 분할에 며칠을 투자한다.

## ML 파이프라인의 테스트 — 연구 코드에도 테스트가 필요하다

박사 학생의 ML 코드의 대부분은 테스트가 없다. 모델이 "돌아간다"는 것을 확인하고 결과를 논문에 쓴다. 모델이 실제로 의도한 것을 하는지 확인하지 않는다. 전처리 스크립트에 버그가 있어서 데이터의 한 부분이 잘못 처리되어도 모른다. 평가 지표 계산에 오류가 있어도 모른다. 박사 논문의 결과가 버그 때문에 잘못되어도 모른다.

<div class="highlight-box warning">
  <span class="highlight-box-icon">⚠️</span>
  <div class="highlight-box-content">
    <p><strong>테스트 없는 ML의 실제 위험</strong></p>
    <p>2015년 이후 여러 ML 논문의 재현 실패의 상당 부분이 "코드의 버그"에서 기인했다. 유명 논문의 저자가 버그를 논문 출판 2-3년 후에 발견하고 논문을 취소(retraction)한 경우가 여러 번 있었다. 박사 논문이 이 운명을 맞지 않도록 코드에 테스트를 작성한다. 테스트가 "엔지니어의 취미"가 아니라 과학적 엄밀성의 한 부분이다.</p>
  </div>
</div>

### ML 코드의 5가지 테스트 레벨

ML 코드의 테스트는 일반 소프트웨어 테스트와 다르다. ML 파이프라인의 각 부분에 다른 종류의 테스트를 적용한다.

**레벨 1: 단위 테스트 (Unit Tests)**

가장 기본적인 테스트. 한 함수의 한 동작을 확인.

테스트할 대상:
- 데이터 전처리 함수 (정규화, 원핫 인코딩, 결측치 처리)
- 특성 공학 함수 (특성 추출, 변환)
- 평가 지표 계산 함수 (정밀도, 재현율, F1)
- 데이터 로더 함수

예시 (pytest):
```python
def test_normalize():
    # 알려진 입력과 출력으로 테스트
    x = np.array([1, 2, 3, 4, 5])
    result = normalize(x)
    expected = np.array([-1.414, -0.707, 0, 0.707, 1.414])
    assert np.allclose(result, expected, atol=1e-3)

def test_precision_recall_edge_case():
    # 극단 사례 테스트
    # 모든 예측이 정답인 경우
    y_true = [1, 1, 0, 0]
    y_pred = [1, 1, 0, 0]
    assert precision(y_true, y_pred) == 1.0
    assert recall(y_true, y_pred) == 1.0
```

단위 테스트가 함수의 "정상 경로(happy path)"와 "극단 사례(edge case)"를 모두 확인.

**레벨 2: 데이터 테스트 (Data Tests)**

데이터의 구조와 통계적 속성을 확인.

테스트할 대상:
- 데이터의 크기 (예상 샘플 수)
- 데이터의 형식 (형태, 데이터 타입)
- 결측치 (허용 한도)
- 값의 범위 (센서 값이 물리적으로 가능한 범위인가)
- 클래스 분포 (예상 비율)

예시:
```python
def test_data_shape():
    data = load_dataset()
    assert data.shape == (10000, 50), f"예상 (10000, 50), 실제 {data.shape}"

def test_no_missing_values():
    data = load_dataset()
    missing = data.isnull().sum().sum()
    assert missing == 0, f"{missing}개의 결측치 발견"

def test_temperature_range():
    # 센서 데이터의 온도가 물리적으로 가능한 범위인가
    data = load_dataset()
    temps = data['temperature']
    assert temps.min() > -50, f"너무 낮은 온도: {temps.min()}"
    assert temps.max() < 200, f"너무 높은 온도: {temps.max()}"

def test_class_balance():
    # 클래스 분포가 예상 범위 내인가
    labels = load_labels()
    class_ratio = labels.value_counts(normalize=True)
    assert 0.4 < class_ratio[0] < 0.6, "클래스 불균형 발생"
```

데이터 테스트는 데이터의 "안전 검사". 데이터가 예상과 다르면 테스트가 알려준다.

도구: `pytest`, `Great Expectations`, `pandera`, `Pydantic`.

**레벨 3: 모델 테스트 (Model Tests)**

모델 자체의 동작을 확인.

테스트할 대상:
- 모델이 입력을 받아 예상 형태의 출력을 만드는가
- 모델의 파라미터 수가 예상과 일치하는가
- 모델이 그래디언트를 계산할 수 있는가
- 모델이 한 에폭 학습 후 손실이 감소하는가 (학습 능력 확인)
- 모델이 매우 작은 데이터에서 과적합할 수 있는가 (overfit test)

예시 (PyTorch):
```python
def test_model_output_shape():
    model = MyModel()
    x = torch.randn(32, 3, 224, 224)  # 배치 크기 32
    output = model(x)
    assert output.shape == (32, 10), f"예상 (32, 10), 실제 {output.shape}"

def test_model_can_overfit():
    # 본인의 모델이 10개 샘플에 과적합 가능한가?
    # 가능하지 않으면 본인의 모델에 본인이 버그가 있다.
    model = MyModel()
    x = torch.randn(10, 3, 224, 224)
    y = torch.randint(0, 10, (10,))
    
    optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
    criterion = nn.CrossEntropyLoss()
    
    for _ in range(200):
        optimizer.zero_grad()
        loss = criterion(model(x), y)
        loss.backward()
        optimizer.step()
    
    # 학습 후 손실이 매우 작아야 한다
    assert loss.item() < 0.01, f"과적합 실패: loss={loss.item()}"
```

**"모델이 10개 샘플에 과적합할 수 있는가"** 테스트는 ML 학계의 "디버깅의 황금 표준"이다. 모델이 매우 작은 데이터에 과적합하지 못하면 모델 구조에 버그가 있다. 이 테스트를 모든 새 모델에 적용한다.

**레벨 4: 통합 테스트 (Integration Tests)**

전체 파이프라인의 한 끝에서 다른 끝까지 실행을 확인.

테스트할 대상:
- 데이터 로딩 → 전처리 → 학습 → 평가의 전체 흐름
- 작은 샘플 데이터로 전체 파이프라인을 한 번 실행
- 저장된 모델을 다시 로드하여 예측을 만들기

예시:
```python
def test_end_to_end_pipeline():
    # 작은 데이터로 전체 파이프라인 실행
    with tempfile.TemporaryDirectory() as tmpdir:
        # 데이터 생성
        data = create_tiny_dataset(n=100)
        data.to_csv(f"{tmpdir}/data.csv")
        
        # 파이프라인 실행
        pipeline = MLPipeline(
            data_path=f"{tmpdir}/data.csv",
            model_dir=tmpdir
        )
        pipeline.train()
        results = pipeline.evaluate()
        
        # 결과가 합리적인가?
        assert 0 <= results['accuracy'] <= 1
        assert results['loss'] > 0
```

통합 테스트는 "파이프라인이 돌아간다"는 것을 매번 확인하는 도구.

**레벨 5: 재현성 테스트 (Reproducibility Tests)**

같은 입력과 같은 시드로 두 번 실행했을 때 같은 결과가 나오는가를 확인.

예시:
```python
def test_reproducibility():
    # 같은 시드로 두 번 학습
    set_seed(42)
    model1 = train_model(data)
    pred1 = model1.predict(test_data)
    
    set_seed(42)
    model2 = train_model(data)
    pred2 = model2.predict(test_data)
    
    # 예측이 일치해야 한다
    assert np.allclose(pred1, pred2), "재현성 실패"
```

재현성 테스트가 실패하면 코드에 비결정적 요소가 있다 (시드 고정 누락, 병렬 처리의 순서, GPU 비결정성). 비결정성을 찾아 수정.

### ML 테스트의 5가지 실전 원칙

**원칙 1: 빠른 테스트를 유지**

테스트 전체 실행이 5-10분을 넘으면 테스트를 실행하지 않는다. 테스트를 빠르게 유지:

- 작은 합성 데이터 사용 (10-100 샘플)
- 작은 모델 사용 (2-3 layer)
- 매우 짧은 학습 (1-5 에폭)

전체 테스트가 30초-2분 이내.

**원칙 2: CI/CD에 통합**

테스트를 Git push마다 자동 실행. GitHub Actions, GitLab CI.

예시 (GitHub Actions):
```yaml
name: Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      - run: pip install -r requirements.txt
      - run: pytest tests/
```

CI가 실패 시 본인에게 알림. 버그를 즉시 수정.

**원칙 3: 테스트 커버리지 측정**

코드의 얼마의 부분이 테스트되었는가. `pytest-cov` 도구.

```bash
pytest --cov=src tests/
```

커버리지가 50-70%가 되도록 목표. 100%는 현실적이지 않다.

**원칙 4: 버그를 발견할 때마다 테스트 추가**

코드에서 버그를 발견하면 버그를 재현하는 테스트를 먼저 작성한다. 그 다음 버그를 수정. 테스트가 미래의 같은 버그를 방지.

이것이 Test-Driven Development의 한 형태. 테스트가 "버그의 박물관"이 된다.

**원칙 5: 박사 논문 제출 전 전체 테스트 실행**

박사 논문을 심사 위원회에 제출하기 전에 전체 테스트를 실행. 모든 테스트가 통과하는가를 확인. 테스트의 결과를 박사 논문의 부록에 포함.

박사 논문의 "코드가 테스트를 통과한다"가 과학적 엄밀성의 증거.

### 테스트를 시작하는 법 — 박사 과정의 단계별 접근

박사 1년차에 테스트를 완벽하게 적용할 필요가 없다. 박사 과정의 단계에 맞게 테스트를 점진적으로 도입.

**박사 1-2년차: 기초 테스트**
- 데이터 로딩 함수에 단위 테스트
- 평가 지표 함수에 단위 테스트
- "모델이 10개 샘플에 과적합할 수 있는가" 테스트

목표: 코드의 가장 기본적인 부분에 버그가 없음을 보장.

**박사 2-4년차: 중급 테스트**
- 전처리 파이프라인의 통합 테스트
- 데이터 테스트 (크기, 형식, 결측치)
- 재현성 테스트
- CI/CD 통합

목표: 박사 논문의 핵심 결과가 버그에서 자유로움을 보장.

**박사 4-7년차: 고급 테스트**
- 전체 파이프라인의 End-to-End 테스트
- 회귀 테스트 (이전 결과가 변경되지 않는가)
- 배포 테스트 (모델이 실전 환경에서 작동하는가)

목표: 박사 논문의 재현성과 박사 후의 코드 유지보수 가능성.

### ML 테스트의 흔한 변명과 답

**변명 1: "연구 코드는 한 번만 쓰고 버릴 것이다"**

답: 연구 코드가 "한 번만 쓰고 버릴" 것이라면 코드의 결과를 박사 논문에 사용하면 안 된다. 박사 논문의 결과는 평생 이름과 연결된다. 코드가 박사 논문의 결과의 기초이면 코드의 정확성이 중요하다.

**변명 2: "테스트 작성이 시간이 너무 많이 걸린다"**

답: 코드의 버그로 박사 졸업이 6개월 지연되는 것보다 테스트 작성에 1-2주를 투자하는 것이 더 빠르다. 테스트가 "미래의 시간 절약".

**변명 3: "ML 코드는 테스트가 어렵다"**

답: ML 코드가 일반 소프트웨어보다 테스트가 어려운 것은 사실. 그러나 불가능하지 않다. 위의 5가지 레벨 중 시작 가능한 것부터 시작한다. 데이터 테스트와 단위 테스트가 가장 쉽다.

**변명 4: "지도교수가 테스트를 요구하지 않는다"**

답: 지도교수가 테스트를 요구하지 않아도 리뷰어가 재현성을 요구한다. 2020년대 중반 이후의 ML 저널은 재현성 체크리스트를 요구. 체크리스트의 한 항목이 "코드에 테스트가 있는가?"

**변명 5: "테스트를 작성하는 법을 모른다"**

답: 학습의 한 부분으로 테스트 작성을 배운다. pytest 튜토리얼은 2-3시간의 학습으로 기초를 제공. 투자가 박사 과정 전체에 걸쳐 본인에게 이익을 준다.

### 박사 논문의 테스트 섹션 구성

**구성 1: 테스트 전략 설명**

Appendix 또는 Methodology 섹션에 테스트 전략을 설명. 5가지 레벨 중 적용한 것들.

**구성 2: 테스트 커버리지**

코드의 몇 %가 테스트되었는지 명시. `pytest-cov`의 결과를 보고.

**구성 3: 핵심 테스트의 예시**

**구성 4: 테스트 실행 결과**

박사 논문 제출 시점의 모든 테스트가 통과함을 명시. 테스트 결과의 스크린샷 또는 요약 표.

**구성 5: 코드와 테스트의 공개**

박사 논문 출판 시점에 코드와 테스트를 GitHub에 공개. 박사 논문의 URL을 명시. ch38(오픈 사이언스)과 연결.

## 마지막 — 테스트는 박사의 과학적 양심의 코드화

박사 학위가 과학이라면 과학의 결과가 정확해야 한다. 결과의 정확성은 코드의 정확성에서 나온다. 코드의 정확성은 테스트에서 나온다. 테스트가 박사의 과학적 양심의 코드화된 형태다.

박사 1-2년차에 테스트를 시작하면 박사 5-7년 동안 테스트가 일상의 한 부분이 된다. 박사 졸업 시점에 테스트 없이는 ML 코드를 쓰지 못하는 사람이 된다. 이 습관이 박사 학위의 한 자산.

박사 졸업 후의 첫 자리 (학계 또는 산업체)에서 테스트 능력이 차별화 요소가 된다. 산업체에서 테스트 없는 코드는 제품으로 배포되지 않는다. 테스트를 박사 과정에서 배우면 박사 후 첫 자리의 적응이 크게 쉬워진다.

박사 학위가 논문의 결과의 합이 아니라 정확한 결과를 만드는 능력이다. 테스트가 이 능력의 한 기둥이다.

## 데이터 라벨링 — 공학 박사의 잊혀진 병목

ML 프로젝트의 교과서는 모델 아키텍처, 학습 알고리즘, 평가 지표에 초점을 맞춘다. 그러나 공학 박사생이 실제로 겪는 가장 큰 시간 소비와 좌절의 원천은 다른 곳에 있다. **데이터 라벨링**이다. "이 이미지에는 균열이 있는가?", "이 진동 신호의 이상 시점은 어디인가?", "이 재료 미세구조의 상(phase) 경계는?" 이런 질문에 답을 붙이는 것이 라벨링이다. 대부분의 공학 ML 논문에서 "우리는 2,000개 샘플을 수집했다"라는 한 줄이 실은 수백 시간의 라벨링 노동을 감추고 있다. 이 병목을 인식하고 전략적으로 다루지 못하면 박사 과정에서 ML 연구는 좌절의 연속이 된다.

<div class="highlight-box highlight-important">

**"라벨링이 곧 연구의 한계다".** 많은 학생이 "모델이 중요하다"고 믿지만, 공학 ML의 현실에서 모델의 차이는 5-10%이고 라벨 품질의 차이는 30-50%다. 라벨이 부정확하면 아무리 좋은 모델도 부정확한 학습을 한다. 라벨이 부족하면 아무리 좋은 알고리즘도 일반화하지 못한다. 박사 ML 연구의 성공은 모델 설계보다 라벨링 전략에 더 크게 의존한다. 이 사실을 늦게 깨닫는 학생이 많다.

</div>

**라벨링의 현실 — 5가지 숨은 비용.**

라벨링의 비용은 단순히 "시간"만이 아니다.

**비용 1: 시간.**
한 이미지에 균열 마스크를 그리는 데 2-5분. 10,000장이면 500-800시간. 박사 한 학기의 절반이다. 시계열 이상 탐지 라벨링은 더 오래 걸릴 수 있다.

**비용 2: 도메인 전문성.**
많은 공학 라벨링은 전문가만 할 수 있다. 학부 알바나 크라우드소싱으로는 불가능. 본인, 동료 박사생, 또는 실무 엔지니어만 가능. 이것이 확장성을 제한한다.

**비용 3: 정신적 피로.**
라벨링은 반복적이고 집중력을 요한다. 4-5시간 후 정확도가 급격히 떨어진다. 하루 4시간 이상의 라벨링은 품질을 해친다.

**비용 4: 불확실성.**
"이것이 균열인가 아닌가"가 항상 명확하지 않다. 모호한 경우들이 많다. 이것이 라벨러 간 불일치(inter-annotator disagreement)로 이어진다.

**비용 5: 재작업 비용.**
라벨링 기준이 중간에 바뀌면 이미 라벨링한 것을 다시 해야 한다. 이것이 가장 흔한 낭비. 기준을 먼저 정의하지 않고 시작하는 학생이 많다.

이 비용들을 모두 고려하면, 라벨링은 단순히 "시간 있을 때 해버리면 되는" 작업이 아니라 박사 연구의 전략적 결정이다.

**라벨링 전략의 5가지 질문.**

라벨링을 시작하기 전에 반드시 답해야 할 질문들.

**질문 1: 라벨링 가이드라인이 있는가?**

라벨링 시작 전에 상세한 가이드라인을 작성한다. 이것 없이 시작하면 재작업이 불가피.

**가이드라인에 포함할 것**:
- 라벨 정의 (각 클래스의 정확한 의미)
- 경계 케이스의 처리 규칙 (모호한 경우)
- 실제 예시 (좋은 라벨과 나쁜 라벨)
- 특수 상황의 처리 (예: 부분적으로 가려진 경우, 여러 개가 겹친 경우)
- 품질 기준 (어느 정도 정확해야 하는가)

**예시 — 균열 검출 라벨링 가이드라인**:
```
균열: 폭 0.1mm 이상, 길이 5mm 이상의 선형 결함
     - 표면 긁힘은 포함 안 함 (깊이 0.1mm 미만)
     - 분기 균열은 하나의 연속된 객체로 라벨링
     - 그림자는 균열 아님
     - 50% 이상 가려진 균열은 라벨링 제외
```

가이드라인이 10-20페이지가 될 수도 있다. 이것이 재작업 시간을 10배 이상 절약한다.

**질문 2: 몇 명이 라벨링할 것인가?**

단일 라벨러 vs 다중 라벨러의 선택.

**단일 라벨러**:
- 장점: 일관성, 빠름, 단순
- 단점: 주관성, 오류 검증 불가, 병목

**다중 라벨러 (2-3명)**:
- 장점: 오류 검증, 객관성, 합의 가능
- 단점: 느림, 조율 필요, 불일치 처리

**결정 기준**: 라벨이 객관적이면 단일, 주관적이면 다중. 박사 논문의 중요한 실험은 다중 권장.

**질문 3: 라벨러 간 합의(agreement)를 어떻게 측정할 것인가?**

다중 라벨러를 사용하면 합의도를 측정해야 한다. 이것이 라벨 품질의 지표.

**측정 지표**:
- **Cohen's Kappa**: 두 라벨러의 합의. 0(우연)에서 1(완전 일치).
- **Fleiss' Kappa**: 세 명 이상의 라벨러.
- **IoU (Intersection over Union)**: 영역 기반 라벨링 (세그멘테이션)의 합의.
- **Dice coefficient**: 세그멘테이션의 또 다른 합의 지표.

**해석 기준**:
- κ > 0.8: 매우 좋음
- 0.6 < κ < 0.8: 좋음
- 0.4 < κ < 0.6: 중간
- κ < 0.4: 문제 있음 (가이드라인 재검토 필요)

**박사 논문에서의 보고**: "라벨러 간 Cohen's Kappa = 0.84"는 데이터 품질에 대한 신뢰를 준다.

**질문 4: 라벨링 도구는 무엇인가?**

손으로 마우스로 모든 것을 그리는 시대는 지났다. 전문 라벨링 도구를 사용한다.

**주요 도구**:
- **Label Studio**: 오픈소스, 다양한 데이터 유형 지원. 가장 인기.
- **CVAT (Intel)**: 이미지/비디오 특화. 강력한 기능.
- **LabelImg**: 간단한 이미지 bounding box. 초보용.
- **VIA (VGG Image Annotator)**: 브라우저 기반, 설치 불필요.
- **Prodigy**: 상용, 액티브 러닝 통합.
- **Roboflow**: 상용, 클라우드 기반.
- **SuperAnnotate**: 상용, 협업 중심.

**선택 기준**:
- 본인 데이터의 유형과 매칭
- 협업 기능 (다중 라벨러)
- 내보내기 형식 (COCO, Pascal VOC, YOLO 등)
- 가격

공학 박사에는 Label Studio 또는 CVAT가 가장 적합하다.

**질문 5: 라벨링 계획의 타임라인은?**

라벨링을 "나중에 하면 되지"라고 미루면 항상 부족하다. 명시적 계획 필요.

**권장 타임라인**:
- **박사 1-2년차**: 라벨링 가이드라인 작성, 파일럿 라벨링 (100-500 샘플)
- **박사 2-3년차**: 본격 라벨링 (1000-5000 샘플), 품질 검증
- **박사 3-4년차**: 추가 라벨링 필요 판단, 액티브 러닝 적용
- **박사 4-5년차**: 최종 확장, 공개 준비

라벨링은 누적 작업이다. 조금씩 꾸준히가 한 번에 몰아서보다 효과적.

**효율적 라벨링의 6가지 기법.**

라벨링 시간을 줄이는 실전 기법들.

**기법 1: 파일럿 라벨링.**

본격 라벨링 전에 100-200 샘플로 파일럿. 목적:
- 가이드라인의 문제점 발견
- 예상 시간의 현실적 추정
- 라벨링 도구의 효율성 테스트
- 라벨러의 훈련

파일럿 후에 가이드라인과 프로세스를 수정. 이것이 본격 라벨링의 품질과 효율을 크게 향상.

**기법 2: 사전 라벨링 (Pre-labeling).**

처음부터 손으로 라벨링하지 말고, 기존 모델(단순한 것도 포함)로 초기 라벨을 생성하고 수정만. "수정"이 "처음부터 그리기"보다 5-10배 빠르다.

**구체적 방법**:
1. 먼저 100-500개의 샘플을 손으로 라벨링
2. 이것으로 단순 모델 학습 (fine-tune)
3. 이 모델로 나머지 데이터에 초기 라벨 생성
4. 오류만 수정
5. 필요하면 이 과정을 반복 (Label → Train → Predict → Refine)

이것이 액티브 러닝의 기본 아이디어.

**기법 3: 액티브 러닝(Active Learning).**

모든 샘플을 라벨링하지 말고, "모델이 가장 불확실한" 샘플만 골라서 라벨링. 이것이 같은 시간으로 더 좋은 모델을 만든다.

**기본 전략**:
1. 초기 모델 학습 (작은 라벨셋)
2. 미라벨 데이터에 대한 모델의 예측 불확실성 계산
3. 불확실성이 가장 큰 샘플 선택
4. 이 샘플만 라벨링
5. 모델 재학습
6. 반복

**효과**: 10,000개 중 2,000개만 라벨링해도 전체를 다 한 것과 비슷한 성능.

**주의**: 액티브 러닝은 "편향된 샘플"을 선택할 수 있다. 테스트 세트는 별도로 무작위 샘플링.

**기법 4: 약한 라벨(Weak Labels)의 활용.**

완벽한 라벨 대신 "대략적 라벨"을 쉽게 생성하고, 그것으로 모델을 학습.

**예시**:
- 이미지 분류에서 image-level 라벨만으로 pixel-level 세그멘테이션 학습
- 텍스트 분류에서 키워드 매칭으로 약한 라벨 생성
- 휴리스틱으로 자동 라벨링 (예: "밝기가 임계값 이상이면 결함")

약한 라벨은 정확도가 떨어지지만 대량으로 생성 가능. 이것을 작은 정확한 라벨셋과 결합하면 효과적.

**관련 기법**: Snorkel 프레임워크, Noisy label learning.

**기법 5: 학부생/인턴의 활용.**

모든 라벨링을 직접 할 필요는 없다. 학부생이나 인턴이 할 수 있는 부분이 있다.

**위임 가능한 것**:
- 명확한 기준의 단순 분류
- Bounding box 그리기 (엄밀한 마스크가 아닌)
- 초벌 라벨링 (검수)
- 반복적 작업

**위임 불가능한 것**:
- 고도의 도메인 지식이 필요한 판단
- 최종 품질 검수
- 경계 케이스의 결정

**관리 방법**:
- 명확한 가이드라인 제공
- 초반 1-2일 함께 작업하며 훈련
- 정기적인 품질 검수 (5-10% 샘플링)
- 적절한 보상 (연구실 식사, 소정의 급여)

**주의**: 저자 포함 여부를 미리 명확히 한다. 단순 라벨링만 한 경우 보통 저자가 아니지만, 기여도가 크면 Acknowledgment에 포함.

**기법 6: 라벨링 세션의 설계.**

긴 라벨링 마라톤 대신 짧은 세션을 분산.

**권장 세션**:
- 한 번에 1-2시간
- 사이에 10분 휴식
- 하루 총 4-5시간 이하
- 주간 총 20-25시간 이하
- 같은 시간대에 (오전이 좋음)

**환경**:
- 조용한 곳
- 편한 의자와 키보드
- 큰 모니터 (이미지 라벨링)
- 음악 없음 (집중력 유지)

**피로 관리**: 피로할 때 라벨링하면 오류가 증가. 차라리 쉬는 것이 낫다. ch04의 에너지 관리와 연결.

**라벨 품질 관리 — 품질을 보장하는 4가지 방법.**

라벨을 생성하는 것과 별개로, 품질을 유지하는 시스템이 필요하다.

**방법 1: 중복 라벨링(Double Labeling).**
핵심 테스트 세트의 5-20%를 두 명이 독립적으로 라벨링. 불일치 분석. 이것이 라벨 품질의 기준 지표.

**방법 2: 본인 라벨의 재라벨링.**
라벨링 시작 후 1개월, 초기에 라벨링한 50개를 다시 라벨링. 일관성 확인. 큰 불일치가 있으면 가이드라인 재검토.

**방법 3: 이상치 탐지.**
라벨링된 데이터에서 통계적 이상치를 찾는다. 예: "이 사진은 라벨이 A인데 다른 A들과 모양이 완전히 다르다" → 라벨 오류 가능성.

**방법 4: 모델의 불일치 사용.**
학습된 모델이 높은 확신으로 틀린 예측을 하는 샘플들. 이것은 종종 라벨 오류다. 이런 샘플을 재검토.

이 4가지 방법을 조합하면 라벨 오류율을 5% 이하로 유지할 수 있다.

**박사 논문에서의 라벨링 보고.**

박사 논문이나 ML 논문에서 라벨링에 대한 투명한 보고가 점점 요구된다.

**보고할 항목**:
- 라벨러의 수와 전문성
- 라벨링 가이드라인 (요약 또는 부록)
- 라벨러 간 합의도 (Kappa, IoU 등)
- 라벨링 도구
- 품질 관리 절차
- 예외 사항 처리
- 추정된 라벨 오류율

이 투명한 보고가 데이터의 신뢰성을 증명. 많은 논문이 이 부분을 간과하는데, 정직한 보고가 오히려 리뷰어에게 긍정적 인상을 준다.

**라벨링의 장기적 가치 — 연구 자산.**

박사 과정에서 만든 라벨셋은 단지 한 논문의 도구가 아니라 연구 자산이다.

**장기 가치**:
1. **후속 논문의 기반**: 같은 라벨셋으로 여러 논문.
2. **데이터 논문(Data Paper)의 출판**: ch07에서 다룬 것처럼, 라벨셋 자체를 독립 출판.
3. **후배에게 전수**: 다음 세대가 라벨셋 위에서 연구.
4. **공개 기여**: Zenodo, figshare에 공개해 분야에 기여. ch38의 오픈 사이언스.
5. **포닥/취업의 증거물**: "양질의 데이터셋을 만든 경험"이 강한 신호.

따라서 라벨링을 "귀찮은 일"이 아니라 "장기 자산 구축"으로 본다. 이 관점이 라벨링에 대한 태도를 바꾼다.

**라벨링의 심리학 — 동기 유지의 기술.**

라벨링은 지루하고 반복적이다. 수백 시간의 라벨링을 버티는 심리적 기법.

1. **진척 가시화**: 매일 라벨링한 수를 기록. 그래프로 시각화. 누적량이 동기 부여.
2. **마일스톤 설정**: "1,000개 라벨링 완료"에 작은 보상.
3. **동료와 함께**: 가능하면 다른 학생과 함께 라벨링 세션. 커피숍 라벨링 파티.
4. **목적 상기**: 라벨링이 무엇을 위한 것인지 주기적으로 상기. 큰 그림 유지.
5. **완벽주의 회피**: "이 라벨이 95% 좋으면 충분"을 받아들인다. 100%는 불가능.
6. **다양화**: 하루 한 가지만 하지 말고, 라벨링 1시간 후 다른 작업 1시간.

이 기법들이 본인을 라벨링 마라톤에서 완주하게 한다.

> 데이터 라벨링은 공학 ML 박사의 숨은 병목이다. 모델, 알고리즘, 평가에 대한 책과 논문은 많지만 라벨링에 대한 실전 지식은 선배에게서 배우거나 스스로 고생하며 얻는다. 이 섹션의 원칙들을 일찍 적용하면 박사 ML 연구의 시간과 좌절을 크게 줄일 수 있다. 라벨셋의 품질이 박사의 품질이다. 모델보다 라벨에 먼저 투자하라.

## ML 모델 버전 관리와 실험 분기 — 수백 실험의 관리

박사 ML 연구의 어느 시점에 본인은 한 가지를 깨닫는다. "내가 했던 실험이 수백 개인데 어느 것이 어느 것인지 모르겠다." 모델 파일, 체크포인트, 로그, 설정 파일, 결과 그림. 모두 섞여 있고 구분할 수 없다. "3주 전 Reynolds 1000 실험"을 찾으려면 폴더 수십 개를 뒤져야 한다. "이 모델이 어떤 설정으로 훈련되었는지" 기억이 안 난다. 이 혼란이 축적되면 연구는 멈춘다. 체계적 **버전 관리와 실험 분기**가 필수다. 코드의 Git과 유사하게, ML 실험도 고유한 버전 관리가 필요하다. 이 섹션은 수백 실험을 관리하는 체계를 다룬다.

<div class="highlight-box highlight-important">

**"ML 실험은 코드보다 복잡하다".** 코드의 Git은 텍스트 파일의 변경을 추적한다. 그러나 ML 실험은 여러 차원을 가진다: 코드, 데이터, 하이퍼파라미터, 환경, 결과, 체크포인트. 이 모든 것이 실험의 정체성을 구성한다. 단순한 Git으로는 충분하지 않다. ML 실험 관리는 별도의 체계가 필요하다. 박사 초기에 이 체계를 구축하면 박사 5-7년 동안 수백 실험을 안전하게 관리할 수 있다. 구축하지 않으면 본인은 "어느 설정이었지?"를 반복하며 시간을 잃는다.

</div>

**ML 실험의 "6가지 차원"**.

한 실험을 정의하는 요소들.

**차원 1: 코드 버전**.
실험 당시 사용한 코드. Git commit 해시.

**차원 2: 데이터 버전**.
훈련/테스트에 사용한 데이터의 버전. 전처리 포함.

**차원 3: 하이퍼파라미터**.
학습률, 배치 크기, 옵티마이저, 모델 크기 등.

**차원 4: 환경**.
PyTorch 버전, CUDA 버전, 의존성 라이브러리.

**차원 5: 시드 (Seed)**.
랜덤 시드. 재현성의 핵심.

**차원 6: 결과**.
메트릭, 체크포인트, 로그, 시각화.

**이 6가지가 일치해야** 실험이 재현 가능하다. 하나라도 빠지면 "대략 비슷한" 결과만 가능.

**문제의 규모 — 박사 과정의 실험 수**.

박사 과정에서 실행하는 실험의 대략적 수.

**보수적 추정**:
- 박사 1년차: 50-100개
- 박사 2년차: 200-500개
- 박사 3-4년차: 500-2000개
- 박사 5-6년차: 1000-3000개
- **박사 전체**: 2000-5000개

이 중 논문에 나오는 것은 20-50개. 나머지는 탐색, 실패, 반복.

**왜 이렇게 많은가**:
- 하이퍼파라미터 튜닝 (100-500개)
- ablation 연구 (50-100개)
- 랜덤 시드 반복 (각 실험 x 5)
- 실패와 재시도 (많음)
- 탐색적 실험 (많음)

**체계 없이는** 이 많은 실험을 추적 불가능.

**계층적 관리 — 실험의 구조**.

실험을 계층으로 조직.

**계층 1: 프로젝트 (Project)**.
박사의 한 연구 주제. 논문 1-3편에 해당.

**예시**: "PINN for crack detection"

**계층 2: 실험 시리즈 (Experiment Series)**.
한 프로젝트 내의 관련 실험들. 한 연구 질문에 답함.

**예시**: 
- "Series A: Architecture comparison"
- "Series B: Loss weight tuning"
- "Series C: Data size sensitivity"

**계층 3: 개별 실험 (Individual Run)**.
한 구성의 한 실행.

**예시**: "Series B, λ_pde=1, λ_bc=10, seed=42"

**계층 4: Epoch/Checkpoint**.
한 실험 내의 시점.

**조직 예시**:
```
phd-research/
├── project-pinn-crack/
│   ├── series-A-architecture/
│   │   ├── run-001-resnet/
│   │   ├── run-002-transformer/
│   │   └── run-003-pinn/
│   ├── series-B-loss-weights/
│   │   ├── run-001-w1-1-w2-10/
│   │   ├── run-002-w1-1-w2-100/
│   │   └── ...
│   └── README.md
└── project-ml-fatigue/
    └── ...
```

**원칙**: 프로젝트 → 시리즈 → 실행의 3단계. 검색과 관리가 쉬워진다.

**실험의 고유 식별자**.

각 실험에 고유한 이름을 부여.

**좋은 이름의 요소**:
1. **날짜**: YYYYMMDD
2. **프로젝트**: 짧은 코드
3. **시리즈**: 시리즈 이름
4. **핵심 설정**: 주요 하이퍼파라미터
5. **실행 번호**: 순서

**예시**:
```
20260415_pinn_crack_Slw_lambda1_w10_seed42
```

- 프로젝트: pinn_crack
- 시리즈: Slw (Series loss weight)
- 설정: λ1=1, λ2=10
- 시드: 42

**대안 — UUID 또는 해시**:
```

```
- 완전 고유
- 짧음
- 해석 불가능 (메타데이터 별도 필요)

**선택**: 해석 가능한 이름이 좋음. 극단적으로 많으면 UUID.

**체크포인트 관리**.

ML 모델의 체크포인트는 용량이 크다. 관리 필요.

**체크포인트의 유형**:

**유형 1: 정기 체크포인트**.
매 N epoch마다 저장. 실패 시 재개.

```python
if epoch % 10 == 0:
    torch.save({
        'epoch': epoch,
        'model_state_dict': model.state_dict(),
        'optimizer_state_dict': optimizer.state_dict(),
        'loss': loss,
    }, f'checkpoint_epoch_{epoch}.pt')
```

**유형 2: Best 체크포인트**.
가장 좋은 검증 성능의 모델만.

```python
if val_loss < best_val_loss:
    best_val_loss = val_loss
    torch.save(model.state_dict(), 'best_model.pt')
```

**유형 3: 최종 체크포인트**.
훈련 종료 시의 모델.

**보관 전략**:
- **Hot**: 최근 실험의 모든 체크포인트 (로컬)
- **Warm**: 중요 체크포인트 (연구실 서버)
- **Cold**: 논문용 최종 모델 (장기 백업)

**용량 관리**:
- 한 체크포인트: 10MB-1GB
- 박사 과정: 총 100-1000GB
- 정기 청소 필요

**청소 기준**:
- 실패한 실험: 즉시 삭제
- 탐색 실험: 1개월 후 삭제
- 논문 실험: 영구 보관
- Best 체크포인트만 보관 (대부분의 경우)

**실험 분기 (Branching)**.

"실험의 분기"는 Git의 브랜치와 유사.

**상황**: 실험이 두 갈래로 나뉨. "어떤 방향이 나을까?"

**예시**:
- 메인 모델: GCN 기반
- 분기 A: GAT로 교체
- 분기 B: GCN + attention 추가
- 분기 C: 완전히 새로운 아키텍처

**관리**:
각 분기를 독립적으로 실행. 결과 비교 후 방향 결정.

**도구**:
- Git 브랜치 (코드의 분기)
- 별도 config 파일 (분기별)
- 실험 추적 도구의 그룹 (W&B의 project/group)

**원칙**:
1. 분기는 명확한 질문으로 시작
2. 각 분기에 충분한 자원 투입
3. 공정한 비교 (같은 시드, 같은 데이터)
4. 결과로 결정
5. 실패한 분기도 기록 (교훈)

**분기의 보관**:
- 선택된 분기: 주 라인으로
- 탐색된 분기: 별도 보관 (부록이나 보충 자료)
- 실패한 분기: 교훈 노트

**실험 일지 (Experiment Log)**.

실험마다 메타 정보를 기록.

**일지의 구조**:
```markdown
# Experiment Log: 20260415_pinn_crack_Slw_001

## 목적
손실 가중치 λ_bc=10이 λ_bc=1보다 수렴을 개선하는지 확인.

## 가설
λ_bc=10이면 경계 조건 위반이 줄어 전체 오차 감소.

## 설정
- 모델: PINN 5층 MLP
- 데이터: Crack dataset v2
- 학습률: 0.001
- 에포크: 1000
- 시드: 42
- 하드웨어: RTX 4090

## 결과
- 최종 손실: 0.0023
- 검증 정확도: 94.2%
- 훈련 시간: 2.5시간

## 관찰
- 수렴이 λ_bc=1 대비 빠름
- 그러나 과적합 경향 (epoch 600 이후)
- Best 모델: epoch 580

## 결론
λ_bc=10이 유용하나 early stopping 필요.

## 다음 단계
- λ_bc=5로도 시도
- Regularization 추가
```

**이 일지가 없으면**: 몇 달 후 "왜 λ_bc=10을 선택했지?"에 답할 수 없다.

**자동화 vs 수동**:
- 자동: 설정, 메트릭, 시간
- 수동: 목적, 가설, 관찰, 결론

**결합 도구**: Markdown + 실험 추적 도구. Markdown으로 해석, 추적 도구로 데이터.

**실험 결과의 비교**.

여러 실험을 비교하는 방법.

**방법 1: 추적 도구의 대시보드**.
W&B, MLflow의 비교 뷰. 여러 실험을 나란히.

**장점**: 시각적, 쉬움.
**단점**: 깊은 분석 어려움.

**방법 2: 데이터 추출 + 분석**.
추적 도구에서 데이터를 내보내 Pandas로 분석.

```python
import pandas as pd

# W&B에서 실험 가져오기
import wandb
api = wandb.Api()
runs = api.runs("project-name")

data = []
for run in runs:
    data.append({
        'name': run.name,
        'lr': run.config.get('lr'),
        'batch_size': run.config.get('batch_size'),
        'final_loss': run.summary.get('loss'),
        'val_acc': run.summary.get('val_acc'),
    })

df = pd.DataFrame(data)
print(df.sort_values('val_acc', ascending=False).head(10))
```

**장점**: 유연한 분석.
**단점**: 코드 작성 필요.

**방법 3: 통계적 비교**.
여러 시드의 평균과 표준편차.

```python
# 같은 설정의 여러 시드
runs_same_config = df[df['config_hash'] == 'abc123']
mean_acc = runs_same_config['val_acc'].mean()
std_acc = runs_same_config['val_acc'].std()
print(f"Accuracy: {mean_acc:.3f} ± {std_acc:.3f}")
```

**장점**: 과학적 신뢰.
**단점**: 많은 실행 필요.

**논문용 결과의 재생산**.

박사 후반의 특수한 과제: 논문에 쓴 실험을 리비전이나 심사 때 재현.

**재생산의 요구 사항**:
1. 원본 config
2. 원본 코드 (git commit)
3. 원본 데이터
4. 원본 체크포인트 (옵션)
5. 원본 환경

**재생산 체크리스트**:
- [ ] 코드 commit 해시를 기록했는가?
- [ ] config 파일이 보존되었는가?
- [ ] 데이터셋의 정확한 버전이 있는가?
- [ ] 랜덤 시드가 고정되었는가?
- [ ] 환경이 재현 가능한가 (Docker, requirements)?

**이것들이 모두 있으면** 1-2시간 내 재현 가능. 하나라도 없으면 며칠 걸릴 수 있다.

**논문 리비전의 경험**: "이 실험을 추가해 주세요"가 오면 이미 지나간 설정을 정확히 복원해야. 이것이 체계적 관리의 진가.

**스토리지 관리 — 용량의 현실**.

수백 실험은 많은 저장 공간이 필요하다.

**용량 추정**:
- 코드: < 100MB
- 로그: 1-10MB per run
- 메트릭: 1-10MB per run
- 체크포인트: 10MB-10GB per run
- 시각화: 1-100MB per run

**박사 총합**: 100GB-10TB

**관리 전략**:

**단계 1: 계층 저장**.
- **Hot**: 최근 1개월 (로컬 SSD)
- **Warm**: 최근 1년 (학교 서버)
- **Cold**: 논문용 (장기 백업)

**단계 2: 정기 청소**.
- 월 1회: 실패한 실험 삭제
- 분기 1회: 탐색 실험 정리
- 연 1회: 대규모 아카이브

**단계 3: 선택적 보관**.
모든 체크포인트가 아닌 핵심만:
- Best 모델
- 논문에 사용된 모델
- 재현에 필요한 모델

**단계 4: 압축**.
오래된 실험의 로그를 압축:
```bash
tar -czf old_experiments.tar.gz old_runs/
rm -rf old_runs/
```

**Git-LFS와 DVC**.

대용량 파일을 Git과 함께 관리.

**Git-LFS**:
- 큰 파일을 별도 저장소
- Git과 통합
- 간단한 설정

```bash
git lfs track "*.pt"
git lfs track "*.pth"
git add .gitattributes
git add model.pt
git commit -m "Add model"
```

**DVC (Data Version Control)**:
- 데이터와 모델의 버전 관리
- Git과 유사한 인터페이스
- 원격 스토리지 (S3, GCS)

```bash
dvc init
dvc add data/raw
git add data/raw.dvc .gitignore
git commit -m "Add raw data"
dvc push
```

**권장**: 박사 초반은 단순 폴더 구조. 중반 이후 DVC 도입.

**실험의 부모-자식 관계**.

실험이 이전 실험에서 파생될 때 관계 추적.

**예시**:
```
exp001 (baseline)
├── exp002 (다른 lr)
├── exp003 (다른 batch_size)
└── exp004 (다른 아키텍처)
    ├── exp005 (exp004 + lr 변경)
    └── exp006 (exp004 + dropout 추가)
```

**추적 방법**:
- 실험 일지에 "parent_exp" 필드
- Git 브랜치 사용
- 실험 추적 도구의 tag

**이점**: 실험의 진화를 추적. "이 설정에 왜 도달했나?"에 답.

**팀 협업 시의 실험 관리**.

여러 명이 같은 프로젝트.

**문제**:
- 실험 이름 충돌
- 중복 실험
- 결과 공유 어려움

**해결**:
1. **공유 네임스페이스**: 실험 이름에 사용자 prefix
2. **공유 추적 도구**: W&B의 team project
3. **정기 공유 미팅**: 매주 진행 공유
4. **문서화**: 공유 일지

**원칙**: 명확한 소통 + 체계적 도구.

**실험의 "재탐색" 방지**.

흔한 함정: 같은 실험을 여러 번 반복.

**원인**:
- 이전 실험을 잊음
- 검색이 어려움
- 결과를 찾을 수 없음

**방지**:
1. **통합 검색 가능한 데이터베이스**: 모든 실험 한 곳에
2. **태그와 라벨**: 쉬운 필터링
3. **정기 리뷰**: 월 1회 이전 실험 훑기
4. **Failure log**: 실패한 접근의 기록

**시간 절약**: 중복 방지가 박사 과정의 수십 시간 이상 절약.

**환경의 재현성**.

"내 컴퓨터에서는 되는데 다른 곳에서는 안 돼"의 방지.

**도구**:
- **requirements.txt / environment.yml**: Python 의존성
- **Docker**: 완전한 환경 컨테이너
- **Singularity**: HPC용 컨테이너

**예시 Dockerfile**:
```dockerfile
FROM pytorch/pytorch:2.0.0-cuda11.7-cudnn8-runtime

RUN pip install wandb hydra-core scikit-learn

WORKDIR /workspace
COPY . .

CMD ["python", "train.py"]
```

**장점**: 언제 어디서나 같은 환경.
**단점**: 크기, 설정 복잡성.

**박사 전략**: 박사 1-2년차는 requirements.txt. 3년차 이후 Docker.

**실험 관리의 단계적 도입**.

모든 것을 한 번에 도입하지 말자.

**박사 1년차**:
- 기본 폴더 구조
- 간단한 config (dict)
- 수동 일지
- Git for 코드

**박사 2년차**:
- YAML config
- W&B 사용 시작
- 매일 커밋
- 체크포인트 관리

**박사 3년차**:
- Hydra 도입
- 실험 추적 고도화
- 자동화된 분석
- 체계적 일지

**박사 4년차**:
- Docker 환경
- DVC 또는 Git-LFS
- 팀 협업 도구
- 재현성 파이프라인

**박사 5-6년차**:
- 전체 시스템 정착
- 후배에게 전수
- 논문 재현성 보장

이 단계적 접근이 과부하를 방지.

**박사 논문의 실험 섹션**.

박사 논문에 실험 관리를 반영.

**방법론 섹션에**:
"All experiments were tracked using Weights & Biases. Configurations were managed via Hydra. Each experiment was run with 5 random seeds, and we report mean ± standard deviation."

**부록에**:
- 전체 하이퍼파라미터 목록
- 실험 수와 시간
- 재현 지침
- Git 저장소 링크

**이것이** 논문의 재현성을 보장.

**흔한 실수 6가지**.

**실수 1: 시드 고정 안 함**.
결과가 매번 다름. 비교 불가.
**수정**: 항상 `torch.manual_seed(42)`, `np.random.seed(42)`.

**실수 2: Config 하드코딩**.
코드에 파라미터 삽입. 변경 시 재컴파일.
**수정**: 외부 config 파일.

**실수 3: 결과 저장 안 함**.
실행 후 결과를 나중에 보존 안 함.
**수정**: 자동 로깅.

**실수 4: 이름 없는 실험**.
"test", "run1", "new" 같은 이름.
**수정**: 의미 있는 이름.

**실수 5: 과거 실험 잊음**.
한 달 전 실험을 찾을 수 없음.
**수정**: 검색 가능한 시스템.

**실수 6: 전환 비용 두려움**.
"지금은 이대로 쓰자, 나중에 바꾸자."
**수정**: 일찍 투자. 나중에 바꾸기 어렵다.

**체크리스트 — 실험 관리 시스템**.

- [ ] 랜덤 시드가 고정되고 기록되는가?
- [ ] 실험 추적 도구 (W&B, MLflow)를 쓰는가?
- [ ] 각 실험에 고유한 이름이 있는가?
- [ ] 하이퍼파라미터가 자동 로깅되는가?
- [ ] 메트릭이 자동 로깅되는가?
- [ ] 체크포인트 보관 전략이 있는가?

- [ ] 실험 일지를 작성하는가?
- [ ] 환경 (Python/CUDA 버전)이 기록되는가?
- [ ] 논문 실험의 재현이 가능한가?

이 12가지가 실험 관리 시스템의 기반.

**장기 관점 — 실험 관리의 가치**.

박사 과정 전체의 관점.

**박사 초기**: 체계가 없어도 관리 가능. 실험 수 적음.
**박사 중반**: 체계 없으면 혼란 시작. 1-2주 낭비/월.
**박사 후반**: 체계 없으면 재앙. 논문 리비전 시 재현 불가.

**투자의 회수**:
- 체계 구축 시간: 박사 초기 1-2주
- 절약 시간: 박사 전체 수백 시간
- ROI: 매우 높음

**박사 이후에도**:
- 포스닥에서 같은 시스템 사용
- 산업체로 전이
- 평생 기술

**실천 가이드 — 오늘 시작할 수 있는 것**.

1. **현재 실험 폴더의 정리**: 한 프로젝트의 폴더 구조 개선
2. **YAML config 작성**: 하나의 실험의 설정을 파일로
3. **W&B 계정 생성**: 무료, 5분
4. **첫 실험 로깅**: wandb.init()과 wandb.log()
5. **Git 커밋**: 매일 1회 이상
6. **실험 일지**: 현재 진행 중인 실험 기록

이 6가지를 하루에. 내일부터 실험 관리가 체계화 시작.

> ML 실험 관리는 박사 과정의 "조용한 기반"이다. 눈에 띄지 않지만, 박사 5-7년의 생산성을 결정한다. 수백 수천 실험을 안전하게 관리하는 체계가 없으면 "실험 찾기"에 시간을 낭비한다. 반대로 체계가 있으면 실험의 주인이 된다. 박사 1-2년차에 체계 구축에 1-2주를 투자하고, 이후 점진적으로 고도화하라. 박사 졸업 시점에는 "내 모든 실험을 5분 안에 찾을 수 있는" 연구자가 된다. 이것이 박사의 품질을 높이고, 논문 재현성을 보장하고, 후배에게 전수할 수 있는 자산을 만든다.
