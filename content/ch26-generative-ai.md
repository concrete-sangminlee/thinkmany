# 생성형 AI 활용법

2022년 말 ChatGPT가 등장한 이후, 세상이 바뀌었다. 과장이 아니다. 코드를 작성하고, 논문을 교정하고, 아이디어를 정리하고, 수식을 풀고, 데이터를 분석하는 방식이 근본적으로 변하고 있다. 대학원생의 일상도 예외가 아니다.

생성형 AI(Generative AI)란 텍스트, 이미지, 코드, 음악 등의 콘텐츠를 "생성"하는 AI를 말한다. 대규모 언어 모델(Large Language Model, LLM)이 그 핵심이다. ChatGPT(OpenAI), Claude(Anthropic), Gemini(Google)가 대표적인 LLM 기반 서비스이고, GitHub Copilot은 코드 생성에 특화된 도구다.

이 장에서는 생성형 AI를 공학 연구에 어떻게 활용할 수 있는지를 구체적으로 다룬다. 어떤 작업에 써야 하고, 어떤 작업에 쓰면 안 되는지, 어떻게 써야 효과적인지, 그리고 무엇을 조심해야 하는지를 솔직하게 이야기한다.

한 가지 먼저 분명히 해둔다. 생성형 AI는 도구(tool)다. 아무리 뛰어난 도구라도 사용자의 판단을 대체하지는 못한다. 망치가 아무리 좋아도 못을 어디에 박을지는 목수가 결정한다. 생성형 AI가 아무리 똑똑해도, 연구의 방향과 판단은 연구자의 몫이다.

## 연구 활용 4가지

공학 대학원생이 생성형 AI를 활용할 수 있는 핵심 영역은 네 가지다. 가장 먼저 시도할 것은 "코딩 어시스턴트"다. 디버깅이나 코드 설명은 결과를 즉시 검증할 수 있어 할루시네이션 위험이 낮고, 시간 절약 효과가 가장 크다.

### 문헌조사 보조

논문을 찾고, 읽고, 정리하는 과정에서 AI가 큰 도움이 된다. 하지만 여기서 "보조"라는 단어를 강조한다. AI가 문헌조사를 "대신" 해주는 것이 아니다.

구체적인 활용법을 보자. 첫째, 논문 요약이다. 긴 논문의 핵심 내용을 빠르게 파악하고 싶을 때, 논문의 Abstract와 Conclusion을 AI에게 보여주고 "이 논문의 주요 기여와 한계를 요약해달라"고 요청할 수 있다. 전체 논문을 붙여넣고 "방법론 부분만 한국어로 요약해달라"고 할 수도 있다. 초기 스크리닝 단계에서 수십 편의 논문을 빠르게 훑는 데 유용하다.

둘째, 개념 설명이다. 논문에서 낯선 개념이나 기법이 나왔을 때, AI에게 물어볼 수 있다. "Variational Inference가 뭔지 대학원 1년차 수준으로 설명해달라", "Attention mechanism이 CNN의 pooling과 뭐가 다른지 비교해달라" 같은 질문이다. 교과서를 찾아보는 것보다 훨씬 빠르게 직관적 이해를 얻을 수 있다.

셋째, 연구 동향 파악이다. "PINN의 최근 3년 연구 동향을 분야별로 정리해달라", "Transfer learning in structural health monitoring 분야의 주요 연구 그룹과 논문을 알려달라" 같은 요청으로 분야의 큰 그림을 파악할 수 있다. 단, AI가 제시하는 논문 제목과 저자는 반드시 Google Scholar에서 실제로 존재하는지 확인해야 한다. 이것은 아래 할루시네이션 절에서 다시 다룬다.

### 코딩 어시스턴트

이것이 아마 가장 즉각적이고 실용적인 활용처일 것이다. AI는 코드 작성, 디버깅, 최적화, 설명에서 놀라운 능력을 보여준다.

코드 작성 보조: "PyTorch로 1D CNN 모델을 작성해줘. 입력은 길이 1000의 시계열, 출력은 3개 클래스 분류야"라고 요청하면, 실행 가능한 코드 초안을 받을 수 있다. 이 코드를 기반으로 수정하면, 처음부터 작성하는 것보다 훨씬 빠르다.

디버깅: 에러 메시지와 관련 코드를 붙여넣고 "이 에러가 왜 나는지 설명하고 수정해달라"고 요청한다. 특히 PyTorch의 텐서 차원 불일치(shape mismatch) 에러나, 복잡한 데이터 전처리 코드의 버그를 찾는 데 매우 효과적이다.

코드 설명: 다른 연구자의 GitHub 코드를 이해해야 할 때, 코드를 붙여넣고 "이 함수가 뭘 하는 건지 한 줄씩 설명해달라"고 요청한다. 문서화가 잘 안 된 오픈소스 코드를 분석할 때 특히 유용하다.

코드 최적화: "이 코드를 vectorize해서 속도를 개선해달라", "이 반복문을 NumPy 연산으로 바꿔달라" 같은 요청으로 코드 효율을 높일 수 있다.

### 글쓰기 보조

영어 논문 작성에서 AI의 도움이 크다. 특히 영어가 모국어가 아닌 연구자에게 유용하다.

문법 교정과 표현 개선: 작성한 영문 단락을 AI에게 보여주고 "학술적 톤으로 문법과 표현을 교정해달라"고 요청한다. 어색한 표현을 자연스럽게 바꿔주고, 학술 논문에 적합한 어휘를 제안해준다. Grammarly보다 맥락을 더 잘 이해하는 경우가 많다.

초고 작성 보조: "이 실험 조건과 결과로 Methodology 섹션의 초안을 작성해달라"고 할 수 있다. 하지만 AI가 작성한 초안을 그대로 사용하면 안 된다. AI의 초안은 구조와 흐름을 잡는 데 활용하고, 내용의 정확성과 표현은 반드시 직접 검토하고 수정해야 한다.

표현 제안: "increase와 enhance와 improve의 차이가 뭔지, 논문에서 각각 어떤 맥락에서 쓰이는지 알려달라" 같은 질문으로 적확한 학술 표현을 찾을 수 있다.

### 아이디어 브레인스토밍

연구 아이디어를 구상하는 단계에서 AI를 토론 상대로 활용할 수 있다. "PINN을 구조 건전성 모니터링에 적용하려면 어떤 접근이 가능할까?", "이 데이터셋에서 시도해볼 만한 ML 기법이 뭐가 있을까?" 같은 열린 질문을 던지고, AI의 제안을 비판적으로 검토한다.

AI의 제안이 모두 좋은 것은 아니다. 오히려 대부분은 이미 알려진 것이거나, 실현 가능성이 낮은 것이다. 하지만 한두 가지 생각지 못한 관점을 발견할 수 있고, 그것이 연구의 실마리가 될 수 있다. AI를 "정답을 주는 존재"가 아니라 "질문을 던져주는 존재"로 활용하는 것이 핵심이다.

어떤 도구를 쓸지는 용도에 따라 다르다. 코딩 작업이 많으면 GitHub Copilot이나 Cursor를 IDE에 설치하는 것이 최우선이다. 논문 검색에는 Elicit이나 Semantic Scholar를 쓰되, ChatGPT/Claude는 개념 이해와 아이디어 정리에 활용한다. 영어 글쓰기에는 ChatGPT/Claude로 교정하되, 최종적으로 Grammarly나 Writefull로 한 번 더 확인한다.

중요한 것은, 하나의 도구에 의존하지 말라는 것이다. 각 도구의 강점이 다르므로, 작업에 따라 도구를 바꿔가며 사용하는 유연함이 필요하다.

## 프롬프트 엔지니어링

생성형 AI의 출력 품질은 입력(프롬프트)의 품질에 크게 의존한다. 같은 질문이라도 어떻게 묻느냐에 따라 답변의 수준이 천차만별이다. 효과적인 프롬프트를 작성하는 네 가지 원칙을 정리한다. 오늘 당장 하나만 시도한다면, "맥락 제공"부터 해보자. 평소에 던지던 짧은 질문에 배경 정보를 세 줄만 추가해도 답변 품질이 눈에 띄게 달라진다.

### 역할 지정

AI에게 역할을 부여하면 답변의 수준과 관점이 달라진다. "너는 구조공학 전공 교수야. 대학원 1년차 학생에게 설명하듯이 답변해줘"라고 시작하면, 일반적인 답변보다 전문적이면서도 이해하기 쉬운 답변을 받을 수 있다. "너는 Python 시니어 개발자야. 클린 코드와 에러 핸들링을 중시해줘"라고 하면, 더 견고한 코드를 받을 수 있다.

### 맥락 제공

질문에 충분한 배경 정보를 제공해야 한다. "CNN 성능이 안 나와"보다 "ImageNet 사전학습된 ResNet-50으로 콘크리트 균열 이미지 500장을 파인튜닝하고 있는데, validation accuracy가 70%에서 멈춘다. 학습률은 1e-3, 배치 크기 32, 데이터 증강은 수평 반전만 적용했다"가 훨씬 유용한 답변을 이끌어낸다.

### 단계별 지시

복잡한 작업은 한 번에 요청하지 말고 단계별로 나눠서 요청한다. "이 데이터를 분석하고 시각화하고 모델을 학습시키고 결과를 해석해줘"보다, 각 단계를 개별적으로 요청하는 것이 결과가 좋다. "1단계: 이 데이터의 기본 통계량을 계산하고 분포를 시각화해줘"로 시작하고, 결과를 확인한 후 다음 단계를 요청한다.

### 출력 형식 지정

원하는 출력의 형식을 명시한다. "표로 정리해줘", "코드 블록으로 작성해줘", "장단점을 비교하는 형식으로 답변해줘", "500자 이내로 요약해줘" 등이다. 형식을 지정하지 않으면 AI가 자기 판단으로 출력 형식을 결정하는데, 원하는 형식과 다를 수 있다.

**코드 디버깅 프롬프트 예시:**

"다음 PyTorch 코드에서 RuntimeError: mat1 and mat2 shapes cannot be multiplied 에러가 발생한다. 모델 구조와 데이터 로딩 코드를 보고 원인을 찾아 수정해달라."

이후 에러 메시지 전문과 관련 코드를 붙여넣는다. 핵심은 에러 메시지를 생략하지 않는 것이다. 전체 traceback을 포함시키면 AI가 정확한 위치를 파악할 수 있다. 코드도 관련 부분만이 아니라, 데이터 로딩부터 모델 정의, 학습 루프까지 전체 맥락을 제공하는 것이 좋다.

디버깅 결과를 받으면, AI의 수정 사항을 이해하고 나서 적용한다. "왜 이 수정이 문제를 해결하는가?"를 이해하지 못하면, 비슷한 에러가 다시 발생했을 때 또 AI에게 의존하게 된다.

</div>

**논문 교정 프롬프트 예시:**

"다음은 내 논문의 Methodology 섹션의 한 단락이다. 학술 영어 톤으로 문법, 표현, 논리적 흐름을 교정해달라. 원래 의미를 바꾸지 말고, 수정한 부분에는 [수정]이라고 표시해달라."

한 번에 전체 논문을 넣지 말고, 한 섹션 또는 한 단락씩 처리하는 것이 품질이 높다. 전체를 한 번에 넣으면 AI가 세부적인 표현까지 주의를 기울이지 못하고 대충 넘어가는 경향이 있다.

수정된 결과를 받으면, 수정 사항을 하나하나 확인한다. AI가 의미를 잘못 해석하여 원래와 다른 뜻으로 바꾸는 경우가 있다. 특히 기술 용어가 포함된 문장에서 이런 일이 발생할 수 있다.

**문헌조사 프롬프트 예시:**

"나는 PINN(Physics-Informed Neural Networks)을 열전달 역문제에 적용하는 연구를 하고 있다. 이 분야의 주요 연구 동향, 핵심 논문, 현재 미해결 문제를 정리해달라. 단, 논문을 언급할 때는 저자, 연도, 학술지를 명시해달라."

AI가 제시하는 논문 목록을 받으면, 반드시 Google Scholar에서 각 논문이 실제로 존재하는지 확인한다. AI는 그럴듯하지만 존재하지 않는 논문을 만들어내는 경우가 빈번하다. 저자 이름이 맞지만 제목이 다르거나, 제목이 비슷하지만 학술지가 다르거나, 아예 존재하지 않는 논문을 자신 있게 인용하는 경우가 있다.

문헌조사에서 AI의 가장 안전한 활용법은 "검색 키워드 제안"이다. "이 주제로 검색할 때 어떤 키워드 조합을 쓰면 좋을까?"라고 물으면, 여러 키워드 조합을 제안해준다. 이 키워드로 Google Scholar에서 직접 검색하는 것이 정확하다.

**수식/이론 질문 프롬프트 예시:**

"가우시안 프로세스 회귀에서 커널 함수의 하이퍼파라미터를 최적화하는 방법을 설명해달라. marginal likelihood를 최대화하는 과정을 수식과 함께 단계별로 설명해줘. 대학원 1년차 수준으로 부탁한다."

수식 관련 답변을 받으면, 반드시 직접 검산한다. AI는 수식의 전반적인 구조는 맞추지만, 부호, 계수, 적분 범위 같은 세부사항에서 실수를 하는 경우가 있다. 특히 복잡한 유도 과정에서 중간 단계가 빠지거나, 가정을 명시하지 않는 경우가 흔하다.

교과서의 해당 부분과 대조하거나, 특수한 경우(예: 1차원, 단순 커널)에서 수식이 알려진 결과와 일치하는지 확인하는 것이 좋은 검증 방법이다.

## 할루시네이션 탐지

생성형 AI의 가장 위험한 특성은 할루시네이션(hallucination)이다. AI가 사실이 아닌 내용을 자신 있게 말하는 현상이다. 거짓말을 하려는 의도가 아니라, 모델의 구조적 한계 때문에 발생한다. LLM은 "그럴듯한 다음 단어"를 생성하도록 학습되었지, "사실인 다음 단어"를 생성하도록 학습되지 않았다.

공학 연구에서 할루시네이션은 치명적일 수 있다. 존재하지 않는 논문을 인용하면 학술적 신뢰가 무너지고, 틀린 수식을 사용하면 연구 결과 전체가 무효가 된다. 할루시네이션을 탐지하고 방지하는 구체적인 방법을 알아야 한다.

### 인용 논문 검증

AI가 언급하는 모든 논문을 Google Scholar에서 확인한다. 다음 세 단계를 매번 실행한다.

1. AI가 제시한 논문 제목을 그대로 복사하여 Google Scholar에 붙여넣고 검색한다.
2. 저자, 제목, 학술지, 연도가 모두 일치하는지 하나씩 대조한다.
3. 한 가지라도 다르면 해당 인용을 사용하지 않는다.

AI가 "Smith et al. (2020), Journal of Computational Physics"라고 말했는데, 실제로 검색해보면 그런 논문이 없거나 제목이 다른 경우가 흔하다.

Semantic Scholar나 Elicit 같은 학술 검색 도구를 함께 사용하면 검증이 더 쉽다. 이들은 실제 논문 데이터베이스에 기반하므로 할루시네이션이 거의 없다.

### 수식 검산

AI가 제시한 수식을 반드시 직접 검산한다. 간단한 특수 케이스(예: 1차원, 선형, 단일 변수)에서 수식이 알려진 결과를 재현하는지 확인한다. 차원 분석(dimensional analysis)을 수행하여 양변의 단위가 일치하는지 확인한다. 수식의 극한값(0, 무한대 등)에서 물리적으로 합리적인 결과가 나오는지 확인한다.

### 코드 실행 검증

AI가 작성한 코드를 반드시 실행하여 결과를 확인한다. 컴파일/실행이 되는 것만 확인하는 것이 아니라, 알려진 입력에 대해 예상되는 출력이 나오는지 테스트한다. 특히 경계 조건(빈 입력, 매우 큰 입력, 음수 등)에서의 동작을 확인한다.

AI가 작성한 코드에서 흔히 발생하는 문제: 존재하지 않는 라이브러리 함수 호출, 더 이상 사용되지 않는(deprecated) API 사용, 잘못된 인자 순서, 특정 버전에서만 작동하는 문법 등이 있다.

<div class="highlight-box warning">
<span class="highlight-box-icon">⚠️</span>
<div class="highlight-box-content">
<p><strong>할루시네이션 경고 신호</strong></p>
<p>AI의 답변에서 다음과 같은 신호가 보이면 할루시네이션을 의심한다. 1) 매우 구체적인 숫자를 자신 있게 제시한다 ("이 방법은 정확도를 23.7% 향상시켰다"). 2) 잘 알려지지 않은 논문을 마치 유명한 것처럼 인용한다. 3) 질문의 전제에 맞추려고 사실을 왜곡한다 ("X가 Y보다 좋다고 알려져 있다"는 질문에, 실제로는 그렇지 않은데도 동의하는 경우). 4) 여러 번 물었을 때 다른 답을 한다. 같은 질문을 다시 했을 때 이전과 다른 논문을 인용하면 둘 다 신뢰할 수 없다.</p>

## 윤리 가이드라인

생성형 AI를 연구에 사용할 때의 윤리적 기준은 아직 학계에서 정립 중이다. 하지만 이미 많은 학술지와 학회가 가이드라인을 발표하고 있다. 현재 시점에서의 핵심 원칙을 정리한다.

<li>AI가 생성한 내용을 반드시 검증한다 (논문 존재 여부, 수식 정확도, 코드 실행)</li>
<li>AI 사용 사실을 논문에 명시한다 (많은 학술지가 이를 요구한다)</li>
<li>AI의 도움을 받은 부분과 직접 작성한 부분을 구분한다</li>

### AI 사용 명시

현재 Nature, Science, IEEE, ASCE 등 주요 학술지는 AI 사용을 논문에 명시하도록 요구하고 있다. 일반적으로 Acknowledgements 섹션이나 별도의 AI Declaration 섹션에 기술한다. 아래는 바로 사용할 수 있는 선언문 예시다.

> During the preparation of this manuscript, the authors used [ChatGPT / Claude / GitHub Copilot] for [grammar correction / code debugging / literature keyword suggestion]. The authors reviewed and edited all AI-generated content and take full responsibility for the final publication.

이 문구를 자신의 사용 범위에 맞게 수정하여 논문에 포함한다.

AI를 공저자(co-author)로 표기하는 것은 대부분의 학술지에서 금지하고 있다. 공저자는 연구의 책임을 질 수 있는 인간이어야 한다는 것이 기본 원칙이다.

### 기밀 데이터 주의

상용 AI 서비스(ChatGPT, Claude 등)에 입력한 데이터는 서비스 제공자의 서버로 전송된다. 미발표 실험 데이터, 산업체의 기밀 정보, 특허 출원 전의 아이디어 등을 AI에 입력하면 정보 유출 위험이 있다. 대부분의 서비스가 "사용자 데이터를 학습에 사용하지 않는다"고 명시하고 있지만, 서비스 약관을 반드시 확인하고, 기밀 데이터는 가능하면 로컬에서 실행되는 모델(오픈소스 LLM)을 사용하는 것이 안전하다.

## 공학 연구에서의 한계

생성형 AI의 한계를 솔직하게 짚어본다. 이 한계를 알아야 도구를 현명하게 사용할 수 있다.

### 수식 정확도

현재의 LLM은 수학적 추론에서 여전히 약점을 보인다. 단순한 미적분이나 선형대수는 대체로 맞추지만, 복잡한 유도 과정에서 부호 오류, 계수 오류, 논리적 비약이 발생하는 경우가 적지 않다. 텐서 미적분, 변분법, 복소 해석 같은 고급 수학에서는 신뢰도가 더 떨어진다.

구체적인 예를 들면, 나비에-스토크스 방정식의 비압축성 조건에서 특정 경계조건을 적용한 유도를 요청하면, 전반적인 흐름은 맞지만 중간 단계에서 항이 빠지거나, 적분 상수 처리가 잘못되는 경우가 있다. 이런 오류는 최종 결과에 영향을 미치므로, AI가 유도한 수식을 논문에 직접 사용하는 것은 매우 위험하다.

수식은 AI의 도움을 받되, 반드시 교과서나 원논문으로 검증해야 한다. AI는 "이런 유도 과정이 있다"는 방향을 알려주는 데 유용하고, 정확한 수식 자체를 제공하는 데는 아직 불안정하다.

### 최신 논문 미반영

LLM은 학습 데이터의 시점까지만 알고 있다. 학습 데이터 이후에 발표된 논문, 새로 등장한 기법, 최근의 벤치마크 결과 등은 알지 못한다. "2025년에 발표된 논문 중..."이라는 질문에 AI가 답변을 한다면, 그것은 높은 확률로 할루시네이션이다.

최신 연구 동향은 Google Scholar, arXiv, Semantic Scholar에서 직접 검색해야 한다. AI는 "기존에 알려진 일반적 지식"을 정리하는 데 활용하고, "최신 연구"는 직접 확인한다.

### 도메인 지식의 깊이

AI는 다양한 분야에 대해 폭넓은 지식을 가지고 있지만, 특정 분야의 깊이는 해당 분야 전문가에 미치지 못한다. "CFRP 복합재의 층간 분리 진전 메커니즘에서 Mode I과 Mode II의 혼합 비율이 피로 수명에 미치는 영향"같은 구체적이고 전문적인 질문에는 피상적인 답변을 하거나 잘못된 정보를 제시할 수 있다.

깊은 도메인 지식이 필요한 부분은 교과서, 전문 논문, 지도교수에게 확인해야 한다 (분야별 핵심 교과서와 추천 학습 경로는 ch27 '교과서, 강의, 필독 논문'을 참고한다.). AI는 넓지만 얕은 지식을 제공한다고 생각하면 된다.

### 재현성 문제

같은 프롬프트를 입력해도 AI의 답변이 매번 다를 수 있다(temperature 설정에 따라). 이는 연구의 재현성에 문제가 된다. AI가 생성한 코드나 분석 결과를 논문에 포함한다면, 최종적으로 사용한 코드를 별도로 저장하고 공유해야 한다. "AI에게 이렇게 물어서 받은 코드를 사용했다"는 것만으로는 다른 연구자가 같은 결과를 재현할 수 없다.

## 생성형 AI를 잘 쓰는 연구자의 특징

생성형 AI를 효과적으로 활용하는 연구자들에게는 공통된 특징이 있다.

첫째, 자기 분야의 기초가 탄탄하다. 기초가 없으면 AI의 답변이 맞는지 틀린지 판단할 수 없다. AI가 제시한 수식이 물리적으로 타당한지, 제안한 방법론이 내 문제에 적합한지를 판단하려면 해당 분야의 기본 지식이 있어야 한다. AI는 기초 없이 연구하는 지름길이 아니다. 오히려 기초가 있는 사람이 AI를 더 잘 활용한다 (기초를 다지기 위한 교과서, 강의, 필독 논문은 ch27 '교과서, 강의, 필독 논문'에서 정리한다.).

둘째, 비판적으로 사고한다. AI의 모든 출력을 의심하고 검증한다. "AI가 이렇게 말했으니까 맞겠지"라는 태도는 위험하다. "AI가 이렇게 말했는데, 정말 그런가?"라고 질문하는 습관이 필요하다.

셋째, AI를 대화 상대로 활용한다. 일방적으로 질문하고 답을 받는 것이 아니라, 후속 질문을 던지고, 답변의 약점을 지적하고, 다른 관점을 요구하는 식으로 대화를 이어간다. "그 방법의 단점은 뭔가?", "다른 접근법은 없는가?", "이 가정이 틀리면 어떻게 되나?" 같은 후속 질문이 AI의 가치를 극대화한다.

넷째, 결과를 자기 것으로 만든다. AI가 작성한 코드를 그대로 쓰는 것이 아니라, 이해하고 수정하고 개선한다. AI가 제안한 아이디어를 그대로 따르는 것이 아니라, 비판적으로 검토하고 자기 연구의 맥락에 맞게 재해석한다.

<div class="highlight-box info">
<span class="highlight-box-icon">ℹ️</span>
<div class="highlight-box-content">
<p><strong>AI 활용의 80/20 법칙</strong></p>
<p>생성형 AI가 연구의 80%를 5분 만에 처리해 줄 것 같지만, 현실은 다르다. AI가 생산적으로 도울 수 있는 것은 전체 연구의 20% 정도(초안 작성, 코드 스캐폴딩, 개념 정리)이며, 나머지 80%(실험 설계, 데이터 분석, 결과 해석, 논문의 핵심 기여)는 여전히 연구자의 몫이다. 하지만 그 20%에서 절약되는 시간이 상당하므로, AI를 잘 활용하는 것은 분명히 경쟁력이 된다.</p>

## AI와 함께 연구하는 미래

생성형 AI는 빠르게 발전하고 있다. 현재의 한계(수식 오류, 할루시네이션, 최신 정보 부족)는 점차 개선될 것이다. 멀티모달 AI가 이미지, 그래프, 표를 이해하고 분석하는 능력도 향상되고 있다. AI가 논문을 직접 읽고, 실험 데이터를 분석하고, 시뮬레이션 결과를 해석하는 날이 먼 미래가 아니다.

하지만 AI가 아무리 발전해도, 연구의 핵심은 변하지 않는다. "어떤 문제를 풀어야 의미가 있는가?"를 판단하는 것, "이 결과가 기존 지식에 어떤 기여를 하는가?"를 해석하는 것, "이 연구가 사회에 어떤 영향을 미치는가?"를 고민하는 것. 이것들은 인간 연구자의 고유한 역할이다.

AI 시대에 경쟁력 있는 연구자가 되는 방법은 두 가지다. 첫째, AI 도구를 능숙하게 사용할 줄 아는 것. 프롬프트를 잘 작성하고, 적절한 도구를 선택하고, 할루시네이션을 걸러내는 능력이다. 둘째, AI가 할 수 없는 것을 잘하는 것. 깊은 도메인 지식, 창의적 문제 정의, 비판적 사고, 윤리적 판단이다.

도구를 잘 쓰되, 도구에 지배당하지 않는 것. 이것이 AI 시대 연구자의 자세다.

## 프롬프트 실패와 성공: Before/After

앞서 프롬프트 엔지니어링의 원칙을 다뤘다. 원칙은 알겠는데, 실제로 뭐가 달라지는지 감이 안 올 수 있다. 여기서는 세 가지 실전 사례를 Before(나쁜 프롬프트)와 After(개선 프롬프트)로 비교한다. 같은 질문이라도 어떻게 묻느냐에 따라 답변의 품질이 완전히 달라지는 것을 직접 확인할 수 있다.

### 사례 1: 코드 디버깅

**Before (나쁜 프롬프트):**

> "파이썬 코드 에러 고쳐줘"

이 프롬프트에는 세 가지가 빠져 있다. 어떤 에러인지, 어떤 코드인지, 어떤 환경인지. AI는 "어떤 에러가 발생하셨나요?"라고 되물을 수밖에 없고, 여러 번 주고받아야 겨우 핵심에 접근할 수 있다.

**After (개선 프롬프트):**

> "Python 3.10, PyTorch 2.1 환경에서 아래 코드를 실행하면 RuntimeError: mat1 and mat2 shapes cannot be multiplied (64x128 and 256x10) 에러가 발생한다. 모델은 1D CNN으로, 입력 크기는 (batch, 1, 1000)이고, Linear 레이어에서 에러가 난다. 에러 traceback 전문과 모델 정의 코드를 아래에 첨부한다. 원인을 분석하고 수정 코드를 제시해달라."

**무엇이 달라졌는가:** 환경(Python/PyTorch 버전), 에러 메시지 전문, 입력 형태, 에러 발생 위치를 명시했다. AI가 한 번에 정확한 원인(CNN 출력 크기와 Linear 입력 크기 불일치)을 짚어내고, `nn.Flatten()` 후의 텐서 크기를 계산하여 수정 코드를 제시할 수 있다. 맥락이 충분하면 AI의 디버깅 정확도가 비약적으로 높아진다.

### 사례 2: 논문 문장 교정

**Before (나쁜 프롬프트):**

> "이 문장 고쳐줘: The result is showed that the method is better than the previous ones."

AI는 문법만 고쳐서 "The results showed that the method is better than the previous ones."라고 돌려준다. 문법적으로 틀리진 않지만, 학술 논문에 쓰기엔 여전히 모호하고 약하다.

**After (개선 프롬프트):**

> "아래는 내 논문 Results 섹션의 문장이다. 학술 영어 톤(수동태 허용, 과거시제)으로 교정해달라. 원래 의미를 바꾸지 말고, 수정한 부분마다 왜 바꿨는지 간단히 설명해달라. 비교 대상은 기존 FEM 해석법이고, 개선 지표는 계산 시간 단축이다.
> 원문: The result is showed that the method is better than the previous ones."

**무엇이 달라졌는가:** 섹션(Results), 톤(학술), 시제(과거), 수정 이유 설명 요청, 비교 대상과 지표를 명시했다. AI는 "The results demonstrated that the proposed method reduced the computational time by X% compared to the conventional FEM approach."처럼 구체적이고 학술적인 문장을 제안하고, 각 수정에 대한 이유(showed→demonstrated: 더 강한 학술 동사, better→reduced computational time: 구체적 지표)를 설명해준다. 맥락을 주면 AI가 "학술 영어의 관행"에 맞춰 교정할 수 있다.

### 사례 3: 데이터 분석 요청

**Before (나쁜 프롬프트):**

> "데이터 분석해줘"

AI가 할 수 있는 것이 거의 없다. 어떤 데이터인지, 어떤 분석인지, 어떤 목적인지 전부 모른다.

**After (개선 프롬프트):**

> "아래 CSV 데이터는 콘크리트 배합 실험 결과다. 열은 [W/C ratio, Cement(kg), Aggregate(kg), Curing_days, Compressive_strength(MPa)]이다. 총 150개 데이터 포인트가 있다. 다음 분석을 Python(pandas, matplotlib)으로 수행해달라.
> 1. 기술 통계량(평균, 표준편차, 최대, 최소)을 표로 정리
> 2. 각 변수와 Compressive_strength 간의 상관계수 히트맵 시각화
> 3. W/C ratio와 Compressive_strength의 산점도에 회귀선 추가
> 각 코드 블록에 주석을 달아달라."

**무엇이 달라졌는가:** 데이터의 구조(열 이름, 데이터 수), 사용 도구(pandas, matplotlib), 분석 내용(3단계), 출력 형식(표, 히트맵, 산점도), 부가 요청(주석)을 모두 명시했다. AI는 즉시 실행 가능한 코드를 단계별로 생성한다. "뭘 분석할까요?"라는 역질문 없이 바로 결과물이 나온다.

세 사례의 공통점은 명확하다. **맥락, 구체성, 출력 형식.** 이 세 가지를 프롬프트에 포함하면 AI의 답변 품질이 극적으로 달라진다. "좋은 프롬프트를 쓰는 데 시간이 걸린다"고 생각할 수 있지만, 나쁜 프롬프트로 여러 번 주고받는 시간을 합치면 좋은 프롬프트 한 번이 훨씬 빠르다.

## AI 사용 윤리 결정 트리

"이 상황에서 AI를 써도 되는가?" 연구 중에 이 질문이 떠오를 때, 아래의 흐름도를 따라가면 판단 기준을 세울 수 있다. 직감이 아니라 체계적인 점검이 필요하다.

### 질문 1: 기밀 데이터인가?

AI에 입력하려는 데이터가 기밀인지 먼저 확인한다. 미발표 실험 결과, 산업체 비공개 데이터, 특허 출원 전 아이디어, 공동연구자가 공유 제한을 건 데이터가 해당된다.

**"예"라면:** 상용 AI 서비스(ChatGPT, Claude 등)에 입력하면 안 된다. 데이터가 서버로 전송되므로 정보 유출 위험이 있다. 반드시 로컬에서 실행되는 오픈소스 모델(Llama, Mistral 등)을 사용하거나, AI 사용 자체를 보류한다.

**"아니오"라면:** 다음 질문으로 넘어간다.

### 질문 2: 저널/학회의 AI 정책을 확인했는가?

투고하려는 학술지나 학회의 AI 사용 정책을 확인해야 한다. Nature, Science, IEEE, ASCE 등 주요 학술지는 이미 AI 사용에 관한 구체적인 가이드라인을 발표했다. 일부는 AI로 생성된 텍스트를 허용하되 명시를 요구하고, 일부는 특정 용도(문법 교정 등)만 허용하고, 극히 일부는 AI 사용을 전면 금지한다.

**"확인하지 않았다면":** 논문 투고 전에 반드시 확인한다. "Instructions for Authors" 또는 "Editorial Policy" 페이지에 AI 관련 조항이 있다. 정책을 모른 채 사용하면 투고 후 리젝 사유가 될 수 있다.

**"확인했다면":** 해당 정책의 범위 안에서만 사용하고, 다음 질문으로 넘어간다.

### 질문 3: AI 결과를 독립적으로 검증할 수 있는가?

AI가 생성한 코드, 수식, 텍스트, 분석 결과를 본인이 직접 검증할 수 있는 역량이 있는가? 코드를 실행해서 결과를 확인할 수 있는가? 수식을 직접 유도하거나 검산할 수 있는가? AI가 인용한 논문이 실제로 존재하는지 확인했는가?

**"검증할 수 없다면":** 사용을 보류한다. 검증할 수 없는 AI 출력을 논문에 포함하는 것은 학술적으로 매우 위험하다. 틀린 수식, 존재하지 않는 인용, 잘못된 분석이 논문에 들어가면 학술적 신뢰가 무너진다.

**"검증할 수 있다면":** 반드시 검증을 수행한 후에 사용하고, 다음 질문으로 넘어간다.

### 질문 4: 사용 사실을 명시할 수 있는가?

AI를 사용한 사실을 논문의 Acknowledgements나 별도 AI Declaration 섹션에 명시할 수 있는가? "이 과정에서 AI를 사용했다"고 솔직하게 밝힐 수 있는가?

**"명시하기 꺼려진다면":** 해당 용도의 AI 사용이 윤리적으로 적절한지 다시 생각해야 한다. 밝히기 꺼려지는 사용은 대부분 부적절한 사용이다. 예를 들어, 핵심 기여(방법론 설계, 결과 해석)를 AI에 전적으로 의존한 경우, 이를 명시하면 저자로서의 기여가 의심받을 수 있다. 그 경우, AI에 의존한 부분을 직접 수행하는 것이 맞다.

**"명시할 수 있다면":** AI를 사용하고, 사용 범위와 목적을 논문에 명확히 기술한다.

## Diffusion Models: 생성 AI의 새로운 표준

ch26 본문은 주로 텍스트 기반 LLM(ChatGPT, Claude)을 다뤘다. 하지만 생성 AI의 또 다른 큰 영역은 이미지, 비디오, 분자 구조의 생성이다. 이 분야는 **Diffusion Models**가 사실상 표준이 되었다. Stable Diffusion, DALL-E 3, Midjourney 모두 diffusion 기반이다. 본인이 합성 데이터를 만들거나 시각적 자료를 생성하려면 diffusion model을 알 가치가 있다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>Diffusion Model의 핵심 직관</strong></p>
    <p>"이미지에 노이즈를 점진적으로 추가하여 완전한 노이즈로 만든 후, 그 과정을 거꾸로 학습한다." 이것이 diffusion의 한 줄 요약이다. 노이즈에서 이미지를 복원하는 능력을 학습하면, 무작위 노이즈에서 시작해도 그럴듯한 이미지를 만들 수 있다. 이 단순한 아이디어가 이전 GAN 기반 방법보다 안정적이고 다양한 결과를 만들어 낸다.</p>
  </div>
</div>

**Diffusion vs GAN vs VAE.**

| 측면 | GAN | VAE | Diffusion |
|------|-----|-----|-----------|
| 학습 안정성 | 불안정(mode collapse) | 안정 | 매우 안정 |
| 샘플 품질 | 매우 높음 | 낮음~중간 | 매우 높음 |
| 다양성 | 낮을 수 있음 | 좋음 | 매우 좋음 |
| 학습 속도 | 빠름 | 빠름 | 느림 |
| 추론 속도 | 빠름 (단일 패스) | 빠름 | 느림 (수십~수백 단계) |
| 조건부 생성 | 가능하지만 어려움 | 가능 | 매우 강력 |

Diffusion은 거의 모든 측면에서 우수하지만, 학습과 추론이 느리다는 단점이 있다. 최근 연구들이 이 단점을 해결하고 있다(예: DDIM, Latent Diffusion, Consistency Models).

**공학 연구에서의 활용.**

첫째, **합성 데이터 생성**. 본인의 데이터가 부족할 때, diffusion model로 합성 데이터를 만들어 학습 데이터를 보강. 단, 합성 데이터의 한계(분포 편향, 비현실적 패턴)를 인지하고 검증해야 한다. 둘째, **Conditional Generation**. "X 조건의 부품 사진", "Y 결함 종류의 표면" 같은 조건부 생성. 본인의 데이터에서 부족한 사례를 보강할 수 있다. 셋째, **Domain Adaptation**. 시뮬레이션 이미지를 실제 이미지처럼 변환(sim-to-real). 자율주행, 로봇 학습에서 활용. 넷째, **Inpainting과 복원**. 손상된 영상을 복원하거나, 일부가 가려진 데이터를 채워 넣음. 다섯째, **분자 구조 생성**. 신약 개발, 신소재 설계에서 새로운 분자를 생성하는 데 활용. AlphaFold, RFdiffusion, GenMol 같은 모델.

**Diffusion model을 사용하는 실전 도구.**

| 도구 | 용도 |
|------|------|
| Hugging Face Diffusers | Python 라이브러리, 가장 표준 |
| Stable Diffusion (SDXL, SD3) | 오픈소스 이미지 생성 |
| ComfyUI | GUI 기반, 복잡한 워크플로 |
| Automatic1111 WebUI | 가장 인기 있는 SD 프론트엔드 |
| Replicate API | 클라우드 추론, 코딩 없이 사용 |
| RFdiffusion | 단백질 설계 특화 |

**기본 코드 예시 (Hugging Face Diffusers).**

```python
from diffusers import StableDiffusionPipeline
import torch

pipe = StableDiffusionPipeline.from_pretrained(
    "stabilityai/stable-diffusion-2-1",
    torch_dtype=torch.float16
).to("cuda")

prompt = "high-resolution microscopy image of concrete crack, scientific photo"
image = pipe(prompt, num_inference_steps=50, guidance_scale=7.5).images[0]
image.save("crack.png")
```

**합성 데이터 사용 시 주의.** 합성 데이터로 학습한 모델이 실제 데이터에서 어떻게 작동하는지 반드시 검증한다. 합성 데이터만으로 학습하면 종종 실제 데이터와 미묘한 차이로 성능이 떨어진다. 가능하면 실제 데이터와 합성 데이터를 함께 사용하고, 평가는 항상 실제 데이터로 한다.

## 고급 프롬프트 엔지니어링

ch26 본문에서 기본 프롬프트 엔지니어링(역할 지정, 맥락 제공, 단계별 지시, 출력 형식)을 다뤘다. 여기서는 더 정교한 기법들을 소개한다. 이 기법들은 본인이 LLM을 단순한 질의응답이 아니라 복잡한 추론 도구로 사용할 때 강력하다.

**Chain-of-Thought (CoT): 단계별 사고를 유도.** LLM에게 "정답을 즉시 말해 줘"라고 하면 종종 틀린다. 대신 "단계별로 생각해 봐"라고 하면 정확도가 크게 올라간다. 특히 수학, 논리, 다단계 추론에 효과적이다.

**기본 CoT 예시.**

```
Q: 한 콘크리트 시료의 압축 강도가 25 MPa이다. 이 강도가 ACI 318
기준의 보통 콘크리트 분류에 해당하는가? 단계별로 분석하라.

A: 단계별로 생각해 보겠다.
1단계: ACI 318의 보통 콘크리트 분류 기준 확인.
   - 보통 콘크리트: 17 MPa ≤ f'c < 35 MPa
2단계: 주어진 강도 25 MPa이 이 범위에 속하는지 확인.
   - 17 MPa ≤ 25 MPa < 35 MPa → 참
3단계: 결론.
   - 25 MPa는 ACI 318의 보통 콘크리트 분류에 해당한다.
```

**Few-shot Learning: 예시로 학습시키기.** LLM에게 1-5개의 예시를 보여주면, 그 패턴을 따라 새로운 입력에 응답한다. 학습 없이도 새 작업을 가르칠 수 있다.

```
다음 형식으로 데이터를 변환하라.

입력: "콘크리트 강도 25 MPa, 시료 28일"
출력: {"property": "compressive strength", "value": 25, "unit": "MPa", "age_days": 28}

입력: "강철 인장 강도 450 MPa, 표본 50개"
출력: {"property": "tensile strength", "value": 450, "unit": "MPa", "n_samples": 50}

입력: "알루미늄 항복점 270 MPa, 시험 5회"
출력:
```

LLM은 위 예시 두 개를 보고 세 번째 출력을 패턴에 맞춰 생성한다.

**Self-Consistency: 여러 답을 생성하고 다수결.** 같은 질문에 LLM을 여러 번 호출하여 여러 답을 얻고, 가장 자주 나오는 답을 선택. 단일 호출보다 정확도가 높지만 비용이 더 든다. 중요한 결정에 사용한다.

**ReAct: 추론과 행동의 결합.** LLM이 단순히 답하는 것이 아니라, 추론(Reason)과 행동(Action)을 번갈아 한다. 행동은 외부 도구 사용(웹 검색, 계산기, 데이터베이스 조회)이다. 본인이 LLM을 에이전트로 사용할 때 표준 패턴이다.

```
Question: 2024년 콘크리트 분야의 가장 인용된 논문은 무엇인가?

Thought: 2024년 정보가 필요하다. 내 지식은 오래되었을 수 있다. 검색이 필요하다.
Action: search_google_scholar("most cited concrete papers 2024")
Observation: [검색 결과...]
Thought: 결과를 확인했다. 가장 인용된 논문은 X et al. (2024)이다.
Answer: X et al. (2024) "Title..." which has Y citations as of...
```

이 패턴은 LangChain, LlamaIndex 같은 라이브러리에서 표준으로 구현되어 있다.

**Tree-of-Thoughts (ToT): 여러 추론 경로 탐색.** CoT가 단일 추론 체인이라면, ToT는 트리 구조로 여러 가능한 추론 경로를 탐색한다. 복잡한 문제(수학 증명, 플래닝)에 효과적. 하지만 계산 비용이 매우 크다.

**프롬프트 엔지니어링의 일반 원칙.** 첫째, **명시적이어라**. 모호한 요청은 모호한 답을 받는다. 둘째, **예시를 제공하라**. 한두 개의 예시가 백 마디 설명보다 효과적이다. 셋째, **출력 형식을 정의하라**. JSON, 리스트, 표 등의 형식을 명시. 넷째, **에러를 허용하라**. "확신이 없으면 '모름'이라고 답하라"고 명시하면 hallucination이 줄어든다. 다섯째, **반복하라**. 첫 응답이 만족스럽지 않으면 후속 질문으로 개선한다. 여섯째, **컨텍스트 윈도우를 관리하라**. 너무 긴 컨텍스트는 중간 정보가 잊힌다. 핵심 정보를 끝에 배치한다.

## RAG: 본인의 문서로 LLM 활용하기

LLM의 큰 한계는 **본인이 가진 정보를 모른다**는 것이다. 본인의 박사 논문, 연구실의 내부 문서, 최근 논문은 LLM의 학습 데이터에 없다. 본인의 자료로 질의응답을 하려면 **RAG(Retrieval-Augmented Generation)**가 필요하다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>RAG의 핵심 아이디어</strong></p>
    <p>본인의 문서를 작은 조각(chunk)으로 나누어 벡터 데이터베이스에 저장한다. 사용자가 질문하면, 그 질문과 가장 관련 있는 조각들을 검색한다. 검색된 조각들을 LLM에게 컨텍스트로 제공하고, 그것을 기반으로 답하게 한다. LLM의 일반 지식 + 본인의 특수 지식을 결합한다.</p>
  </div>
</div>

**RAG의 5단계 파이프라인.**

1단계: **문서 수집과 전처리**. 본인의 PDF 논문, Word 문서, 텍스트 노트, 웹 페이지를 수집. 텍스트로 변환.

2단계: **청킹(Chunking)**. 긴 문서를 작은 조각(보통 200-1000 토큰)으로 나눔. 청크 크기는 트레이드오프: 작으면 정확하지만 컨텍스트 부족, 크면 컨텍스트 풍부하지만 부정확.

3단계: **임베딩(Embedding)**. 각 청크를 벡터로 변환. OpenAI text-embedding-3, sentence-transformers, BGE-large 같은 임베딩 모델 사용.

4단계: **벡터 데이터베이스 저장**. 벡터 데이터베이스에 청크와 임베딩을 저장. ChromaDB, FAISS, Pinecone, Weaviate, Qdrant 등.

5단계: **질의 처리**. 사용자 질문을 임베딩으로 변환 → 가장 가까운 청크 K개 검색 → 청크들을 LLM 프롬프트에 컨텍스트로 추가 → LLM이 답 생성.

**기본 코드 예시 (LangChain).**

```python
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain.llms import OpenAI

# 1. 문서 로드
loader = PyPDFLoader("my_thesis.pdf")
docs = loader.load()

# 2. 청킹
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

# 3-4. 임베딩과 벡터 DB
embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
vectordb = Chroma.from_documents(chunks, embeddings)

# 5. 질의 응답
qa = RetrievalQA.from_chain_type(llm=OpenAI(), retriever=vectordb.as_retriever())
answer = qa.run("내 논문의 핵심 기여는 무엇인가?")
print(answer)
```

**RAG의 장점.** 첫째, **본인의 데이터에 접근**. 본인의 박사 논문, 연구실 노트, 분야 핵심 논문 100편을 LLM에게 알게 할 수 있다. 둘째, **최신성**. 새 문서를 추가하면 즉시 반영된다. LLM 재학습이 필요 없다. 셋째, **출처 추적**. 답변이 어느 문서/페이지에서 나왔는지 인용 가능. 환각(hallucination) 가능성이 줄어든다. 넷째, **저비용**. LLM 파인튜닝보다 훨씬 싸다.

**연구자에게 RAG의 활용 예시.** 첫째, **본인의 박사 논문 챗봇**. 자기 논문의 모든 내용에 대해 질문 가능. 둘째, **분야 핵심 논문 검색 어시스턴트**. 100-1000편의 분야 논문을 RAG로 만들어 "X 방법을 사용한 논문을 찾아 줘" 같은 질의 가능. 셋째, **연구실 지식 베이스**. 연구실의 내부 문서, 매뉴얼, 과거 학생들의 노트를 RAG로 만들어 새 학생의 적응을 도움. 넷째, **저널 정책 검색**. 본인이 투고하려는 저널의 가이드라인 PDF를 RAG로 만들어 빠른 정책 확인.

**RAG의 한계와 함정.** 첫째, **검색 품질이 핵심**. 검색이 잘못되면 LLM이 잘못된 정보를 받아 잘못된 답을 한다. 임베딩 모델, 청크 크기, 검색 알고리즘을 조정해야 한다. 둘째, **컨텍스트 초과**. 검색된 청크가 LLM의 컨텍스트 윈도우를 초과하면 일부가 잘리거나 무시된다. 셋째, **추론의 한계**. RAG는 검색된 텍스트에 기반한 답만 한다. 검색된 정보를 종합하여 새로운 통찰을 만드는 능력은 제한적. 넷째, **수치/표 데이터**. 텍스트 임베딩은 수치 데이터나 표를 잘 처리하지 못한다. 표는 별도 처리가 필요.

## LLM 파인튜닝과 LoRA

RAG가 본인의 정보를 LLM에 제공하는 방법이라면, **파인튜닝**은 LLM 자체를 본인의 데이터에 맞게 수정하는 방법이다. 둘은 보완적이다. RAG는 빠르고 저비용이지만 깊은 도메인 적응이 어렵다. 파인튜닝은 비용이 들지만 특정 도메인에 깊이 적응한다.

**언제 파인튜닝이 필요한가.** 첫째, 본인의 도메인이 LLM의 일반 지식과 매우 다를 때(예: 특수 공학 분야의 전문 용어). 둘째, 본인이 원하는 출력 형식이 매우 일관되어야 할 때. 셋째, 본인의 도메인 데이터가 충분히 많을 때(최소 1000+ 예시). 넷째, RAG로는 충분한 성능을 얻지 못할 때.

**언제 RAG를 우선해야 하는가.** 첫째, 본인의 자료가 자주 업데이트될 때. 둘째, 출처 인용이 중요할 때. 셋째, 데이터가 적을 때(100-1000 예시). 넷째, 빠르게 시작하고 싶을 때.

**LoRA(Low-Rank Adaptation): 효율적인 파인튜닝.** 큰 LLM(7B-70B 파라미터)을 전체 파인튜닝하면 막대한 GPU와 메모리가 필요하다. LoRA는 모델의 가중치는 고정하고, 작은 추가 행렬(rank r의 저차원 행렬)만 학습한다. 학습 파라미터가 1-10%로 줄어 한 GPU(24GB)로 7B 모델을 파인튜닝할 수 있다. 추론 시 LoRA 가중치를 원본 모델과 결합하면 원본과 같은 속도로 작동한다.

**파인튜닝 도구와 라이브러리.**

| 도구 | 특징 |
|------|------|
| Hugging Face PEFT | LoRA, Prefix Tuning 등 표준 라이브러리 |
| Hugging Face TRL | RLHF, DPO 같은 정렬 학습 |
| Axolotl | 파인튜닝 설정 자동화 |
| Unsloth | 학습 속도와 메모리 효율 최적화 |
| LLaMA Factory | GUI 기반 파인튜닝 |

**기본 LoRA 파인튜닝 코드.**

```python
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")

lora_config = LoraConfig(
    r=8,  # rank
    lora_alpha=32,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.1,
    bias="none",
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# 출력 예: trainable params: 4,194,304 || all params: 6,738,415,616 || trainable: 0.062%
```

**파인튜닝의 단계별 진행.**

1단계: **데이터 준비**. 본인의 도메인 데이터를 LLM 학습 형식(JSONL)으로 변환. 보통 instruction-response 쌍 또는 chat format. 최소 500-1000개 예시.

2단계: **베이스 모델 선택**. 본인의 작업에 적합한 모델 선택. 한국어 작업이면 EXAONE, KoLLaMA 등 한국어 사전학습 모델. 영어면 LLaMA 3, Mistral, Qwen 등.

3단계: **LoRA 설정**. rank(보통 8-64), target modules, learning rate(보통 1e-4) 설정.

4단계: **학습**. 1-3 에포크면 충분. 너무 많이 학습하면 과적합되어 일반 능력이 저하된다.

5단계: **평가**. 본인의 테스트 세트에서 평가. 베이스 모델과 비교하여 개선이 있는지 확인.

6단계: **추론과 배포**. 학습된 LoRA 가중치를 베이스 모델과 결합하여 사용.

**파인튜닝 vs RAG의 결합.** 가장 강력한 시스템은 두 가지를 결합한다. LoRA로 도메인 언어와 출력 형식을 학습 → RAG로 최신 정보 제공. 본인의 박사 연구가 LLM 응용이라면 둘 다 시도해 볼 가치가 있다.

**현실적 조언.** 박사 과정 학생이 처음 LLM을 활용한다면, 먼저 RAG부터 시도한다. 빠르게 결과를 볼 수 있고 저비용. RAG로 충분하지 않으면 그때 파인튜닝을 검토. 파인튜닝은 GPU 자원, 데이터 준비, 디버깅에 상당한 시간이 든다. 본인의 박사 연구의 핵심이 LLM 자체가 아니라면, RAG로 충분한 경우가 많다.
