# 생성형 AI 활용법

2022년 말 ChatGPT가 등장한 이후, 세상이 바뀌었다. 과장이 아니다. 코드를 작성하고, 논문을 교정하고, 아이디어를 정리하고, 수식을 풀고, 데이터를 분석하는 방식이 근본적으로 변하고 있다. 대학원생의 일상도 예외가 아니다.

생성형 AI(Generative AI)란 텍스트, 이미지, 코드, 음악 등의 콘텐츠를 "생성"하는 AI를 말한다. 대규모 언어 모델(Large Language Model, LLM)이 그 핵심이다. ChatGPT(OpenAI), Claude(Anthropic), Gemini(Google)가 대표적인 LLM 기반 서비스이고, GitHub Copilot은 코드 생성에 특화된 도구다.

이 장에서는 생성형 AI를 공학 연구에 어떻게 활용할 수 있는지를 구체적으로 다룬다. 어떤 작업에 써야 하고, 어떤 작업에 쓰면 안 되는지, 어떻게 써야 효과적인지, 그리고 무엇을 조심해야 하는지를 솔직하게 이야기한다.

한 가지 먼저 분명히 해둔다. 생성형 AI는 도구(tool)다. 아무리 뛰어난 도구라도 사용자의 판단을 대체하지는 못한다. 망치가 아무리 좋아도 못을 어디에 박을지는 목수가 결정한다. 생성형 AI가 아무리 똑똑해도, 연구의 방향과 판단은 연구자의 몫이다.

## 연구 활용 4가지

공학 대학원생이 생성형 AI를 활용할 수 있는 핵심 영역은 네 가지다. 가장 먼저 시도할 것은 "코딩 어시스턴트"다. 디버깅이나 코드 설명은 결과를 즉시 검증할 수 있어 할루시네이션 위험이 낮고, 시간 절약 효과가 가장 크다.

### 문헌조사 보조

논문을 찾고, 읽고, 정리하는 과정에서 AI가 큰 도움이 된다. 하지만 여기서 "보조"라는 단어를 강조한다. AI가 문헌조사를 "대신" 해주는 것이 아니다.

구체적인 활용법을 보자. 첫째, 논문 요약이다. 긴 논문의 핵심 내용을 빠르게 파악하고 싶을 때, 논문의 Abstract와 Conclusion을 AI에게 보여주고 "이 논문의 주요 기여와 한계를 요약해달라"고 요청할 수 있다. 전체 논문을 붙여넣고 "방법론 부분만 한국어로 요약해달라"고 할 수도 있다. 초기 스크리닝 단계에서 수십 편의 논문을 빠르게 훑는 데 유용하다.

둘째, 개념 설명이다. 논문에서 낯선 개념이나 기법이 나왔을 때, AI에게 물어볼 수 있다. "Variational Inference가 뭔지 대학원 1년차 수준으로 설명해달라", "Attention mechanism이 CNN의 pooling과 뭐가 다른지 비교해달라" 같은 질문이다. 교과서를 찾아보는 것보다 훨씬 빠르게 직관적 이해를 얻을 수 있다.

셋째, 연구 동향 파악이다. "PINN의 최근 3년 연구 동향을 분야별로 정리해달라", "Transfer learning in structural health monitoring 분야의 주요 연구 그룹과 논문을 알려달라" 같은 요청으로 분야의 큰 그림을 파악할 수 있다. 단, AI가 제시하는 논문 제목과 저자는 반드시 Google Scholar에서 실제로 존재하는지 확인해야 한다. 이것은 아래 할루시네이션 절에서 다시 다룬다.

### 코딩 어시스턴트

이것이 아마 가장 즉각적이고 실용적인 활용처일 것이다. AI는 코드 작성, 디버깅, 최적화, 설명에서 놀라운 능력을 보여준다.

코드 작성 보조: "PyTorch로 1D CNN 모델을 작성해줘. 입력은 길이 1000의 시계열, 출력은 3개 클래스 분류야"라고 요청하면, 실행 가능한 코드 초안을 받을 수 있다. 이 코드를 기반으로 수정하면, 처음부터 작성하는 것보다 훨씬 빠르다.

디버깅: 에러 메시지와 관련 코드를 붙여넣고 "이 에러가 왜 나는지 설명하고 수정해달라"고 요청한다. 특히 PyTorch의 텐서 차원 불일치(shape mismatch) 에러나, 복잡한 데이터 전처리 코드의 버그를 찾는 데 매우 효과적이다.

코드 설명: 다른 연구자의 GitHub 코드를 이해해야 할 때, 코드를 붙여넣고 "이 함수가 뭘 하는 건지 한 줄씩 설명해달라"고 요청한다. 문서화가 잘 안 된 오픈소스 코드를 분석할 때 특히 유용하다.

코드 최적화: "이 코드를 vectorize해서 속도를 개선해달라", "이 반복문을 NumPy 연산으로 바꿔달라" 같은 요청으로 코드 효율을 높일 수 있다.

### 글쓰기 보조

영어 논문 작성에서 AI의 도움이 크다. 특히 영어가 모국어가 아닌 연구자에게 유용하다.

문법 교정과 표현 개선: 작성한 영문 단락을 AI에게 보여주고 "학술적 톤으로 문법과 표현을 교정해달라"고 요청한다. 어색한 표현을 자연스럽게 바꿔주고, 학술 논문에 적합한 어휘를 제안해준다. Grammarly보다 맥락을 더 잘 이해하는 경우가 많다.

초고 작성 보조: "이 실험 조건과 결과로 Methodology 섹션의 초안을 작성해달라"고 할 수 있다. 하지만 AI가 작성한 초안을 그대로 사용하면 안 된다. AI의 초안은 구조와 흐름을 잡는 데 활용하고, 내용의 정확성과 표현은 반드시 직접 검토하고 수정해야 한다.

표현 제안: "increase와 enhance와 improve의 차이가 뭔지, 논문에서 각각 어떤 맥락에서 쓰이는지 알려달라" 같은 질문으로 적확한 학술 표현을 찾을 수 있다.

### 아이디어 브레인스토밍

연구 아이디어를 구상하는 단계에서 AI를 토론 상대로 활용할 수 있다. "PINN을 구조 건전성 모니터링에 적용하려면 어떤 접근이 가능할까?", "이 데이터셋에서 시도해볼 만한 ML 기법이 뭐가 있을까?" 같은 열린 질문을 던지고, AI의 제안을 비판적으로 검토한다.

AI의 제안이 모두 좋은 것은 아니다. 오히려 대부분은 이미 알려진 것이거나, 실현 가능성이 낮은 것이다. 하지만 한두 가지 생각지 못한 관점을 발견할 수 있고, 그것이 연구의 실마리가 될 수 있다. AI를 "정답을 주는 존재"가 아니라 "질문을 던져주는 존재"로 활용하는 것이 핵심이다.

어떤 도구를 쓸지는 용도에 따라 다르다. 코딩 작업이 많으면 GitHub Copilot이나 Cursor를 IDE에 설치하는 것이 최우선이다. 논문 검색에는 Elicit이나 Semantic Scholar를 쓰되, ChatGPT/Claude는 개념 이해와 아이디어 정리에 활용한다. 영어 글쓰기에는 ChatGPT/Claude로 교정하되, 최종적으로 Grammarly나 Writefull로 한 번 더 확인한다.

중요한 것은, 하나의 도구에 의존하지 말라는 것이다. 각 도구의 강점이 다르므로, 작업에 따라 도구를 바꿔가며 사용하는 유연함이 필요하다.

## 프롬프트 엔지니어링

생성형 AI의 출력 품질은 입력(프롬프트)의 품질에 크게 의존한다. 같은 질문이라도 어떻게 묻느냐에 따라 답변의 수준이 천차만별이다. 효과적인 프롬프트를 작성하는 네 가지 원칙을 정리한다. 오늘 당장 하나만 시도한다면, "맥락 제공"부터 해보자. 평소에 던지던 짧은 질문에 배경 정보를 세 줄만 추가해도 답변 품질이 눈에 띄게 달라진다.

### 역할 지정

AI에게 역할을 부여하면 답변의 수준과 관점이 달라진다. "너는 구조공학 전공 교수야. 대학원 1년차 학생에게 설명하듯이 답변해줘"라고 시작하면, 일반적인 답변보다 전문적이면서도 이해하기 쉬운 답변을 받을 수 있다. "너는 Python 시니어 개발자야. 클린 코드와 에러 핸들링을 중시해줘"라고 하면, 더 견고한 코드를 받을 수 있다.

### 맥락 제공

질문에 충분한 배경 정보를 제공해야 한다. "CNN 성능이 안 나와"보다 "ImageNet 사전학습된 ResNet-50으로 콘크리트 균열 이미지 500장을 파인튜닝하고 있는데, validation accuracy가 70%에서 멈춘다. 학습률은 1e-3, 배치 크기 32, 데이터 증강은 수평 반전만 적용했다"가 훨씬 유용한 답변을 이끌어낸다.

### 단계별 지시

복잡한 작업은 한 번에 요청하지 말고 단계별로 나눠서 요청한다. "이 데이터를 분석하고 시각화하고 모델을 학습시키고 결과를 해석해줘"보다, 각 단계를 개별적으로 요청하는 것이 결과가 좋다. "1단계: 이 데이터의 기본 통계량을 계산하고 분포를 시각화해줘"로 시작하고, 결과를 확인한 후 다음 단계를 요청한다.

### 출력 형식 지정

원하는 출력의 형식을 명시한다. "표로 정리해줘", "코드 블록으로 작성해줘", "장단점을 비교하는 형식으로 답변해줘", "500자 이내로 요약해줘" 등이다. 형식을 지정하지 않으면 AI가 자기 판단으로 출력 형식을 결정하는데, 원하는 형식과 다를 수 있다.

**코드 디버깅 프롬프트 예시:**

"다음 PyTorch 코드에서 RuntimeError: mat1 and mat2 shapes cannot be multiplied 에러가 발생한다. 모델 구조와 데이터 로딩 코드를 보고 원인을 찾아 수정해달라."

이후 에러 메시지 전문과 관련 코드를 붙여넣는다. 핵심은 에러 메시지를 생략하지 않는 것이다. 전체 traceback을 포함시키면 AI가 정확한 위치를 파악할 수 있다. 코드도 관련 부분만이 아니라, 데이터 로딩부터 모델 정의, 학습 루프까지 전체 맥락을 제공하는 것이 좋다.

디버깅 결과를 받으면, AI의 수정 사항을 이해하고 나서 적용한다. "왜 이 수정이 문제를 해결하는가?"를 이해하지 못하면, 비슷한 에러가 다시 발생했을 때 또 AI에게 의존하게 된다.

</div>

**논문 교정 프롬프트 예시:**

"다음은 내 논문의 Methodology 섹션의 한 단락이다. 학술 영어 톤으로 문법, 표현, 논리적 흐름을 교정해달라. 원래 의미를 바꾸지 말고, 수정한 부분에는 [수정]이라고 표시해달라."

한 번에 전체 논문을 넣지 말고, 한 섹션 또는 한 단락씩 처리하는 것이 품질이 높다. 전체를 한 번에 넣으면 AI가 세부적인 표현까지 주의를 기울이지 못하고 대충 넘어가는 경향이 있다.

수정된 결과를 받으면, 수정 사항을 하나하나 확인한다. AI가 의미를 잘못 해석하여 원래와 다른 뜻으로 바꾸는 경우가 있다. 특히 기술 용어가 포함된 문장에서 이런 일이 발생할 수 있다.

**문헌조사 프롬프트 예시:**

"나는 PINN(Physics-Informed Neural Networks)을 열전달 역문제에 적용하는 연구를 하고 있다. 이 분야의 주요 연구 동향, 핵심 논문, 현재 미해결 문제를 정리해달라. 단, 논문을 언급할 때는 저자, 연도, 학술지를 명시해달라."

AI가 제시하는 논문 목록을 받으면, 반드시 Google Scholar에서 각 논문이 실제로 존재하는지 확인한다. AI는 그럴듯하지만 존재하지 않는 논문을 만들어내는 경우가 빈번하다. 저자 이름이 맞지만 제목이 다르거나, 제목이 비슷하지만 학술지가 다르거나, 아예 존재하지 않는 논문을 자신 있게 인용하는 경우가 있다.

문헌조사에서 AI의 가장 안전한 활용법은 "검색 키워드 제안"이다. "이 주제로 검색할 때 어떤 키워드 조합을 쓰면 좋을까?"라고 물으면, 여러 키워드 조합을 제안해준다. 이 키워드로 Google Scholar에서 직접 검색하는 것이 정확하다.

**수식/이론 질문 프롬프트 예시:**

"가우시안 프로세스 회귀에서 커널 함수의 하이퍼파라미터를 최적화하는 방법을 설명해달라. marginal likelihood를 최대화하는 과정을 수식과 함께 단계별로 설명해줘. 대학원 1년차 수준으로 부탁한다."

수식 관련 답변을 받으면, 반드시 직접 검산한다. AI는 수식의 전반적인 구조는 맞추지만, 부호, 계수, 적분 범위 같은 세부사항에서 실수를 하는 경우가 있다. 특히 복잡한 유도 과정에서 중간 단계가 빠지거나, 가정을 명시하지 않는 경우가 흔하다.

교과서의 해당 부분과 대조하거나, 특수한 경우(예: 1차원, 단순 커널)에서 수식이 알려진 결과와 일치하는지 확인하는 것이 좋은 검증 방법이다.

## 할루시네이션 탐지

생성형 AI의 가장 위험한 특성은 할루시네이션(hallucination)이다. AI가 사실이 아닌 내용을 자신 있게 말하는 현상이다. 거짓말을 하려는 의도가 아니라, 모델의 구조적 한계 때문에 발생한다. LLM은 "그럴듯한 다음 단어"를 생성하도록 학습되었지, "사실인 다음 단어"를 생성하도록 학습되지 않았다.

공학 연구에서 할루시네이션은 치명적일 수 있다. 존재하지 않는 논문을 인용하면 학술적 신뢰가 무너지고, 틀린 수식을 사용하면 연구 결과 전체가 무효가 된다. 할루시네이션을 탐지하고 방지하는 구체적인 방법을 알아야 한다.

### 인용 논문 검증

AI가 언급하는 모든 논문을 Google Scholar에서 확인한다. 다음 세 단계를 매번 실행한다.

1. AI가 제시한 논문 제목을 그대로 복사하여 Google Scholar에 붙여넣고 검색한다.
2. 저자, 제목, 학술지, 연도가 모두 일치하는지 하나씩 대조한다.
3. 한 가지라도 다르면 해당 인용을 사용하지 않는다.

AI가 "Smith et al. (2020), Journal of Computational Physics"라고 말했는데, 실제로 검색해보면 그런 논문이 없거나 제목이 다른 경우가 흔하다.

Semantic Scholar나 Elicit 같은 학술 검색 도구를 함께 사용하면 검증이 더 쉽다. 이들은 실제 논문 데이터베이스에 기반하므로 할루시네이션이 거의 없다.

### 수식 검산

AI가 제시한 수식을 반드시 직접 검산한다. 간단한 특수 케이스(예: 1차원, 선형, 단일 변수)에서 수식이 알려진 결과를 재현하는지 확인한다. 차원 분석(dimensional analysis)을 수행하여 양변의 단위가 일치하는지 확인한다. 수식의 극한값(0, 무한대 등)에서 물리적으로 합리적인 결과가 나오는지 확인한다.

### 코드 실행 검증

AI가 작성한 코드를 반드시 실행하여 결과를 확인한다. 컴파일/실행이 되는 것만 확인하는 것이 아니라, 알려진 입력에 대해 예상되는 출력이 나오는지 테스트한다. 특히 경계 조건(빈 입력, 매우 큰 입력, 음수 등)에서의 동작을 확인한다.

AI가 작성한 코드에서 흔히 발생하는 문제: 존재하지 않는 라이브러리 함수 호출, 더 이상 사용되지 않는(deprecated) API 사용, 잘못된 인자 순서, 특정 버전에서만 작동하는 문법 등이 있다.

<div class="highlight-box warning">
<span class="highlight-box-icon">⚠️</span>
<div class="highlight-box-content">
<p><strong>할루시네이션 경고 신호</strong></p>
<p>AI의 답변에서 다음과 같은 신호가 보이면 할루시네이션을 의심한다. 1) 매우 구체적인 숫자를 자신 있게 제시한다 ("이 방법은 정확도를 23.7% 향상시켰다"). 2) 잘 알려지지 않은 논문을 마치 유명한 것처럼 인용한다. 3) 질문의 전제에 맞추려고 사실을 왜곡한다 ("X가 Y보다 좋다고 알려져 있다"는 질문에, 실제로는 그렇지 않은데도 동의하는 경우). 4) 여러 번 물었을 때 다른 답을 한다. 같은 질문을 다시 했을 때 이전과 다른 논문을 인용하면 둘 다 신뢰할 수 없다.</p>

## 윤리 가이드라인

생성형 AI를 연구에 사용할 때의 윤리적 기준은 아직 학계에서 정립 중이다. 하지만 이미 많은 학술지와 학회가 가이드라인을 발표하고 있다. 현재 시점에서의 핵심 원칙을 정리한다.

<li>AI가 생성한 내용을 반드시 검증한다 (논문 존재 여부, 수식 정확도, 코드 실행)</li>
<li>AI 사용 사실을 논문에 명시한다 (많은 학술지가 이를 요구한다)</li>
<li>AI의 도움을 받은 부분과 직접 작성한 부분을 구분한다</li>

### AI 사용 명시

현재 Nature, Science, IEEE, ASCE 등 주요 학술지는 AI 사용을 논문에 명시하도록 요구하고 있다. 일반적으로 Acknowledgements 섹션이나 별도의 AI Declaration 섹션에 기술한다. 아래는 바로 사용할 수 있는 선언문 예시다.

> During the preparation of this manuscript, the authors used [ChatGPT / Claude / GitHub Copilot] for [grammar correction / code debugging / literature keyword suggestion]. The authors reviewed and edited all AI-generated content and take full responsibility for the final publication.

이 문구를 자신의 사용 범위에 맞게 수정하여 논문에 포함한다.

AI를 공저자(co-author)로 표기하는 것은 대부분의 학술지에서 금지하고 있다. 공저자는 연구의 책임을 질 수 있는 인간이어야 한다는 것이 기본 원칙이다.

### 기밀 데이터 주의

상용 AI 서비스(ChatGPT, Claude 등)에 입력한 데이터는 서비스 제공자의 서버로 전송된다. 미발표 실험 데이터, 산업체의 기밀 정보, 특허 출원 전의 아이디어 등을 AI에 입력하면 정보 유출 위험이 있다. 대부분의 서비스가 "사용자 데이터를 학습에 사용하지 않는다"고 명시하고 있지만, 서비스 약관을 반드시 확인하고, 기밀 데이터는 가능하면 로컬에서 실행되는 모델(오픈소스 LLM)을 사용하는 것이 안전하다.

## 공학 연구에서의 한계

생성형 AI의 한계를 솔직하게 짚어본다. 이 한계를 알아야 도구를 현명하게 사용할 수 있다.

### 수식 정확도

현재의 LLM은 수학적 추론에서 여전히 약점을 보인다. 단순한 미적분이나 선형대수는 대체로 맞추지만, 복잡한 유도 과정에서 부호 오류, 계수 오류, 논리적 비약이 발생하는 경우가 적지 않다. 텐서 미적분, 변분법, 복소 해석 같은 고급 수학에서는 신뢰도가 더 떨어진다.

구체적인 예를 들면, 나비에-스토크스 방정식의 비압축성 조건에서 특정 경계조건을 적용한 유도를 요청하면, 전반적인 흐름은 맞지만 중간 단계에서 항이 빠지거나, 적분 상수 처리가 잘못되는 경우가 있다. 이런 오류는 최종 결과에 영향을 미치므로, AI가 유도한 수식을 논문에 직접 사용하는 것은 매우 위험하다.

수식은 AI의 도움을 받되, 반드시 교과서나 원논문으로 검증해야 한다. AI는 "이런 유도 과정이 있다"는 방향을 알려주는 데 유용하고, 정확한 수식 자체를 제공하는 데는 아직 불안정하다.

### 최신 논문 미반영

LLM은 학습 데이터의 시점까지만 알고 있다. 학습 데이터 이후에 발표된 논문, 새로 등장한 기법, 최근의 벤치마크 결과 등은 알지 못한다. "2025년에 발표된 논문 중..."이라는 질문에 AI가 답변을 한다면, 그것은 높은 확률로 할루시네이션이다.

최신 연구 동향은 Google Scholar, arXiv, Semantic Scholar에서 직접 검색해야 한다. AI는 "기존에 알려진 일반적 지식"을 정리하는 데 활용하고, "최신 연구"는 직접 확인한다.

### 도메인 지식의 깊이

AI는 다양한 분야에 대해 폭넓은 지식을 가지고 있지만, 특정 분야의 깊이는 해당 분야 전문가에 미치지 못한다. "CFRP 복합재의 층간 분리 진전 메커니즘에서 Mode I과 Mode II의 혼합 비율이 피로 수명에 미치는 영향"같은 구체적이고 전문적인 질문에는 피상적인 답변을 하거나 잘못된 정보를 제시할 수 있다.

깊은 도메인 지식이 필요한 부분은 교과서, 전문 논문, 지도교수에게 확인해야 한다 (분야별 핵심 교과서와 추천 학습 경로는 ch27 '교과서, 강의, 필독 논문'을 참고한다.). AI는 넓지만 얕은 지식을 제공한다고 생각하면 된다.

### 재현성 문제

같은 프롬프트를 입력해도 AI의 답변이 매번 다를 수 있다(temperature 설정에 따라). 이는 연구의 재현성에 문제가 된다. AI가 생성한 코드나 분석 결과를 논문에 포함한다면, 최종적으로 사용한 코드를 별도로 저장하고 공유해야 한다. "AI에게 이렇게 물어서 받은 코드를 사용했다"는 것만으로는 다른 연구자가 같은 결과를 재현할 수 없다.

## 생성형 AI를 잘 쓰는 연구자의 특징

생성형 AI를 효과적으로 활용하는 연구자들에게는 공통된 특징이 있다.

첫째, 자기 분야의 기초가 탄탄하다. 기초가 없으면 AI의 답변이 맞는지 틀린지 판단할 수 없다. AI가 제시한 수식이 물리적으로 타당한지, 제안한 방법론이 내 문제에 적합한지를 판단하려면 해당 분야의 기본 지식이 있어야 한다. AI는 기초 없이 연구하는 지름길이 아니다. 오히려 기초가 있는 사람이 AI를 더 잘 활용한다 (기초를 다지기 위한 교과서, 강의, 필독 논문은 ch27 '교과서, 강의, 필독 논문'에서 정리한다.).

둘째, 비판적으로 사고한다. AI의 모든 출력을 의심하고 검증한다. "AI가 이렇게 말했으니까 맞겠지"라는 태도는 위험하다. "AI가 이렇게 말했는데, 정말 그런가?"라고 질문하는 습관이 필요하다.

셋째, AI를 대화 상대로 활용한다. 일방적으로 질문하고 답을 받는 것이 아니라, 후속 질문을 던지고, 답변의 약점을 지적하고, 다른 관점을 요구하는 식으로 대화를 이어간다. "그 방법의 단점은 뭔가?", "다른 접근법은 없는가?", "이 가정이 틀리면 어떻게 되나?" 같은 후속 질문이 AI의 가치를 극대화한다.

넷째, 결과를 자기 것으로 만든다. AI가 작성한 코드를 그대로 쓰는 것이 아니라, 이해하고 수정하고 개선한다. AI가 제안한 아이디어를 그대로 따르는 것이 아니라, 비판적으로 검토하고 자기 연구의 맥락에 맞게 재해석한다.

<div class="highlight-box info">
<span class="highlight-box-icon">ℹ️</span>
<div class="highlight-box-content">
<p><strong>AI 활용의 80/20 법칙</strong></p>
<p>생성형 AI가 연구의 80%를 5분 만에 처리해 줄 것 같지만, 현실은 다르다. AI가 생산적으로 도울 수 있는 것은 전체 연구의 20% 정도(초안 작성, 코드 스캐폴딩, 개념 정리)이며, 나머지 80%(실험 설계, 데이터 분석, 결과 해석, 논문의 핵심 기여)는 여전히 연구자의 몫이다. 하지만 그 20%에서 절약되는 시간이 상당하므로, AI를 잘 활용하는 것은 분명히 경쟁력이 된다.</p>

## AI와 함께 연구하는 미래

생성형 AI는 빠르게 발전하고 있다. 현재의 한계(수식 오류, 할루시네이션, 최신 정보 부족)는 점차 개선될 것이다. 멀티모달 AI가 이미지, 그래프, 표를 이해하고 분석하는 능력도 향상되고 있다. AI가 논문을 직접 읽고, 실험 데이터를 분석하고, 시뮬레이션 결과를 해석하는 날이 먼 미래가 아니다.

하지만 AI가 아무리 발전해도, 연구의 핵심은 변하지 않는다. "어떤 문제를 풀어야 의미가 있는가?"를 판단하는 것, "이 결과가 기존 지식에 어떤 기여를 하는가?"를 해석하는 것, "이 연구가 사회에 어떤 영향을 미치는가?"를 고민하는 것. 이것들은 인간 연구자의 고유한 역할이다.

AI 시대에 경쟁력 있는 연구자가 되는 방법은 두 가지다. 첫째, AI 도구를 능숙하게 사용할 줄 아는 것. 프롬프트를 잘 작성하고, 적절한 도구를 선택하고, 할루시네이션을 걸러내는 능력이다. 둘째, AI가 할 수 없는 것을 잘하는 것. 깊은 도메인 지식, 창의적 문제 정의, 비판적 사고, 윤리적 판단이다.

도구를 잘 쓰되, 도구에 지배당하지 않는 것. 이것이 AI 시대 연구자의 자세다.

## 프롬프트 실패와 성공: Before/After

앞서 프롬프트 엔지니어링의 원칙을 다뤘다. 원칙은 알겠는데, 실제로 뭐가 달라지는지 감이 안 올 수 있다. 여기서는 세 가지 실전 사례를 Before(나쁜 프롬프트)와 After(개선 프롬프트)로 비교한다. 같은 질문이라도 어떻게 묻느냐에 따라 답변의 품질이 완전히 달라지는 것을 직접 확인할 수 있다.

### 사례 1: 코드 디버깅

**Before (나쁜 프롬프트):**

> "파이썬 코드 에러 고쳐줘"

이 프롬프트에는 세 가지가 빠져 있다. 어떤 에러인지, 어떤 코드인지, 어떤 환경인지. AI는 "어떤 에러가 발생하셨나요?"라고 되물을 수밖에 없고, 여러 번 주고받아야 겨우 핵심에 접근할 수 있다.

**After (개선 프롬프트):**

> "Python 3.10, PyTorch 2.1 환경에서 아래 코드를 실행하면 RuntimeError: mat1 and mat2 shapes cannot be multiplied (64x128 and 256x10) 에러가 발생한다. 모델은 1D CNN으로, 입력 크기는 (batch, 1, 1000)이고, Linear 레이어에서 에러가 난다. 에러 traceback 전문과 모델 정의 코드를 아래에 첨부한다. 원인을 분석하고 수정 코드를 제시해달라."

**무엇이 달라졌는가:** 환경(Python/PyTorch 버전), 에러 메시지 전문, 입력 형태, 에러 발생 위치를 명시했다. AI가 한 번에 정확한 원인(CNN 출력 크기와 Linear 입력 크기 불일치)을 짚어내고, `nn.Flatten()` 후의 텐서 크기를 계산하여 수정 코드를 제시할 수 있다. 맥락이 충분하면 AI의 디버깅 정확도가 비약적으로 높아진다.

### 사례 2: 논문 문장 교정

**Before (나쁜 프롬프트):**

> "이 문장 고쳐줘: The result is showed that the method is better than the previous ones."

AI는 문법만 고쳐서 "The results showed that the method is better than the previous ones."라고 돌려준다. 문법적으로 틀리진 않지만, 학술 논문에 쓰기엔 여전히 모호하고 약하다.

**After (개선 프롬프트):**

> "아래는 내 논문 Results 섹션의 문장이다. 학술 영어 톤(수동태 허용, 과거시제)으로 교정해달라. 원래 의미를 바꾸지 말고, 수정한 부분마다 왜 바꿨는지 간단히 설명해달라. 비교 대상은 기존 FEM 해석법이고, 개선 지표는 계산 시간 단축이다.
> 원문: The result is showed that the method is better than the previous ones."

**무엇이 달라졌는가:** 섹션(Results), 톤(학술), 시제(과거), 수정 이유 설명 요청, 비교 대상과 지표를 명시했다. AI는 "The results demonstrated that the proposed method reduced the computational time by X% compared to the conventional FEM approach."처럼 구체적이고 학술적인 문장을 제안하고, 각 수정에 대한 이유(showed→demonstrated: 더 강한 학술 동사, better→reduced computational time: 구체적 지표)를 설명해준다. 맥락을 주면 AI가 "학술 영어의 관행"에 맞춰 교정할 수 있다.

### 사례 3: 데이터 분석 요청

**Before (나쁜 프롬프트):**

> "데이터 분석해줘"

AI가 할 수 있는 것이 거의 없다. 어떤 데이터인지, 어떤 분석인지, 어떤 목적인지 전부 모른다.

**After (개선 프롬프트):**

> "아래 CSV 데이터는 콘크리트 배합 실험 결과다. 열은 [W/C ratio, Cement(kg), Aggregate(kg), Curing_days, Compressive_strength(MPa)]이다. 총 150개 데이터 포인트가 있다. 다음 분석을 Python(pandas, matplotlib)으로 수행해달라.
> 1. 기술 통계량(평균, 표준편차, 최대, 최소)을 표로 정리
> 2. 각 변수와 Compressive_strength 간의 상관계수 히트맵 시각화
> 3. W/C ratio와 Compressive_strength의 산점도에 회귀선 추가
> 각 코드 블록에 주석을 달아달라."

**무엇이 달라졌는가:** 데이터의 구조(열 이름, 데이터 수), 사용 도구(pandas, matplotlib), 분석 내용(3단계), 출력 형식(표, 히트맵, 산점도), 부가 요청(주석)을 모두 명시했다. AI는 즉시 실행 가능한 코드를 단계별로 생성한다. "뭘 분석할까요?"라는 역질문 없이 바로 결과물이 나온다.

세 사례의 공통점은 명확하다. **맥락, 구체성, 출력 형식.** 이 세 가지를 프롬프트에 포함하면 AI의 답변 품질이 극적으로 달라진다. "좋은 프롬프트를 쓰는 데 시간이 걸린다"고 생각할 수 있지만, 나쁜 프롬프트로 여러 번 주고받는 시간을 합치면 좋은 프롬프트 한 번이 훨씬 빠르다.

## AI 사용 윤리 결정 트리

"이 상황에서 AI를 써도 되는가?" 연구 중에 이 질문이 떠오를 때, 아래의 흐름도를 따라가면 판단 기준을 세울 수 있다. 직감이 아니라 체계적인 점검이 필요하다.

### 질문 1: 기밀 데이터인가?

AI에 입력하려는 데이터가 기밀인지 먼저 확인한다. 미발표 실험 결과, 산업체 비공개 데이터, 특허 출원 전 아이디어, 공동연구자가 공유 제한을 건 데이터가 해당된다.

**"예"라면:** 상용 AI 서비스(ChatGPT, Claude 등)에 입력하면 안 된다. 데이터가 서버로 전송되므로 정보 유출 위험이 있다. 반드시 로컬에서 실행되는 오픈소스 모델(Llama, Mistral 등)을 사용하거나, AI 사용 자체를 보류한다.

**"아니오"라면:** 다음 질문으로 넘어간다.

### 질문 2: 저널/학회의 AI 정책을 확인했는가?

투고하려는 학술지나 학회의 AI 사용 정책을 확인해야 한다. Nature, Science, IEEE, ASCE 등 주요 학술지는 이미 AI 사용에 관한 구체적인 가이드라인을 발표했다. 일부는 AI로 생성된 텍스트를 허용하되 명시를 요구하고, 일부는 특정 용도(문법 교정 등)만 허용하고, 극히 일부는 AI 사용을 전면 금지한다.

**"확인하지 않았다면":** 논문 투고 전에 반드시 확인한다. "Instructions for Authors" 또는 "Editorial Policy" 페이지에 AI 관련 조항이 있다. 정책을 모른 채 사용하면 투고 후 리젝 사유가 될 수 있다.

**"확인했다면":** 해당 정책의 범위 안에서만 사용하고, 다음 질문으로 넘어간다.

### 질문 3: AI 결과를 독립적으로 검증할 수 있는가?

AI가 생성한 코드, 수식, 텍스트, 분석 결과를 본인이 직접 검증할 수 있는 역량이 있는가? 코드를 실행해서 결과를 확인할 수 있는가? 수식을 직접 유도하거나 검산할 수 있는가? AI가 인용한 논문이 실제로 존재하는지 확인했는가?

**"검증할 수 없다면":** 사용을 보류한다. 검증할 수 없는 AI 출력을 논문에 포함하는 것은 학술적으로 매우 위험하다. 틀린 수식, 존재하지 않는 인용, 잘못된 분석이 논문에 들어가면 학술적 신뢰가 무너진다.

**"검증할 수 있다면":** 반드시 검증을 수행한 후에 사용하고, 다음 질문으로 넘어간다.

### 질문 4: 사용 사실을 명시할 수 있는가?

AI를 사용한 사실을 논문의 Acknowledgements나 별도 AI Declaration 섹션에 명시할 수 있는가? "이 과정에서 AI를 사용했다"고 솔직하게 밝힐 수 있는가?

**"명시하기 꺼려진다면":** 해당 용도의 AI 사용이 윤리적으로 적절한지 다시 생각해야 한다. 밝히기 꺼려지는 사용은 대부분 부적절한 사용이다. 예를 들어, 핵심 기여(방법론 설계, 결과 해석)를 AI에 전적으로 의존한 경우, 이를 명시하면 저자로서의 기여가 의심받을 수 있다. 그 경우, AI에 의존한 부분을 직접 수행하는 것이 맞다.

**"명시할 수 있다면":** AI를 사용하고, 사용 범위와 목적을 논문에 명확히 기술한다.

## Diffusion Models: 생성 AI의 새로운 표준

ch26 본문은 주로 텍스트 기반 LLM(ChatGPT, Claude)을 다뤘다. 하지만 생성 AI의 또 다른 큰 영역은 이미지, 비디오, 분자 구조의 생성이다. 이 분야는 **Diffusion Models**가 사실상 표준이 되었다. Stable Diffusion, DALL-E 3, Midjourney 모두 diffusion 기반이다. 본인이 합성 데이터를 만들거나 시각적 자료를 생성하려면 diffusion model을 알 가치가 있다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>Diffusion Model의 핵심 직관</strong></p>
    <p>"이미지에 노이즈를 점진적으로 추가하여 완전한 노이즈로 만든 후, 그 과정을 거꾸로 학습한다." 이것이 diffusion의 한 줄 요약이다. 노이즈에서 이미지를 복원하는 능력을 학습하면, 무작위 노이즈에서 시작해도 그럴듯한 이미지를 만들 수 있다. 이 단순한 아이디어가 이전 GAN 기반 방법보다 안정적이고 다양한 결과를 만들어 낸다.</p>
  </div>
</div>

**Diffusion vs GAN vs VAE.**

| 측면 | GAN | VAE | Diffusion |
|------|-----|-----|-----------|
| 학습 안정성 | 불안정(mode collapse) | 안정 | 매우 안정 |
| 샘플 품질 | 매우 높음 | 낮음~중간 | 매우 높음 |
| 다양성 | 낮을 수 있음 | 좋음 | 매우 좋음 |
| 학습 속도 | 빠름 | 빠름 | 느림 |
| 추론 속도 | 빠름 (단일 패스) | 빠름 | 느림 (수십~수백 단계) |
| 조건부 생성 | 가능하지만 어려움 | 가능 | 매우 강력 |

Diffusion은 거의 모든 측면에서 우수하지만, 학습과 추론이 느리다는 단점이 있다. 최근 연구들이 이 단점을 해결하고 있다(예: DDIM, Latent Diffusion, Consistency Models).

**공학 연구에서의 활용.**

첫째, **합성 데이터 생성**. 본인의 데이터가 부족할 때, diffusion model로 합성 데이터를 만들어 학습 데이터를 보강. 단, 합성 데이터의 한계(분포 편향, 비현실적 패턴)를 인지하고 검증해야 한다. 둘째, **Conditional Generation**. "X 조건의 부품 사진", "Y 결함 종류의 표면" 같은 조건부 생성. 본인의 데이터에서 부족한 사례를 보강할 수 있다. 셋째, **Domain Adaptation**. 시뮬레이션 이미지를 실제 이미지처럼 변환(sim-to-real). 자율주행, 로봇 학습에서 활용. 넷째, **Inpainting과 복원**. 손상된 영상을 복원하거나, 일부가 가려진 데이터를 채워 넣음. 다섯째, **분자 구조 생성**. 신약 개발, 신소재 설계에서 새로운 분자를 생성하는 데 활용. AlphaFold, RFdiffusion, GenMol 같은 모델.

**Diffusion model을 사용하는 실전 도구.**

| 도구 | 용도 |
|------|------|
| Hugging Face Diffusers | Python 라이브러리, 가장 표준 |
| Stable Diffusion (SDXL, SD3) | 오픈소스 이미지 생성 |
| ComfyUI | GUI 기반, 복잡한 워크플로 |
| Automatic1111 WebUI | 가장 인기 있는 SD 프론트엔드 |
| Replicate API | 클라우드 추론, 코딩 없이 사용 |
| RFdiffusion | 단백질 설계 특화 |

**기본 코드 예시 (Hugging Face Diffusers).**

```python
from diffusers import StableDiffusionPipeline
import torch

pipe = StableDiffusionPipeline.from_pretrained(
    "stabilityai/stable-diffusion-2-1",
    torch_dtype=torch.float16
).to("cuda")

prompt = "high-resolution microscopy image of concrete crack, scientific photo"
image = pipe(prompt, num_inference_steps=50, guidance_scale=7.5).images[0]
image.save("crack.png")
```

**합성 데이터 사용 시 주의.** 합성 데이터로 학습한 모델이 실제 데이터에서 어떻게 작동하는지 반드시 검증한다. 합성 데이터만으로 학습하면 종종 실제 데이터와 미묘한 차이로 성능이 떨어진다. 가능하면 실제 데이터와 합성 데이터를 함께 사용하고, 평가는 항상 실제 데이터로 한다.

## 고급 프롬프트 엔지니어링

ch26 본문에서 기본 프롬프트 엔지니어링(역할 지정, 맥락 제공, 단계별 지시, 출력 형식)을 다뤘다. 여기서는 더 정교한 기법들을 소개한다. 이 기법들은 본인이 LLM을 단순한 질의응답이 아니라 복잡한 추론 도구로 사용할 때 강력하다.

**Chain-of-Thought (CoT): 단계별 사고를 유도.** LLM에게 "정답을 즉시 말해 줘"라고 하면 종종 틀린다. 대신 "단계별로 생각해 봐"라고 하면 정확도가 크게 올라간다. 특히 수학, 논리, 다단계 추론에 효과적이다.

**기본 CoT 예시.**

```
Q: 한 콘크리트 시료의 압축 강도가 25 MPa이다. 이 강도가 ACI 318
기준의 보통 콘크리트 분류에 해당하는가? 단계별로 분석하라.

A: 단계별로 생각해 보겠다.
1단계: ACI 318의 보통 콘크리트 분류 기준 확인.
   - 보통 콘크리트: 17 MPa ≤ f'c < 35 MPa
2단계: 주어진 강도 25 MPa이 이 범위에 속하는지 확인.
   - 17 MPa ≤ 25 MPa < 35 MPa → 참
3단계: 결론.
   - 25 MPa는 ACI 318의 보통 콘크리트 분류에 해당한다.
```

**Few-shot Learning: 예시로 학습시키기.** LLM에게 1-5개의 예시를 보여주면, 그 패턴을 따라 새로운 입력에 응답한다. 학습 없이도 새 작업을 가르칠 수 있다.

```
다음 형식으로 데이터를 변환하라.

입력: "콘크리트 강도 25 MPa, 시료 28일"
출력: {"property": "compressive strength", "value": 25, "unit": "MPa", "age_days": 28}

입력: "강철 인장 강도 450 MPa, 표본 50개"
출력: {"property": "tensile strength", "value": 450, "unit": "MPa", "n_samples": 50}

입력: "알루미늄 항복점 270 MPa, 시험 5회"
출력:
```

LLM은 위 예시 두 개를 보고 세 번째 출력을 패턴에 맞춰 생성한다.

**Self-Consistency: 여러 답을 생성하고 다수결.** 같은 질문에 LLM을 여러 번 호출하여 여러 답을 얻고, 가장 자주 나오는 답을 선택. 단일 호출보다 정확도가 높지만 비용이 더 든다. 중요한 결정에 사용한다.

**ReAct: 추론과 행동의 결합.** LLM이 단순히 답하는 것이 아니라, 추론(Reason)과 행동(Action)을 번갈아 한다. 행동은 외부 도구 사용(웹 검색, 계산기, 데이터베이스 조회)이다. 본인이 LLM을 에이전트로 사용할 때 표준 패턴이다.

```
Question: 2024년 콘크리트 분야의 가장 인용된 논문은 무엇인가?

Thought: 2024년 정보가 필요하다. 내 지식은 오래되었을 수 있다. 검색이 필요하다.
Action: search_google_scholar("most cited concrete papers 2024")
Observation: [검색 결과...]
Thought: 결과를 확인했다. 가장 인용된 논문은 X et al. (2024)이다.
Answer: X et al. (2024) "Title..." which has Y citations as of...
```

이 패턴은 LangChain, LlamaIndex 같은 라이브러리에서 표준으로 구현되어 있다.

**Tree-of-Thoughts (ToT): 여러 추론 경로 탐색.** CoT가 단일 추론 체인이라면, ToT는 트리 구조로 여러 가능한 추론 경로를 탐색한다. 복잡한 문제(수학 증명, 플래닝)에 효과적. 하지만 계산 비용이 매우 크다.

**프롬프트 엔지니어링의 일반 원칙.** 첫째, **명시적이어라**. 모호한 요청은 모호한 답을 받는다. 둘째, **예시를 제공하라**. 한두 개의 예시가 백 마디 설명보다 효과적이다. 셋째, **출력 형식을 정의하라**. JSON, 리스트, 표 등의 형식을 명시. 넷째, **에러를 허용하라**. "확신이 없으면 '모름'이라고 답하라"고 명시하면 hallucination이 줄어든다. 다섯째, **반복하라**. 첫 응답이 만족스럽지 않으면 후속 질문으로 개선한다. 여섯째, **컨텍스트 윈도우를 관리하라**. 너무 긴 컨텍스트는 중간 정보가 잊힌다. 핵심 정보를 끝에 배치한다.

## RAG: 본인의 문서로 LLM 활용하기

LLM의 큰 한계는 **본인이 가진 정보를 모른다**는 것이다. 본인의 박사 논문, 연구실의 내부 문서, 최근 논문은 LLM의 학습 데이터에 없다. 본인의 자료로 질의응답을 하려면 **RAG(Retrieval-Augmented Generation)**가 필요하다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>RAG의 핵심 아이디어</strong></p>
    <p>본인의 문서를 작은 조각(chunk)으로 나누어 벡터 데이터베이스에 저장한다. 사용자가 질문하면, 그 질문과 가장 관련 있는 조각들을 검색한다. 검색된 조각들을 LLM에게 컨텍스트로 제공하고, 그것을 기반으로 답하게 한다. LLM의 일반 지식 + 본인의 특수 지식을 결합한다.</p>
  </div>
</div>

**RAG의 5단계 파이프라인.**

1단계: **문서 수집과 전처리**. 본인의 PDF 논문, Word 문서, 텍스트 노트, 웹 페이지를 수집. 텍스트로 변환.

2단계: **청킹(Chunking)**. 긴 문서를 작은 조각(보통 200-1000 토큰)으로 나눔. 청크 크기는 트레이드오프: 작으면 정확하지만 컨텍스트 부족, 크면 컨텍스트 풍부하지만 부정확.

3단계: **임베딩(Embedding)**. 각 청크를 벡터로 변환. OpenAI text-embedding-3, sentence-transformers, BGE-large 같은 임베딩 모델 사용.

4단계: **벡터 데이터베이스 저장**. 벡터 데이터베이스에 청크와 임베딩을 저장. ChromaDB, FAISS, Pinecone, Weaviate, Qdrant 등.

5단계: **질의 처리**. 사용자 질문을 임베딩으로 변환 → 가장 가까운 청크 K개 검색 → 청크들을 LLM 프롬프트에 컨텍스트로 추가 → LLM이 답 생성.

**기본 코드 예시 (LangChain).**

```python
from langchain.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.embeddings import HuggingFaceEmbeddings
from langchain.vectorstores import Chroma
from langchain.chains import RetrievalQA
from langchain.llms import OpenAI

# 1. 문서 로드
loader = PyPDFLoader("my_thesis.pdf")
docs = loader.load()

# 2. 청킹
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)

# 3-4. 임베딩과 벡터 DB
embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
vectordb = Chroma.from_documents(chunks, embeddings)

# 5. 질의 응답
qa = RetrievalQA.from_chain_type(llm=OpenAI(), retriever=vectordb.as_retriever())
answer = qa.run("내 논문의 핵심 기여는 무엇인가?")
print(answer)
```

**RAG의 장점.** 첫째, **본인의 데이터에 접근**. 본인의 박사 논문, 연구실 노트, 분야 핵심 논문 100편을 LLM에게 알게 할 수 있다. 둘째, **최신성**. 새 문서를 추가하면 즉시 반영된다. LLM 재학습이 필요 없다. 셋째, **출처 추적**. 답변이 어느 문서/페이지에서 나왔는지 인용 가능. 환각(hallucination) 가능성이 줄어든다. 넷째, **저비용**. LLM 파인튜닝보다 훨씬 싸다.

**연구자에게 RAG의 활용 예시.** 첫째, **본인의 박사 논문 챗봇**. 자기 논문의 모든 내용에 대해 질문 가능. 둘째, **분야 핵심 논문 검색 어시스턴트**. 100-1000편의 분야 논문을 RAG로 만들어 "X 방법을 사용한 논문을 찾아 줘" 같은 질의 가능. 셋째, **연구실 지식 베이스**. 연구실의 내부 문서, 매뉴얼, 과거 학생들의 노트를 RAG로 만들어 새 학생의 적응을 도움. 넷째, **저널 정책 검색**. 본인이 투고하려는 저널의 가이드라인 PDF를 RAG로 만들어 빠른 정책 확인.

**RAG의 한계와 함정.** 첫째, **검색 품질이 핵심**. 검색이 잘못되면 LLM이 잘못된 정보를 받아 잘못된 답을 한다. 임베딩 모델, 청크 크기, 검색 알고리즘을 조정해야 한다. 둘째, **컨텍스트 초과**. 검색된 청크가 LLM의 컨텍스트 윈도우를 초과하면 일부가 잘리거나 무시된다. 셋째, **추론의 한계**. RAG는 검색된 텍스트에 기반한 답만 한다. 검색된 정보를 종합하여 새로운 통찰을 만드는 능력은 제한적. 넷째, **수치/표 데이터**. 텍스트 임베딩은 수치 데이터나 표를 잘 처리하지 못한다. 표는 별도 처리가 필요.

---

## LLM 파인튜닝과 LoRA

RAG가 본인의 정보를 LLM에 제공하는 방법이라면, **파인튜닝**은 LLM 자체를 본인의 데이터에 맞게 수정하는 방법이다. 둘은 보완적이다. RAG는 빠르고 저비용이지만 깊은 도메인 적응이 어렵다. 파인튜닝은 비용이 들지만 특정 도메인에 깊이 적응한다.

**언제 파인튜닝이 필요한가.** 첫째, 본인의 도메인이 LLM의 일반 지식과 매우 다를 때(예: 특수 공학 분야의 전문 용어). 둘째, 본인이 원하는 출력 형식이 매우 일관되어야 할 때. 셋째, 본인의 도메인 데이터가 충분히 많을 때(최소 1000+ 예시). 넷째, RAG로는 충분한 성능을 얻지 못할 때.

**언제 RAG를 우선해야 하는가.** 첫째, 본인의 자료가 자주 업데이트될 때. 둘째, 출처 인용이 중요할 때. 셋째, 데이터가 적을 때(100-1000 예시). 넷째, 빠르게 시작하고 싶을 때.

**LoRA(Low-Rank Adaptation): 효율적인 파인튜닝.** 큰 LLM(7B-70B 파라미터)을 전체 파인튜닝하면 막대한 GPU와 메모리가 필요하다. LoRA는 모델의 가중치는 고정하고, 작은 추가 행렬(rank r의 저차원 행렬)만 학습한다. 학습 파라미터가 1-10%로 줄어 한 GPU(24GB)로 7B 모델을 파인튜닝할 수 있다. 추론 시 LoRA 가중치를 원본 모델과 결합하면 원본과 같은 속도로 작동한다.

**파인튜닝 도구와 라이브러리.**

| 도구 | 특징 |
|------|------|
| Hugging Face PEFT | LoRA, Prefix Tuning 등 표준 라이브러리 |
| Hugging Face TRL | RLHF, DPO 같은 정렬 학습 |
| Axolotl | 파인튜닝 설정 자동화 |
| Unsloth | 학습 속도와 메모리 효율 최적화 |
| LLaMA Factory | GUI 기반 파인튜닝 |

**기본 LoRA 파인튜닝 코드.**

```python
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b-hf")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b-hf")

lora_config = LoraConfig(
    r=8,  # rank
    lora_alpha=32,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.1,
    bias="none",
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# 출력 예: trainable params: 4,194,304 || all params: 6,738,415,616 || trainable: 0.062%
```

**파인튜닝의 단계별 진행.**

1단계: **데이터 준비**. 본인의 도메인 데이터를 LLM 학습 형식(JSONL)으로 변환. 보통 instruction-response 쌍 또는 chat format. 최소 500-1000개 예시.

2단계: **베이스 모델 선택**. 본인의 작업에 적합한 모델 선택. 한국어 작업이면 EXAONE, KoLLaMA 등 한국어 사전학습 모델. 영어면 LLaMA 3, Mistral, Qwen 등.

3단계: **LoRA 설정**. rank(보통 8-64), target modules, learning rate(보통 1e-4) 설정.

4단계: **학습**. 1-3 에포크면 충분. 너무 많이 학습하면 과적합되어 일반 능력이 저하된다.

5단계: **평가**. 본인의 테스트 세트에서 평가. 베이스 모델과 비교하여 개선이 있는지 확인.

6단계: **추론과 배포**. 학습된 LoRA 가중치를 베이스 모델과 결합하여 사용.

**파인튜닝 vs RAG의 결합.** 가장 강력한 시스템은 두 가지를 결합한다. LoRA로 도메인 언어와 출력 형식을 학습 → RAG로 최신 정보 제공. 본인의 박사 연구가 LLM 응용이라면 둘 다 시도해 볼 가치가 있다.

**현실적 조언.** 박사 과정 학생이 처음 LLM을 활용한다면, 먼저 RAG부터 시도한다. 빠르게 결과를 볼 수 있고 저비용. RAG로 충분하지 않으면 그때 파인튜닝을 검토. 파인튜닝은 GPU 자원, 데이터 준비, 디버깅에 상당한 시간이 든다. 본인의 박사 연구의 핵심이 LLM 자체가 아니라면, RAG로 충분한 경우가 많다.

## LLM 출력의 체계적 평가 — 평가 없는 사용은 신뢰할 수 없다

본인이 LLM을 연구에 활용하든, 연구 대상 자체로 삼든, 반드시 맞닥뜨리는 질문이 있다. "**이 LLM이 본인의 작업에서 얼마나 잘 작동하는가?**" 이 질문은 느낌이나 몇 개의 예시로 답할 수 없다. 체계적 평가가 필요하다. 놀랍게도 많은 박사생이 LLM을 쓰면서 이 단계를 건너뛴다. ChatGPT의 답변이 "그럴듯해 보이면" 그것으로 끝이다. 하지만 논문으로 발표하거나 의사결정에 사용하려면, 성능을 수치로 정량화해야 한다.

<div class="highlight-box highlight-warning">

**"그럴듯함"의 위험.** LLM은 자신 있게 틀린 답을 생성하는 능력이 뛰어나다. 본인이 한두 개의 예시를 보고 "잘 작동한다"고 판단하면, 100개 중 30개가 틀린 시스템을 배포할 수 있다. 숫자로 측정하지 않은 LLM 시스템은 본인의 논문에서도, 실제 응용에서도 신뢰할 수 없다.

</div>

**평가의 3가지 축.**

LLM 성능은 단일 지표로 측정할 수 없다. 최소한 다음 세 축을 함께 본다.

첫째, **정확성(Correctness)**: 답이 사실인가? 주어진 입력에 대해 정답이 있는 경우 직접 비교한다. 정답이 없는 경우(예: 요약, 창의적 작성)에는 참조 답변과의 유사도, 또는 인간 평가자의 판단을 사용한다.

둘째, **일관성(Consistency)**: 같은 질문을 여러 번 했을 때 같은 답이 나오는가? LLM은 본질적으로 확률적이므로 같은 프롬프트에도 다른 답을 내기도 한다. 특히 의사결정에 사용할 경우 일관성이 중요하다. 같은 질문을 10회 반복하여 답변의 분산을 측정한다.

셋째, **강건성(Robustness)**: 입력을 약간 바꿨을 때 답이 급변하지 않는가? "균열을 탐지해"와 "균열 검출해줘"는 의미적으로 같지만, LLM은 다른 답을 낼 수 있다. 본인의 프롬프트 변형(paraphrasing)에 대한 성능 변화를 측정한다.

**평가 데이터셋(Evaluation Set) 구성.**

체계적 평가의 첫 단계는 평가 데이터셋을 만드는 것이다. 이것은 ML 연구의 test set과 같은 역할이다. 다음 원칙을 따른다.

1. **최소 규모**: 30-50개의 평가 샘플이 최소. 100-500개가 이상적. 10개 미만은 통계적으로 의미가 없다.

2. **다양성**: 본인이 LLM에 기대하는 작업의 다양한 경우를 포함한다. 쉬운 경우, 어려운 경우, 경계 사례(edge cases), 함정이 있는 경우. 쉬운 것만 있으면 실제 성능을 과대평가한다.

3. **현실성**: 실제 사용할 때 올 수 있는 입력 분포를 반영한다. 학술적으로 깨끗한 문장만 있으면 현장에서 사용할 때 실패한다.

4. **정답 또는 기준**: 각 샘플에 대한 "정답" 또는 "좋은 답의 기준"을 미리 정의한다. 창의적 작업이라면 루브릭(rubric)을 정의한다 (예: 정확성 1-5, 명확성 1-5, 관련성 1-5).

5. **격리 유지**: 평가 데이터셋을 LLM의 학습 데이터에 포함시키지 않는다. ChatGPT 같은 상용 모델에서는 본인의 평가 데이터가 미래 모델의 학습에 사용될 수 있으므로, 민감한 평가는 로컬 모델에서 수행하거나 API의 "학습 사용 안 함" 옵션을 활성화한다.

**평가 방법의 4가지 계층.**

**1. 자동 정확 일치(Exact Match).** 정답이 명확한 경우 가장 단순. 분류, 추출, 숫자 답. 구현이 간단하고 노이즈가 없다. 단점은 "의미적으로 같지만 표현이 다른" 답을 틀렸다고 판정한다.

**2. 자동 유사도 측정(Automatic Metrics).** 생성된 텍스트와 참조 텍스트의 유사도를 측정한다. BLEU(기계번역), ROUGE(요약), BERTScore(의미 유사도). 빠르고 재현 가능하지만, 인간의 판단과 불일치할 때가 있다.

**3. LLM-as-Judge.** 다른(보통 더 강력한) LLM을 판정자로 사용한다. 예를 들어 GPT-4나 Claude에게 "다음 답변이 질문에 얼마나 잘 답하는가? 1-5점으로 평가하라"고 묻는다. 2023년 이후 표준이 되고 있는 방법. 장점은 자동화되고 비교적 정확하다. 단점은 판정자 LLM 자체의 편향이 있을 수 있다. 평가 편향을 줄이기 위해 2개 이상의 다른 LLM을 판정자로 사용하는 것도 방법이다.

**4. 인간 평가(Human Evaluation).** 궁극의 기준. 본인이 직접 또는 다른 사람(연구실 동료, 전문가)이 각 답변을 평가한다. 가장 신뢰할 수 있지만 느리고 비싸다. 보통 자동 평가로 초기 필터링 후, 상위/하위 샘플만 인간 평가로 확인하는 계층 구조를 사용한다. 인간 평가자 간의 일치도(inter-rater agreement)를 Cohen's kappa 또는 Krippendorff's alpha로 측정한다.

**LLM-as-Judge의 실전 패턴.**

```python
judge_prompt = """
다음은 공학 개념에 대한 질문과 AI가 생성한 답변이다.
답변을 다음 기준으로 평가하라.

1. 정확성 (Accuracy): 사실이 맞는가? (1-5)
2. 완전성 (Completeness): 질문의 모든 측면을 다루는가? (1-5)
3. 명확성 (Clarity): 이해하기 쉬운가? (1-5)

각 항목의 점수와 간단한 이유를 JSON으로 출력하라.

질문: {question}
답변: {answer}

출력 형식:
{{
  "accuracy": <점수>,
  "accuracy_reason": "<이유>",
  "completeness": <점수>,
  "completeness_reason": "<이유>",
  "clarity": <점수>,
  "clarity_reason": "<이유>"
}}
"""
```

판정자 LLM에게 구조화된 JSON 출력을 요구하면 후속 분석이 쉽다. 100개의 답변을 평가하는 데 몇 분이 걸리고, 결과를 표로 정리할 수 있다.

**평가 편향의 함정.**

LLM 평가에는 여러 편향이 숨어 있다. 이를 이해하지 못하면 잘못된 결론을 내린다.

- **Positional Bias**: 판정자 LLM은 두 답변을 비교할 때 먼저 제시된 답변을 선호하는 경향이 있다. 대응: A-B 순서와 B-A 순서로 두 번 평가하여 평균.

- **Verbosity Bias**: 판정자 LLM은 긴 답변을 더 높이 평가하는 경향이 있다. 대응: 길이를 점수에 반영하지 말라고 명시적으로 지시.

- **Self-Preference Bias**: GPT-4는 GPT-4 출력을, Claude는 Claude 출력을 선호한다. 대응: 여러 다른 판정자 LLM 사용.

- **Confirmation Bias**: 본인이 특정 답을 기대하면 평가 프롬프트를 무의식적으로 그쪽으로 유도한다. 대응: 블라인드 평가(누가 생성했는지 모르게).

**회귀 테스트(Regression Testing).**

본인이 프롬프트를 개선하거나 모델을 업데이트할 때, 새 버전이 기존에 잘 작동하던 입력에서 갑자기 실패하지 않는지 확인해야 한다. 이것을 위해 "회귀 테스트 셋"을 유지한다. 과거에 성공적으로 처리된 입력을 저장해 두고, 새 버전을 배포하기 전에 모두 통과하는지 확인한다.

소프트웨어 엔지니어링의 unit test와 같은 역할이다. LLM 시스템을 진지하게 운영하려면 필수다.

**평가 결과의 통계적 신뢰성.**

100개 샘플에서 측정한 "85% 정확도"는 어떤 신뢰 구간을 가지는가? 표본이 작으면 실제 성능의 범위가 꽤 넓다. 예를 들어 100개 중 85개 정답이면 95% 신뢰 구간은 대략 76%-91%다. 본인의 두 버전이 85%와 87%라고 해서 진짜로 87% 쪽이 더 나은지 확신할 수 없다. 통계적 유의성을 확인하려면 McNemar's test, paired t-test, bootstrap 같은 방법을 사용한다(ch35 참조).

**LLM 성능 추적 대시보드.**

본인의 LLM 시스템을 여러 번 개선할 때, 각 버전의 평가 결과를 쌓아 둔다. 엑셀이나 W&B(Weights & Biases)에 버전별 점수를 기록한다. 어느 변경이 성능을 올리고 어느 것이 내렸는지 추적할 수 있다. 이것이 있으면 "이번 주의 프롬프트 변경이 효과가 있었는가"를 객관적으로 판단할 수 있다.

**평가를 논문에 녹이는 법.**

본인이 LLM 기반 시스템을 논문으로 쓴다면, 평가 섹션은 다음 요소를 포함한다.

1. **평가 데이터셋 설명**: 얼마나 많은 샘플, 어떻게 수집했는지, 어떤 다양성을 포함하는지.
2. **평가 지표의 정의**: 정확성·일관성·강건성을 어떻게 측정했는지.
3. **평가 프로토콜**: 자동/LLM-as-Judge/인간 평가 중 무엇을 썼는지. 판정자 LLM이 있다면 어떤 모델인지.
4. **베이스라인 비교**: GPT-3.5 vs GPT-4, 또는 기존 방법 vs 제안 방법.
5. **통계적 유의성**: p-value, 신뢰 구간.
6. **실패 사례 분석**: 시스템이 어디서 실패하는지 (ch23 오류 분석 참조).
7. **평가의 한계**: 평가 방법의 편향이나 한계를 정직하게 인정.

**실전 팁: 평가를 일찍 시작하라.** 많은 학생이 "시스템이 완성되면 평가하겠다"고 미룬다. 이것은 실수다. 평가 프레임워크를 먼저 만들고, 각 개발 단계에서 평가한다. 이렇게 해야 본인이 옳은 방향으로 가고 있는지 매 단계에서 확인할 수 있다. "개선했다고 생각했는데 평가해 보니 더 나빠졌다"는 경험은 누구나 한다. 초기에 평가 인프라를 만드는 것이 장기적으로 시간을 절약한다.

> LLM을 연구에 사용하면서 평가를 하지 않는 것은 저울 없이 요리하는 것과 같다. "대충 맛있어 보인다"로는 재현할 수 없고, 개선할 수 없고, 출판할 수 없다. 30분의 평가 인프라 구축이 수개월의 헛된 최적화를 막는다. 본인의 LLM 연구에 가장 먼저 투자해야 할 것은 평가 시스템이다. 그다음에 프롬프트 엔지니어링, RAG, 파인튜닝이 온다.

## AI 에이전트 시스템과 도구 사용 — LLM을 실행자로 쓰기

2024년 이후 AI 활용의 중심이 "LLM에게 질문하기"에서 **"LLM에게 작업을 시키기"**로 이동했다. Claude Code, Cursor의 agent 모드, AutoGen, LangGraph, OpenAI Assistants API 등이 이 전환을 이끌었다. 이 시스템들에서 LLM은 단순히 답을 생성하는 것이 아니라, 외부 도구(파일 읽기, 코드 실행, 웹 검색, API 호출)를 스스로 사용하여 목표를 달성한다. 박사 학생이 이 패러다임을 이해하면 연구 생산성이 근본적으로 바뀐다. 반대로 모르면 2023년식 "채팅 기반 AI" 사용에 머무른다.

<div class="highlight-box highlight-important">

**"LLM에게 물어보기"의 한계.** ChatGPT에 "이 데이터를 분석해 줘"라고 물으면 LLM은 분석 **방법**을 설명한다. 실제 분석을 하지 않는다. 본인이 그 설명을 받아 코드를 쓰고, 실행하고, 결과를 다시 LLM에 붙여넣어 해석을 받는다. 이 왕복이 비효율적이다. AI 에이전트 시스템에서는 LLM이 직접 코드를 쓰고, 실행하고, 결과를 보고, 수정한다. 사람은 "목표"만 주면 된다. 이 차이가 생산성의 10배 차이를 만든다.

</div>

**AI 에이전트의 3가지 핵심 구성 요소.**

**1. LLM (두뇌).**
추론과 계획을 담당. GPT-4, Claude 3.5 Sonnet, Gemini 같은 강력한 모델이 필요. 약한 모델은 다단계 작업에서 실수가 누적된다.

**2. 도구(Tools).**
LLM이 호출할 수 있는 함수들. 각 도구는 이름, 설명, 입력 스키마, 실행 함수로 정의된다.

예시 도구들:
- `read_file(path)`: 파일 읽기
- `write_file(path, content)`: 파일 쓰기
- `execute_python(code)`: Python 코드 실행
- `search_web(query)`: 웹 검색
- `query_database(sql)`: 데이터베이스 쿼리

**3. 실행 루프(Agent Loop).**
"LLM 호출 → 도구 사용 결정 → 도구 실행 → 결과를 LLM에 반환 → 반복"의 사이클. 목표 달성까지 여러 번 반복.

```
사용자 목표: "이 CSV의 데이터를 분석하고 그래프를 그려줘"
  ↓
에이전트 사이클:
  1. LLM: "먼저 파일을 읽어야겠다" → read_file("data.csv")
  2. LLM: "데이터 구조를 봤다. Python으로 통계 분석하겠다"
     → execute_python("import pandas; df = pd.read_csv(...); df.describe()")
  3. LLM: "이상치가 있다. 시각화해서 확인하겠다"
     → execute_python("import matplotlib; ...")
  4. LLM: "분석 완료. 결과를 보고서로 정리한다"
     → write_file("report.md", "...")
```

사람은 한 문장만 주고, 에이전트가 나머지를 수행한다.

**주요 AI 에이전트 시스템들.**

**1. Claude Code (Anthropic, 2024).**
Claude Code는 Anthropic이 만든 터미널 기반 에이전트. 파일 시스템, 코드 실행, Git 등 개발 환경 전반에 접근할 수 있다.

**특징**:
- 터미널에서 직접 사용 (IDE 없이)
- 파일 생성/편집/삭제
- 코드 실행과 테스트
- Git 작업 (커밋, PR 등)
- 프로젝트 이해 (수백 개 파일에 걸친 맥락)

**공학 박사에게 가장 유용한 사용 예시**:
- 새 Python 프로젝트 설정 ("PyTorch 기반 이미지 분류 프로젝트를 만들어줘")
- 논문 재현 ("이 논문의 GitHub 코드를 내 데이터에 맞게 조정해줘")
- 리팩토링 ("이 노트북을 모듈화된 Python 패키지로 바꿔줘")
- 디버깅 ("이 에러의 원인을 찾고 고쳐줘")
- 문서 작성 ("이 코드베이스의 README를 작성해줘")

**2. Cursor Agent 모드 (Cursor, 2024).**
Cursor는 VS Code 기반의 AI IDE. Agent 모드에서는 여러 파일에 걸친 변경을 자동으로 수행.

**특징**:
- IDE 통합 (실시간 코드 편집)
- 멀티 파일 편집
- 코드베이스 검색
- Git diff 제안

**3. AutoGen (Microsoft).**
여러 AI 에이전트가 서로 대화하면서 문제를 해결하는 프레임워크. "Developer 에이전트"와 "Critic 에이전트"가 대화하면서 코드를 개선.

**4. LangGraph (LangChain).**
AI 에이전트의 실행 흐름을 그래프로 정의. 복잡한 다단계 워크플로에 적합.

**5. OpenAI Assistants API.**
OpenAI의 공식 에이전트 API. Code Interpreter, File Search, Function Calling 내장.

**6. Devin (Cognition Labs).**
"AI 소프트웨어 엔지니어"로 불리는 완전 자동화 에이전트. 복잡한 코딩 작업을 장시간 자동 수행.

**공학 연구에서 AI 에이전트의 5가지 활용 방안.**

**활용 1: 데이터 전처리 자동화.**
"이 원시 CSV에서 결측치 처리하고, 이상치 제거하고, 정규화해서 저장해줘."

에이전트가 수행하는 작업:
1. 파일 읽기
2. 데이터 탐색 (describe, head)
3. 결측치 패턴 파악
4. 적절한 전처리 파이프라인 작성
5. 실행과 결과 저장
6. 처리 로그 작성

사람은 검토만 한다.

**활용 2: 논문 재현.**
"이 GitHub 저장소의 코드를 우리 환경에서 실행할 수 있게 만들어줘."

에이전트가 수행:
1. 저장소 clone
2. 요구사항 분석 (requirements.txt, README)
3. 의존성 충돌 감지와 해결
4. 경로, 데이터 형식 조정
5. 실행 후 결과 확인

전통적으로 며칠 걸리던 "다른 사람 코드 돌리기"가 몇 시간으로 단축.

**활용 3: 실험 스크립트 생성.**
"이 모델로 10가지 하이퍼파라미터 조합 실험하는 스크립트를 만들어줘. 결과를 W&B에 기록하게."

에이전트가 수행:
1. 모델 코드 이해
2. 실험 스크립트 작성
3. W&B 통합
4. 배치 실행 스크립트 작성
5. 테스트 실행으로 검증

**활용 4: 문헌 조사 자동화.**
"이 주제에 대한 최근 5년 논문을 검색하고, 주요 방법들을 비교하는 표를 만들어줘."

에이전트가 수행:
1. 웹 검색 (arXiv, Google Scholar)
2. 논문 PDF 다운로드
3. 각 논문 요약
4. 방법론 추출과 비교
5. Markdown 표 작성

다만 "제대로 된 문헌 검토"는 여전히 사람의 판단이 필요하다. 에이전트는 **초안**을 빠르게 만드는 데 유용.

**활용 5: 논문 그림 생성.**
"이 데이터에서 그림 5와 비슷한 그래프를 만들어줘. 다만 학술 논문 스타일로."

에이전트가 수행:
1. 데이터 로드와 탐색
2. 기존 그림 스타일 참조
3. matplotlib/seaborn 코드 작성
4. 학술 스타일 적용 (폰트, 색상, 축 라벨)
5. 저장과 미리보기

**에이전트 사용의 실전 팁.**

**팁 1: 명확한 목표 설정.**
"~을 개선해줘"보다 "~을 X 방식으로 개선하여 Y 결과를 얻어줘"가 더 좋다. 에이전트는 구체적 지시에 더 잘 반응한다.

**팁 2: 작은 단위로 나누기.**
큰 목표("논문 전체를 써줘")는 실패할 가능성이 높다. "먼저 Methods 섹션의 한 단락을 써줘"처럼 작게 나눈다.

**팁 3: 중간 확인.**
에이전트가 여러 단계를 수행한 후, 결과를 중간 확인한다. "지금까지 한 것을 보여줘"라고 물어본다. 잘못된 방향으로 가고 있으면 수정.

**팁 4: 검증 요구.**
에이전트에게 결과를 검증하도록 요구한다. "코드를 실행해서 결과가 맞는지 확인해줘." 에이전트가 자체 테스트를 하면 오류가 줄어든다.

**팁 5: 버전 관리와 백업.**
에이전트가 파일을 수정하기 전에 백업 또는 Git commit. "실수로 중요한 파일을 덮어썼다"를 방지.

**팁 6: 민감 정보 보호.**
에이전트에게 API 키, 비밀번호, 개인정보가 노출되지 않도록 한다. `.env` 파일 사용(ch16 참조).

**에이전트의 한계와 주의사항.**

**한계 1: 오류의 누적.**
다단계 작업에서 초기 오류가 뒤로 전파된다. 에이전트가 "잘못된 가정"으로 시작하면 그 위에 더 많은 잘못된 작업이 쌓인다. 주기적 검증이 필수.

**한계 2: 맥락 창(Context Window)의 한계.**
에이전트는 대화 이력과 파일 내용을 "context window"에 저장한다. 이 창이 가득 차면 오래된 정보를 잊는다. 긴 작업에서는 문제가 될 수 있다.

**한계 3: 비용.**
GPT-4 수준의 모델을 에이전트로 사용하면 API 비용이 빠르게 쌓인다. 한 번의 복잡한 작업에 $5-$50이 나올 수 있다. 박사 학생에게는 부담.

**한계 4: 재현성.**
에이전트의 출력은 비결정적이다. 같은 작업을 두 번 실행하면 약간 다른 결과가 나올 수 있다. 과학적 재현성에 문제가 될 수 있다.

**한계 5: "블랙박스" 실행.**
에이전트가 많은 작업을 자동 수행하면 본인이 "무엇을 했는지" 완전히 이해하지 못할 수 있다. 논문에 "에이전트가 이렇게 했다"고 쓸 수는 없다. 에이전트의 작업을 본인이 이해하고 검증해야 한다.

**한계 6: 책임 소재.**
에이전트가 실수로 중요한 파일을 삭제하거나 실험을 망치면 누구 책임인가? 본인이다. 에이전트를 사용해도 결과에 대한 책임은 본인에게 있다.

**학생이 에이전트를 "잘못" 사용하는 패턴.**

**패턴 1: 맹신.**
에이전트가 하는 모든 것을 맞다고 가정. 결과적으로 에이전트의 실수가 본인의 논문에 들어간다. **해결**: 에이전트의 출력을 항상 검증.

**패턴 2: 본인 능력의 대체로 오해.**
"에이전트가 다 해주니까 나는 배울 필요 없다." 위험한 태도. 본인이 기초를 이해하지 못하면 에이전트의 실수를 감지할 수 없다. **해결**: 에이전트는 생산성 도구이지 학습 도구가 아니다. 기초 학습은 별도로 한다.

**패턴 3: 학술 논문에 에이전트의 글을 그대로 사용.**
에이전트가 쓴 introduction을 그대로 논문에 붙여넣기. 이것은 여러 문제(스타일, 정확성, 윤리)를 만든다. **해결**: 에이전트의 출력은 항상 본인이 다시 쓴다.

**패턴 4: 복잡한 목표를 그대로 던지기.**
"내 박사 논문 실험을 모두 해줘." 에이전트가 성공할 리 없다. **해결**: 작은 단계로 나누고, 각 단계를 검증한다.

**패턴 5: 비용 무시.**
월 $500의 API 비용을 모른 채 사용. 학생 예산으로 감당 불가. **해결**: 사용량 모니터링 설정. 월 한도 설정. 저렴한 모델(Haiku, GPT-4o mini)부터 시도.

**에이전트 사용의 논문 작성 윤리.**

본인이 에이전트를 사용해서 연구를 수행했을 때, 논문에 어떻게 보고할 것인가?

**1. 사용한 도구 공개.**
"We used Claude Code (Anthropic) to assist with code development and data preprocessing." 이런 공개가 표준이 되어가고 있다.

**2. 검증의 명시.**
"All AI-generated code was reviewed and tested by the authors before use." 본인이 검증했음을 명시.

**3. 창의적 기여는 본인의 것.**
에이전트는 실행 도구일 뿐, 연구의 핵심 아이디어와 결정은 본인의 것이어야 한다. 저자는 사람이다.

**4. 저널 정책 확인.**
각 저널의 AI 사용 정책이 다르다. Nature, Science, IEEE 등은 각자의 가이드라인을 가지고 있다. 투고 전 확인.

ch29 (연구 윤리)와 밀접하게 연결된다.

**박사 학생을 위한 에이전트 학습 로드맵.**

**1주차**: Claude Code 설치, 간단한 작업 시도 ("내 Python 스크립트 디버깅").
**2주차**: 다단계 작업 시도 ("데이터 전처리 파이프라인 만들기").
**3주차**: 프로젝트 수준 작업 ("기존 논문 재현").
**4주차**: 본인의 박사 연구에 통합. 어떤 반복 작업을 에이전트에게 맡길 수 있는지 식별.
**5주차 이후**: 정기적 사용, 워크플로에 자연스럽게 통합.

이 한 달의 투자가 본인의 박사 과정의 남은 시간을 크게 단축한다.

**에이전트 평가의 척도.**

본인이 에이전트의 효과를 어떻게 측정할 것인가?

- **시간 절약**: 같은 작업을 에이전트와 수동으로 비교. 에이전트가 1/5 시간이면 크게 유용.
- **정확도**: 에이전트의 결과가 올바른가? 검증 필요.
- **학습 기회**: 에이전트가 본인의 학습을 방해하는가 촉진하는가?
- **비용 대비 효과**: API 비용 vs 시간 가치

본인의 사용 패턴이 생산적인지 정기적으로 돌아본다.

**AI 에이전트의 미래와 준비.**

2024-2026년은 AI 에이전트의 폭발적 성장기다. 2027-2030년에는 현재보다 훨씬 강력한 에이전트가 등장할 것이다. 현재 에이전트의 한계(비용, 재현성, 긴 작업)는 빠르게 개선되고 있다.

박사 학생으로서 본인은 이 변화에 어떻게 대비해야 하는가?

1. **현재 에이전트를 지금 사용한다**. 완벽하지 않지만 학습 곡선을 빠르게 오른다.
2. **에이전트에 의존하되 기초를 놓지 않는다**. 에이전트 없이도 할 수 있어야 한다.
3. **"에이전트가 할 수 없는 것"을 배운다**. 창의적 연구, 문제 정의, 학술적 판단. 이것이 본인의 차별화.
4. **에이전트의 발전을 따라간다**. 새 도구가 매달 나온다. 주기적 탐색.

> AI 에이전트는 2024-2025년의 가장 큰 도구 혁명이다. 단순한 채팅 AI에서 실행하는 AI로의 전환은 박사 연구의 생산성에 근본적 변화를 가져온다. 본인이 이것을 거부하면 2023년식 연구에 갇힌다. 반면 맹목적으로 의존하면 본인의 학술적 성장이 멈춘다. 도구로 활용하되, 주인은 본인이어야 한다. 이 균형을 잡는 능력이 AI 시대 박사생의 필수 역량이다.

## 코드 생성 AI — Cursor, Copilot, Claude Code의 박사 연구 활용법

ChatGPT에게 코드를 복사-붙여넣기 하는 시대는 지났다. 2024-2026년의 실전은 IDE에 내장된 코드 생성 AI다. GitHub Copilot, Cursor, Claude Code, Windsurf, Aider 등은 박사 연구의 코딩 시간을 2-5배 단축시킨다. 그러나 모든 도구가 같지 않고, 모든 사용법이 올바른 것도 아니다.

**다섯 가지 활용 패턴.**

첫째, **자동완성 모드**다. 함수 이름이나 주석을 쓰면 AI가 나머지를 제안한다. 반복적인 전처리 코드나 플롯 코드에 효과적이다. 생각의 흐름을 끊지 않는다.

둘째, **주석 기반 생성**이다. `# Compute RMS of signal with windowing`처럼 구체적인 주석을 쓰고 함수 본문을 생성시킨다. 연구 코드의 의도가 주석에 남는 부작용이 좋다.

셋째, **대화형 편집**이다. 선택한 코드에 대해 "이 루프를 벡터화해줘", "에러 처리를 추가해줘"라고 지시한다. 리팩토링에 가장 많이 쓰는 패턴이다.

넷째, **여러 파일 수정**이다. Cursor의 Composer나 Aider는 "데이터 로더를 HDF5에서 Zarr로 바꾸고 관련된 모든 파일을 수정해줘"같은 지시를 처리한다. 수작업이면 한나절 걸릴 변경이 수분으로 줄어든다.

다섯째, **에이전트 모드**다. Claude Code처럼 "이 버그를 찾아 고쳐줘"라고 지시하면 AI가 파일을 탐색하고 원인을 찾고 수정한다. 박사의 시간을 가장 크게 아껴주지만, 검증 책임은 온전히 본인에게 있다.

**효과적으로 쓰기 위한 다섯 원칙.**

1. **생성된 코드를 반드시 이해하라.** "동작하지만 설명할 수 없는 코드"는 논문 심사에서 재앙이다. AI 코드의 모든 줄을 읽고 왜 그런지 말할 수 있어야 한다.
2. **검증을 건너뛰지 마라.** 단위 테스트, 경계값 확인, 물리적 타당성 점검은 AI 사용 여부와 무관하게 필수다. AI는 자신감 있게 틀린 답을 준다.
3. **Git 커밋을 세밀하게 하라.** AI 변경 전후를 나눠 커밋하면 문제 발생 시 되돌리기 쉽다. ch16에서 다룬 원칙이 여기서 더 중요해진다.
4. **기본기를 놓지 마라.** AI 없이도 핵심 알고리즘을 구현할 수 있어야 한다. 자동차가 있어도 걷는 법은 잊지 말아야 한다.
5. **학위논문에 공개하라.** "이 코드는 AI 어시스턴트의 도움을 받아 작성되었다"를 명시하는 것이 점점 표준이 되고 있다. ch29의 윤리 원칙과 일관된 태도다.

**다섯 가지 함정.**

첫째, **환각 API다**. AI는 존재하지 않는 함수나 매개변수를 그럴듯하게 제시한다. 특히 최신이나 마이너 라이브러리에서. 공식 문서로 확인하라.

둘째, **구식 API다**. 학습 데이터가 1-2년 전인 AI는 deprecated 함수를 제안한다. PyTorch, TensorFlow, scikit-learn의 메이저 변경에 취약하다.

셋째, **과도한 복잡성이다**. AI는 필요 이상으로 추상화한 코드를 만드는 경향이 있다. 연구 코드는 단순함이 덕목이다. 복잡해 보이면 단순화를 지시하라.

넷째, **라이선스 문제다**. AI가 학습 데이터의 GPL 코드를 거의 그대로 생성할 수 있다. 상업화 가능성이 있는 연구에서는 주의해야 한다. ch40의 IP 관점에서도 중요하다.

다섯째, **의존성이다**. AI 없이 코딩을 못 하게 되는 것이 가장 큰 위험이다. 의존성은 본인의 성장을 멈추게 한다. 주 1-2회는 AI 없이 코딩하는 시간을 가져라.

## 로컬 LLM — 프라이버시와 재현성을 위한 자체 호스팅

2023년까지 LLM을 쓰려면 OpenAI나 Anthropic의 API를 호출해야 했다. 2025년 현재, 랩톱에서 Llama, Qwen, Mistral을 돌리는 것이 일상적이다. 박사 연구의 특정 시나리오에서는 로컬 실행이 단순한 선호가 아니라 필수 선택이다.

**다섯 가지 이유.**

첫째, **프라이버시**다. 민감한 데이터를 클라우드로 보내지 않는다. 의료 데이터, 기업 기밀, 미발표 연구는 IRB 승인과 NDA 때문에 클라우드 전송이 금지될 수 있다. ch29와 ch38의 윤리적 요구사항과 직결된다.

둘째, **재현성**이다. OpenAI는 모델을 업데이트하고 폐기한다. 본인이 2024년에 GPT-4로 얻은 결과를 2027년에 재현할 수 있을까? 로컬 모델은 정확한 가중치를 보관할 수 있어 재현 가능하다.

셋째, **비용**이다. 대규모 실험에서 API 비용은 박사 예산을 초과할 수 있다. 하드웨어 한 번 투자로 무제한 호출이 가능해진다. 연구 주제에 따라 1-2년에 회수된다.

넷째, **인터넷 독립성**이다. 출장 중 비행기, 보안 연구실, 현장 데이터 수집 환경 등 오프라인 상황에서도 AI를 쓸 수 있다.

다섯째, **커스터마이징**이다. 로컬 모델은 자유롭게 fine-tune 할 수 있다. 분야별 특화 모델을 만들 수 있다. API는 이런 자유를 허용하지 않는다.

**다섯 도구.**

**하드웨어 레벨.**

첫째, **노트북 레벨**이다. M1-M3 맥북(16GB 이상) 또는 Windows 노트북(RTX 4060 이상)에서 7B 양자화 모델이 돌아간다. 초안 작성, 번역, 요약 등에 충분하다.

둘째, **게이밍 PC 레벨**이다. RTX 4090(24GB) 기준 13B-34B 모델이 가능하다. 본격적인 연구 보조로 쓸 수 있다. 기존 게이밍 PC를 연구 장비로 활용하는 학생이 많다.

셋째, **워크스테이션 레벨**이다. RTX 6000 Ada 또는 A6000(48GB) 수준이면 70B 모델까지 가능. 연구실 예산으로 한 대 확보하면 팀 전체가 쓸 수 있다.

넷째, **서버 레벨**이다. H100 또는 다중 GPU 구성. 405B급 거대 모델 또는 fine-tuning에 필요. 대학 공용 GPU 클러스터나 교수 과제비가 필요하다.

**박사 연구의 다섯 활용 사례.**

1. **대용량 문헌 요약.** 500편의 논문을 로컬 Qwen이나 Llama로 요약. API로는 비용이 부담되는 규모가 하드웨어가 있으면 공짜다.
2. **민감 데이터 분석.** 환자 데이터, 기업 내부 문서, 보안 자료에 LLM을 적용. 클라우드는 금지되지만 로컬은 허용되는 경우가 많다.
3. **재현 가능한 파이프라인.** 특정 모델 버전을 가중치로 고정해 5년 후에도 동일한 결과를 낸다. 학위논문 재현성 보장.
4. **도메인 특화 fine-tuning.** 연구 분야의 논문 수천 편으로 모델을 미세조정해 전문가급 보조 도구로 만든다. ch22의 LoRA와 직접 연결된다.
5. **24시간 실험.** API 한도 걱정 없이 밤새 수천 건의 추론을 돌린다. 대규모 평가 실험에 필수.

**주의할 점 셋.**

첫째, 로컬 모델은 여전히 GPT-4나 Claude Opus보다 약하다. 추론력, 정확도, 안전성 모두에서. 어려운 작업은 클라우드가 여전히 낫다.

둘째, 초기 설정 시간이 든다. CUDA 드라이버, 의존성, 양자화 선택 등 학습 곡선이 있다. 시간을 아끼려는 목적이라면 처음에는 손해다.

셋째, 전기료와 발열을 무시할 수 없다. 24시간 추론 서버를 집에서 돌리면 여름에 에어컨 비용이 뛴다. 연구실 환경이 더 적합하다.

> 코드 생성 AI와 로컬 LLM은 2024-2026년 박사 연구의 두 실전 축이다. 전자는 코딩 생산성을, 후자는 프라이버시와 재현성을 담당한다. 이 장의 앞부분에서 다룬 프롬프트 엔지니어링, RAG, LoRA, AI 에이전트와 함께 쓰면 본인만의 AI 연구 인프라가 완성된다. 완벽한 도구는 없다. 본인의 연구 맥락에 맞는 조합을 찾는 것이 박사의 몫이다.

## LLM으로 체계적 문헌 검토 자동화 — 박사의 새 도구

박사 과정의 가장 시간이 드는 작업 중 하나가 **체계적 문헌 검토(systematic literature review)**다. 수백 편의 논문을 읽고, 핵심 정보를 추출하고, 트렌드를 파악하고, 표로 정리한다. 이것이 박사 1-2년차의 배경 연구, 리뷰 논문 작성, 박사 논문의 이론적 배경 챕터 등에 필요하다. 전통적으로 이것은 수백 시간의 노동이다. 그러나 2024-2026년의 LLM은 이 작업을 보조할 수 있다. 올바르게 쓰면 본인이 혼자 하는 것보다 5-10배 빠르고, 어떤 경우는 더 철저한 검토가 가능하다. 그러나 잘못 쓰면 존재하지 않는 논문이나 왜곡된 요약으로 가득 찬 가짜 검토가 된다. 이 섹션은 LLM을 문헌 검토에 체계적으로 활용하는 실전 방법이다.

<div class="highlight-box highlight-important">

**"LLM은 본인의 읽기를 대체하지 못한다, 확장한다".** LLM에게 "이 분야의 문헌을 검토해줘"라고 하면 말이 안 되는 요약이 나온다. 그러나 "이 100개 논문의 초록에서 특정 정보를 추출해줘"라고 하면 놀랍게 정확하다. 차이는 본인이 LLM에게 "구체적이고 검증 가능한 작업"을 주는 것이다. LLM을 문헌 검토의 주인공이 아닌 도구로, 본인의 판단을 대체하지 않는 보조로 쓴다면 박사 과정의 새 강력한 동맹이 된다.

</div>

**LLM 활용의 4단계 문헌 검토 파이프라인.**

체계적 문헌 검토를 LLM 지원으로 하는 파이프라인.

**단계 1: 초기 검색과 수집.**
키워드 기반으로 Google Scholar, Scopus, Web of Science에서 후보 논문 수집. 이 단계는 전통적 도구 사용.

**LLM의 역할**: 검색어 제안. "나는 X 분야를 검토하고 싶다. 어떤 키워드 조합이 좋을까?" LLM이 본인이 생각하지 못한 관련 용어나 동의어를 제안.

**예시 프롬프트**:
```
I'm doing a systematic literature review on physics-informed 
neural networks (PINNs) for structural health monitoring. 
Suggest 10-15 search term combinations that would capture 
relevant papers from 2019-2026, including synonyms and 
related terms from adjacent fields.
```

**단계 2: 제목/초록 기반 선별 (Title/Abstract Screening).**
수집한 수백 편에서 실제로 관련 있는 논문을 고른다. 이것이 가장 시간이 많이 드는 단계.

**LLM의 역할**: 각 논문의 초록을 읽고 본인의 검토 기준에 맞는지 판단. 포함/제외 추천.

**프롬프트 예시**:
```
Below is the abstract of a paper. My review focuses on 
physics-informed neural networks specifically for structural 
health monitoring (e.g., damage detection, crack identification 
in civil structures). 

Please classify this paper into one of three categories:
1. HIGHLY RELEVANT: Directly addresses PINN for SHM
2. SOMEWHAT RELEVANT: PINN in other domains or SHM without PINN
3. NOT RELEVANT: Neither PINN nor SHM

Also provide a one-sentence justification.

Abstract: [붙여넣기]
```

**효율화**: 이것을 자동화 스크립트로 만들어 수백 개의 초록을 연속으로 처리.

**주의**: LLM의 분류를 맹신하지 말자. 결과의 일부(예: 10%)를 본인이 직접 확인해 정확도 측정. 80% 이상이면 쓸 만함.

**단계 3: 전문(Full-text) 기반 정보 추출.**
선별된 논문(보통 30-100편)의 전문을 읽고 구체적 정보 추출.

**추출할 정보 예시** (PINN for SHM):
- 사용한 PINN 아키텍처
- 대상 구조 유형
- 훈련 데이터의 종류와 양
- 달성한 정확도
- 주요 한계
- 기여의 독창성

**LLM의 역할**: 논문의 전문(또는 주요 섹션)을 받아서 구조화된 표로 정보 추출.

**프롬프트 예시**:
```
Below is a research paper on PINN for structural health 
monitoring. Extract the following information in JSON format:

{
  "title": "",
  "year": "",
  "pinn_architecture": "",
  "structure_type": "",
  "training_data_type": "",
  "training_data_size": "",
  "main_contribution": "",
  "accuracy_metric": "",
  "accuracy_value": "",
  "key_limitation": ""
}

If a field is not mentioned, use "NOT_REPORTED". Base your 
extraction strictly on what is written in the paper; do not 
infer or guess.

Paper text: [붙여넣기]
```

**결과**: 수십 편의 논문이 구조화된 JSON 데이터로 변환. 이것을 Excel이나 Pandas로 분석 가능.

**단계 4: 종합 분석과 검증.**
추출된 정보로 트렌드, 패턴, 공백을 분석.

**LLM의 역할**: 추출된 구조화 데이터로부터 고수준 통찰 도출.

**예시 분석**:
- "어떤 구조 유형에 가장 많은 연구가 있는가?"
- "시간에 따른 정확도의 변화는?"
- "자주 언급되는 한계는?"
- "연구의 공백 영역은?"

**본인의 역할**: LLM의 통찰을 검증. 실제 논문으로 돌아가서 본인이 직접 확인. LLM이 놓친 관점이나 잘못 해석한 부분을 수정.

**프롬프트 설계의 원칙 — 문헌 검토 특화.**

일반 프롬프트와 달리 문헌 검토에는 특수한 원칙이 있다.

**원칙 1: "Don't infer" 명시**.
LLM이 논문에 없는 내용을 추측하면 안 된다. 프롬프트에 명시: "Base your answer strictly on the text. Do not infer or add information."

**원칙 2: 구조화된 출력**.
JSON 또는 표 형식으로 요청. 자유 형식 텍스트는 분석이 어렵다.

**원칙 3: 예시 제공 (Few-shot)**.
완벽한 예시를 1-2개 주면 정확도가 크게 올라간다. "Here's an example of good extraction: ..."

**원칙 4: 불확실성 표현**.
모르는 것에 대해 "NOT_REPORTED", "UNCLEAR", "N/A" 같은 명시적 표현 사용. LLM이 억지로 답을 만들지 않게.

**원칙 5: 검증 가능성**.
LLM의 답변마다 "어디에 나와 있는가"의 참조 요청. "Quote the exact text that supports this claim."

**환각(Hallucination) 방지 전략.**

문헌 검토에서 LLM의 가장 큰 위험은 환각이다. 존재하지 않는 논문, 잘못된 인용, 왜곡된 요약.

**방지 전략 1: 실제 텍스트만 제공**.
LLM에게 논문 제목만 주면 안 된다. 반드시 실제 초록이나 전문을 함께 제공. 제목만으로는 LLM이 "추측"할 수밖에 없다.

**방지 전략 2: "없는 것은 없다고 말하라"**.
프롬프트에 "If the paper does not mention X, say 'NOT MENTIONED'" 포함. 억지로 답을 만들지 않게.

**방지 전략 3: 인용 검증**.
LLM이 논문을 인용하면 반드시 본인이 DOI나 Google Scholar로 실제 논문 존재 확인. 특히 저자 이름과 연도.

**방지 전략 4: 랜덤 샘플 검증**.
LLM의 추출 결과 10-20%를 본인이 직접 원본과 비교. 오류율 측정. 5% 이상이면 프롬프트 개선 또는 다른 모델 시도.

**방지 전략 5: 여러 모델 비교**.
중요한 정보는 GPT-4, Claude, Gemini 여러 모델에게 물어서 일치하는지 확인. 불일치하면 본인이 원본 확인.

**어떤 LLM을 쓸 것인가 — 문헌 검토 용도**.

2026년 기준 주요 LLM의 문헌 검토 적합성.

**GPT-4 / GPT-4 Turbo**:
- 장점: 긴 맥락 처리, 정확한 추출
- 단점: 비용, 일부 도메인 지식 부족
- 권장: 일반 문헌 검토

**Claude Opus / Sonnet**:
- 장점: 정직성, 긴 문서 처리, 한국어 강점
- 단점: 비용
- 권장: 정밀한 정보 추출

**Gemini Pro**:
- 장점: Google Scholar 연동 가능, 긴 맥락
- 단점: 환각 경향
- 권장: 탐색적 검색

**Perplexity AI**:
- 장점: 인용 표시, 실시간 검색
- 단점: 학술 논문보다 일반 웹 중심
- 권장: 초기 탐색

**전용 도구 — Elicit, ResearchRabbit, Connected Papers**:
- 장점: 학술 논문 특화
- 단점: 유료, 제한된 데이터베이스
- 권장: 학술 전용 워크플로

**권장**: GPT-4 또는 Claude를 주 도구로, Perplexity를 초기 탐색, Elicit을 학술 데이터베이스 검색 보조로.

**구체적 도구 — Elicit.**

Elicit는 학술 논문에 특화된 LLM 도구다. 2023년 이후 많은 박사생이 사용.

**주요 기능**:
1. **자연어 질문으로 논문 검색**: "What are effective methods for detecting cracks in concrete?"
2. **자동 요약**: 각 논문의 핵심을 한두 문장으로
3. **비교 표 생성**: 여러 논문의 특정 측면을 표로 비교
4. **관련 논문 추천**: 유사한 논문 자동 발견

**장점**:
- 학술 데이터베이스 (Semantic Scholar) 연결
- 인용 검증 내장
- 연구자 친화적 UI

**한계**:
- 유료 (무료 제한 있음)
- 일부 분야 데이터 부족
- 심층 분석은 여전히 본인이 해야

**무료 대안**: Connected Papers (그래프 기반), ResearchRabbit (추천 기반).

**박사 과정 단계별 활용.**

박사 과정의 어느 시기에 어떻게 쓸 것인가.

**박사 1년차 — 분야 파악**:
- "이 분야의 주요 연구 그룹은?"
- "최근 5년의 주요 발전은?"
- "기초 개념은 어떻게 시작되었나?"

LLM으로 빠르게 전체 지도를 얻는다. 그러나 핵심 논문은 본인이 직접 읽는다.

**박사 2-3년차 — 심층 배경 연구**:
- 본인의 박사 논문의 이론적 배경 장 준비
- 100-300편의 체계적 검토
- 정보 추출과 표 정리

LLM의 구조화된 추출을 적극 활용. 표로 만들어 본인의 논문 Related Work 섹션의 기초.

**박사 3-4년차 — 리뷰 논문 작성**:
- 본인의 분야의 리뷰 논문 작성 (ch07과 연결)
- 수백 편의 체계적 분석
- 트렌드와 공백 파악

LLM이 대량 처리를 가능하게 해서 리뷰 논문이 현실적 목표가 된다.

**박사 4-5년차 — 경쟁 논문 모니터링**:
- 리비전 중 외부 변화 추적 (ch12와 연결)
- 새 논문의 빠른 평가
- 본인의 연구와의 관계 판단

**박사 5-6년차 — 학위논문 업데이트**:
- 학위논문의 배경 챕터 갱신
- 최신 문헌 포함
- 본인의 기여를 맥락에 정확히 위치

**체계적 검토 vs 빠른 검토**.

LLM은 두 종류의 검토 모두에 도움이 된다. 차이를 이해하자.

**체계적 검토 (Systematic Review)**:
- 엄격한 프로토콜 (PRISMA)
- 재현 가능한 검색 전략
- 모든 관련 논문 포함
- 수개월의 작업

LLM의 역할: 초록 선별, 정보 추출, 속도 향상. 본인의 판단은 핵심 결정에만.

**빠른 검토 (Rapid Review)**:
- 시간 제약
- 대표 논문만
- 트렌드 파악 목적
- 몇 주의 작업

LLM의 역할: 더 큰 비중. 본인이 검증만 해도 진행 가능.

박사 과정에서는 둘 다 필요. 리뷰 논문이나 학위논문은 체계적, 일상적 배경 연구는 빠른.

**문헌 검토의 전통 vs LLM 증강 — 비교**.

같은 작업을 두 방식으로 비교하면 이점이 명확하다.

**100편의 논문을 검토한다고 가정**:

**전통 방식**:
- 초록 읽기: 100편 × 10분 = 1000분 (약 17시간)
- 선별: 50편 선정
- 전문 읽기: 50편 × 1시간 = 50시간
- 정보 추출 (수기): 50편 × 30분 = 25시간
- 종합 분석: 20시간
- 총: 약 110시간 (3주 전일)

**LLM 증강 방식**:
- LLM으로 초록 선별: 2시간 (자동 + 검증 10%)
- LLM으로 정보 추출: 5시간 (자동 + 검증)
- 본인이 중요한 20편 직접 읽기: 20시간
- LLM과 본인의 종합 분석: 10시간
- 총: 약 40시간 (1주)

**비율**: 2.5배 빠르다. 단, 본인의 읽기는 여전히 필요 (20편).

**품질**: 제대로 한다면 동등하거나 더 좋을 수 있다. LLM은 본인이 지친 후반에도 일관성 유지.

**문헌 검토의 윤리 — LLM 사용의 투명성**.

박사 논문이나 학술지의 리뷰 논문에 LLM 사용을 공개할 의무가 점점 커진다.

**공개할 사항**:
1. 어떤 LLM을 사용했는가 (모델과 버전)
2. 어떤 단계에서 사용했는가 (검색/선별/추출/분석)
3. 본인이 검증한 정도
4. 어떤 논문을 본인이 직접 읽었는가

**예시 문구**:
```
Methodology: Literature search was conducted using [database] 
with the search terms suggested by GPT-4. Title/abstract 
screening was performed with Claude 3 Opus (verified by 
manual review of 15% sample, 93% agreement). Information 
extraction from full-text papers used the same model with 
a structured JSON schema, verified against original papers. 
Final synthesis and interpretation are the authors' own, 
informed by direct reading of 50 key papers.
```

이 수준의 투명성이 리뷰어와 독자의 신뢰를 얻는다.

**하지 말 것**: "본인이 직접 모든 논문을 읽었다"고 거짓말. 들키면 학술 부정행위.

**체크리스트 — LLM 문헌 검토의 품질 보증**.

**LLM 문헌 검토 체크리스트**

- [ ] 검색 전략을 명확히 문서화했는가?
- [ ] LLM에게 제공하는 텍스트가 실제 원본인가 (제목만 아님)?
- [ ] 추출 결과의 10-20%를 본인이 직접 검증했는가?
- [ ] 환각 방지 프롬프트 (don't infer)를 포함했는가?
- [ ] 구조화된 출력 형식을 사용했는가?
- [ ] 불확실성을 명시적으로 표현하게 했는가?
- [ ] 핵심 논문 (20-30편)은 본인이 직접 읽었는가?
- [ ] 여러 LLM으로 중요한 부분을 교차 검증했는가?
- [ ] 인용의 존재를 DOI로 확인했는가?
- [ ] 박사 논문이나 출판 시 LLM 사용을 공개할 준비가 되었는가?

**고급 기법 — RAG로 본인만의 문헌 데이터베이스**.

ch26의 앞부분에서 다룬 RAG를 문헌 검토에 결합.

**구축 방법**:
1. 본인이 수집한 논문 PDF들을 하나의 폴더에
2. RAG 시스템 구축 (LangChain, LlamaIndex 등)
3. 각 논문을 청크로 나누고 임베딩
4. 본인의 질문을 이 데이터베이스에 검색

**장점**:
- 본인이 모은 논문 전체에 자연어 질문
- "PINN의 손실 가중치를 다룬 논문은?"
- 답변에 출처 표시 (환각 방지)

**도구**:
- **LangChain + Chroma**: 오픈소스
- **LlamaIndex**: 문서 중심 RAG
- **NotebookLM (Google)**: Google의 무료 도구, 사용하기 쉬움
- **Perplexity Spaces**: 본인의 문서로 검색

**권장 도구**: 시작으로 NotebookLM. 50개 문서까지 무료로 올리고 질문 가능.

**LLM 문헌 검토의 미래와 박사 전략**.

2024-2026년의 트렌드를 보면 LLM 문헌 검토는 빠르게 발전 중.

**예상되는 발전**:
1. **더 긴 맥락**: 한 번에 100-1000 페이지 처리 가능
2. **자동 그래프 생성**: 논문 관계를 자동 그래프로
3. **전문 도구의 확산**: 학술 특화 LLM 서비스
4. **인용 품질의 개선**: 환각이 줄어듦
5. **통합 워크플로**: 검색-선별-추출-분석이 한 도구에

**박사 전략**:
- 2024년 시작한 학생은 이 도구들을 박사 내내 써야 함
- 본인의 프롬프트 라이브러리 구축 (재사용)
- 도구의 변화에 따라가되 본질은 변하지 않음 (본인의 판단이 핵심)
- 후배에게 전수할 수 있는 워크플로 만들기

> LLM은 문헌 검토의 게임 체인저다. 예전에 수백 시간이 걸렸던 작업이 수십 시간으로 줄어든다. 그러나 LLM이 본인의 읽기와 판단을 대체하지는 않는다. 올바른 사용은 "LLM + 본인의 검증"이다. 박사 과정에서 이 조합을 익히면 본인이 훨씬 많은 논문을 훨씬 체계적으로 다룰 수 있다. 그 결과 본인의 박사 논문의 배경이 더 탄탄해지고, 리뷰 논문이 현실적 목표가 되고, 경쟁 논문 모니터링이 자동화된다. 도구로서 LLM을 받아들이되, 본인의 학술적 판단을 잃지 말자.
