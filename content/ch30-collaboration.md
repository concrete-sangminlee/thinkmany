# 협업과 공동연구

대학원 생활의 낭만적인 이미지가 있다. 고독한 천재가 연구실에 혼자 앉아 밤새 연구하다가 유레카를 외치는 장면. 현실은 다르다. 현대 연구의 대부분은 협업으로 이루어진다. Nature에 실리는 논문의 평균 저자 수는 꾸준히 증가해왔고, 공학 분야에서 단독 저자 논문은 사실상 찾기 어렵다. 혼자서 하드웨어를 설계하고, 소프트웨어를 개발하고, 실험을 수행하고, 데이터를 분석하고, 논문까지 쓸 수 있는 사람은 거의 없다.

협업은 선택이 아니라 필수다. 문제는 협업의 필요성을 인식하는 것이 아니라, 협업을 잘 하는 것이다. 좋은 협업은 개인이 혼자서 도달할 수 없는 성과를 만들어내고, 나쁜 협업은 혼자 하는 것보다 못한 결과를 낳으면서 인간관계까지 손상시킨다.

이 장에서는 대학원생이 마주하는 다양한 협업 상황, 즉 연구실 내 협업, 다학제 공동연구, 국제 협업에서 어떻게 효과적으로 협력하고, 갈등을 관리하고, 성과를 공정하게 나누는지를 다룬다.

## 왜 협업하는가

### 연구 문제의 복잡성

현대 공학 문제는 한 분야의 전문성만으로 풀 수 없는 경우가 대부분이다. 자율주행차를 예로 들면, 센서 공학, 컴퓨터 비전, 제어 공학, 기계학습, 법률, 윤리가 모두 필요하다. 스마트 빌딩은 구조공학, 건축환경, IoT, 에너지 공학, 데이터 사이언스가 얽혀 있다. 이런 문제를 한 사람이 모두 커버하는 것은 불가능하다.

대학원생 수준에서도 마찬가지다. 자신이 시뮬레이션을 잘한다면, 실험 데이터를 가진 사람과 협업해서 시뮬레이션을 검증하는 것이 연구의 질을 높인다. 자신이 알고리즘을 개발했다면, 실제 현장 데이터를 가진 연구자와 협업해서 알고리즘의 실용성을 검증하는 것이 논문의 임팩트를 높인다.

### 협업의 실질적 이점

연구 품질의 향상 외에도 협업은 여러 실질적인 이점이 있다. 첫째, 시간 절약이다. 혼자서 모든 것을 배우고 수행하는 대신, 이미 전문성을 가진 사람과 역할을 나누면 연구 속도가 빨라진다. 둘째, 새로운 시각이다. 같은 데이터를 다른 분야 사람이 보면 전혀 다른 해석이 나올 수 있다. 이것이 새로운 연구 질문으로 이어지기도 한다. 셋째, 자원 접근이다. 자신의 연구실에 없는 장비, 데이터, 소프트웨어를 가진 연구실과 협업하면 자원의 제약을 극복할 수 있다. 넷째, 네트워크 확장이다. 공동연구를 통해 다른 분야의 연구자들과 관계를 맺게 되고, 이것이 미래의 기회로 이어진다.

## 공동저자의 역할

### CRediT (Contributor Roles Taxonomy)

효과적인 협업의 출발점은 역할 분담이다. 누가 무엇을 하는지가 처음부터 명확해야 나중에 문제가 생기지 않는다.

CRediT은 연구에서의 기여를 14가지 역할로 분류하는 국제 표준이다. 점점 더 많은 저널이 CRediT 기반의 저자 기여 명세를 요구하고 있으며, 이 분류 체계를 프로젝트 시작 시점에 활용하면 역할 분담이 훨씬 명확해진다.

이 외에도 검토 및 편집(Writing - Review & Editing), 시각화(Visualization), 관리 감독(Supervision), 프로젝트 관리(Project Administration), 연구비 확보(Funding Acquisition), 자원(Resources)이 있다.

프로젝트를 시작할 때 "누가 어떤 CRediT 역할을 맡는가"를 표로 만들어두면, 나중에 저자 순서를 정할 때도, 논문에 기여 명세를 작성할 때도 훨씬 수월하다.

### 역할 분담의 원칙

역할은 프로젝트 시작 시점에 논의하고 합의해야 한다. "나중에 자연스럽게 정해지겠지"는 위험한 생각이다. 프로젝트가 진행되면서 역할이 바뀔 수 있고, 예상보다 어떤 역할의 비중이 커질 수도 있다. 이런 변화는 중간중간 논의하고 조정해야 한다.

역할 분담에서 흔한 문제는 "누가 논문을 쓸 것인가"이다. 데이터 수집이나 분석은 열심히 하지만 논문 작성은 하기 싫어하는 사람, 반대로 쓰는 것만 하고 싶어하는 사람이 있다. 1저자가 논문 작성의 주된 책임을 지는 것이 관행이지만, 1저자가 글쓰기에 약하다면 공동저자가 함께 작성하는 구조를 처음부터 합의해야 한다.

또 다른 흔한 문제는 "보이지 않는 노동"이다. 데이터 정리, 코드 정돈, 그림 제작, 참고문헌 정리 같은 작업은 시간이 많이 들지만 기여로 인정받기 어렵다. 이런 작업을 누가 하는지, 그 기여를 어떻게 인정할지를 미리 논의한다.

## 효과적인 협업 도구

### 코드 협업: Git과 GitHub

코드를 동반하는 연구에서 Git은 선택이 아니라 필수다. 협업 시 Git 없이 코드를 관리하면 반드시 문제가 생긴다. "최종_최종_진짜최종_v3.py" 같은 파일명이 난무하고, 누가 어떤 변경을 했는지 추적할 수 없게 된다.

Git/GitHub를 이용한 코드 협업의 기본 규칙은 다음과 같다. main 브랜치는 항상 작동하는 상태를 유지한다. 새로운 기능이나 실험은 별도의 브랜치에서 작업한다. Pull Request를 통해 코드 리뷰 후 main에 병합한다. 커밋 메시지를 명확하게 작성한다("fix bug"가 아니라 "fix boundary condition handling in thermal simulation").

### 문서 협업: Overleaf과 Google Docs

논문 작성에서의 협업은 Overleaf(LaTeX 기반)이나 Google Docs가 표준이다.

Overleaf는 LaTeX 문서를 웹에서 공동 편집할 수 있는 플랫폼이다. 실시간으로 여러 사람이 동시에 편집할 수 있고, 변경 이력이 남는다. 수학 수식이 많은 공학 논문에는 LaTeX가 거의 필수이므로, Overleaf가 특히 유용하다. 무료 버전은 동시 편집자 수 등의 제한이 있지만, 대부분의 학교가 기관 라이선스를 제공한다.

Google Docs는 초기 브레인스토밍, 초고 작성, 비-LaTeX 문서에 적합하다. 댓글 기능이 잘 되어 있어서 피드백을 주고받기 쉽다. 다만 수학 수식 지원이 약하고, 최종 논문 형식으로는 부적합한 경우가 많다.

협업 문서에서 지켜야 할 규칙이 있다. 다른 사람이 쓴 부분을 수정할 때는 트래킹 기능을 사용하거나 댓글로 제안한다. 자신이 담당하지 않은 섹션을 대폭 수정하기 전에는 해당 저자와 먼저 상의한다. 문서의 포맷팅 규칙(폰트, 레이아웃, 참고문헌 스타일 등)을 처음에 합의하고 일관되게 적용한다.

### 소통 도구: Slack, Teams, 이메일

비동기 소통(asynchronous communication)은 협업에서 매우 중요하다. 모든 논의를 대면 미팅에서만 하면 비효율적이고, 기록이 남지 않는다.

Slack이나 Microsoft Teams 같은 메시징 플랫폼은 빠른 질문과 일상적인 소통에 적합하다. 채널을 프로젝트별로 나누고, 관련 파일과 링크를 공유하고, 중요한 결정은 핀(pin)해둔다. 하지만 중요한 결정이나 복잡한 논의는 이메일이나 미팅으로 하는 것이 좋다. 메시지는 빠르게 흘러가서 나중에 찾기 어렵기 때문이다.

이메일은 공식적인 의사소통, 외부 협력자와의 소통, 중요한 결정의 기록에 적합하다. "지난 미팅에서 논의한 내용을 정리합니다"라는 이메일을 미팅 후에 보내면, 모든 참여자가 동일한 이해를 가지고 있는지 확인할 수 있다.

### 프로젝트 관리: Notion, Trello, GitHub Issues

프로젝트가 복잡해지면 할 일(task) 관리가 필요하다. Notion, Trello, GitHub Issues 등의 도구를 사용하면 누가 무엇을 언제까지 해야 하는지를 시각적으로 관리할 수 있다.

GitHub Issues는 코드 중심 프로젝트에 특히 적합하다. 버그, 기능 요청, 할 일을 이슈로 등록하고, 담당자를 지정하고, 진행 상태를 추적한다. 이슈와 Pull Request를 연결하면 "이 코드 변경이 어떤 문제를 해결하는 것인지"가 명확해진다.

어떤 도구를 사용하든, 핵심은 팀 전체가 같은 도구를 사용하고, 일관되게 업데이트하는 것이다. 절반의 팀원만 사용하는 프로젝트 관리 도구는 없느니만 못하다.

## 코드/데이터 공유 프로토콜

### 공유 저장소의 규칙

협업에서 코드와 데이터를 공유할 때, 명확한 규칙이 없으면 혼란이 빠르게 증가한다. 다음은 공유 저장소를 운영할 때 최소한으로 합의해야 할 항목이다.

디렉토리 구조를 통일한다. 프로젝트 루트에 어떤 폴더가 있고, 각 폴더에 무엇이 들어가는지를 README에 명시한다. 예를 들어, `src/`에 소스 코드, `data/`에 데이터, `results/`에 결과, `docs/`에 문서, `scripts/`에 실행 스크립트를 둔다는 규칙이다.

파일 명명 규칙을 정한다. 날짜 형식(YYYY-MM-DD), 버전 표기 방법, 데이터 파일 명명법 등을 합의한다. "experiment_20250315_v2_final.csv"와 "exp_03-15_ver2_FINAL.csv"가 혼재하면 어떤 파일이 최신인지 알 수 없다.

대용량 데이터는 별도로 관리한다. Git은 코드 관리에 최적화되어 있으며, 대용량 파일에는 적합하지 않다. 데이터는 Google Drive, 학교 서버, AWS S3 같은 저장소에 두고, 접근 경로를 README에 기록한다. Git LFS(Large File Storage)를 사용하는 방법도 있다.

### README의 중요성

공유 저장소에서 README는 가장 중요한 파일이다. 프로젝트에 새로 참여하는 사람이 가장 먼저 읽는 문서이며, 코드를 재현하려는 외부 연구자에게도 첫 번째 안내서다.

좋은 README에는 다음이 포함된다. 프로젝트의 간단한 설명(이 코드는 무엇을 하는가), 설치 방법(필요한 패키지, 환경 설정), 실행 방법(어떤 스크립트를 어떤 순서로 실행하는가), 데이터 설명(데이터 형식, 출처, 접근 방법), 결과 재현 방법(논문의 그림/표를 어떻게 재생성하는가), 기여자 목록과 연락처이다.

README는 프로젝트가 진행되면서 업데이트해야 한다. "나중에 정리하자"는 말은 "영원히 정리하지 않겠다"와 같은 뜻이다. 코드를 변경할 때 README도 함께 업데이트하는 습관을 들여야 한다.

### 데이터 형식 표준화

협업에서 데이터를 주고받을 때, 형식이 통일되어 있지 않으면 변환 작업에 불필요한 시간이 소요된다. 프로젝트 시작 시 데이터 형식을 합의해야 한다.

숫자 데이터는 CSV나 HDF5 같은 범용 형식을 사용한다. Excel 파일(.xlsx)은 사람이 직접 보기에는 편하지만, 프로그래밍으로 처리할 때 문제가 생길 수 있다(시트 이름 문제, 자동 형식 변환 등). 이미지 데이터는 무손실 형식(PNG, TIFF)을 기본으로 하고, JPEG는 손실 압축이 있으므로 원본 보존 시 주의한다. 메타데이터(데이터 수집 날짜, 조건, 장비 정보 등)를 함께 기록하는 것도 중요하다.

## 다학제 연구

### 다른 세계의 언어

다학제(interdisciplinary) 연구는 다른 분야의 연구자들과 협업하는 것이다. 이것은 같은 분야 내 협업보다 훨씬 더 어렵고, 동시에 훨씬 더 보상이 크다.

가장 큰 장벽은 언어다. 같은 단어가 분야마다 다른 뜻을 가질 수 있다. "모델"이라는 단어는 토목공학에서는 물리적 축소 모형이나 수학적 수식을, 컴퓨터 과학에서는 머신러닝 알고리즘을, 사회과학에서는 이론적 프레임워크를 의미할 수 있다. "검증(validation)"의 기준도 분야마다 다르다. 처음 만났을 때 서로의 용어를 정의하는 시간을 갖는 것이 중요하다. 용어 사전(glossary)을 공유 문서에 만들어두면 소통이 훨씬 수월해진다.

### 기대치의 차이

분야마다 연구의 속도, 논문의 길이, 발표 형식, 심사 기준이 다르다. 컴퓨터 과학에서는 학회 논문이 주된 출판 경로이고 리뷰 기간이 수 개월이지만, 토목공학에서는 저널 논문이 주류이고 리뷰 기간이 6개월-1년이 넘을 수 있다. 실험 과학에서는 한 실험에 몇 주가 걸리지만, 시뮬레이션 분야에서는 하루에 수백 개의 실험을 돌릴 수 있다.

이런 차이를 이해하지 못하면 갈등이 생긴다. "왜 아직 결과가 안 나왔어요?"라는 질문이, 한쪽에서는 합리적이고 다른 쪽에서는 무례하게 느껴질 수 있다. 프로젝트 시작 시 각 분야의 현실적인 타임라인을 공유하고, 정기적으로 진행 상황을 업데이트하는 것이 갈등을 예방한다.

### 공통 언어 찾기

다학제 협업의 성공은 공통 언어를 찾는 데 달려 있다. 이것은 한쪽이 다른 쪽의 언어를 배우라는 뜻이 아니다. 양쪽 모두가 이해할 수 있는 수준에서 소통하는 방법을 찾아야 한다.

시각적 표현이 큰 도움이 된다. 복잡한 개념을 설명할 때 말이나 수식보다 그림이 훨씬 효과적이다. 화이트보드에 다이어그램을 그리며 토론하면, 전문 용어의 벽을 넘을 수 있다. 또한 정기 미팅에서 각자의 진행 상황을 비전공자도 이해할 수 있는 수준으로 설명하는 연습을 하면, 연구를 더 넓은 맥락에서 바라보는 능력도 함께 발전한다.

## 국제 협업

### 시차와 스케줄링

국제 공동연구에서 가장 현실적인 어려움은 시차다. 한국과 미국 동부의 시차는 14시간(썸머타임 기준 13시간)이다. 한국 오전 9시가 미국 저녁 7시다. 양쪽 모두에게 편한 시간을 찾기가 어렵다.

해결 방법은 다음과 같다. 첫째, 정기 미팅 시간을 합의하고 지킨다. 양쪽 모두에게 약간 불편한 시간이 될 수 있지만, 그것이 공정하다. 둘째, 비동기 소통을 적극 활용한다. 모든 것을 실시간 미팅으로 논의할 필요는 없다. 이메일이나 Slack으로 할 수 있는 것은 비동기로 처리한다. 셋째, 공유 문서에 진행 상황을 기록한다. 미팅 없이도 상대방이 어디까지 했는지 알 수 있게 한다.

### 언어와 문화의 차이

영어가 모국어가 아닌 상황에서 영어로 소통하는 것은 에너지가 많이 드는 일이다. 특히 화상 미팅에서 빠르게 진행되는 토론을 따라가기 어려울 수 있다.

몇 가지 실용적인 팁이 있다. 미팅 전에 안건을 공유하면, 미리 생각을 정리하고 필요한 표현을 준비할 수 있다. 미팅 중에 이해가 안 되면 바로 물어본다. "I'm sorry, could you repeat that?"는 전혀 부끄러운 말이 아니다. 미팅 후에 논의 내용을 이메일로 정리해서 보내면, 잘못된 이해를 사전에 잡을 수 있다. 중요한 결정은 반드시 글로 확인한다. 구두 합의만으로는 나중에 "그때 그렇게 말한 게 아니었는데"라는 상황이 생길 수 있다.

문화적 차이도 있다. 한국의 학술 문화에서는 선배/후배, 교수/학생 간의 위계가 명확하지만, 서구 학술 문화에서는 상대적으로 수평적이다. 미국이나 유럽의 공동연구자에게 자신의 의견을 직접적으로 말하는 것, 교수에게 동의하지 않는 부분을 정중하게 표현하는 것은 무례가 아니라 기대되는 행동이다.

## 갈등 해결

### 갈등은 정상이다

협업에서 갈등이 생기지 않으면, 그것은 좋은 협업이 아니라 한쪽이 일방적으로 양보하고 있는 것일 수 있다. 의견 차이, 기여도 인정 문제, 연구 방향 불일치는 건강한 협업에서도 발생한다. 문제는 갈등의 존재가 아니라, 갈등을 다루는 방법이다.

### 저자 순서 분쟁

가장 흔하고 가장 감정적인 갈등이다. "내가 더 많이 했는데 왜 2저자인가?" "코드를 내가 다 짰는데 실험 데이터 제공한 사람이 1저자라고?" 이런 갈등은 프로젝트 시작 시 합의가 없었기 때문에 발생한다.

예방이 최선이다. 프로젝트를 시작할 때, 아직 결과가 나오지 않았고 감정이 개입되지 않은 상태에서, 저자 순서의 기준을 논의한다. "1저자는 논문 작성의 주된 책임을 지는 사람" "기여도가 동등하면 공동 1저자로 한다" "교신저자는 PI가 맡는다" 같은 합의를 미리 해둔다. 이것을 이메일이나 공유 문서에 기록으로 남기는 것이 중요하다.

만약 이미 갈등이 발생했다면, 감정이 아닌 사실에 기반하여 논의한다. 각자가 구체적으로 무엇을 했는지를 리스트로 만들고, CRediT 분류에 따라 기여를 정리한다. 합의가 안 되면 지도교수나 제3자의 중재를 요청한다.

### 연구 방향 불일치

"이 방법이 더 좋다" vs "아니, 저 방법이 더 좋다." 연구 방향에 대한 의견 차이는 학술적 토론의 일부이며, 건전한 것이다. 하지만 합의에 이르지 못하면 프로젝트가 멈출 수 있다.

이런 상황에서는 데이터로 이야기하는 것이 가장 효과적이다. 두 가지 접근 모두를 작은 규모로 시도해보고, 결과를 비교한다. 결과가 명확하면 토론이 해결되고, 두 접근 모두 장단점이 있다면 논문에서 두 가지를 모두 다루는 것도 하나의 방법이다.

최악의 대처는 갈등을 회피하고 애매한 상태로 두는 것이다. 불일치를 해소하지 않고 각자 다른 방향으로 작업하면, 나중에 합칠 때 더 큰 문제가 된다. 불편하더라도 직접 대화하고, 합의를 기록으로 남기는 것이 좋다.

### 기여도 인정

"그 사람은 미팅에만 참석하고 실제로 아무것도 안 했다." 이런 불만은 기여도 인정이 불명확할 때 생긴다. 정기적으로 각자의 진행 상황을 공유하면, 누가 얼마나 기여하고 있는지가 자연스럽게 드러난다. 미팅 기록에 각자의 할 일과 완료 현황을 남기면, 기여도의 객관적인 근거가 된다.

## 지도교수 간 공동연구

### 두 교수 사이에 끼인 학생

지도교수와 다른 교수의 공동연구 프로젝트에 학생이 투입되는 경우가 있다. 이 상황은 기회이기도 하고 어려움이기도 하다.

기회인 이유: 두 분야의 전문성을 동시에 배울 수 있다. 네트워크가 확장된다. 공동 연구 경험이 이력에 좋다.

어려운 이유: 두 교수의 기대가 다를 수 있다. A 교수는 "시뮬레이션을 더 돌려라"고 하고, B 교수는 "논문을 먼저 써라"고 할 수 있다. 두 교수 사이의 의견 불일치가 학생에게 전가될 수 있다. 보고를 누구에게 해야 하는지 혼란스러울 수 있다.

대처 방법은 명확하다. 첫째, 자신의 1차 지도교수가 누구인지를 확인한다. 두 교수의 지시가 충돌할 때 누구의 우선순위를 따를지가 명확해야 한다. 둘째, 두 교수가 함께 참석하는 미팅을 정기적으로 갖는다. 학생이 두 교수 사이에서 메시지를 전달하는 역할을 맡으면 안 된다. 교수들이 직접 논의해야 할 사항은 교수들끼리 논의하게 한다. 셋째, 모든 결정과 지시를 기록한다. "A 교수님은 이렇게 말씀하셨고, B 교수님은 이렇게 말씀하셨는데, 어떻게 진행할까요?"라는 이메일을 두 교수 모두에게 보내면, 불일치가 조기에 드러나고 해결된다.

## 좋은 협업자가 되기

협업에 관한 모든 기술과 도구를 알아도, 결국 가장 중요한 것은 신뢰다. 신뢰는 작은 행동의 축적으로 만들어진다.

약속한 기한을 지킨다. 지키지 못할 것 같으면 미리 알린다. "마감일이 지난 후에 아무 말 없이 제출하지 않는 것"은 기한을 지키는 것보다 더 나쁘다. 기한 전에 "이 부분이 예상보다 오래 걸리고 있어서 이틀 더 필요하다"고 솔직하게 말하는 것이 신뢰를 유지하는 방법이다.

자신의 실수를 인정한다. 코드에 버그가 있었다면, 데이터 처리에 오류가 있었다면, 빨리 인정하고 수정한다. 실수를 숨기려다 더 큰 문제를 만드는 것보다 훨씬 낫다.

다른 사람의 기여를 인정한다. "이 아이디어는 누구와의 토론에서 나왔다" "이 코드의 핵심 부분은 누구가 작성했다"를 명확히 하는 것이 건강한 협업 문화를 만든다.

반응을 빨리 한다. 이메일이나 메시지에 답장하는 데 며칠씩 걸리면 협업이 멈춘다. 바로 답하기 어려운 질문이라도 "확인했고, 며칠 내에 답하겠다"는 짧은 반응을 보내는 것만으로도 상대방의 불안을 줄일 수 있다.

결국 협업은 기술의 문제가 아니라 태도의 문제다. 상대방을 존중하고, 투명하게 소통하고, 합의를 지키는 기본적인 태도가 모든 협업 도구와 프로토콜보다 중요하다. 좋은 협업자라는 평판은 학계에서 가장 가치 있는 자산 중 하나다.

그런데 모든 협업 관계 중에서 가장 중요하고, 가장 오래 지속되며, 가장 큰 영향을 미치는 관계가 하나 있다. 지도교수와의 관계다. 공동연구자는 프로젝트가 끝나면 관계가 느슨해질 수 있지만, 지도교수와의 관계는 졸업 후에도 추천서, 학술 네트워크, 공동연구를 통해 계속된다. 다음 장에서는 이 관계를 어떻게 시작하고, 유지하고, 위기에서 지켜내는지를 다룬다.

## 협업에서 흔한 갈등 시나리오

위에서 갈등 해결의 원칙을 다루었다. 여기서는 대학원생이 실제로 자주 마주하는 구체적인 갈등 시나리오 세 가지를 살펴본다. 각 시나리오에 대해 현실적인 대처 방법을 정리한다.

### 시나리오 1: 공동 연구자가 중간에 빠진다

프로젝트가 한창 진행 중인데, 공동 연구자가 갑자기 참여를 중단하는 경우가 있다. 졸업, 취업, 개인 사정, 다른 프로젝트 우선순위 변경 등 이유는 다양하다. 감정적으로 배신감을 느낄 수 있지만, 감정보다 현실적 대응이 중요하다.

남은 사람이 해야 할 일은 다음과 같다.

역할 재분배를 즉시 수행한다. 빠진 사람이 맡고 있던 작업의 목록을 정리하고, 남은 인원이 현실적으로 가져갈 수 있는지 판단한다. 모든 작업을 한 사람이 떠안으면 프로젝트 전체가 위험해지므로, 범위를 축소하는 것도 하나의 선택지다.

기여도를 재조정한다. 빠진 사람이 이미 의미 있는 기여를 한 경우, 그 기여를 인정하되 저자 순서는 실제 기여에 맞게 조정한다. 프로젝트 초기에만 참여하고 이후 기여가 없었다면, 감사의 글(Acknowledgment)에 이름을 넣는 것이 적절할 수 있다.

지도교수에게 즉시 보고한다. 상황을 숨기고 혼자 해결하려 하면 더 큰 문제가 된다. 지도교수는 경험에서 나온 해결책을 제시할 수 있고, 필요하다면 다른 학생을 투입하거나 프로젝트 방향을 조정하는 결정을 내릴 수 있다.

빠진 사람과의 관계를 손상시키지 않는다. 학계는 좁다. 지금 함께 못 하더라도 미래에 다시 만날 가능성이 있다. 감정적 대응 대신 "상황을 이해한다, 나중에 기회가 되면 다시 함께 하자"는 태도가 장기적으로 유리하다.

### 시나리오 2: 방법론에 대한 의견 불일치

"유한요소법(FEM)으로 해야 한다" vs "딥러닝으로 충분하다." 연구 방법론에 대한 의견 차이는 건전한 학술적 토론이지만, 합의에 이르지 못하면 프로젝트가 정체된다.

해결법의 첫 번째 원칙은 데이터로 비교하는 것이다. 두 방법론 모두 소규모 실험(파일럿 스터디)을 먼저 수행해본다. 같은 데이터셋에 두 방법을 적용하고, 정확도, 계산 시간, 확장성 등의 기준으로 객관적으로 비교한다. 결과가 명확하면 토론이 자연스럽게 해결된다.

두 방법 모두 장단점이 있다면, 논문에서 양쪽을 모두 다루는 것도 좋은 전략이다. 비교 연구(comparative study) 자체가 의미 있는 기여가 될 수 있다. "방법 A는 이런 조건에서 우수하고, 방법 B는 저런 조건에서 우수하다"는 결론은 실용적이고 인용 가능성도 높다.

합의가 안 되면 지도교수의 중재를 요청한다. 지도교수는 두 방법론에 대한 경험이 있을 수 있고, 제3자의 관점에서 판단을 내릴 수 있다. 중재를 요청하는 것은 약점이 아니라, 프로젝트를 전진시키기 위한 합리적 행동이다.

절대 피해야 할 것은 "합의 없이 각자 다른 방법으로 작업"하는 것이다. 나중에 결과를 합치려 할 때 훨씬 큰 문제가 발생한다. 불편하더라도 결정을 내리고, 결정의 근거를 기록으로 남긴다.

### 시나리오 3: 한 사람이 90% 했는데 저자 순서가 분쟁이 된다

가장 감정적으로 격해지기 쉬운 갈등이다. "코드를 다 짜고, 데이터를 모으고, 논문 초고까지 작성했는데, 아이디어만 낸 사람이 1저자를 주장한다." 이런 상황은 프로젝트 초기에 합의가 없었기 때문에 발생한다.

해결의 핵심은 CRediT(Contributor Roles Taxonomy)을 활용한 기여도의 명문화다. 각자가 14가지 역할 중 어떤 항목에 기여했는지를 구체적으로 리스트로 만든다. "개념화는 누가 했는가, 방법론 설계는 누가 했는가, 코드 작성은 누가 했는가, 논문 작성은 누가 했는가." 이렇게 정리하면 감정이 아닌 사실에 기반한 논의가 가능해진다.

가장 중요한 교훈은 초기 합의의 중요성이다. 프로젝트를 시작할 때, 아무리 가까운 사이라도, 저자 순서의 기준을 명확하게 논의하고 문서로 남겨야 한다. "이 프로젝트에서 1저자는 코드 개발과 논문 작성을 주도한 사람이 맡는다"와 같은 합의가 이메일이나 공유 문서에 기록되어 있으면, 분쟁의 소지가 크게 줄어든다.

이미 갈등이 발생한 경우, 당사자 간 해결이 어려우면 지도교수에게 중재를 요청한다. 감정적으로 대응하거나, 관계를 끊겠다고 협박하는 것은 어떤 상황에서도 도움이 되지 않는다.

<p>세 가지 시나리오 모두에서 공통된 교훈은 "프로젝트 초기에 합의하고 문서화한다"는 것이다. 역할 분담, 저자 순서, 이탈 시 처리 방법, 의견 불일치 시 결정 방식을 프로젝트 시작 시 논의하고 기록으로 남긴다. 관계가 좋을 때 합의해두면, 문제가 생겼을 때 감정이 아닌 합의에 기반하여 해결할 수 있다.</p>

## 산학 협력의 실무

산학 협력(industry-academia collaboration)은 대학원생이 자주 접하는 협업 형태다. 본인의 연구실이 기업과 과제를 수행하면, 박사 과정 중 산업체 연구원과 직접 일할 기회가 생긴다. 학술 동료와의 협업과는 규칙이 다르다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>산학 협력의 본질적 차이</strong></p>
    <p>학술 협업은 "지식 생산"이 목적이고, 산학 협력은 "기업 가치 창출"이 목적이다. 같은 연구를 해도 평가 기준이 다르다. 학술 동료는 "이 연구가 새로운가?"를 묻고, 기업 파트너는 "이 연구가 얼마나 빨리 우리 제품에 적용되는가?"를 묻는다. 이 차이를 인정하고 양쪽 기대치를 동시에 관리하는 것이 산학 협력의 핵심 기술이다.</p>
  </div>
</div>

**비밀유지협약(NDA)을 가장 먼저 확인한다.** 기업과 협력이 시작되면 거의 대부분 비밀유지협약(Non-Disclosure Agreement)을 체결한다. NDA는 본인이 협력 중에 알게 된 기업의 정보를 외부에 공개하지 않겠다는 약속이다. 학생이라도 직접 서명해야 하는 경우가 많다. 서명 전에 반드시 다음을 확인한다. 첫째, **비밀 정보의 범위가 어디까지인가**. 너무 광범위하면("이 분야의 모든 정보") 본인의 박사 논문에도 쓸 수 없게 될 수 있다. 둘째, **유효 기간**이 합리적인가. 보통 2-5년이 일반적이고, 평생 비밀 유지를 요구하는 NDA는 거부할 수 있다. 셋째, **본인의 학위 논문 사용 권리가 명시되어 있는가**. NDA가 본인의 졸업을 막아서는 안 된다. 넷째, **위반 시 책임 범위**. 일부 NDA는 위반 시 막대한 손해 배상을 요구한다. 학생이 감당할 수 없는 수준이면 거부해야 한다. NDA 서명 전에 지도교수, 학교 산학협력단, 가능하면 법률 자문을 거친다.

**기업 파트너와의 의사소통.** 첫째, **보고는 비즈니스 언어로**. 기업 파트너에게 보고할 때는 학술 용어보다 실용적 언어를 쓴다. "We achieved 95% accuracy"보다 "정확도가 95%이며, 이는 현재 사용하는 X 방법보다 10%p 높습니다"가 더 효과적이다. 그들이 의사 결정을 내릴 수 있는 정보 형태로 전달한다. 둘째, **진행 상황을 정기적으로 공유한다**. 학술 협업과 달리 기업 파트너는 자주 진행 상황을 알고 싶어 한다. 격주 또는 월 1회 짧은 진도 보고서를 만든다. "What we did, What we found, What's next, Issues" 4가지 항목으로 1-2페이지면 충분하다. 셋째, **기대치를 미리 설정한다**. "이 연구는 학술 연구이므로 즉시 제품에 적용 가능한 결과를 보장하지 않는다"는 점을 처음부터 명확히 한다. 후반에 가서 "왜 아직 안 되느냐"는 갈등을 피할 수 있다.

**산학 과제의 IP 처리.** ch40에서 다룬 것처럼, 산학 과제에서 발생한 발명/소프트웨어의 소유권은 사전에 합의된 계약에 따른다. 보통 다음 패턴이 흔하다. 첫째, **공동 소유**: 대학과 기업이 공동으로 소유. 라이선싱과 사용 권리를 분담. 둘째, **기업 단독 소유**: 기업이 연구비를 전액 부담한 경우 흔함. 대학과 학생은 학술 발표 권리만 보유. 셋째, **대학 소유 + 기업의 우선 라이선싱 권리**: 대학이 소유하지만, 기업이 일정 기간 우선적으로 사용할 권리를 가짐. 학생 입장에서 가장 중요한 것은 "본인의 박사 논문에 이 결과를 사용할 수 있는가"와 "본인이 학술 논문으로 발표할 수 있는가"이다. 두 가지를 사전에 명확히 한다.

**논문 발표 시 기업의 사전 검토.** 산학 과제의 결과를 학술 논문으로 발표할 때, 기업이 사전 검토(pre-publication review)를 요구하는 경우가 많다. 보통 30-60일의 검토 기간을 가진다. 이 기간 동안 기업은 비밀 정보가 노출되는지, 본인의 사업에 불리한 내용이 있는지를 검토한다. 검토 후 수정 요청이 올 수 있다(예: 특정 수치 삭제, 특정 표현 변경). 합리적인 수정은 받아들이되, 핵심 결과를 수정하라는 요청은 학술 윤리 관점(ch29)에서 거부할 수 있다. 이 부분에서 갈등이 생기면 지도교수와 학교 산학협력단의 도움을 받는다.

## 국제 공동연구의 실무

국제 공동연구는 박사 과정 중 가장 가치 있는 경험 중 하나다. 본인의 연구가 한국 밖의 시야를 얻고, 글로벌 네트워크가 만들어진다. 다만 시간과 행정 부담이 한국 내 협업의 2-3배는 든다.

**국제 공동연구의 시작.** 가장 흔한 시작은 본인의 지도교수가 해외 동료와 공동 과제를 받거나, 본인이 학회에서 만난 해외 연구자와 자연스럽게 공동 작업을 시작하는 것이다. 학생 본인이 주도하여 시작하는 경우는 드물지만 불가능하지 않다. 예를 들어, 본인의 박사 연구와 정확히 보완적인 작업을 하는 해외 연구실이 있다면, 지도교수에게 "이 사람과 협업할 가능성이 있을까요?"라고 제안할 수 있다.

**MOU와 공식 협약.** 한국 대학과 해외 대학 사이에 공동연구가 시작되면, 보통 양해각서(Memorandum of Understanding, MOU)나 공동연구 협약(Collaborative Research Agreement)을 체결한다. MOU는 협력의 의지를 표현하는 비교적 가벼운 문서이고, 공동연구 협약은 더 구체적인 권리와 의무를 명시한다. 학생 입장에서 MOU 체결 자체에 직접 관여하는 일은 드물지만, MOU에 본인의 졸업과 관련된 조항(데이터 사용 권리, 논문 발표 권리)이 적절히 포함되어 있는지 지도교수에게 확인한다.

**국제 공동 펀딩.** 국제 공동연구가 본격화되면 양국에서 공동으로 연구비를 신청할 수 있다. 한국의 NRF는 한-미, 한-EU, 한-일, 한-독 등 양자 공동 연구 사업을 운영한다(ch39 참조). EU의 Horizon Europe도 한국 연구자가 파트너로 참여할 수 있다. 본인이 박사 과정 학생이라면 직접 지원자가 되기는 어렵지만, 지도교수의 신청서 작성을 돕고 결과를 활용할 수 있다.

**데이터 공유 협정.** 국제 공동연구에서 가장 까다로운 부분 중 하나는 데이터 공유다. 데이터에는 개인 정보, 보안 정보, 기업 정보가 포함될 수 있고, 국가별 규제가 다르다. 유럽의 GDPR(General Data Protection Regulation)은 매우 엄격하여, 유럽 시민의 데이터를 한국으로 가져오는 것 자체에 절차가 필요하다. 미국은 분야별 규제(HIPAA for medical data, ITAR for defense)가 있다. 데이터 공유 전에 양국의 규제를 확인하고, 필요하면 데이터 공유 협정(Data Use Agreement)을 체결한다.

**시차와 의사소통의 실전.** 한국과 유럽은 6-8시간, 한국과 미국 동부는 13시간, 미국 서부는 16시간 차이가 난다. 화상 회의 시간을 잡을 때 양쪽 모두 합리적인 시간(둘 다 9-19시 사이)을 찾는 것이 어렵다. 한국이 오전이면 미국은 전날 저녁이거나 새벽이다. 현실적인 전략은 다음과 같다. 첫째, **회의 빈도를 줄인다**. 격주 또는 월 1회로 충분하다. 회의보다 비동기 소통(이메일, Slack, GitHub issues)을 늘린다. 둘째, **양쪽이 번갈아 양보한다**. 이번 회의는 미국에 편한 시간(한국 새벽 6시), 다음 회의는 한국에 편한 시간(미국 저녁 8시)으로 번갈아 잡는다. 한쪽만 계속 양보하면 관계가 손상된다. 셋째, **회의록을 반드시 작성한다**. 양쪽이 같은 시간에 모두 참석하지 못할 수 있으므로, 결정 사항을 문서로 남겨 모두가 같은 정보를 가지게 한다.

**문화적 차이의 인지.** 같은 영어로 대화해도 문화에 따라 의사소통 방식이 다르다. 미국 연구자는 직설적이고 빠른 결정을 선호한다. 독일 연구자는 정확하고 체계적인 절차를 중시한다. 일본 연구자는 간접적이고 합의 기반의 의사 결정을 선호한다. 본인의 협업 파트너의 문화적 배경을 의식하면 갈등을 줄일 수 있다. 다만 "모든 미국인이 직설적이다" 같은 일반화는 위험하다. 개인차가 항상 더 크다. 결국은 개인 대 개인으로 신뢰를 쌓는다.

**방문 연구원(Visiting Researcher) 경험.** 국제 공동연구가 깊어지면, 학생이 해당 연구실로 단기 방문(1-6개월)하는 경우가 있다. 이것을 **방문 연구원** 또는 **연구 인턴**이라고 한다. 방문 연구원 경험은 박사 과정에서 가장 변화시키는 경험 중 하나다. 본인의 시야가 넓어지고, 새로운 방법론을 직접 배우고, 평생 동료가 되는 사람을 만난다. 비용은 보통 본인이 부담하지만(여행, 숙박), 일부는 NRF의 박사과정생 해외연수 프로그램이나 BK21로 지원받을 수 있다. 본인이 박사 과정 중 한 번이라도 방문 연구원 경험을 가질 수 있다면, 적극 추천한다.

## 컨소시엄과 다기관 프로젝트

대형 정부 과제는 단일 연구실이 아니라 여러 기관이 함께 수행하는 컨소시엄(consortium) 형태가 흔하다. 한국의 대형 R&D 사업, EU의 Horizon Europe, 미국의 DARPA 같은 사업들이 그렇다. 박사 과정 중 컨소시엄 프로젝트에 참여하면 학습 곡선이 가파르지만, 그만큼 배우는 것이 많다.

**컨소시엄 구조의 이해.** 컨소시엄은 보통 다음 역할로 구성된다. **주관 기관(Lead Institution)**: 전체 과제의 책임자, PI(Principal Investigator)가 소속된 곳. **참여 기관(Partner Institutions)**: 서브 태스크를 담당하는 다른 대학/연구소/기업. **자문 위원(Advisory Board)**: 외부 전문가로 구성된 자문 그룹. 학생 본인은 보통 한 참여 기관의 학생 신분으로 컨소시엄에 참여한다.

**다기관 프로젝트의 도전.** 첫째, **의사 결정이 느리다**. 5개 기관이 모두 동의해야 하는 결정은 한 달 이상 걸리는 경우가 흔하다. 본인의 박사 연구가 컨소시엄의 결정에 종속되면, 이 느림이 본인의 졸업을 지연시킬 수 있다. 둘째, **의사소통 부담이 크다**. 매월 정기 회의, 분기별 보고서, 연간 종합 보고서 같은 행정 업무가 본인의 시간을 잡아먹는다. 본인이 박사 과정 학생이라면 이 부담을 지도교수나 포닥이 대부분 가져가도록 협상할 수 있다. 학생이 행정에 빠지면 본인의 박사 연구 진도가 늦어진다. 셋째, **저자권 협상이 복잡하다**. 5개 기관이 함께 쓴 논문에서 누가 어떤 위치에 들어갈지 합의가 어렵다. CRediT(앞서 다룸)을 더 엄격하게 적용해야 한다.

**컨소시엄에서 학생의 위치.** 학생은 보통 컨소시엄의 가장 낮은 계층에 있다. 컨소시엄의 핵심 결정에 직접 참여하지 않고, 본인의 지도교수나 포닥의 지시를 받아 작업을 수행한다. 이것이 답답할 수 있지만, 박사 과정에서는 자연스럽다. 컨소시엄의 큰 그림은 보면서 배우되, 본인의 박사 연구는 컨소시엄과 정렬된 작은 부분에 집중한다. 본인이 박사 졸업 후 PI가 되면 그때 컨소시엄 운영에 직접 참여할 기회가 온다.

**컨소시엄의 가치.** 어렵더라도 컨소시엄 참여는 가치가 크다. 첫째, 본인 분야의 핵심 연구자들과 정기적으로 만난다. 평생의 네트워크가 만들어진다. 둘째, 본인의 박사 연구가 더 큰 맥락에서 의미를 가진다. "이 연구가 왜 중요한가"의 답이 분명해진다. 셋째, 대형 과제 운영의 노하우를 가까이서 본다. 본인이 미래에 PI가 될 때 큰 자산이 된다.

**컨소시엄 학회/워크숍.** 컨소시엄은 보통 정기적으로 모든 참여 기관이 모이는 학회나 워크숍을 개최한다. 이 자리에서는 외부 학회와 다른 분위기로 본인의 진행 상황을 발표한다. 일반 학회는 "최고의 결과만" 발표하지만, 컨소시엄 학회는 "현재 진행 중인 작업과 어려움"을 솔직하게 공유한다. 이 자리에서 다른 기관의 연구자가 본인의 문제를 해결할 통찰을 줄 수 있다. 본인의 어려움을 숨기지 말고 솔직하게 공유한다.

## 연구 인수인계 — 누군가 프로젝트를 떠날 때

연구 협업에서 피할 수 없는 순간이 있다. 누군가가 프로젝트를 떠나는 때다. 선배가 졸업하면서 본인에게 프로젝트를 넘기고 가거나, 본인이 졸업하면서 후배에게 진행 중인 일을 넘겨야 하거나, 포닥이 다른 자리로 옮기거나, 공동연구자가 다른 주제로 옮길 수 있다. 이 **인수인계(handover)**가 잘 이루어지지 않으면, 떠나는 사람의 수년간 투자가 사라지고, 남는 사람은 아무것도 없는 상태에서 다시 시작해야 한다. 하지만 대부분의 연구실에서 인수인계는 문화로 자리 잡지 않았고, 그 결과 엄청난 지식 손실이 반복된다.

<div class="highlight-box highlight-warning">

**지식 손실의 비용.** 박사생 한 명이 4-6년간 쌓은 프로젝트의 암묵적 지식(실험 노하우, 데이터 세부 사항, 시도해 본 실패들)은 졸업 후 2-3주면 대부분 휘발된다. 다음 사람이 같은 작업을 재현하려면 보통 6개월-1년이 걸린다. 이것은 개인의 손실이 아니라 연구실 전체의 손실이다. 체계적 인수인계는 이 손실을 80% 이상 줄일 수 있다.

</div>

**인수인계 3가지 시나리오.**

첫째, **졸업하는 선배에서 후배로**. 가장 흔한 경우. 선배는 졸업 직전 수 주간 매우 바쁘고, 후배는 아직 본인의 주제에 집중하느라 여유가 없다. 선배가 떠난 후에야 "그때 물어볼 걸"이라는 후회가 온다. 선배의 졸업 최소 2-3개월 전에 인수인계를 시작해야 한다.

둘째, **본인이 떠나는 상황**. 본인이 졸업, 취업, 다른 주제로 전환할 때다. 본인의 작업을 후배나 동료에게 넘긴다. 이 경우 본인이 주도적으로 인수인계 문서를 만들어야 한다. "떠나는 사람이 성의껏 인수인계하는 것"이 연구실의 문화를 만든다.

셋째, **공동연구자가 떠나는 경우**. 포닥이 다른 기관으로 이직하거나, 다른 학교의 협력자가 주제를 바꾸는 경우다. 이 경우는 더 민감하다. 공식적 위계 관계가 없으므로 인수인계를 "요구"하기 어렵다. 미리 합의된 관행이 있어야 한다.

**인수인계 문서의 핵심 7가지 요소.**

모든 인수인계는 문서로 남겨야 한다. 구두 인수인계는 실패한다. 다음 요소를 포함한다.

1. **프로젝트 개요와 맥락**
   - 연구 목표와 현재까지의 진행 상황
   - 왜 이 주제를 선택했는지(배경)
   - 다른 프로젝트/논문과의 관계
   - 현재의 이해관계자(지도교수, 공저자, 펀딩 기관)

2. **데이터와 파일 위치**
   - 모든 데이터의 저장 위치와 접근 방법
   - 폴더 구조와 명명 규칙
   - 백업 위치
   - 각 데이터의 출처와 라이선스 (ch28 참조)

3. **코드와 환경 설정**
   - 주요 스크립트와 실행 방법
   - 필요한 라이브러리와 버전
   - 환경 재현 방법 (Docker, conda env, requirements.txt)
   - API 키나 자격증명의 저장 위치 (실제 값은 제외, ch16 참조)

4. **실험 기록과 결과**
   - 시도한 실험들의 리스트
   - 성공한 것과 실패한 것, 그 이유
   - 결과 그래프와 데이터의 위치
   - 논문 원고의 최신 버전

5. **외부 리소스와 연락처**
   - 협력자 연락처와 각자의 역할
   - 데이터 제공자, 현장 담당자
   - 중요한 이메일 스레드
   - 슬랙/디스코드 채널

6. **알려진 문제와 TODO**
   - 현재 해결되지 않은 버그
   - 미완료 작업
   - 다음 단계로 추천하는 것
   - 피해야 할 함정 ("X는 시도해 봤는데 안 됨")

7. **암묵적 지식**
   - 문서화되지 않은 노하우
   - "이걸 모르면 하루 헤맨다" 같은 팁
   - 연구실 외부 사람에게는 당연해 보이지만 내부에서만 통용되는 관행

이 7가지를 별도의 문서 `HANDOVER.md`로 만들고, 각 섹션을 2-5쪽 정도로 쓴다. 총 20-50쪽이 일반적이다. 처음에는 이 양이 많아 보이지만, 직접 써 보면 실제로 필요한 정보가 이만큼 있다는 것을 깨닫는다.

**인수인계의 이상적 타임라인.**

졸업 또는 이직 6-8주 전부터 시작하는 것이 이상적이다. 다음 스케줄을 참고한다.

- **D-8주**: 인수인계 대상자 확정. 후배가 이미 정해져 있지 않다면 지도교수와 논의.
- **D-6주**: 첫 인수인계 미팅. 프로젝트 개요 설명. 후배와 함께 주요 파일을 둘러본다.
- **D-4주**: `HANDOVER.md` 초안 작성. 주요 섹션들을 채운다.
- **D-3주**: 후배가 본인의 코드를 직접 실행해 본다. 문제가 생기면 그 자리에서 해결하고 문서에 기록한다.
- **D-2주**: 후배가 첫 번째 작은 실험을 스스로 수행한다. 본인은 옆에서 관찰만 한다.
- **D-1주**: 마지막 정리. 남아 있는 질문에 답한다. 문서를 최종화한다.
- **D-0**: 떠나는 날. 연락처를 공유하고, 향후 질문이 있을 때 연락 가능한 채널(이메일)을 확정한다.

이 타임라인이 빡빡해 보이지만, 실제로 해보면 "충분히 시간을 썼다"고 느끼지 못할 때가 많다. 여유 있게 시작하는 것이 좋다.

**인수인계를 받는 사람의 책임.**

후배나 신규 담당자가 되는 경우, 본인도 수동적으로만 받아서는 안 된다. 다음을 적극적으로 한다.

1. **질문을 적극적으로 한다**: 선배가 설명하는 것을 듣고 이해가 안 되면 즉시 묻는다. 한 달 후에 묻는 것보다 훨씬 낫다.

2. **본인이 직접 실행해 본다**: 설명을 듣는 것과 직접 실행하는 것은 다르다. 선배가 있을 때 코드를 본인의 환경에서 돌려 본다. 작동하지 않으면 그 자리에서 해결한다.

3. **본인의 말로 요약한다**: 선배의 설명을 듣고 본인의 언어로 다시 정리한다. 선배가 이 요약을 검토하게 한다. 오해한 부분이 있으면 이 단계에서 교정된다.

4. **미래의 본인이 이해할 문서를 만든다**: 선배가 만든 `HANDOVER.md`에 본인의 관점에서 추가 메모를 남긴다. "선배는 당연하다고 했지만 나는 여기서 3시간 헤맸다" 같은 메모가 가치 있다.

5. **연락 창구 유지**: 졸업 후에도 선배에게 가끔 이메일을 보내 관계를 유지한다. "잘 지내시나요? 선배 코드에 대해 한 가지 질문이 있습니다" 정도의 가벼운 접촉이다.

**인수인계를 어렵게 만드는 심리적 요인.**

인수인계가 잘 안 되는 데는 심리적 이유가 있다. 이것을 이해하고 극복해야 한다.

- **떠나는 사람의 "필요성 유지" 욕구**: 본인이 떠난 후에도 "그 사람이 필요했다"는 평가를 원해서, 무의식적으로 문서화를 부실하게 한다. 이것은 장기적으로 본인의 평판에 해롭다. 완벽한 인수인계를 한 사람이 오히려 "저 사람 일 잘했다"고 기억된다.

- **남는 사람의 "본인 색깔" 욕구**: 후배는 선배의 코드와 방식을 "낡은 것"으로 보고 처음부터 다시 하고 싶어 할 수 있다. 이것은 때때로 합리적이지만, 종종 순수한 자존심의 문제다. 선배의 유산을 존중하면서 점진적으로 본인의 색깔을 더하는 것이 현명하다.

- **지도교수의 개입 부족**: 인수인계는 학생들 간의 일로만 치부되고 교수가 관여하지 않는 경우가 많다. 교수가 인수인계의 품질을 챙기지 않으면 지식 손실이 계속된다. 교수에게 "인수인계 미팅에 참석해 달라"고 요청하는 것이 학생의 권리다.

- **"급하게 떠나야 해서"의 합리화**: 졸업이 임박하면 마지막 몇 주에 논문 마무리, 졸업 준비, 새 직장 시작 같은 것들이 겹쳐 매우 바쁘다. 인수인계를 "시간이 없어서 제대로 못 했다"고 합리화한다. 하지만 앞서 말한 6-8주 전부터 시작하면 이 문제는 대부분 피할 수 있다.

**디지털 아카이브로서의 인수인계.**

최근에는 인수인계를 단순한 문서가 아니라 검색 가능한 디지털 아카이브로 만드는 연구실이 늘고 있다. Notion, Obsidian, GitHub Wiki 같은 플랫폼에 연구실 전체의 지식 베이스를 구축하고, 각 학생이 자신의 프로젝트 섹션을 관리한다. 졸업할 때 이 섹션을 마무리하고 후배가 이어받는다. 이 접근의 장점은 여러 명의 졸업생의 인수인계가 축적되어, 10년 후에는 연구실의 집단 지식이 된다는 것이다. 본인이 속한 연구실에 이런 시스템이 없다면, 본인이 시작하는 것도 좋은 기여다.

**연락 관계의 장기적 유지.**

인수인계는 떠나는 날에 끝나지 않는다. 이상적으로는 졸업 후에도 가끔 연락을 주고받으며 질문에 답할 수 있어야 한다. 떠나는 사람은 졸업 후 최소 6개월 동안은 후배의 질문에 답할 의무를 느껴야 한다. 이것은 계약이 아니라 매너의 문제다. 본인이 받은 인수인계에 감사한다면, 본인이 떠날 때도 같은 수준의 지원을 제공한다. 이 작은 선순환이 연구실의 문화를 만든다.

> 연구 인수인계는 눈에 잘 띄지 않는 협업의 형태지만, 연구실의 장기적 생산성을 결정한다. 본인이 떠나는 날, 후배가 본인의 프로젝트를 원활하게 이어갈 수 있다면, 본인의 박사 과정은 단순히 본인의 학위뿐 아니라 연구실의 자산을 키운 시간이 된다. 인수인계를 의무가 아닌 선물로 보는 것이 건강한 태도다. 본인도 같은 선물을 받았거나 앞으로 받을 것이기 때문이다.

## 원격 협업의 실전 — 다른 시간대, 다른 나라, 다른 언어

2020년 이후 연구 협업의 상당 부분이 원격으로 진행된다. 본인의 공동연구자가 옆 건물에 있을 수도 있지만, 미국, 유럽, 일본에 있을 가능성도 높다. 국제 협력이 점점 표준이 되는 시대에, **원격 협업의 기술**은 박사 과정의 필수 역량이다. 그러나 많은 학생이 이것을 배운 적이 없다. 대면 협업의 패턴을 그대로 원격에 적용하려다가 실패한다. 원격 협업은 대면 협업의 "화면 버전"이 아니다. 완전히 다른 규칙과 기법을 따른다.

<div class="highlight-box highlight-important">

**"화상회의만 하면 된다"는 오해.** 많은 학생이 원격 협업을 "가끔 Zoom으로 만나면 되는 것"으로 생각한다. 그러나 성공적인 원격 협업의 80%는 **비동기(async) 소통**에서 일어나고, 20%만이 실시간 회의다. 이 비율을 반대로 생각하면 원격 협업이 망가진다. 화면 회의를 아무리 잘해도, async 워크플로가 없으면 소통이 병목이 된다.

</div>

**동기(sync) vs 비동기(async) 소통의 구분.**

원격 협업의 첫 번째 원칙은 두 가지를 명확히 구분하는 것이다.

**동기 소통 (Sync)**: 실시간으로 주고받는 것. 예: Zoom 회의, 전화, 라이브 채팅.
- 장점: 빠른 의사결정, 감정적 뉘앙스 전달, 복잡한 토론
- 단점: 모든 사람의 시간을 한 번에 점유, 시차 문제, 집중 시간 방해

**비동기 소통 (Async)**: 답을 즉시 기대하지 않는 것. 예: 이메일, Slack 메시지, GitHub Issue, 문서 코멘트.
- 장점: 각자의 리듬 유지, 시차 무관, 기록이 남음, 깊이 있는 생각
- 단점: 답장이 늦을 수 있음, 오해 가능성, 긴급한 결정에 부적합

**원칙**: **동기는 꼭 필요할 때만, 비동기를 기본으로**. 많은 학생이 "잠깐 Zoom으로 이야기하자"를 남발하는데, 이것이 협업을 비효율적으로 만든다. "먼저 이메일이나 메시지로 시도하고, 그래도 해결 안 되면 회의"가 올바른 순서.

**어떤 소통이 동기, 어떤 것이 비동기인가.**

**동기가 필요한 경우**:
- 복잡한 연구 방향 결정 (말로 설명하는 것이 글보다 빠름)
- 갈등이나 오해 해결 (감정적 뉘앙스)
- 브레인스토밍 (실시간 상호작용)
- 팀 빌딩과 관계 형성
- 처음 만나는 협력자와의 친밀감 형성

**비동기가 더 나은 경우**:
- 코드 리뷰
- 논문 draft에 대한 피드백
- 데이터 공유와 분석
- 상태 업데이트 (누가 무엇을 했는지)
- 질문과 답변 (단순한 것)
- 결정 사항 공유
- 진도 보고

원격 협업의 80%가 이 두 번째 카테고리에 속한다.

**시차 관리의 실전.**

본인의 공동연구자가 미국에 있다면 시차가 13-16시간. 유럽은 7-8시간. 이 시차를 다루는 구체적 기법.

**기법 1: 겹치는 시간대 식별.**
본인이 한국(KST, UTC+9)에 있고 협력자가 미국 동부(EST, UTC-5)에 있다면, 본인의 밤 10시가 상대의 오전 8시다. 한국 아침은 미국 저녁. 이 "겹치는 시간대"를 식별하고, 동기 회의는 이 시간대에 잡는다.

**World Time Buddy** (worldtimebuddy.com) 같은 도구로 여러 시간대의 겹침을 시각적으로 확인.

**기법 2: 시차를 "24시간 릴레이"로 활용.**
시차가 장애물이 아니라 장점이 될 수 있다. 본인이 낮에 작업한 것을 상대방이 밤(본인의 밤)에 이어서 작업한다. 이렇게 하면 프로젝트가 "24시간 진행"된다. 마감이 임박한 시기에 특히 유용.

**기법 3: 회의 시간의 공정한 분배.**
항상 같은 시간에 회의하면 한쪽이 항상 불편하다. 매번 "저녁 10시 회의"는 본인만 힘들다. 돌아가며 불편한 시간을 분담하는 것이 공정하다. "이번 달은 본인이 새벽 회의, 다음 달은 상대가 새벽 회의".

**기법 4: 동기 회의의 최소화.**
시차가 큰 경우 동기 회의 빈도를 낮춘다. 주 1회가 이상적, 격주도 가능. 나머지는 async 문서로 소통.

**기법 5: 핸드오프(Handoff) 시점 명시.**
하루의 끝에 "오늘 이걸 했고, 내일 본인이 이걸 이어가면 된다"를 명확히 적는다. 이것이 24시간 릴레이를 가능하게 한다.

**원격 협업의 기본 도구 스택.**

본인의 협력 팀이 써야 할 도구들.

**1. 프로젝트 관리**: Notion, Asana, Trello, Linear
- 할 일 목록, 진행 상황, 마일스톤

**2. 문서 공동 편집**: Google Docs, Notion, Overleaf (LaTeX)
- 실시간 협업 문서 작성

**3. 코드 협업**: GitHub, GitLab
- Git 기반 버전 관리, PR 리뷰

**4. 메시징**: Slack, Microsoft Teams, Discord
- 비동기 채팅, 채널별 주제 분리

**5. 화상 회의**: Zoom, Google Meet, Microsoft Teams
- 정기 동기 회의

**6. 화면 공유와 설명**: Loom, CleanShot X
- 녹화된 비디오 설명 (async 시각적 소통)

**7. 지식 관리**: Notion, Obsidian, Confluence
- 팀의 집단 지식 저장

**8. 데이터 공유**: Google Drive, Dropbox, 학교 클라우드
- 파일 공유

본인의 팀이 이 모든 도구를 쓸 필요는 없다. 보통 3-5개의 핵심 도구로 충분. 중요한 것은 팀 전체가 같은 도구를 일관되게 쓰는 것.

**"Loom" 중심 비동기 소통.**

Loom이나 비슷한 화면 녹화 도구는 원격 협업의 "숨은 보석"이다. 본인이 5분간 화면 공유와 목소리로 설명하는 것이, 긴 이메일보다 훨씬 명확하다.

**Loom을 쓰는 경우**:
- 코드 리뷰 ("이 함수는 이렇게 작동합니다")
- 데이터 분석 설명
- 논문 draft 피드백 ("여기서 이 논리를 이해하기 어려웠어요")
- 실험 결과 보고
- 튜토리얼 공유

**Loom의 장점**:
- 동기 회의 없이 복잡한 내용 전달
- 상대방이 본인의 편한 시간에 시청
- 1.5배속으로 빠르게 재생 가능
- 기록으로 남음

5-10분 Loom 비디오 하나가 30분 Zoom 회의를 대체할 수 있다.

**원격 미팅의 기술.**

동기 회의를 해야 할 때, 원격 회의는 대면 회의와 다른 규칙이 필요하다.

**원칙 1: 명확한 아젠다.**
회의 전에 아젠다를 공유한다. 참석자가 준비할 시간을 준다. "그냥 이야기해 보자"는 원격에서 비효율적.

**원칙 2: 시간 엄수.**
원격 회의는 30분 또는 45분이 적절. 60분을 넘으면 집중력이 떨어진다. 대면보다 짧게 잡는다.

**원칙 3: 카메라 켜기.**
카메라를 켜야 비언어적 신호를 볼 수 있다. 예외는 있지만 기본은 카메라 on.

**원칙 4: 한 명씩 말하기.**
원격에서는 여러 명이 동시에 말하면 소리가 겹친다. 손들기 기능이나 명시적 "다음 본인이 말하겠습니다"를 활용.

**원칙 5: 기록 담당 지정.**
매 회의에 "이번 회의의 노트를 누가 작성할 것인가"를 정한다. 회의 후 노트를 공유한다. 이것이 async 팔로업의 기반.

**원칙 6: 결정 사항 명시.**
회의 끝에 "오늘 결정한 것"을 명확히 정리. "오늘 토론했지만 결정 못한 것"도 명시. 막연히 끝내지 않는다.

**원칙 7: 액션 아이템 할당.**
"누가 무엇을 언제까지 할 것인가"를 구체적으로. 애매한 "다음에 고려하자"는 실행되지 않는다.

**원격 회의 후의 Follow-up.**

회의는 끝이 아니라 시작이다. 회의 후 24시간 내에:

1. **회의 노트 공유**: 참석자 모두에게 공유
2. **액션 아이템 확인**: 각자의 할 일을 명확히
3. **다음 회의 일정**: 필요하면 미리 잡기
4. **대기 중인 질문**: 회의에서 해결 못한 것 리스트

이 팔로업이 없으면 회의의 가치가 절반 이하로 떨어진다.

**국제 협업의 언어 문제.**

영어가 모국어가 아닌 한국 학생에게, 국제 협업의 큰 장애물은 언어다. 대응 방법.

**대응 1: 비동기 중심.**
영어 말하기는 어렵지만 쓰기는 상대적으로 쉽다. 복잡한 설명은 글로 쓰고, 회의는 최소화한다. 글로 쓰면 본인이 생각할 시간이 있고, 문법 교정도 가능.

**대응 2: AI 도구 활용.**
본인이 쓴 영어를 DeepL, Grammarly, ChatGPT로 교정한다. 이것이 부끄럽거나 편법이 아니다. 요즘 모든 국제 연구자들이 AI 도구를 쓴다. 다만 AI의 출력을 그대로 복사하지 말고, 본인이 이해한 후 검토한다.

**대응 3: 회의 전 준비.**
본인이 회의에서 말할 것을 미리 글로 써본다. 중요한 발표는 스크립트를 준비한다. 즉흥적으로 말하는 것보다 준비된 내용이 훨씬 정확하게 전달된다.

**대응 4: 회의 중 채팅 활용.**
말로 설명하기 어려운 것은 채팅에 쓰거나, "잠깐, 글로 보내겠습니다"라고 말한다. 구어와 문어를 병행.

**대응 5: 회의 후 확인.**
회의 후 "제가 이해한 것은 X인데, 맞나요?"라고 글로 확인한다. 오해를 방지.

**대응 6: 완벽함 추구하지 않기.**
본인의 영어가 완벽하지 않아도 괜찮다. 국제 협력자들은 본인이 모국어가 아니라는 것을 안다. 의미 전달이 목표이지 문법적 완벽이 아니다.

**문화적 차이의 관리.**

언어뿐 아니라 문화도 원격 협업에 영향을 준다.

**한국 문화의 특징과 국제 협업**:
- **위계에 대한 강한 인식**: 한국 학생은 교수에게 매우 공손하게 소통. 서양에서는 이것이 "과도하게 공식적"으로 느껴질 수 있다.
- **"네"의 의미**: 한국에서 "네"는 "들었다"의 의미일 수 있는데, 서양에서는 "동의한다"로 해석된다. 오해의 원인.
- **부정적 피드백**: 한국인은 직접적 비판을 피하는 경향. 서양 협력자는 명확한 "No"를 기대한다.
- **침묵**: 한국에서 회의 중 침묵은 생각하는 시간. 서양에서는 "아무 할 말이 없다"로 해석.
- **시간 약속**: 한국은 시간 엄수를 중요시. 일부 지역(지중해, 남미)은 더 유연.

**적응 팁**:
- 본인의 의사를 더 명확하게 표현 (특히 "No" 또는 불확실성)
- 회의 중 본인의 생각을 적극적으로 말하기 (침묵이 기본값이 아니게)
- 문화적 차이가 있을 때 묻기: "본인의 문화에서는 X가 일반적인데, 귀하의 문화는 어떤가요?"

**원격 협업의 관계 형성.**

대면 협업에서는 커피 브레이크와 점심 식사가 관계 형성의 장이다. 원격에서는 이것이 없다. 의식적으로 관계 형성 시간을 만들어야 한다.

**전략 1: "Virtual Coffee Chat".**
월 1회, 업무 이야기 없이 30분 수다. 본인의 일상, 취미, 가족 이야기. 이것이 유대감을 만든다.

**전략 2: 회의 시작의 5분 잡담.**
정기 회의 시작 전 5분을 "how was your weekend?" 같은 잡담에 쓴다. 이것이 인간적 연결.

**전략 3: 개인적 관심 공유.**
본인이 읽고 있는 책, 본 영화, 관심 있는 주제를 공유. 작은 것부터 시작.

**전략 4: 기념일 축하.**
협력자의 생일, 승진, 논문 채택 등을 기억하고 축하. 작은 메시지가 큰 영향.

**전략 5: 대면 만남의 기회 활용.**
학회에서 만날 기회가 있으면 반드시 만난다. 한 번의 대면 만남이 이후 원격 협업을 훨씬 수월하게 만든다.

**원격 협업의 흔한 실패 패턴.**

**실패 1: 소통 빈도가 너무 낮음.**
"이메일을 보냈는데 답장이 없으니 기다린다"고 2주 보내기. 원격에서는 의식적으로 자주 소통해야 한다. 주 1회는 최소.

**실패 2: 너무 많은 회의.**
반대로 매일 회의를 잡음. 각자의 집중 시간을 파괴. "이 회의가 정말 필요한가?"를 매번 묻는다.

**실패 3: 시차 무시.**
상대의 새벽 3시에 회의를 잡기. 배려 부족의 신호. 상대의 시간대를 확인하고 공정한 시간을 제안.

**실패 4: 글 없는 소통.**
모든 것을 말로만 결정하고 기록하지 않음. 며칠 후 "무엇을 결정했는지" 아무도 기억 못 함. 회의록은 필수.

**실패 5: 피드백의 부재.**
상대의 작업을 받고 "잘 받았다" 외에 반응 없음. 오랜 시간이 지난 후 지적하면 관계가 망가진다. 빠르고 명확한 피드백.

**실패 6: 격식의 과잉 또는 부족.**
너무 격식적이어서 딱딱해지거나, 너무 캐주얼해서 무례해지기. 상대의 스타일을 관찰하고 맞춘다.

**실패 7: 갈등의 회피.**
문제가 있을 때 말하지 않고 넘김. 쌓이면 폭발한다. 작은 불편함도 초기에 소통.

**원격 협업의 장점 활용.**

단점만 있는 것이 아니다. 원격 협업의 장점을 활용한다.

1. **넓은 협력자 풀**: 지역에 얽매이지 않고 세계 최고 전문가와 협력 가능.
2. **유연한 시간**: 본인의 최적 집중 시간에 일할 수 있음.
3. **기록의 자연스러움**: 모든 소통이 글로 남아 나중에 참고 가능.
4. **비용 절감**: 출장비, 이동 시간 절약.
5. **다양성**: 다른 문화와 관점을 배울 수 있음.

이 장점을 의식적으로 활용하면 대면 협업보다 더 효과적일 수 있다.

**원격 협업 기술의 박사 과정 이후 가치.**

원격 협업은 박사 과정뿐 아니라 졸업 후에도 필수 기술이다. 포닥(해외), 산업체 R&D, 컨설팅, 스타트업 모두 원격 요소가 있다. 박사 과정 중 이 기술을 익히면 평생의 자산이 된다.

> 원격 협업은 2020년 이후 연구의 기본 모드가 되었다. 본인이 "대면만 선호"하는 태도로는 국제 협력의 기회를 놓친다. async와 sync의 구분, 시차 관리, 원격 회의 기술, 문화적 차이 대응 — 이런 기술들은 학교에서 가르치지 않지만 배워야 한다. 박사 과정 중 최소 한 번은 국제 원격 협업을 경험해 볼 것을 권한다. 그 경험이 본인의 연구자 정체성을 글로벌 수준으로 끌어올린다.

## 본인의 협업 포트폴리오 — 어떤 협업에 본인의 시간을 쓸 것인가

박사 과정 중반부터 본인에게 협업 제안이 늘어난다. 본인의 지도교수의 프로젝트, 본인의 연구실의 다른 박사의 프로젝트, 본인의 외부 협력자의 제안, 본인의 학회에서 만난 사람의 초대, 본인의 산학 과제의 참여. 본인이 모든 협업 제안을 수락하면 본인의 박사 연구가 본인의 외부 협업에 잠식된다. 본인이 모든 제안을 거절하면 본인이 본인의 분야의 네트워크를 잃는다. 본인이 본인의 협업 포트폴리오를 의식적으로 관리해야 한다.

### 협업 포트폴리오의 4가지 카테고리

본인의 모든 협업이 같은 가치를 주지 않는다. 본인이 본인의 협업을 4가지 카테고리로 분류하고 각 카테고리에 시간을 배분한다.

**카테고리 1: 핵심 협업 (Core Collaboration) — 본인의 박사 논문에 직접 기여**

본인의 박사 학위 논문의 본인의 핵심 장의 결과를 만드는 협업. 본인의 지도교수와의 협업, 본인의 박사 논문의 공동저자와의 협업.

- 본인의 시간 비율: 50-60%
- 본인의 수락 기준: 본인의 박사 논문의 진전에 직접 기여하면 수락.
- 본인의 예시: 본인의 박사 논문의 장 1, 장 2, 장 3의 각 논문.

**카테고리 2: 학습 협업 (Learning Collaboration) — 본인이 새 기법을 배우는 것**

본인이 본인의 분야의 새 기법, 새 도구, 새 시각을 학습하는 협업. 본인의 박사 논문에 직접 기여하지 않지만 본인의 능력을 확장.

- 본인의 시간 비율: 15-20%
- 본인의 수락 기준: 본인의 새 학습이 본인의 박사 후반 또는 본인의 박사 후의 본인의 연구에 활용 가능하면 수락.
- 본인의 예시: 본인의 분야 외의 연구실과 본인의 공동 프로젝트, 본인의 새 데이터셋의 탐색.

**카테고리 3: 네트워크 협업 (Network Collaboration) — 본인의 관계를 만드는 것**

본인의 분야의 시니어 학자, 본인의 잠재적 박사 후 멘토, 본인의 박사 후 첫 자리의 잠재적 상사와의 협업. 본인의 박사 논문에 직접 기여보다 본인의 관계의 가치가 더 크다.

- 본인의 시간 비율: 10-15%
- 본인의 수락 기준: 본인의 협업 상대가 본인의 박사 후의 본인의 경력에 중요하면 수락.
- 본인의 예시: 본인의 분야의 유명 학자와의 본인의 짧은 공동 프로젝트, 본인의 워크숍의 본인의 공동 조직.

**카테고리 4: 봉사 협업 (Service Collaboration) — 본인의 연구실/공동체에 기여**

본인의 연구실의 후배의 프로젝트 지원, 본인의 학회의 조직 위원, 본인의 커뮤니티의 자원봉사. 본인의 박사 논문의 기여는 적지만 본인의 공동체의 건강에 기여.

- 본인의 시간 비율: 5-10%
- 본인의 수락 기준: 본인이 본인의 공동체의 한 부분이라는 것을 유지할 정도만 수락.
- 본인의 예시: 본인의 연구실 후배의 첫 논문 지원, 본인의 학회의 학생 위원, 본인의 리뷰 봉사 (ch42와 연결).

### 포트폴리오 설계의 5가지 원칙

**원칙 1: 본인의 박사 시기에 맞춰 비율 조정**

본인의 박사 1-2년차: 핵심 40%, 학습 30%, 네트워크 10%, 봉사 20%. 본인이 본인의 박사 주제가 아직 정해지지 않은 시기. 본인이 학습에 더 큰 비율.

본인의 박사 3-4년차: 핵심 60%, 학습 15%, 네트워크 15%, 봉사 10%. 본인이 본인의 박사 논문의 핵심 결과를 만드는 시기. 본인이 핵심에 집중.

본인의 박사 5-7년차: 핵심 70%, 학습 5%, 네트워크 20%, 봉사 5%. 본인이 본인의 박사 논문을 완성하고 본인의 박사 후의 자리를 준비하는 시기. 본인이 네트워크를 키운다.
