# AI 입문: 제로에서 시작하기

"나는 코딩을 한 번도 해본 적이 없는데, AI를 할 수 있을까?"

이 질문을 하는 공학 대학원생이 생각보다 많다. 구조공학을 전공하는데 유한요소 해석은 상용 소프트웨어로 돌리고, 데이터 정리는 엑셀로 하고, 그래프는 Origin이나 Excel로 그린다. 코드 한 줄 안 짜도 석사 논문을 쓸 수 있다. 실제로 많은 선배들이 그렇게 졸업했다.

그런데 요즘 학회에 가면 분위기가 다르다. 발표 슬라이드마다 "deep learning", "machine learning", "neural network"가 등장하고, 리뷰어가 "AI 기반 비교 분석은 없는가?"라고 묻는다. 교수님도 "요즘 트렌드를 좀 반영해 보자"라고 한다. 갑자기 불안해진다. 나만 뒤처지는 건 아닌가?

이 장은 바로 그 상황에 있는 사람을 위해 썼다. 프로그래밍 경험이 전무한 공학 대학원생이, AI를 연구 도구로 활용할 수 있는 수준까지 도달하는 현실적인 로드맵을 제시한다. 핵심 메시지는 이것이다. "AI를 모르면 뒤처진다"가 아니라, **"AI를 알면 새로운 문이 열린다."**

## AI가 공학 연구에 왜 필요한가

먼저 냉정하게 질문하자. AI 없이도 공학 연구를 할 수 있는가? 물론 할 수 있다. 지난 수십 년간 공학자들은 물리 기반 모델, 해석적 해법, 실험 데이터만으로 훌륭한 연구를 해왔다. 유한요소법은 AI 없이도 구조물의 거동을 예측하고, 열전달 방정식은 AI 없이도 온도 분포를 계산한다. AI가 없다고 연구를 못하는 것은 아니다.

그러면 왜 굳이 AI를 배워야 하는가? 세 가지 이유가 있다.

**첫째, 데이터의 폭증이다.** 센서 기술이 발전하면서 공학 시스템에서 수집되는 데이터의 양이 폭발적으로 증가했다. 교량 하나에 수백 개의 센서가 붙어 있고, 매초 진동 데이터가 쌓인다. 공정 라인 하나에서 수천 개의 공정 변수가 실시간으로 기록된다. 이 데이터를 엑셀로 정리하고 눈으로 패턴을 찾는 것은 물리적으로 불가능하다. AI는 이 대량 데이터에서 인간이 발견하지 못하는 패턴을 찾아낸다.

**둘째, 전통적 방법의 한계가 보이기 시작했다.** 유한요소 해석은 강력하지만, 한 번 돌리는 데 수 시간에서 수일이 걸리는 경우가 있다. 설계 최적화를 하려면 수백, 수천 번을 돌려야 하는데, 시간과 계산 비용이 감당이 안 된다. AI 기반 대리 모델(surrogate model, 복잡한 시뮬레이션을 대신하여 빠르게 결과를 근사하는 간소화된 모델)은 유한요소 해석 결과를 학습해서, 새로운 입력에 대한 결과를 밀리초 단위로 예측한다. 정확도를 약간 희생하는 대신 속도를 수천 배 높인다. 이것은 전통적 방법으로는 불가능했던 대규모 최적화를 가능하게 한다.

**셋째, AI로 가능해진 완전히 새로운 연구 영역이 있다.** 이미지에서 균열을 자동 탐지하는 것, 시계열 센서 데이터로 설비 고장을 예측하는 것, 재료의 미세 구조 이미지에서 물성을 예측하는 것. 이런 문제들은 전통적 공학 방법론만으로는 접근하기 어려웠다. AI는 이런 문제를 풀 수 있는 새로운 도구 상자를 제공한다.

여기서 강조하고 싶은 것이 있다. AI는 공학의 물리 기반 지식을 대체하는 것이 아니다. 오히려 보완하는 것이다. 물리를 아는 공학자가 AI를 도구로 쓰면, AI만 아는 사람이나 물리만 아는 사람보다 훨씬 강력하다. 구조역학을 이해하는 사람이 딥러닝으로 구조물 손상을 탐지하면, 물리적으로 말이 안 되는 예측을 걸러낼 수 있다. 이것이 공학 대학원생이 AI를 배워야 하는 진짜 이유다. 당신은 이미 도메인 지식이라는 강력한 무기를 가지고 있다. AI는 그 무기의 사거리를 넓혀주는 도구다.

## 프로그래밍 경험 제로에서 시작하기

솔직하게 말하자. 프로그래밍을 한 번도 안 해본 상태에서 AI를 하겠다는 것은, 수영을 한 번도 안 해본 사람이 바다를 건너겠다는 것과 비슷하다. 불가능한가? 아니다. 하지만 물에 먼저 들어가야 한다. 영상만 보고 수영을 배울 수 없듯이, 코딩도 직접 해봐야 배운다.

좋은 소식은, 2020년대의 프로그래밍 학습 환경은 10년 전과 비교할 수 없을 정도로 좋아졌다는 것이다. 무료 강의, 대화형 학습 플랫폼, 그리고 무엇보다 AI 코딩 도우미가 있다. 과거에는 에러 메시지 하나를 해독하는 데 30분을 구글링해야 했다면, 지금은 AI에게 "이 에러가 뭐야?"라고 물으면 즉시 답을 준다.

### Python을 선택해야 하는 이유

공학 연구에서 프로그래밍 언어 선택은 사실상 정해져 있다. Python이다. C++이 더 빠르고, MATLAB이 공학 분야에서 오래 쓰였고, Julia가 더 현대적이라는 주장은 모두 맞다. 하지만 AI/ML 생태계는 Python 중심으로 구축되어 있다. TensorFlow, PyTorch, scikit-learn, pandas, numpy 등 이 모든 핵심 라이브러리가 Python이다. AI를 하겠다면 Python이 사실상 유일한 선택이다.

MATLAB에서 Python으로 넘어오는 것이 걱정되는 사람도 있을 것이다. 문법이 다르고, 인덱스가 0부터 시작하고, 행렬 연산 방식이 다르다. 하지만 핵심 개념은 같다. 변수, 반복문, 함수, 배열. MATLAB에서 이것들을 다뤄봤다면 Python 전환은 며칠이면 된다.

### Python 설치: 첫 관문

Python 설치는 의외로 많은 초보자가 막히는 지점이다. Python 공식 사이트에서 다운받을 수도 있고, Anaconda라는 배포판을 쓸 수도 있다. 추천은 Anaconda다. Python뿐 아니라 과학 계산에 필요한 라이브러리(numpy, pandas, matplotlib 등)가 한 번에 설치되고, 가상 환경 관리도 편리하다.

설치 후 해야 할 첫 번째 일은 터미널(또는 Anaconda Prompt)을 열고 `python --version`을 입력하는 것이다. 버전 번호가 뜨면 설치 성공이다. 이 한 줄을 성공적으로 실행하는 것이 프로그래밍 여정의 첫 발자국이다.

그 다음은 Jupyter Notebook을 열어보는 것이다. 터미널에서 `jupyter notebook`을 입력하면 웹 브라우저에 인터랙티브 코딩 환경이 열린다. 셀에 코드를 쓰고, 실행하고, 결과를 바로 확인할 수 있다. 이것이 초보자에게 가장 적합한 코딩 환경이다.

### 첫 코드: Hello World를 넘어서

대부분의 프로그래밍 교재가 `print("Hello World")`로 시작한다. 이것은 전통이지, 공학 대학원생에게 실질적 도움은 안 된다. 대신 이렇게 시작해 보자.

```python
# 보의 최대 처짐 계산 (단순 지지보, 중앙 집중하중)
P = 10000      # 하중 (N)
L = 5.0        # 스팬 (m)
E = 200e9      # 탄성계수 (Pa) - 강재
I = 8.33e-6    # 단면 2차 모멘트 (m^4)

delta_max = (P * L**3) / (48 * E * I)
print(f"최대 처짐: {delta_max*1000:.2f} mm")
```

이 코드는 단 6줄이지만, 프로그래밍의 핵심 개념 세 가지를 담고 있다. 변수에 값을 저장하고(`P = 10000`), 수식을 계산하고(`delta_max = ...`), 결과를 출력한다(`print(...)`). 무엇보다 자기 전공의 문제를 풀고 있다는 점에서 동기부여가 된다.

## 바이브 코딩이란?

2024년 이후 프로그래밍의 풍경이 근본적으로 바뀌고 있다. "바이브 코딩(Vibe Coding)"이라는 개념이 등장했다. 이것은 자연어로 원하는 것을 설명하면 AI가 코드를 생성해 주는 방식이다. 코드를 한 줄 한 줄 직접 작성하는 대신, "이 데이터로 Random Forest 모델을 학습시키고, 교차 검증(cross-validation, 데이터를 여러 묶음으로 나눠서 번갈아 학습과 검증을 반복하는 방법) 결과를 그래프로 보여줘"라고 말하면 AI가 전체 코드를 만들어준다.

이것이 가능해진 것은 대규모 언어 모델(LLM)의 발전 덕분이다. ChatGPT, Claude, GitHub Copilot 같은 도구들이 프로그래밍 코드를 놀라울 정도로 정확하게 생성할 수 있게 되었다. 특히 Python 과학 계산 코드는 이 도구들이 매우 잘하는 영역이다.

바이브 코딩이 공학 대학원생에게 의미하는 바는 크다. 과거에는 "코딩을 못하면 AI 연구를 못한다"였지만, 이제는 "문제를 명확하게 설명할 수 있으면 AI 연구 코드를 만들 수 있다"로 바뀌고 있다. 이것은 진입 장벽을 극적으로 낮춘다.

하지만 여기서 반드시 짚어야 할 것이 있다. **바이브 코딩은 코딩을 완전히 대체하지 않는다.** AI가 생성한 코드가 올바른지 검증할 수 있어야 한다. AI가 scikit-learn의 `train_test_split`(데이터를 학습용과 검증용으로 나누는 함수)에서 `random_state`(난수 고정값)를 설정하지 않았다면, 실행할 때마다 결과가 달라져서 재현이 되지 않는다는 것을 알아야 한다. AI가 데이터 정규화를 테스트 세트에도 적용해서 데이터 누수(data leakage, 검증용 데이터의 정보가 학습 과정에 유출되어 성능이 실제보다 부풀려지는 문제)를 일으켰다면, 그것을 잡아낼 수 있어야 한다.

바이브 코딩의 올바른 사용법은 이렇다. 기본적인 프로그래밍 이해(변수, 함수, 자료구조)는 갖추되, 반복적이고 시간 소모적인 코딩 작업은 AI에게 맡긴다. 코드의 큰 구조와 논리를 설계하는 것은 연구자의 몫이고, 세부 구현은 AI와 협업한다. AI가 만든 코드를 무비판적으로 사용하는 것이 아니라, 이해하고 검증한 후에 사용한다.

### 1단계: Python 기초

이 단계의 목표는 "코드를 읽고 쓸 수 있는 수준"이다. 유창할 필요 없다. 더듬더듬이라도 된다. 핵심 개념 다섯 가지만 확실히 잡으면 된다.

**변수와 자료형.** `x = 3.14`라고 쓰면 x라는 이름에 3.14라는 값이 저장된다. 숫자(int, float), 문자열(str), 리스트(list), 딕셔너리(dict) 네 가지 자료형만 알면 된다. 리스트는 `[1, 2, 3]`처럼 여러 값을 순서대로 저장하고, 딕셔너리는 `{"name": "concrete", "strength": 30}`처럼 키-값 쌍을 저장한다.

**조건문.** `if strength > 21: print("OK")`처럼 조건에 따라 다른 동작을 하게 한다. `if`, `elif`, `else` 세 키워드만 알면 된다.

**반복문.** `for i in range(10): print(i)`처럼 같은 동작을 반복한다. 리스트의 각 원소에 대해 같은 계산을 하거나, 파라미터를 바꿔가며 시뮬레이션을 돌릴 때 필수적이다.

**함수.** `def calculate_deflection(P, L, E, I): return (P * L**3) / (48 * E * I)`처럼 반복되는 계산을 함수로 묶는다. 한 번 정의하면 어디서든 호출할 수 있다.

**파일 입출력.** `open("data.csv")`로 파일을 열고, 데이터를 읽거나 결과를 저장한다. 나중에 Pandas를 쓰면 훨씬 편해지지만, 기본 원리를 알아두면 좋다.

이 다섯 가지를 합치면 이미 상당한 것을 할 수 있다. 실험 데이터를 읽어서, 조건에 맞는 데이터만 걸러내고, 각 데이터에 대해 계산을 수행하고, 결과를 파일로 저장하는 것이 가능해진다.

### 2단계: NumPy와 Pandas

NumPy는 Python에서 행렬 연산을 가능하게 하는 라이브러리다. MATLAB의 행렬 연산에 해당한다. `import numpy as np`로 불러오고, `np.array([1, 2, 3])`으로 배열을 만든다. 행렬 곱(`np.dot` 또는 `@`), 전치(`A.T`), 역행렬(`np.linalg.inv(A)`) 등 공학 계산에 필수적인 연산을 지원한다.

Pandas는 표 형태의 데이터를 다루는 라이브러리다. Excel의 스프레드시트를 Python으로 가져왔다고 생각하면 된다. `pd.read_csv("experiment_data.csv")`로 CSV 파일을 읽고, `df["strength"].mean()`으로 평균을 구하고, `df[df["temperature"] > 25]`로 조건 필터링을 한다.

이 단계에서 반드시 연습해야 할 것은 자기 실험 데이터를 Pandas로 불러와서 기본적인 통계(평균, 표준편차, 상관관계)를 계산해 보는 것이다. 엑셀에서 하던 것과 같은 작업이지만, 코드로 하면 재현 가능하고, 자동화할 수 있고, 대용량 데이터에도 적용할 수 있다.

### 3단계: Matplotlib

데이터를 분석했으면 시각화해야 한다. Matplotlib은 Python의 가장 기본적인 시각화 라이브러리다. `plt.scatter(x, y)`로 산점도, `plt.plot(x, y)`로 선 그래프, `plt.hist(data)`로 히스토그램을 그린다.

공학 논문에서 Matplotlib이 중요한 이유가 있다. Origin이나 Excel 그래프는 세부 설정에 한계가 있고, 매번 수동으로 그려야 한다. Matplotlib은 코드로 그래프를 만들기 때문에, 데이터가 바뀌어도 코드만 다시 실행하면 동일한 스타일의 그래프가 자동으로 생성된다. 논문 수정 과정에서 그래프를 수십 번 다시 그려야 할 때, 이 자동화는 시간을 엄청나게 절약해 준다.

### 4단계: scikit-learn

scikit-learn은 머신러닝의 "표준 도구 상자"다. 이 라이브러리 하나로 회귀, 분류, 클러스터링, 차원 축소, 전처리, 모델 선택 등 대부분의 전통 ML 작업을 수행할 수 있다.

scikit-learn의 가장 좋은 점은 API(Application Programming Interface, 라이브러리를 사용하기 위해 정해진 함수 호출 규칙)가 일관적이라는 것이다. 어떤 알고리즘을 쓰든 `model.fit(X_train, y_train)`으로 학습하고, `model.predict(X_test)`로 예측한다. Random Forest든 SVM(Support Vector Machine, 데이터를 구분하는 최적의 경계면을 찾는 알고리즘)이든 Gradient Boosting(약한 모델들을 순차적으로 쌓아올려 강한 모델을 만드는 알고리즘)이든 같은 인터페이스다. 알고리즘의 내부 원리를 모르더라도 일단 돌려볼 수 있다.

### 5단계: PyTorch 기초

딥러닝이 필요한 연구를 하게 되면 PyTorch를 배워야 한다. TensorFlow도 있지만, 연구 분야에서는 PyTorch가 사실상 표준이다. 최근 논문의 코드 대부분이 PyTorch로 작성되어 있다.

PyTorch는 scikit-learn보다 배우기 어렵다. 텐서(tensor, 다차원 배열을 GPU에서 빠르게 연산할 수 있게 한 자료 구조) 개념, 자동 미분(autograd, 수식의 미분값을 컴퓨터가 자동으로 계산해주는 기능), 신경망 정의(nn.Module), 학습 루프(forward-backward-update, 예측 후 오차를 역방향으로 전파하고 가중치를 갱신하는 반복 과정) 등을 이해해야 한다. 하지만 기본적인 신경망을 만들고 학습시키는 것 자체는 수십 줄의 코드면 된다. 처음에는 튜토리얼 코드를 따라 치면서 전체 흐름을 이해하고, 점차 자기 데이터에 적용하면 된다.

### 6단계: 내 데이터 적용

가장 중요한 마지막 단계다. 튜토리얼의 예제 데이터가 아니라 자기 연구의 실제 데이터에 ML을 적용해 본다. 여기서 진짜 학습이 일어난다. 예제 데이터는 깨끗하게 정리되어 있지만, 실제 데이터는 지저분하다. 결측치가 있고, 이상치가 있고, 변수의 스케일이 제각각이고, 데이터 양이 부족하다. 이런 현실의 문제를 해결하는 과정에서 실력이 급격히 성장한다.

한 가지 더 강조하고 싶은 것이 있다. **에러를 두려워하지 말아야 한다.** 프로그래밍에서 에러는 실패가 아니라 피드백이다. 경험 많은 개발자도 하루에 수십 번 에러를 만난다. 차이는 에러를 만났을 때의 대응이다. 초보자는 에러가 나면 당황하고, 전체 코드를 지우고 다시 시작하거나, 그냥 포기한다. 숙련자는 에러 메시지를 읽고, 어떤 줄에서 어떤 문제가 발생했는지 파악하고, 그 부분만 수정한다.

Python의 에러 메시지는 사실 매우 친절하다. `TypeError: unsupported operand type(s) for +: 'int' and 'str'`라는 에러 메시지는 "숫자와 문자열을 더하려고 했는데, 그건 안 된다"고 정확하게 알려주고 있다. 에러 메시지를 읽는 습관을 들이면, 디버깅 시간이 절반으로 줄어든다.

에러 메시지를 이해하지 못하겠으면, 그대로 복사해서 AI에게 붙여넣으면 된다. "이 에러 메시지가 무슨 뜻이야? 어떻게 해결해야 해?"라고 물으면, 대부분의 경우 정확한 해결책을 알려준다. 이것이 2020년대 프로그래밍 학습의 가장 큰 장점이다.

## AI 활용 연구 워크플로우

AI를 연구에 활용하는 방식은 "코딩"에만 국한되지 않는다. 연구의 전체 과정, 데이터 수집부터 논문 작성까지에서 AI는 보조 역할을 할 수 있다. 다만, 모든 단계에서 **최종 판단은 반드시 연구자가 해야 한다**는 원칙을 잊으면 안 된다.

### 데이터 전처리 단계

실험 데이터나 시뮬레이션 데이터를 정리하는 것은 연구에서 가장 시간이 많이 드는 작업 중 하나다. 결측치를 어떻게 처리할지, 이상치를 제거할지 말지, 데이터를 어떤 형태로 변환할지. 이런 결정은 연구자가 해야 하지만, 실제 코딩은 AI가 도와줄 수 있다.

"이 CSV 파일에서 결측치가 있는 행을 찾아서 보여줘"라고 AI에게 요청하면, `df.isnull().sum()` 같은 코드를 즉시 생성한다. "결측치를 선형 보간으로 채워줘"라고 하면 `df.interpolate()` 코드를 만든다. 이런 단순 작업을 AI에게 맡기면 전처리 시간을 크게 줄일 수 있다.

### 모델 개발 단계

AI 코딩 도구는 모델 개발에서 특히 강력하다. "이 데이터에 Random Forest를 적용하고, 5-fold 교차 검증으로 평가해 줘"라고 하면 전체 코드를 생성한다. 하이퍼파라미터 튜닝(hyperparameter tuning, 학습률이나 층 수 같은 사전 설정값을 최적으로 조정하는 과정) 코드, 학습 곡선 시각화 코드, 모델 비교 코드 등을 빠르게 만들 수 있다.

하지만 여기서 주의가 필요하다. AI가 생성한 모델 코드가 자기 데이터와 문제에 적합한지를 반드시 확인해야 한다. AI는 "가장 일반적인" 접근법을 제안하는 경향이 있다. 자기 데이터의 특수성(소규모, 불균형, 물리적 제약 등)을 반영하는 것은 연구자의 몫이다.

### 시각화 단계

논문에 들어갈 그래프를 만드는 것은 AI가 잘하는 영역이다. "이 데이터의 산점도를 그려주는데, x축은 예측값, y축은 실측값으로 하고, 대각선을 그려줘. 폰트는 Arial 12pt, 해상도는 300 dpi로 해줘"라고 요청하면, 논문 수준의 그래프 코드를 만들어준다.

Matplotlib의 세부 설정(축 라벨 크기, 범례 위치, 색상 팔레트 등)은 외우기 어렵고 매번 검색해야 하는 부분이다. AI를 쓰면 이런 세부 설정을 일일이 찾을 필요 없이 자연어로 요청할 수 있다.

### 논문 작성 단계

AI를 논문 작성에 활용하는 것은 논란이 있는 영역이다. 하지만 적절하게 활용하면 큰 도움이 된다. 영어 교정(문법, 어색한 표현), 문장 다듬기, 문단 구조 개선 등은 AI가 효과적으로 도와줄 수 있다. 다만 **내용 자체(연구 결과의 해석, 주장, 결론)는 반드시 연구자가 직접 작성해야 한다.** AI가 생성한 학술적 문장을 그대로 쓰면 표절 이슈가 발생할 수 있고, 무엇보다 자기 연구를 자기 언어로 설명할 수 있어야 한다.

<div class="highlight-box warning">
<span class="highlight-box-icon">⚠️</span>
<div class="highlight-box-content">
<p><strong>AI는 도구이지, 연구자가 아니다</strong></p>
<p>AI가 생성한 코드, 분석, 문장을 무비판적으로 사용하는 순간, 연구의 신뢰성이 무너진다. AI는 그럴듯한 틀린 답을 자신 있게 말하는 경우가 있다(hallucination, 환각 현상이라고도 부르며, AI가 사실이 아닌 내용을 마치 사실인 것처럼 생성하는 현상). 특히 도메인 특화 지식이 필요한 공학 문제에서, AI의 일반적 제안이 물리적으로 말이 안 되는 경우가 드물지 않다. 모든 AI 출력물은 반드시 전문가(당신)의 검증을 거쳐야 한다.</p>

## ML을 위한 수학 사전 준비

AI/ML을 시작할 때 흔히 듣는 질문이 "수학을 얼마나 알아야 하는가"이다. 답은 "분야에 따라, 깊이에 따라 다르다"이다. 도구로서 ML을 사용하는 것과 ML 자체를 연구하는 것은 요구되는 수학 수준이 다르다. 공학 연구자가 ML을 도구로 활용하기 위한 수학 수준은 생각보다 높지 않다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>"수학이 부족하다"는 핑계의 함정</strong></p>
    <p>많은 학생이 "수학을 다 익힌 후에 ML을 시작하겠다"고 한다. 이것은 영원히 시작하지 않는 함정이다. 수학을 모두 익히려면 1년 이상 걸리지만, ML 도구로 첫 결과를 내는 데는 한 달이면 충분하다. 수학과 ML을 동시에 학습하는 것이 더 효과적이다. ML 코드를 짜다가 막히면 그때 필요한 수학 개념만 찾아보는 방식이다.</p>
  </div>
</div>

**필수 수학: 학부 1-2학년 수준이면 충분하다.**

| 수학 분야 | 필요한 깊이 | ML에서의 활용 |
|-----------|------------|--------------|
| 선형대수 | 행렬 연산, 행렬 곱셈, 전치, 역행렬, 고윳값 | 모든 데이터가 행렬, 신경망 계산의 기본 |
| 미적분 | 편미분, 체인룰, 그래디언트 | 역전파(backpropagation), 손실 함수 최적화 |
| 확률/통계 | 평균, 분산, 분포, 베이즈 정리 | 데이터 이해, 평가 지표, 베이지안 ML |
| 최적화 기초 | 경사하강법, 볼록 함수 개념 | 모델 학습의 본질 |

이 정도면 충분히 ML을 도구로 사용할 수 있다. PyTorch나 scikit-learn이 복잡한 수학 계산을 모두 자동으로 해 준다. 직접 그래디언트를 계산할 일은 거의 없다.

**고급 수학: 깊이 들어가려면 추가**.

| 수학 분야 | 언제 필요한가 |
|-----------|--------------|
| 다변수 미적분 | 신경망의 역전파 깊이 이해 |
| 행렬 미적분 | 새 모델 아키텍처 설계 |
| 정보 이론 | 손실 함수 설계, 표현 학습 |
| 측도론과 측도이론 | 베이지안 딥러닝, 확률적 ML 이론 |
| 함수해석학 | 일반화 이론, RKHS 기반 방법 |
| 미분기하 | 매니폴드 학습, 정보 기하 |

위의 고급 수학은 ML 자체를 연구할 때 필요하다. 공학 연구자가 ML을 도구로 사용한다면 거의 필요 없다.

**수학 학습의 추천 자원.**

첫째, **선형대수**: Gilbert Strang의 MIT 18.06 강의(YouTube 무료, 한글 자막). 3Blue1Brown의 "Essence of Linear Algebra"(짧고 직관적). 길버트 스트랭의 교과서.

둘째, **미적분**: 3Blue1Brown의 "Essence of Calculus". MIT 18.01/18.02 강의. 한국어로는 김상현 교수의 미적분학 강의(YouTube).

셋째, **확률/통계**: Joseph Blitzstein의 Harvard Stat 110 강의. "Introduction to Probability"(Blitzstein, Hwang). 한국어 자원으로는 김성범 교수의 확률통계 강의.

넷째, **ML을 위한 수학 종합**: "Mathematics for Machine Learning"(Deisenroth et al., 무료 PDF). 처음부터 끝까지 읽지 말고, ML 책에서 모르는 개념이 나올 때 사전처럼 찾아본다.

**수학 학습의 우선순위.** 위의 모든 자원을 다 익히는 데 1년이 걸린다. 이것은 효율적이지 않다. 대신 다음 순서로 한다. 첫째 주: 3Blue1Brown 영상 시청(2-3시간). 직관적 이해가 생긴다. 둘째 주: ML 코드를 짜기 시작한다. 셋째 주 이후: ML 코드에서 막히는 개념을 그때그때 찾아본다. 이 방식이 "수학을 다 익힌 후 ML 시작"보다 10배 효율적이다.

## GPU와 하드웨어: 무엇으로 시작하는가

ML의 학습은 GPU가 있으면 훨씬 빠르다. 단순한 ML(선형 회귀, 결정 트리, 작은 신경망)은 CPU로도 충분하지만, 딥러닝(CNN, Transformer)은 GPU 없이는 불가능에 가깝다. 어떤 하드웨어로 시작할지는 단계와 예산에 따라 다르다.

**4가지 옵션의 비교.**

<div class="card-grid">
<div class="card">
<h4>본인 컴퓨터(CPU)</h4>
<p>가장 쉬운 시작점. 학부 시절 사용하던 노트북도 충분히 사용 가능. NumPy, Pandas, scikit-learn 같은 전통적 ML 도구는 모두 CPU로 작동한다. 작은 데이터셋(수천~수만 건)으로 학습 단계에 적합. 단점은 딥러닝이 매우 느리거나 불가능하다는 것.</p>
</div>
<div class="card">
<h4>본인 컴퓨터(GPU)</h4>
<p>NVIDIA GPU가 있는 게이밍 노트북/데스크탑. RTX 3060 이상이면 입문 딥러닝에 충분. 비용은 100-300만 원. 장점은 인터넷 없이 작업 가능, 데이터 보안. 단점은 초기 비용과 발열/소음. 박사 연구가 ML 중심이라면 투자 가치가 있다.</p>
</div>
<div class="card">
<h4>Google Colab(클라우드)</h4>
<p>가장 흔한 무료 옵션. 브라우저만 있으면 GPU를 무료로 사용 가능. 무료 버전은 GPU 사용 시간 제한이 있고, 세션이 끊기면 다시 시작해야 한다. Pro($10/월)은 더 안정적인 GPU와 긴 사용 시간 제공. Pro+($50/월)은 A100 같은 고급 GPU 가능. 입문자에게 가장 추천.</p>
</div>
<div class="card">
<h4>학교 HPC/GPU 클러스터</h4>
<p>대부분의 대학이 학생에게 무료로 GPU 자원을 제공한다. 본인 학교의 HPC 센터(또는 정보처)에 신청하면 사용 가능. KISTI(한국과학기술정보연구원) 자원도 있다. 장점은 강력한 GPU(V100, A100), 큰 메모리, 무료. 단점은 사용법이 복잡(SSH, SLURM 같은 잡 스케줄러), 박사 연구가 본격화된 후 사용. ch17의 HPC 섹션 참조.</p>

**단계별 추천.**

처음 1-3개월(입문): **본인 컴퓨터 + Google Colab 무료**. 본인 컴퓨터로 코드를 짜고, GPU가 필요한 부분만 Colab으로 옮긴다. 비용 0원.

3-6개월(첫 프로젝트): **Google Colab Pro 검토**. ML을 본격적으로 시작하면서 Colab의 제한이 답답해지면 Pro로 업그레이드. 월 1만 5천 원 정도면 학생 예산으로도 가능.

6개월 이후(본격 연구): **학교 GPU 클러스터 + Colab Pro 병행**. 학교 GPU에 신청하여 큰 학습은 거기서, 빠른 실험은 Colab에서. 본인 컴퓨터는 코드 작성과 가벼운 실험용.

박사 후반(필요시): **개인 GPU 데스크탑 구매 검토**. 매일 GPU를 쓰는 단계가 되면, 학교 GPU의 대기 시간이 답답해진다. 이 단계에서 RTX 4070/4080 데스크탑(150-250만 원)을 구매하는 것이 효율적일 수 있다.

**GPU 선택 시 고려사항.** 첫째, **VRAM(GPU 메모리)**가 가장 중요하다. 모델 크기와 배치 사이즈가 VRAM에 의해 제약된다. 입문에는 8GB, 본격적인 작업에는 16GB+, 큰 모델에는 24GB+가 필요하다. 둘째, **NVIDIA가 사실상 표준**이다. AMD GPU도 ML이 가능하지만, PyTorch와 라이브러리 호환성이 NVIDIA보다 떨어진다. 셋째, **클라우드 vs 자체 구매의 손익분기점**: GPU를 매일 8시간 이상 사용한다면 자체 구매가 1년 안에 본전. 매주 몇 시간만 사용한다면 클라우드가 더 경제적.

## 첫 ML 프로젝트의 5가지 함정

처음 ML 프로젝트를 시작하면 흔히 빠지는 함정이 있다. 이것을 미리 알고 시작하면 시간과 좌절을 크게 줄일 수 있다. 이 함정들은 ch21-23의 ML 기초/파이프라인 챕터에서 다룬 내용과 보완적이다.

**함정 1: 데이터 누출(Data Leakage).** 가장 흔하고 위험한 함정이다. 테스트 데이터의 정보가 학습 과정에 침투하여, 모델이 비현실적으로 좋은 성능을 보이는 것이다. 발견하지 못하면 논문이 출판된 후 재현 실패로 발견될 수 있다. 흔한 형태는 다음과 같다. 첫째, **시간 누출**: 시계열 데이터에서 미래 데이터로 학습하고 과거 데이터로 평가. 반드시 시간순 분할(TimeSeriesSplit) 사용. 둘째, **그룹 누출**: 같은 환자/장비/사용자의 데이터가 학습과 테스트에 모두 들어가는 경우. GroupKFold 사용. 셋째, **전처리 누출**: 학습 데이터와 테스트 데이터를 합쳐서 정규화 후 분할. 정규화는 학습 데이터로만 fit하고 테스트 데이터에 transform 적용. 넷째, **특징 누출**: 라벨 정보가 입력 특징에 간접적으로 들어간 경우(예: "결과를 예측하는 데이터"가 사실은 "결과를 보고 만든 데이터"). 데이터 수집 과정을 다시 확인.

**함정 2: 무의미한 베이스라인.** "내 모델이 95% 정확도를 달성했다!"는 결과가 사실은 전혀 의미 없는 경우가 있다. 클래스가 매우 불균형한 데이터(예: 정상 95%, 비정상 5%)에서 "모두 정상으로 예측"하는 더미 모델도 95% 정확도다. 복잡한 모델이 더미 모델과 같다면 의미가 없다. 항상 다음을 베이스라인으로 비교한다. 첫째, **무작위 예측**: 클래스 분포에 따라 무작위로. 둘째, **다수 클래스 예측**: 항상 가장 많은 클래스로. 셋째, **단순 모델**: 선형 회귀, 결정 트리. 넷째, **기존 방법**: 전통적인 비-ML 방법. 모델이 이 베이스라인들을 명확히 이기지 못하면 가치가 없다.

**함정 3: 과대 평가(Overestimation).** 결과를 너무 좋게 평가하는 경향이다. 흔한 형태: 단일 지표만 보고하기, 단일 데이터셋에서만 평가, 운 좋은 시드로 얻은 결과만 보고. 대안: 여러 지표를 함께 보고, 여러 데이터셋에서 검증, 여러 random seed의 평균과 분산을 보고. 5-10번 다른 시드로 학습하여 평균과 표준편차를 보고하는 것이 정직한 방식이다.

**함정 4: 재현 불가능한 실험.** 6개월 후 같은 결과를 재현할 수 없다면, 그 결과는 사실상 존재하지 않는다. 다음을 항상 한다. 첫째, **모든 시드 고정**: numpy, torch, random 모듈 모두. 둘째, **하이퍼파라미터 기록**: config 파일이나 wandb 같은 도구로. 셋째, **데이터 버전 관리**: 어떤 버전의 데이터를 사용했는지 명확히. 넷째, **환경 기록**: requirements.txt, environment.yml, 또는 Dockerfile. ch16과 ch23에서 자세히 다룬 내용을 따른다.

**함정 5: 잘못된 메트릭 선택.** 문제에 맞지 않는 평가 지표를 사용하는 것이다. 예시: 클래스 불균형 분류에서 정확도(accuracy)만 보기(F1, ROC-AUC, PR-AUC가 더 적합), 회귀 문제에서 R²만 보기(RMSE, MAE도 함께 봐야 함), 시계열 예측에서 단일 시점만 보기(여러 시점 horizon에 걸친 평가 필요). 문제에 어떤 지표가 가장 의미 있는지 신중히 선택한다. 의심스러우면 여러 지표를 모두 보고한다.

## ML 모델이 학습 안 될 때: 디버깅 가이드

ML 코드를 짜다 보면 "왜 학습이 안 되지?"라는 막막한 순간이 온다. 손실(loss)이 줄어들지 않거나, NaN이 되거나, 정확도가 무작위 수준에 머문다. ch22에서 딥러닝 아키텍처별 디버깅을 다뤘다면, 여기서는 입문자 수준에서 가장 자주 만나는 문제를 정리한다.

**디버깅의 첫 단계: 가장 간단한 것부터 의심한다.**

1단계: **데이터부터 확인한다**. 데이터에 NaN, Inf, 이상한 값이 있는지 확인한다. `df.describe()`, `df.isna().sum()`을 실행한다. 데이터 자체가 문제인 경우가 매우 흔하다.

2단계: **아주 작은 데이터로 과적합을 시도한다**. 데이터셋에서 10개 샘플만 골라 모델을 학습한다. 모델이 그 10개에 100% 정확도로 과적합하지 못한다면, 코드에 버그가 있다는 신호다. 작동하는 ML 코드는 작은 데이터에 거의 항상 과적합 가능하다.

3단계: **학습률을 바꿔본다**. 학습률이 너무 크면 발산(loss → NaN), 너무 작으면 학습이 거의 안 일어남. 1e-3, 1e-4, 1e-5 정도를 시도해 본다. Adam optimizer는 보통 1e-3이 좋은 시작점.

4단계: **모델 구조를 단순화한다**. 복잡한 모델로 시작하지 말고 가장 단순한 것부터. 1층 MLP → 2층 MLP → 3층 MLP. 단순한 모델이 작동하면 점차 복잡하게 만든다.

5단계: **출력을 시각화한다**. 학습 과정의 loss curve를 그래프로 그린다. 학습 데이터의 일부 예측 결과를 시각화한다. 그래디언트의 norm을 확인한다. 숫자만 보지 말고 시각화로 확인한다.

**증상별 빠른 진단.**

| 증상 | 가능한 원인 |
|------|-------------|
| Loss가 NaN으로 발산 | 학습률 너무 큼, 데이터에 NaN, log(0) 또는 div(0) |
| Loss가 처음부터 변화 없음 | 학습률 너무 작음, 그래디언트 끊김(detach 실수), 라벨이 학습 데이터에서 분리됨 |
| Loss는 줄어드는데 정확도 안 늘어남 | 라벨 형식 문제(원-핫 vs 인덱스), 라벨과 출력 차원 불일치 |
| Train loss는 줄지만 Val loss는 증가 | 과적합. 더 많은 데이터, 정규화, dropout, early stopping |
| 모든 예측이 같은 값 | 모델 출력이 saturation, 학습 초기 bias가 너무 큼 |
| 매번 다른 결과 | 시드 고정 안 됨, GPU의 비결정적 연산 |

**디버깅의 효과적인 도구.** 첫째, **PyTorch의 `torch.autograd.detect_anomaly()`**: 그래디언트 계산 중 NaN이 발생하면 정확한 위치를 알려준다. 둘째, **wandb 또는 TensorBoard**: 학습 과정의 모든 지표를 시각화. 셋째, **`print` 디버깅**: 모델의 중간 출력, 그래디언트, 가중치를 출력. 가장 단순하지만 가장 효과적. 넷째, **Jupyter Notebook의 인터랙티브 디버깅**: 각 셀에서 변수를 확인하면서 진행.

**언제 도움을 구할 것인가.** 1-2시간 디버깅했는데도 원인을 못 찾으면 도움을 구한다. 첫째, **검색**: 에러 메시지를 그대로 Google에 검색. Stack Overflow, GitHub Issues에 같은 문제의 해결책이 종종 있다. 둘째, **AI 도구**: ChatGPT/Claude에 코드와 에러를 보내고 디버깅 도움을 요청. 종종 빠르게 원인을 찾는다. 셋째, **선배에게 묻기**: 같은 연구실 선배가 같은 문제를 해결한 경험이 있을 수 있다. 넷째, **온라인 커뮤니티**: Stack Overflow, Reddit r/MachineLearning, Discord 서버. 명확한 minimal reproducible example을 만들어 질문한다.

**디버깅에서 배우기.** 디버깅은 시간 낭비가 아니라 학습 기회다. 만난 모든 버그와 해결책을 노트에 기록한다. 다음에 같은 문제를 만나면 즉시 해결할 수 있다. 6개월 후 노트는 본인만의 디버깅 핸드북이 된다.

## AI 초심자를 위한 ML 논문 읽기 전략

공학 배경에서 AI/ML 연구에 진입하면 또 하나의 장벽이 등장한다. ML 논문의 언어와 구조가 전통 공학 논문과 상당히 다르다는 것이다. ch05에서 3-pass 논문 읽기법을 다뤘고, 이 원칙은 ML 논문에도 그대로 적용된다. 다만 ML 논문 특유의 관습과 함정이 있어서, 입문자가 특별히 주의해야 할 것들이 있다.

<div class="highlight-box highlight-info">

**왜 ML 논문이 유독 어려운가.** 첫째, 수식의 밀도가 매우 높다. 한 페이지에 10개 이상의 수식이 나오는 경우가 흔하고, 대부분 별도 설명 없이 지나간다. 둘째, 표준 표기법이 없다. 같은 개념도 논문마다 다른 기호로 쓴다. 셋째, 전제되는 지식이 방대하다. "attention mechanism은 기본"이라고 가정하고 시작한다. 넷째, 결과 섹션의 평가 지표(top-1 accuracy, BLEU, FID 등)가 분야별로 다르다. 이 장벽을 넘지 못하면 ML 논문을 읽을수록 좌절만 쌓인다.

</div>

**입문자가 논문을 고르는 순서.**

ML 논문은 난이도 편차가 극심하다. 첫 논문부터 NeurIPS의 최신 Transformer 논문을 읽으면 거의 확실히 좌절한다. 다음 순서로 논문을 고른다.

1. **교과서 수준의 고전 논문**: 먼저 교과서(예: Goodfellow의 Deep Learning, Bishop의 PRML)의 해당 장을 읽는다. 교과서가 설명한 고전 논문(예: LeNet, AlexNet, ResNet, Transformer 원논문)을 교과서를 옆에 두고 읽는다. 교과서가 맥락을 설명해 주므로 이해가 훨씬 쉽다.

2. **블로그 해설이 있는 논문**: 유명한 논문은 Distill.pub, The Illustrated Transformer, Lilian Weng의 블로그, Yannic Kilcher의 YouTube 해설 등 훌륭한 2차 자료가 있다. 먼저 해설을 30분 읽고, 그 논문을 펼친다. 논문의 핵심 기여가 무엇인지 알고 시작하면 읽는 속도가 5배 빨라진다.

3. **본인 연구와 직접 관련된 최근 논문**: 1-2번을 거쳐 기본기가 생기면, 연구 주제와 직접 관련된 최근 논문(1-2년 내)을 읽기 시작한다. 이때는 교과서나 해설 없이 논문 자체로 들어간다.

**읽지 않아야 하는 논문.** 입문자가 빠지기 쉬운 함정은 "트렌드 논문"에 휩쓸리는 것이다. Twitter/X에서 화제가 된 논문, arXiv의 최신 논문, 이번 주 핫한 LLM 논문을 쫓아 읽으면 맥락 없이 파편만 쌓인다. 기초가 약할 때는 최신 논문보다 3-5년 전의 기초 논문이 훨씬 가치 있다. 최신 트렌드는 기초가 쌓인 후에 따라잡아도 늦지 않다.

**ML 논문의 섹션별 읽기 팁.**

- **Abstract**: 1) 어떤 문제를 풀었나 2) 어떤 방법을 썼나 3) 얼마나 잘 되었나 — 이 3가지만 추출한다. 용어가 이해 안 되어도 괜찮다.

- **Introduction**: 여기서 가장 많은 시간을 쓴다. 문제 정의와 동기가 명확히 설명된다. 입문자에게는 본문보다 Introduction이 더 귀중한 정보원이다.

- **Related Work**: 처음 읽는 논문이라면 이 섹션은 건너뛴다. 이 섹션은 그 분야의 다른 논문을 어느 정도 알고 있어야 가치가 있다. 10편 이상을 읽은 후 돌아오면 Related Work가 훨씬 유용해진다.

- **Method**: 가장 난해한 섹션이다. 첫 읽기에서는 수식을 다 이해하려 하지 말고, "이 논문이 뭘 바꿨는가"만 파악한다. 보통 논문의 핵심 기여는 한두 개의 수식 또는 아키텍처 다이어그램에 집중되어 있다. 그것만 찾는다.

- **Experiments**: 1) 어떤 데이터셋을 썼나 2) 어떤 베이스라인과 비교했나 3) 어떤 지표로 평가했나 4) 얼마나 이겼나. 표 1이 보통 메인 결과다. 각 열이 무엇을 의미하는지 이해한다. 개선 폭이 0.5%인지 5%인지에 따라 논문의 의의가 다르다.

- **Ablation Study**: 입문자가 가장 무시하는 섹션이지만 가장 가치가 높은 섹션이기도 하다. 논문이 제안한 방법의 각 요소가 결과에 얼마나 기여했는지 보여준다. 여기서 배우는 것이 많다.

**수식에 대한 자세.** ML 논문의 수식은 3가지 유형이 있다. 첫째, **핵심 수식**: 논문의 주된 기여를 표현하는 것. 보통 1-3개. 이것은 반드시 손으로 따라 써 가며 이해한다. 둘째, **맥락 수식**: 배경 지식을 리마인드하는 것. 이미 알고 있다면 스킵, 모르면 별도 학습 후 돌아온다. 셋째, **유도 수식**: Appendix로 가는 긴 유도. 첫 읽기에서는 건너뛴다. 모든 수식을 똑같은 깊이로 읽을 필요가 없다.

**공식 코드 저장소의 가치.** ML 논문의 70-80%는 GitHub에 공식 코드를 공개한다. 논문 이해가 막히면 코드를 본다. 수식이 이해 안 되면 그 수식이 구현된 함수를 찾아 들어간다. 코드가 종종 논문보다 명확하다. 특히 PyTorch로 구현된 핵심 레이어 몇십 줄을 읽으면 30페이지 논문보다 이해가 빠를 때가 있다. 입문자에게는 "논문과 코드를 나란히 읽기"가 가장 효과적인 학습 방법이다.

**재현 시도의 학습 효과.** 한 논문을 깊이 이해하는 가장 확실한 방법은 그 논문의 결과를 재현해 보는 것이다. 공식 코드를 클론하고, README대로 실행하고, 논문의 표 1과 같은 수치가 나오는지 확인한다. 대부분 처음에는 잘 안 돌아간다. 환경이 다르거나, 데이터셋이 달라지거나, 세팅이 누락되어 있다. 이 과정에서 ML 실전 감각이 쌓인다. 박사 1년차에 유명 논문 2-3편을 직접 재현해 보는 것은 몇 달을 투자할 가치가 있다.

**읽기 속도에 대한 현실적 기대.** 입문자가 ML 논문 한 편을 "제대로 이해"하려면 보통 4-8시간이 걸린다. 이것이 느린 것이 아니라 정상이다. 유튜브에서 "논문 30분만에 요약" 같은 영상을 보고 자신의 속도를 의심하지 말자. 그런 영상은 이미 그 분야에 10년 있는 사람이 수십 편의 논문을 배경으로 만드는 것이다. 본인도 100편 이상 읽으면 속도가 10배 빨라진다. 처음에는 느려도 된다. 첫 10편이 나머지 100편의 토대가 된다.

**읽은 논문을 것으로 만드는 법.** 읽기만 하면 금방 잊는다. 각 논문마다 한 페이지 요약을 언어로 작성한다. 형식은 다음과 같다.

```
제목:
저자/연도/학회:
한 줄 요약: (본인의 말로)
해결하려는 문제:
기존 방법의 한계:
이 논문의 핵심 아이디어:
핵심 수식 또는 아키텍처 (그림으로):
실험 결과: (숫자 1-2개)
본인의 연구와의 관련성:
의문점:
```

이 한 페이지를 Zotero나 Notion이나 옵시디언에 쌓아 둔다. 6개월 후 돌아보면 본인만의 개인 ML 논문 데이터베이스가 완성된다. 논문을 쓸 때 Related Work를 쓰는 시간이 5배 단축된다.

> ML 논문 읽기의 진짜 어려움은 기술적 난이도보다 "끝이 없다"는 막막함이다. 매주 수십 편의 새 논문이 arXiv에 올라온다. 모두 따라잡으려 하면 번아웃이 온다. 처음부터 "모든 것을 알 필요는 없다"는 자세로 시작한다. 연구에 직접 필요한 것과, 분야의 큰 맥락을 잡는 것. 이 두 가지에 집중한다. 나머지는 포기해도 된다. 이 포기가 입문자의 정신 건강과 장기적 성장을 모두 지킨다.

## 온라인 강의와 MOOC 완주의 기술 — 완주율 10%에서 100%로

AI 입문을 결심한 학생이 가장 먼저 하는 일 중 하나는 온라인 강의를 등록하는 것이다. Coursera의 Andrew Ng 강의, fast.ai, deeplearning.ai, MIT OCW, Stanford CS231n 등. 문제는 **대부분의 학생이 완주하지 못한다**. 연구에 따르면 MOOC의 평균 완주율은 10% 미만이다. 학생들은 "배우려는 의지가 없어서"가 아니라, **완주하는 기술을 모르기 때문에** 포기한다. 이 섹션은 온라인 강의를 끝까지 가는 구체적 방법을 다룬다.

<div class="highlight-box highlight-important">

**완주 실패가 문제가 아닌 이유.** MOOC는 전통 강의와 다른 환경이다. 물리적 출석 의무가 없고, 동료가 없고, 즉각적 피드백이 없다. 이것들이 전통 강의의 강력한 "완주 엔진"이었다. MOOC에서 이것들이 사라지면 완주는 순수한 자기 규율에 의존하는데, 이것은 대부분의 사람에게 어려운 일이다. MOOC를 못 끝내는 것은 결함이 아니라 환경의 구조적 문제다. 해결책은 "의지를 더 강하게"가 아니라 "환경을 재설계하는 것"이다.

</div>

**강의 선택의 원칙 — 완주 확률을 높이는 고르기.**

완주 확률은 강의 선택 시점에 이미 결정된다. 다음 기준으로 고른다.

**기준 1: 강의 길이.**
3-4주 안에 끝낼 수 있는 강의를 선택한다. 12주 짜리 강의는 완주율이 극도로 낮다. 길이가 긴 강의라면 부분으로 나누어 한 번에 한 부분만 본다.

**기준 2: 기초 수준 적합성.**
수준보다 약간 위인 강의를 선택한다. 너무 쉬우면 지루하고, 너무 어려우면 좌절한다. 첫 3개 강의를 보고 "70% 이해"면 적절하다. "30% 이해"면 너무 어렵다.

**기준 3: 실습 중심성.**
이론만 설명하는 강의보다 실습이 있는 강의를 고른다. "강의 → 과제 → 프로젝트"의 사이클이 있어야 기억에 남는다.

**기준 4: 최신성.**
AI/ML 분야는 변화가 빠르다. 3년 이상 된 강의는 일부 내용이 이미 낡았다. 2-3년 내에 제작된 강의를 선택한다.

**기준 5: 커뮤니티의 존재.**
강의가 활발한 학생 커뮤니티(Discord, Slack, 토론 포럼)를 가지고 있으면 완주율이 크게 높아진다. fast.ai, deeplearning.ai가 이런 면에서 강점.

**기준 6: 인증서(Certificate)의 의미.**
완주하면 인증서가 발급되는 강의는 동기 부여에 도움이 된다. 이력서에 추가할 수 있다는 것이 심리적 보상.

**추천 입문 강의 5개**:
- **Andrew Ng의 "Machine Learning Specialization"** (Coursera): ML 기초의 표준
- **"Deep Learning Specialization"** (Coursera, deeplearning.ai): DL 기초
- **fast.ai "Practical Deep Learning"**: 실습 중심, 빠른 결과
- **Stanford CS231n (YouTube)**: CNN과 컴퓨터 비전
- **Stanford CS224n (YouTube)**: NLP

각각의 길이는 4-12주. 수준과 목표에 맞는 하나를 먼저 선택한다. 여러 개를 동시에 시작하지 않는다.

**완주 계획의 설계.**

강의를 선택한 후, 본격적인 계획을 세운다.

**1단계: 전체 일정 그리기.**
강의의 전체 길이와 주당 쓸 수 있는 시간을 계산한다. 예: 12주 강의, 주당 총 6시간 투자. 이것을 캘린더에 고정된 시간으로 블록한다.

**2단계: 고정 시간대 설정.**
"시간 있을 때" 한다고 하면 완주 못 한다. "매주 월/수/금 오전 8-10시"처럼 고정 시간대를 설정한다. 이 시간대는 다른 일정보다 우선한다.

**3단계: 주간 목표.**
매주 몇 개의 비디오를 보고, 몇 개의 과제를 할지 구체적으로 정한다. 목표는 약간 여유가 있어야 한다 (주당 4시간 강의 내용이면 6시간 계획).

**4단계: 마일스톤.**
강의를 4-6개의 마일스톤으로 나눈다. 각 마일스톤 완료 시 작은 보상(커피, 영화, 친구와의 저녁)을 정한다.

**5단계: 중단 대비 계획.**
"2주 공백 발생 시 어떻게 할 것인가?"를 미리 계획한다. 가장 흔한 실패 패턴은 "1주 놓쳐서 따라가기 어려워지고 포기"다. 대비: "1주 놓치면 다음 주에 반나절 추가로 따라잡기. 2주 놓치면 현재 마일스톤을 재시작하지 말고 이어서 진행."

**완주율을 높이는 7가지 기법.**

**기법 1: 공개 선언 (Public Commitment).**
이 강의를 완주할 것이라고 공개한다. Twitter, 블로그, 연구실 그룹 채팅에 "나는 CS231n을 12주 동안 완주할 것이다"라고 선언. 이 공개 약속이 중간에 포기하기 어렵게 만든다.

**기법 2: 공부 파트너.**
같은 강의를 듣는 동료를 찾는다. 매주 진도를 공유하고 질문을 교환한다. 파트너가 있으면 완주율이 2-3배 올라간다. 같은 연구실 동료, SNS에서 같은 강의를 듣는 사람, 온라인 스터디 그룹 등.

**기법 3: 작은 공부 블록.**
한 번에 3시간을 공부하려 하면 지친다. 25분 공부 + 5분 휴식의 포모도로 기법이 효과적. 한 블록에 1-2개 비디오를 본다.

**기법 4: 즉시 실습.**
비디오를 본 직후 30분 이내에 관련 실습을 한다. 시간이 지나면 잊혀지고 실습의 동기가 사라진다. 비디오 → 실습을 세트로 본다.

**기법 5: 노트 작성.**
각 비디오의 핵심을 1-2단락의 노트로 정리한다. 이것이 이해를 돕고, 나중에 복습할 때 유용하다. Notion, Obsidian, 종이 노트 등 본인에게 맞는 도구 선택.

**기법 6: 복습 스케줄.**
매주 첫날에 지난 주의 노트를 10분 복습. 매월 첫 주에 전체 이전 내용을 30분 복습. 이것이 망각 곡선을 이긴다 (ch27 참조).

**기법 7: 진도 시각화.**
진도를 시각적으로 볼 수 있게 한다. 격자 무늬에 완료한 날을 체크, 완료한 비디오/과제를 목록에서 지우기 등. 보상 시스템의 역할을 한다.

**완주를 막는 6가지 함정.**

**함정 1: 완벽주의.**
"이 부분을 완전히 이해한 후에 넘어가겠다"고 하다가 한 비디오에 몇 시간씩 쓴다. 결과적으로 진도가 안 나가고 포기한다. **해결**: 60-70% 이해하면 넘어간다. 나중에 다시 돌아올 수 있다.

**함정 2: 여러 강의 동시 시작.**
욕심에 Andrew Ng + fast.ai + CS231n을 동시에 시작. 결과적으로 어느 것도 끝내지 못한다. **해결**: 한 번에 하나만.

**함정 3: 과제 건너뛰기.**
비디오는 보고 과제는 건너뛴다. "비디오만 봐도 이해된다"고 느끼지만, 과제 없이는 실제 능력이 쌓이지 않는다. **해결**: 과제는 강의의 50%다. 비디오보다 과제에 더 많은 시간을 쓴다.

**함정 4: 맥락 없는 공부.**
"이것이 내 연구와 어떻게 연결되는가?"를 생각하지 않고 무작정 따라간다. 결과적으로 동기가 떨어진다. **해결**: 각 주제를 박사 연구와 연결해 본다.

**함정 5: 자기 비판.**
"왜 나는 이것도 이해 못 할까"라며 자신을 탓한다. 좌절이 쌓이면 포기한다. **해결**: "어려운 것이 정상"이라는 관점. Andrew Ng도 처음에는 어려웠다.

**함정 6: 결과 비교.**
다른 사람의 진도나 이해도와 본인을 비교. SNS에서 "이 강의 3주만에 끝냈다"는 글을 보고 좌절. **해결**: 페이스를 지킨다. 완주하는 것이 빨리 끝내는 것보다 중요하다.

**중단 후 재시작의 기술.**

공부가 중단되었을 때 재시작하는 방법. 중단은 피할 수 없다. 핵심은 "어떻게 돌아오는가"다.

**재시작의 3단계**:

1. **복귀 결정**: "한 주만 다시 해볼까"라고 최소한의 약속부터 한다. "완주를 다시 약속"은 부담스럽다.

2. **간단한 복습**: 바로 새 내용으로 가지 말고, 지난 2-3개 비디오의 노트를 10분 복습. 이것이 맥락을 복원한다.

3. **작은 목표**: 첫날은 1개 비디오만 본다. 탄력이 붙으면 점점 늘린다.

**재시작의 심리**: "놓친 주수만큼 따라잡아야 한다"고 생각하면 부담이 되어 또 포기한다. 대신 "이제부터 매주 꾸준히 한다"로 관점을 바꾼다. 놓친 부분은 포기해도 된다.

**MOOC의 한계 이해하기.**

온라인 강의가 모든 것을 가르치지 않는다. 한계를 알면 기대가 현실적이 된다.

- **깊이의 한계**: MOOC는 대부분 "개요" 수준. 연구에 직접 쓰려면 교과서와 논문으로 심화 필요.
- **질문의 한계**: 즉각적 답변이 어렵다. 막히면 Stack Overflow, Reddit, Discord 커뮤니티에서 묻는다.
- **프로젝트의 한계**: MOOC의 프로젝트는 연구 수준이 아니다. "졸업 후 회사 면접 포트폴리오" 수준이지 "학술 논문" 수준은 아니다.
- **최신성의 한계**: MOOC는 1-2년 주기로 업데이트. 최신 연구(예: 지난 6개월의 arXiv 논문)는 다루지 않는다.

MOOC는 "시작점"이지 "도착점"이 아니다. 완주 후에도 계속 학습이 필요하다.

**MOOC 완주 후 무엇을 할 것인가.**

강의를 끝냈다. 다음은?

**1. 프로젝트 시작.**
MOOC에서 배운 것을 실제 관심 주제에 적용. 강의의 예제 문제가 아니라, 찾은 문제. 이것이 지식을 것으로 만든다.

**2. 중간 난이도 강의로.**
입문 강의를 끝냈다면 중급으로. 예: Andrew Ng 후에 fast.ai, 또는 CS231n 후에 CS224n. 분야를 넓히기.

**3. 논문 읽기 시작.**
ch20의 "ML 논문 읽기" 섹션으로 넘어간다. MOOC는 기초를 주었고, 이제 실제 연구를 본다.

**4. 작은 기여 만들기.**
GitHub에 작은 프로젝트 공개. Kaggle 경진대회 참여. 이것이 능력을 외부에 보여주는 증거가 된다.

**5. 학습 일지 작성.**
무엇을 배웠는지 블로그 또는 노션에 정리. 누군가 읽기 위해서가 아니라 복습과 기록을 위해. 6개월 후 "얼마나 성장했는지" 볼 수 있다.

**"완주 안 해도 되는" 경우.**

모든 강의를 완주할 필요는 없다. 다음 경우는 중간에 멈춰도 좋다.

- **필요에 맞지 않는 강의**: 3주 정도 들었는데 목적과 맞지 않다고 판단. 다른 강의로 바꾼다. 이것은 포기가 아니라 재평가다.
- **너무 어려운 강의**: 수준에 비해 너무 어려워서 아무것도 이해하지 못한다. 더 쉬운 강의로 먼저 가고 나중에 돌아온다.
- **오래된 강의**: 일부 내용이 낡아 현재 표준과 다르다고 느낀다. 최신 강의로 전환.
- **우선순위 변화**: 박사 연구의 방향이 바뀌어서 이 강의가 더 이상 중요하지 않다. 버리는 것이 합리적.

핵심은 "완주 자체가 목표가 아니라 배우는 것이 목표"다. 완주가 배움을 방해하면 포기해도 된다.

**완주의 장기적 가치.**

한 강의를 완주하면 학습 능력 자체가 향상된다. "나는 온라인 강의를 완주할 수 있는 사람"이라는 정체성이 형성된다. 이것이 다음 강의, 다음 학습의 성공 확률을 높인다.

박사 과정 동안 5-10개의 강의를 완주한다면, 졸업 시점에 본인은 "독립 학습자"가 되어 있다. 이것이 박사 학위보다도 오래 남는 능력이다.

> 온라인 강의 완주는 의지의 문제가 아니라 **시스템의 문제**다. 고정 시간, 공개 선언, 공부 파트너, 작은 블록, 노트, 복습, 진도 시각화 — 이런 시스템을 갖추면 완주 확률이 10%에서 80% 이상으로 올라간다. 첫 한 강의를 완주하는 것이 가장 어렵다. 그 다음부터는 "할 수 있다"는 것을 안다. 박사 과정 중 첫 번째 MOOC를 끝까지 가는 것에 의도적으로 노력을 투자한다. 이 투자가 평생의 학습 능력으로 이어진다.

## 공학 분야와 ML의 교차점 찾기

공학 박사 학생이 AI와 ML을 학습할 때 가장 큰 혼란은 "ML 연구자가 되어야 하는가, 아니면 공학 분야에 ML을 적용하는 사람이 되어야 하는가"다. 이 질문에 답하지 못하면 학습이 방향을 잃는다. ML 그 자체를 깊이 학습하는 것과 공학 분야의 문제에 ML을 적용하는 것은 다른 경로다. 공학 분야와 ML의 교차점을 의식적으로 찾으면 박사 학위가 고유한 기여를 만든다.

  <div class="highlight-box-content">
    <p><strong>왜 교차점 찾기가 중요한가</strong></p>
    <p>ML 연구자는 수만 명이다. 그중 세계 탑 ML 연구자는 수백 명이고, 그 수백 명과 경쟁하는 것은 현실적이지 않다. 그러나 공학 분야 (토목, 기계, 재료, 전기, 화학 등) 각각에 ML을 전문적으로 적용하는 사람은 상대적으로 적다. 공학 분야의 깊은 지식과 ML의 실용적 지식을 결합하면 분야의 고유한 기여를 한다. ML에서 경쟁하는 것이 아니라 공학 분야 × ML의 교차점에서 선두가 된다.</p>

### 공학 분야와 ML의 5가지 교차 패턴

공학 분야와 ML은 5가지 서로 다른 방식으로 교차한다. 박사 주제가 어느 패턴에 속하는지 인식하면 학습과 연구의 방향이 명확해진다.

**패턴 1: ML을 예측기로 사용 (Surrogate Models)**

공학 분야의 복잡한 물리 모델을 ML이 학습해서 빠른 예측기로 사용. 유한요소 해석(FEM), 전산유체역학(CFD), 분자동역학(MD) 같은 계산 비용이 높은 해석의 결과를 ML 모델이 수백-수천 배 빠르게 예측.

예시:
- 구조물의 유한요소 해석 결과를 ML이 학습 → 파라미터 변경 시 결과를 즉시 예측.
- 유동 시뮬레이션의 결과를 ML이 학습 → 설계 최적화의 내부 루프에서 활용.
- 재료 물성의 제일원리 계산을 ML이 학습 → 거대한 재료 공간의 탐색.

학습 우선순위: 분야의 계산 도구를 먼저 깊이 안다. ML은 표준 회귀 기법 (신경망, Gaussian Process, Random Forest)만 학습하면 충분. ML의 최신 기법은 필요하지 않다. 데이터의 양과 질이 모델의 정확도를 결정.

관련 챕터: ch17(시뮬레이션), ch24(PINN/서로게이트), ch35(통계).

**패턴 2: ML을 분류기로 사용 (Classification/Detection)**

공학 분야의 센서 데이터, 이미지, 신호에서 상태를 분류하거나 이상을 탐지. 결함 검출, 고장 진단, 품질 검사.

예시:
- 기계 부품의 진동 신호에서 고장 징후 탐지.
- 구조물의 이미지에서 균열 자동 탐지.
- 제조 공정의 센서 데이터에서 공정 이상 감지.
- 의료 공학의 X-ray 이미지의 진단.

학습 우선순위: 컴퓨터 비전 (CNN) 또는 시계열 분석 (LSTM, Transformer)의 표준 모델. 전이 학습 (transfer learning)과 데이터 증강 (data augmentation) 기법. 라벨링된 데이터가 부족한 경우의 대응 (semi-supervised, self-supervised).

관련 챕터: ch22(딥러닝 아키텍처), ch25(컴퓨터 비전), ch37(시계열/신호).

**패턴 3: ML을 역문제 해결기로 사용 (Inverse Problems)**

공학 분야의 관측 데이터에서 원인 또는 파라미터를 역으로 추정. 역설계, 시스템 식별, 파라미터 보정.

예시:
- 구조물의 진동 응답에서 손상 위치 추정.
- 유동의 관측에서 초기 조건 또는 경계 조건 추정.
- 재료의 실험 데이터에서 구성 방정식의 파라미터 식별.
- 제어 시스템의 입출력에서 시스템 모델 학습.

학습 우선순위: 변분 추론(variational inference), MCMC 샘플링, 베이즈 신경망. 물리 기반 신경망(PINN). 불확실성 정량화.

관련 챕터: ch24(PINN), ch35(통계), ch53(확률적 사고).

**패턴 4: ML을 최적화기로 사용 (Design Optimization)**

공학 분야의 설계 공간에서 최적 설계를 찾는 데 ML을 사용. 강화학습 또는 베이즈 최적화.

예시:
- 구조물의 형상 최적화.
- 분자/재료의 역설계 (원하는 물성을 만드는 분자 구조 탐색).
- 로봇의 제어 정책 학습.
- 화학 반응 경로의 최적화.

학습 우선순위: 강화학습(RL)의 기초, 베이즈 최적화, 진화 알고리즘. 생성 모델(VAE, GAN, Diffusion)의 역설계 응용.

관련 챕터: ch26(생성형 AI), ch22(딥러닝 아키텍처).

**패턴 5: ML을 자동화 도구로 사용 (Automation & Workflow)**

공학 분야의 데이터 처리, 문서 분석, 반복 작업을 ML이 자동화. 연구 자체가 아니라 연구의 워크플로우의 한 부분.

예시:
- 분야의 논문 검색과 자동 요약.
- 실험 데이터의 자동 전처리와 이상치 탐지.
- 보고서 작성과 데이터 시각화 자동화.
- 도면의 자동 인식과 디지털화.

학습 우선순위: LLM API 활용, 파이프라인 도구 (Airflow, Prefect), 스크립팅 언어. ML 자체보다 통합 기술.

관련 챕터: ch26(생성형 AI), ch23(ML 파이프라인).

### 박사 주제에서 패턴 식별하기

박사 주제를 5가지 패턴 중 하나 이상에 매핑한다. 주제가 한 패턴에만 속할 수도 있고, 여러 패턴에 걸칠 수도 있다. 주제의 패턴을 식별하면 학습의 우선순위가 명확해진다.

**식별 질문**

질문 1: 박사 주제의 "핵심 문제"가 무엇인가? (예측, 분류, 역추정, 최적화, 자동화)

질문 2: 데이터가 어디에서 오는가? (시뮬레이션, 실험, 관측, 외부 데이터셋)

질문 3: 목표 결과물이 무엇인가? (새 물리 이해, 새 설계, 새 도구, 새 방법론)

질문 4: 평가 기준이 무엇인가? (정확도, 속도, 해석성, 일반화)

답이 패턴을 가리킨다.

### 교차점에서의 학습 전략

**전략 1: 80/20 규칙 — 공학 80%, ML 20%**

공학 분야의 깊이를 박사 학위의 핵심으로 유지한다. ML은 공학 문제의 도구일 뿐. ML 자체를 박사의 목표로 만들면 ML 연구자와 경쟁. 공학 분야의 전문가로 ML을 활용하면 교차점에서 선두.

예외: 박사 주제가 ML 방법론 자체의 새 기여라면 ML 50%, 공학 50%. 그러나 이 경우 ML 연구자와 직접 경쟁.

**전략 2: 분야의 표준 벤치마크 학습**

공학 분야의 표준 벤치마크 문제 (standard test cases)를 학습한다. 분야의 연구자들이 새 방법을 표준 벤치마크에서 평가한다. 벤치마크를 알면 연구를 분야의 맥락에 위치시킬 수 있다.

분야별 예시 벤치마크:
- 토목: ASCE benchmark structures, OpenSees 표준 예제.
- 기계: SDOF/MDOF 시스템, Rotor dynamics 표준 문제.
- 재료: Materials Project 데이터베이스, OQMD.
- 화학공학: OpenFOAM 튜토리얼, Aspen 표준 공정.
- 전기: IEEE test systems (전력), MNIST 같은 신호 처리 벤치마크.

**전략 3: 교차점 논문 10-20편 깊이 읽기**

공학 분야의 탑 저널과 ML 학회에서 교차점 논문 10-20편을 선택해 깊이 읽는다. 교차점 논문이 롤 모델.

교차점 논문의 특징:
- 공학 문제의 명확한 정의 (일반 ML 논문과 다름).
- 분야의 표준 벤치마크에서 평가.
- 분야의 물리적 해석을 함께 제시.
- 기존 공학 방법과 비교.

**전략 4: 분야의 ML 관련 워크숍 참석**

공학 분야의 주요 학회에는 보통 ML 관련 워크숍이 있다 (예: AI4Science, ML4PS, PINN workshop, Machine Learning and the Physical Sciences). 워크숍에 참석하면 교차점의 커뮤니티를 만난다.

워크숍에서 포스터 발표를 우선. ML 대형 학회 (NeurIPS, ICML)의 메인 트랙보다 분야별 워크숍이 교차점 연구의 더 적합한 무대.

**전략 5: 공동 지도 또는 공동 연구**

교차점 연구가 지도교수 한 사람의 전문 영역을 넘을 수 있다. 공학 분야의 지도교수 외에 ML 분야의 공동 지도교수 또는 공동 연구자가 박사에 가치. 박사 중반에 공동 지도의 가능성을 지도교수와 논의.

### 교차점에서 피해야 할 5가지 함정

**함정 1: ML 유행을 좇는 함정**

ML의 새 기법이 나올 때마다 박사 주제를 새 기법으로 변경. 박사 논문이 ML 트렌드의 기록이 되고 분야의 기여가 되지 않는다.

방지: 박사 주제를 공학 문제에 고정. ML 기법은 문제에 맞게 선택. 최신 기법이 문제에 맞지 않으면 오래된 기법을 사용.

**함정 2: ML을 마법으로 보는 함정**

공학 문제에 ML을 적용하면 ML이 문제를 자동으로 해결한다고 가정. 데이터가 부족하거나, 문제가 복잡하거나, 모델이 물리를 위반할 때 실패.

방지: ML의 한계를 먼저 안다. 데이터가 부족하면 물리 기반 방법이 더 낫다. 문제가 해석이 중요하면 단순한 모델이 더 낫다.

**함정 3: 공학 지식을 경시하는 함정**

ML 학습에 너무 집중해 공학 기초를 잊는다. 박사 후반에 분야의 표준 질문에 답하지 못한다.

방지: 박사 과정 동안 공학 기초를 정기적으로 복습. 분야의 표준 교과서, 분야의 클래식 논문. 공학 기초가 ML의 기초보다 더 중요.

**함정 4: 재현성을 무시하는 함정**

ML 실험의 결과를 한 번 얻고 논문에 기록. 다른 연구자가 결과를 재현할 수 없다.

방지: ML 실험의 모든 파라미터 (시드, 하이퍼파라미터, 데이터 분할)를 기록. 코드와 데이터를 공개. ch23(ML 파이프라인), ch38(오픈 사이언스)과 연결.

**함정 5: 교차점 커뮤니티를 찾지 못하는 함정**

교차점 연구를 혼자 한다. 공학 학회의 ML 트랙이나 ML 학회의 응용 트랙의 존재를 모른다. 연구가 분야의 인정을 받지 못한다.

방지: 분야의 ML 관련 워크숍을 조사. 관련 arXiv 카테고리 (예: cs.LG, stat.ML, physics.comp-ph) 구독. 교차점 연구자들의 트위터/X 팔로우.

## 분야의 ML 전문가가 되는 경로

공학 분야 × ML의 교차점에서 박사 학위를 받으면 박사 후에 어느 경로를 가는가.

**경로 1: 학계 (공학 분야의 교수)**

공학 분야의 교수로 임용되어 ML 기반 연구를 주도. 분야의 ML 방향을 이끈다. 분야의 다음 세대의 ML 전문가를 양성.

**경로 2: 학계 (CS/ML 학과의 응용 트랙)**

컴퓨터 과학 학과 또는 ML 학과의 응용 연구원으로 임용. 공학 도메인 지식이 차별화.

**경로 3: 산업체 R&D (분야의 기업의 ML 팀)**

공학 분야의 기업 (건설, 제조, 자동차, 에너지, 제약)의 R&D 팀에서 ML을 적용. 기업이 공학 지식과 ML 지식을 모두 가진 사람을 찾는다.

**경로 4: AI 기업의 도메인 전문가**

AI 기업 (Google DeepMind, OpenAI, Anthropic, 국내 AI 기업)의 과학/공학 응용 팀. AI 기업이 도메인 전문성을 귀하게 본다.

**경로 5: 스타트업 창업**

공학 분야 × ML의 교차점에서 스타트업을 창업. 박사 학위의 기술을 제품으로 전환.

## 마지막 — 교차점이 고유한 자리다

박사 과정에서 ML을 학습할 때 ML 연구자와 경쟁하려 하지 않는다. 공학 분야의 전문가로 ML을 도구로 사용하는 사람이 된다. 교차점이 고유한 자리다.

공학 분야의 깊이와 ML의 실용적 지식을 결합하면 분야의 다음 세대의 한 사람이 된다. 박사 학위가 분야의 ML 적용의 한 기초를 놓는다. 박사 졸업 후 30년 동안 분야의 ML 기반 방법론의 한 부분을 이끈다.

박사 1년차부터 교차점을 의식한다. 공학 기초와 ML 도구를 균형 있게 학습. 박사 졸업 시점에 분야의 "이 사람이 ML의 한 부분을 이끄는 사람이다"로 인식된다. 이 인식이 박사 학위의 가장 큰 자산 중 하나가 된다.

## 박사 중반 이후의 AI 전환 — 3-4년차의 재시작

이 챕터의 대부분은 "박사 1-2년차부터 AI를 배우자"를 전제한다. 그러나 현실에는 다른 경로의 박사생이 많다. 2-3년 동안 전통 방법(시뮬레이션, 실험, 이론)으로 연구하다가 박사 중반에 "AI를 접목해야 할 것 같다"고 느끼는 학생이다. 이것은 패닉과 혼란을 동반한다. "너무 늦었나?", "동기보다 뒤처진 건 아닌가?", "이걸 해서 남은 2-3년 안에 논문이 나올까?" 이 질문들에 대한 명확한 답이 필요하다. 이 섹션은 박사 중반 이후 AI로 전환하려는 학생을 위한 현실적 전략이다.

<div class="highlight-box highlight-important">

**"중반 전환은 불리하지 않다, 다를 뿐이다".** 박사 1년차에 AI를 시작한 학생과 박사 4년차에 시작한 학생의 경로는 근본적으로 다르다. 1년차는 "넓게 배우고 깊이 탐색"할 수 있지만, 4년차는 "좁게 집중해서 빠르게 적용"해야 한다. 두 경로 모두 성공할 수 있다. 다만 중반 전환은 다른 전략이 필요하다. 1년차처럼 접근하면 실패한다. 시간, 경험, 도메인 지식을 최대한 활용하는 맞춤 전략이 필요하다.

</div>

**전환의 4가지 동기.**

박사 중반에 AI 전환을 고려하는 이유는 다양하다. 동기를 명확히 하는 것이 첫 단계다.

**동기 1: 분야의 트렌드 변화.**
분야가 급격히 AI 중심으로 이동. AI 없이는 논문이 거절되거나 무시되기 시작. 예: 재료 과학의 HTP(high-throughput) 연구, 신약 개발, 구조 건강 모니터링.

**동기 2: 지도교수의 요구.**
지도교수가 연구실의 방향을 AI로 바꿈. 본인에게 AI 프로젝트를 배정. 선택의 여지가 제한적.

**동기 3: 자발적 관심.**
AI를 배우고 싶어서. 기존 연구에 한계를 느꼈거나, AI로 더 좋은 결과를 낼 수 있을 것 같다고 판단.

**동기 4: 잡 마켓 고려.**
졸업 후 AI 관련 직업을 원함. 박사 논문에 AI 섹션이 있으면 유리.

각 동기에 따라 전환의 깊이와 범위가 다르다.

- 동기 1: 기존 연구에 AI 보조. 부분 전환.
- 동기 2: 지도교수의 방향 따름. 범위는 지도교수와 협의.
- 동기 3: 열정에 따라 조정. 큰 전환 가능.
- 동기 4: 이력서에 쓸 프로젝트. 최소한의 명확한 성과.

**전환의 3가지 수준.**

전환의 "깊이"를 결정해야 한다.

**수준 1: AI를 도구로 활용 (Light Touch)**.
기존 연구는 유지. AI를 보조 도구로 추가. 예: 시뮬레이션 결과를 ML로 빠르게 추정.

**시간 투자**: 3-6개월.
**기대 결과**: 기존 논문에 "우리는 ML로 X를 가속화했다" 같은 섹션 추가. 또는 짧은 응용 논문 1편.
**적합한 경우**: 동기 1, 4. 대부분의 중반 전환에 적합.

**수준 2: 본격 연구 주제로 편입 (Deep Integration)**.
박사의 한 기여 전체가 AI 관련. 예: 물리 문제에 PINN 적용. 새로운 ML 접근.

**시간 투자**: 1-2년.
**기대 결과**: 박사 논문의 1-2개 챕터. 독립적 AI 논문 1-2편.
**적합한 경우**: 동기 2, 3. 박사 3년차 초에 결정하면 가능.

**수준 3: 박사 방향 전환 (Full Pivot)**.
기존 연구를 일부 포기하고 AI 중심으로 재정향. 예: 실험 연구자가 딥러닝 연구자로.

**시간 투자**: 2-3년.
**기대 결과**: 박사의 대부분이 AI 관련. 졸업 연장 가능성 높음.
**적합한 경우**: 박사 2-3년차 초, 강한 동기와 지도교수 동의가 있을 때.

**시간 계산**: 박사 몇 년차인가와 어떤 수준을 원하는가에 따라 결정.

- 박사 3년차 = 수준 1 쉽게, 수준 2 가능, 수준 3 위험
- 박사 4년차 = 수준 1 가능, 수준 2 어려움, 수준 3 불가
- 박사 5년차 = 수준 1만 가능

**중반 전환의 강점 — 가진 것.**

"늦었다"고 생각하지만, 본인은 1년차 학생에게 없는 자산을 가지고 있다.

**강점 1: 도메인 지식.**
본인은 이미 공학 분야에 대한 깊은 이해가 있다. 1년차 학생이 2년 걸려 배우는 것을 본인은 이미 안다. AI 학생은 이 도메인 지식을 얻기 어렵다. 이것이 차별화.

**강점 2: 문제 의식.**
본인은 분야의 "진짜 문제"가 무엇인지 안다. 1년차는 장난감 문제에 집중하지만, 본인은 실제로 해결 가치 있는 문제를 선택할 수 있다.

**강점 3: 데이터 이해.**
분야의 데이터 특성, 한계, 품질을 안다. AI 모델이 작동하지 않을 때 데이터의 문제인지 모델의 문제인지 빠르게 판단 가능.

**강점 4: 검증 능력.**
분야의 "좋은 결과"가 무엇인지 안다. AI가 내놓은 결과가 물리적/공학적으로 말이 되는지 판단할 수 있다. 이것이 AI 전문가에게 없는 능력.

**강점 5: 연구 성숙도.**
본인은 이미 박사 연구의 여러 단계를 경험했다. 논문 쓰기, 리뷰 대응, 시간 관리. 새로운 학생처럼 모든 것을 다시 배우지 않아도 된다.

이 강점들을 의식적으로 활용하면 "늦은 시작"이 "전략적 집중"이 된다.

**중반 전환의 위험 — 피해야 할 함정.**

중반 전환에는 특유의 위험이 있다.

**함정 1: "1년차처럼" 공부하려 함.**
AI 기초부터 모든 것을 배우려 함. 시간이 부족해서 결국 아무것도 완성하지 못함.

**대응**: 본인에게 필요한 것만 최소한으로. "Just enough" 철학.

**함정 2: 유행 쫓기.**
가장 "핫한" 주제에 뛰어듦. 수개월 후 유행이 바뀌면 처음부터.

**대응**: 분야의 문제에 맞는 것에 집중. 유행과 무관하게.

**함정 3: 자기 의심.**
"동기들은 이미 앞서 있다"는 비교. 동기 상실.

**대응**: 경로가 다르다는 것을 받아들임. 다른 강점을 활용.

**함정 4: 기존 연구의 완전 포기.**
AI에 집중하느라 기존 박사 연구가 멈춤. 결국 박사가 더 늦어짐.

**대응**: 기존 연구를 버리지 말고 AI로 "확장". 시너지를 찾는다.

**함정 5: 지도교수와의 소통 부족.**
AI 전환을 혼자 결정. 지도교수는 여전히 기존 방향을 기대.

**대응**: 지도교수와 솔직한 대화. "제가 이런 생각을 하는데 어떻게 보시나요?" 지도교수의 의견을 반영한 계획.

**중반 전환의 90일 계획.**

박사 중반에 AI 전환을 결정했다면, 첫 90일이 결정적이다.

**Week 1-2: 방향 결정.**
- 전환의 수준 결정 (1/2/3)
- 분야에서 AI의 현재 상태 조사
- 지도교수와 초기 대화
- 가능한 프로젝트 후보 3-5개 리스트

**Week 3-4: 핵심 배경 학습.**
- 특정 목표에 필요한 최소 이론 (전체 ML이 아닌)
- 핵심 프레임워크 선택 (PyTorch 권장)
- 분야의 대표 AI 논문 5-10편 읽기

**Week 5-6: 첫 실험.**
- 가장 간단한 모델로 데이터에 적용
- 목표: "작동한다"의 증거
- 완벽함이 아닌 학습 경험

**Week 7-8: 첫 결과.**
- 첫 실험의 결과를 지도교수에게 공유
- 피드백 수렴
- 다음 단계 계획

**Week 9-10: 확장.**
- 더 복잡한 모델 또는 더 큰 데이터
- 초기 비교 (baseline과)
- 논문 아이디어 구체화

**Week 11-12: 재평가.**
- 처음 계획과 현실의 비교
- 조정: 수준을 낮출까 높일까
- 다음 6개월 계획

이 90일 계획이 "막막함"에서 "방향"으로의 전환을 만든다.

**학습의 우선순위 — 시간이 제한된 학생을 위한.**

박사 1년차는 1년 동안 "기초를 탄탄히"할 수 있지만, 중반 전환자는 그럴 시간이 없다. 우선순위를 명확히.

**반드시 배워야 할 것 (must)**:
1. PyTorch (또는 TensorFlow) 기본 사용법
2. 문제 유형의 표준 모델 (CNN, RNN, Transformer 등)
3. 학습 루프, 손실 함수, 옵티마이저
4. 평가 지표와 결과 해석
5. 과적합과 정규화
6. 결과 시각화

**유용한 것 (nice to have)**:
1. 딥러닝 이론의 수학적 배경
2. 여러 아키텍처의 세부
3. 하이퍼파라미터 튜닝의 최적화
4. 고급 최적화 기법

**나중에 배워도 되는 것 (can skip)**:
1. 최신 논문의 모든 변형
2. Attention의 수학적 상세
3. 고급 정규화 기법
4. 분산 학습 (처음부터 필요 없음)

**원칙**: "꼭 필요한 것만". 첫 프로젝트가 작동하는 데 필요한 것만 배운다. 나머지는 필요할 때.

**지도교수와의 협상.**

중반 전환은 지도교수와의 협상이 필요하다.

**협상의 핵심 메시지**:
1. "저는 기존 연구를 포기하지 않습니다"
2. "AI는 연구를 확장/강화하는 도구입니다"
3. "제가 이것을 할 수 있는 이유는 도메인 지식이 있기 때문입니다"
4. "졸업 일정에 큰 영향이 없도록 계획했습니다"
5. "교수님의 의견을 구합니다"

**지도교수가 찬성할 이유**:
- 연구실의 다양성 확대
- 산업계와 잡 마켓에서의 경쟁력
- 기존 연구와 AI의 시너지
- 새로운 펀딩 기회

**지도교수가 우려할 이유**:
- 박사가 길어짐
- 기존 연구가 흐지부지
- 지도교수 AI에 익숙하지 않음
- 연구실의 자원 분산

제안이 이 우려들을 해소할 수 있어야 한다.

**중반 전환자의 멘토 확보.**

지도교수가 AI 전문가가 아니면 다른 멘토가 필요하다.

**멘토의 유형**:

1. **학교의 AI 교수**: 비공식 조언, 세미나 참석
2. **동료 박사생 중 AI 전문가**: 일상적 질문
3. **온라인 커뮤니티의 전문가**: Reddit, Twitter, Discord
4. **산업체의 AI 엔지니어**: 실전 관점
5. **공동 지도교수 추가**: 공식적 결합 (가장 강력하지만 복잡)

**권장**: 2-3개의 유형을 결합. 지도교수 1명 + 동료 박사생 1-2명 + 온라인 커뮤니티.

**첫 AI 논문의 전략.**

중반 전환자의 첫 AI 논문은 어떻게 될까?

**현실적 목표**:
- 분야의 응용 연구
- 혁신적 방법이 아닌 기존 방법의 새 적용
- 도메인 지식을 차별화 요소로
- 큰 ML 학회(NeurIPS)가 아닌 분야의 학회

**첫 논문의 구조**:
1. **문제**: 분야의 구체적 문제
2. **기존 방법의 한계**: 전통적 접근의 문제
3. **AI 접근**: 표준 방법의 적용
4. **도메인 지식의 통합**: 본인만의 차별화
5. **결과**: 기존 방법과의 비교
6. **한계와 논의**: 정직한 평가

이 구조가 "AI 전문가"가 아닌 "도메인 전문가가 AI를 도구로 활용한" 관점으로 논문을 쓰는 방법이다.

**"실력 부족"의 솔직한 대응.**

중반 전환자는 종종 "실력이 부족하다"고 느낀다. 이것의 솔직한 대응.

**인정할 것**:
- 본인은 AI의 모든 세부를 모른다
- 그것은 정상이다
- 박사 졸업 시점에도 모른다

**하지 말 것**:
- 부족함을 숨기려 과장
- 알지 못하는 것을 안 척
- 완벽주의

**해야 할 것**:
- 모르는 것을 "모른다"고 말하기
- 질문을 두려워하지 않기
- 강점(도메인)을 강조
- 시간이 걸려도 괜찮다는 것을 받아들이기

**공저자의 활용**.

중반 전환자는 AI 전문가와의 공저 논문이 효과적이다.

**공저의 구조**:
- 본인: 도메인 문제 정의, 데이터, 결과 해석
- 공저자: ML 모델 설계, 구현, 최적화

이 분업이 양쪽의 강점을 활용한다. 공저자는 본인 연구실의 다른 학생, 다른 연구실의 AI 학생, 또는 산업체 협력자가 될 수 있다.

**주의**: "이름만 올리는" 저자가 되면 박사 기여로 인정받기 어렵다. 공저에서도 실질적 기여가 명확해야 한다.

**성공 사례의 교훈**.

중반 전환에 성공한 박사생들의 공통점.

1. **명확한 목표**: "AI를 배운다"가 아니라 "X 문제를 AI로 해결한다"
2. **제한된 범위**: 모든 것을 배우려 하지 않음
3. **빠른 실행**: 이론보다 실습 먼저
4. **멘토 활용**: 혼자 하지 않음
5. **도메인 강점 유지**: AI와 기존 전문성의 결합
6. **현실적 기대**: 2년 안에 AI 스타가 되지 않음
7. **지도교수와의 솔직한 소통**: 진행 상황 공유
8. **자기 연민 금지**: "늦었다"는 생각에 빠지지 않음

**성공 사례 vs 실패 사례**:
- 성공: "나는 재료과학의 전문가이고 AI를 활용한다"
- 실패: "나는 AI 전문가가 되고 싶다"

정체성의 차이가 결과를 결정한다.

**장기적 관점 — 졸업 후의 가치.**

중반 전환의 가치는 박사 졸업 후에 더 명확해진다.

**가치 1: 이중 전문성**.
본인은 "도메인만 아는 사람"도 "AI만 아는 사람"도 아니다. 둘을 모두 이해하는 사람이고, 이런 사람은 산업계와 학계에서 매우 귀하다.

**가치 2: 응용의 진정성**.
본인은 AI를 "적용"한 경험이 있다. 단지 공부한 것이 아니다. 이 경험이 실무에서 가치 있다.

**가치 3: 실패를 안 사람**.
본인은 AI가 "만능"이 아니라는 것을 안다. 언제 AI를 쓰고 언제 안 쓸지 판단할 수 있다. 이 판단력이 전문성의 본질.

**가치 4: 가르칠 수 있음**.
본인은 "AI를 처음 배우는 사람"의 관점을 안다. 졸업 후 후배를 가르칠 수 있다.

**가치 5: 다양한 경로**.
본인은 전통 공학 진로와 AI 진로 모두 갈 수 있다. 선택의 폭이 넓다.

**결론 — 중반 전환의 격려.**

만약 박사 3-4년차에 AI 전환을 고민하고 있다면, 이것을 기억하라.

1. **늦지 않았다**. 다를 뿐이다.
2. **강점을 활용하라**. 도메인 지식은 무기다.
3. **작게 시작하라**. 첫 프로젝트는 완벽하지 않아도 된다.
4. **지도교수와 대화하라**. 혼자 결정하지 말자.
5. **시간을 주자**. 1년 안에 전환이 완성되지 않는다.
6. **본인을 비교하지 말자**. 다른 경로는 다른 결과다.
7. **즐기자**. 새로운 것을 배우는 것은 박사의 특권이다.

> 박사 중반 이후의 AI 전환은 불리한 출발이 아니다. 다른 출발이다. 본인은 1년차 학생에게 없는 자산들을 가지고 시작한다. 도메인 지식, 연구 성숙도, 문제 의식, 검증 능력. 이 자산들을 최대한 활용하고 AI를 최소한으로 배워서 문제에 적용한다. 이것이 박사 중반 전환자의 전략이다. 박사 졸업 시점에 본인은 "AI 전문가"는 아니지만 "AI를 도구로 활용하는 도메인 전문가"가 된다. 이것이 많은 경우 더 가치 있는 정체성이다. 박사 과정은 독특한 경로이고, 그 독특함이 차별화가 된다.

## AI 입문자의 환경 구축 — Colab, 로컬, 클라우드의 전략적 선택

AI 학습의 첫 장벽은 이론도 코딩도 아니라 **환경 구축**이다. "Python을 설치했는데 PyTorch가 안 돌아가요", "GPU는 어떻게 쓰죠?", "CUDA 오류가 납니다", "이 튜토리얼이 Colab에서는 되는데 내 컴퓨터에서는 안 됩니다". 이런 문제들이 며칠을 날리고 동기를 꺾는다. 환경 구축의 선택지를 이해하면 이 시간을 크게 줄일 수 있다. 선택은 하나가 아니다. Colab, 로컬 설치, 클라우드 서비스, 학교 서버, 각각이 장단점이 있다. 이 섹션은 AI 입문자가 상황에 맞는 환경을 선택하고 구축하는 실전 가이드다.

<div class="highlight-box highlight-important">

**"환경은 연구의 기반이다".** 환경이 안정적이지 않으면 학습과 실험이 산만해진다. "어제는 되던 코드가 오늘은 안 돼", "GPU가 갑자기 안 잡혀", "라이브러리 버전이 꼬였어"의 경험이 누적되면 에너지가 연구가 아닌 환경 문제 해결에 쓰인다. 박사 초기에 환경을 안정적으로 구축하는 것이 5-7년의 생산성을 크게 좌우한다. 시간을 투자할 가치가 있다.

</div>

공유된 GPU 서버에 SSH로 접속.

**Google Colab — 가장 쉬운 시작**.

Colab은 입문자에게 이상적이다.

**장점**:
1. **설치 없음**: 브라우저만 있으면 즉시 시작
2. **무료 GPU**: T4 GPU 무료 (제한 있음)
3. **Python 환경 미리 설정**: PyTorch, TensorFlow 기본 포함
4. **쉬운 공유**: URL로 노트북 공유
5. **Google Drive 연동**: 데이터와 결과 저장
6. **튜토리얼 호환**: 대부분의 튜토리얼이 Colab에서 작동

**단점**:
1. **세션 시간 제한**: 무료는 12시간, 실제로는 더 짧음
2. **유휴 시간 제한**: 90분 이상 미사용 시 연결 끊김
3. **파일 지속성 부족**: 세션 끝나면 파일 삭제 (Drive 저장 필요)
4. **RAM 제한**: 12GB 기본 (Pro는 25GB)
5. **GPU 가용성**: 무료는 피크 시간에 부족
6. **인터넷 필요**: 오프라인 불가

**Colab의 등급**:
- **Free**: 기본 GPU, 제한 있음
- **Pro ($9.99/월)**: 더 많은 GPU 시간, 더 좋은 GPU
- **Pro+ ($49.99/월)**: 프리미엄 GPU (V100, A100), 더 긴 세션
- **Pay as you go**: 컴퓨팅 단위 구매

**사용 시나리오**:
- **학습과 튜토리얼**: 최적

- **본격 연구**: 부족
- **학위 논문 실험**: 매우 부족

- 피크 시간 경쟁

**사용 방법**:

**단계 1: 접근 요청**.
연구실 IT 담당자에게 계정 신청.

**단계 2: SSH 접속**.
```bash
ssh username@server.university.edu
```

**단계 3: 가용 자원 확인**.

nvidia-smi  # GPU 상태

SLURM의 경우:
```bash
#!/bin/bash

#SBATCH --mem=32G

python train.py
```

```bash
sbatch job.sh
```

**단계 5: 결과 가져오기**.
```bash
scp username@server:/path/to/result.txt ./
```

**주의**:
- **자원 공유**: 모든 GPU를 쓰지 말자. 동료와 조율.
- **공간 관리**: home 디렉토리 용량 제한.
- **시간 제한**: 한 작업의 최대 시간.
- **우선순위**: 일부 시스템은 사용자별 우선순위.

**추천 조합**:

**박사 1-2년차 (학습)**:
- **주**: Colab (무료 또는 Pro)
- **보조**: 가끔 학교 서버
- **언제**: 로컬 개발, 서버에서 실행

**박사 2-3년차 (실험)**:
- **주**: 학교 서버
- **보조**: 로컬 (프로토타이핑)
- **크라우드**: 큰 실험 시

**박사 3-4년차 (본격 연구)**:
- **주**: 학교 서버 + 로컬 GPU
- **보조**: 클라우드 (단기 집중)

**박사 5-6년차 (마무리)**:
- **주**: 학교 서버
- **보조**: 로컬 (논문 그림, 분석)

**일반 원칙**: 가장 저렴하고 충분한 것 먼저. 필요할 때만 upgrade.

**데이터의 관리 — 환경 간 이동**.

여러 환경을 쓰면 데이터 관리가 복잡해진다.

**전략 1: 클라우드 스토리지**.
Google Drive, Dropbox, AWS S3에 데이터 저장. 각 환경에서 다운로드.

**장점**: 환경 간 동기화.
**단점**: 네트워크 속도, 대용량 불편.

**전략 2: 학교 서버를 중심으로**.
학교 서버에 데이터 저장. 로컬/Colab에서 필요시 샘플 다운로드.

**장점**: 큰 데이터 한 곳.
**단점**: 학교 네트워크 의존.

**전략 3: 데이터의 부분 작업**.
로컬에서는 작은 샘플로 개발, 서버에서 전체 실행.

**장점**: 효율적.
**단점**: 두 버전 유지.

**원칙**: "원본은 한 곳, 복사본은 가벼이". 대용량 데이터는 중심 위치에, 개발은 샘플로.

**환경의 재현성**.

환경을 재현 가능하게 만드는 것이 중요하다.

**이유**:
- 다른 사람이 코드 실행
- 다른 컴퓨터에서 실행
- 1년 후에도 같은 결과

**도구**:

**도구 1: requirements.txt**.
```bash
pip freeze > requirements.txt
```
설치할 라이브러리 목록.

**도구 2: environment.yml (Conda)**.
```bash
conda env export > environment.yml
```

**도구 3: Docker**.
완전한 환경을 컨테이너로.
```dockerfile
FROM pytorch/pytorch:2.0.0-cuda11.7-cudnn8-runtime
RUN pip install transformers
WORKDIR /workspace
```

**도구 4: Makefile 또는 설치 스크립트**.
```bash
#!/bin/bash
conda create -n myenv python=3.10
conda activate myenv
pip install -r requirements.txt
```

**권장**: `requirements.txt` 또는 `environment.yml`을 최소한으로. Docker는 고급 단계.

**비용 관리**.

AI 환경은 돈이 든다. 비용 관리 전략.

**예산 추정**:
- **Colab 사용자**: 연 10-100만원 (Pro 구독)
- **로컬 GPU 소유자**: 초기 100-300만원 + 전기세
- **클라우드 위주**: 연 50-500만원 (사용량에 따라)
- **학교 서버**: 무료 (학생에게)

**비용 절감 방법**:

**방법 1: 학교 자원 최대 활용**.
학교 서버가 있으면 우선 사용.

**방법 2: Colab Pro**.
월 ~1.3만원. 가장 경제적 AI 환경.

**방법 3: 중고 GPU**.
RTX 3060, 3070, 3080 중고. 50-70% 가격.

**방법 4: 클라우드의 Spot 인스턴스**.
일반 가격의 30-70%. 단, 언제든 종료될 수 있음. 체크포인팅 필수.

**방법 5: 연구실 공동 구매**.
연구실에서 강력한 GPU 서버 공동 구매. 비용 분담.

**방법 6: 학술 크레딧**.
AWS Educate, GCP, Azure의 학생 크레딧 활용.

**방법 7: NVIDIA 학술 프로그램**.
일부 대학은 NVIDIA와 파트너십. 무료 GPU 시간.

**창의성**: 상황을 분석하고 여러 방법을 조합.

**환경 구축의 흔한 함정**.

입문자가 자주 빠지는 함정.

**함정 1: "최고의 것부터"**.
RTX 4090을 살 돈이 없는데도 갈망. 결과: 아무것도 하지 않음.
**수정**: Colab으로 시작. 필요 확인 후 점진적 업그레이드.

**함정 2: 설치에 지나치게 시간 소모**.
환경 구축에 1-2주. 학습은 뒷전.
**수정**: 첫 1-2일에 기본 환경. 부족하면 나중에 개선.

**함정 3: 여러 환경 혼용**.
Colab, 로컬, 클라우드를 동시에. 결과: 동기화 문제.
**수정**: 한 주 환경에 집중. 다른 것은 백업.

**함정 4: 의존성 관리 무시**.
가상 환경 없이 설치. 1년 후 의존성 지옥.
**수정**: 처음부터 conda 환경.

**함정 5: 백업 없음**.
로컬 GPU에만 데이터. 디스크 고장으로 모두 잃음.
**수정**: 외부 백업 (클라우드, 외장 하드).

**함정 6: 비용 관리 무시**.
클라우드 인스턴스를 끄지 않음. 월말에 수만원 청구.
**수정**: 자동 종료, 알림, 예산 한도 설정.

**함정 7: 네트워크 의존**.
Colab만 쓰다가 인터넷 없으면 작업 불가.
**수정**: 오프라인 작업 가능한 환경도 하나.

**학교 서버 사용의 에티켓**.

학교 서버는 공유 자원이다. 에티켓이 중요.

**에티켓 1: 자원 공유**.
모든 GPU를 쓰지 말자. 최대 1-2개만 점유.

**에티켓 2: 피크 시간 회피**.
낮에 급하지 않으면 밤이나 주말에 실행.

**에티켓 3: 빠른 종료**.
작업이 끝나면 즉시 인스턴스 종료. 다른 사람을 위해.

**에티켓 4: 공간 관리**.
home을 정리. 임시 파일 삭제.

**에티켓 5: 의사 소통**.
문제가 생기면 동료와 IT 담당자에게 빠른 공유.

**에티켓 6: 우선순위 존중**.
동료의 긴급 작업이 있으면 양보.

**이 에티켓이 연구실의 건강한 문화를 만든다**.

**환경 선택의 결정 트리**.

본인에게 맞는 환경을 찾는 결정 트리.

```
시작: AI 학습을 시작하려 하는가?

학생의 상황:
├─ 완전 초보 (Python도 처음)
│  └─ Colab Free
│
├─ 기본 튜토리얼
│  ├─ 빠른 결과 원함: Colab Pro
│  └─ 절약 원함: Colab Free
│
├─ 첫 실험 (작은 규모)
│  ├─ 학교 서버 있음: 학교 서버
│  ├─ 로컬 GPU 있음: 로컬
│  └─ 둘 다 없음: Colab Pro
│
├─ 본격 연구
│  ├─ 학교 서버 있음: 학교 서버 (주) + 로컬 (보조)
│  └─ 없음: 로컬 GPU 구매 (RTX 3080-4070)
│
└─ 대형 실험 (박사 후반)
   ├─ 학교 서버 충분: 학교 서버
   └─ 부족: 클라우드 (단기 집중)
```

**상황에 맞는 경로**를 따른다.

**환경 업그레이드의 타이밍**.

언제 환경을 업그레이드할 것인가.

**업그레이드 신호**:
1. **Colab이 자주 끊김**: Pro로 업그레이드 또는 학교 서버로 이동
2. **로컬 GPU 메모리 부족**: 배치 크기 조정 불가, 모델 크기 한계
3. **실험 시간이 너무 김**: 하나의 실험이 며칠 걸림
4. **병렬 실험 필요**: 동시에 여러 실험
5. **데이터 너무 큼**: 현재 환경에서 처리 불가

**업그레이드 방향**:
- Colab Free → Colab Pro → 학교 서버 → 로컬 GPU → 더 큰 로컬 GPU → 클라우드

**과도 업그레이드 주의**: 현재 환경을 완전히 활용하지 못하는 상태에서 업그레이드하면 낭비.

**환경의 표준화와 공유**.

연구실 내에서 환경을 표준화하면 이익.

**표준화의 이점**:
1. **신입생 빠른 시작**: 같은 환경으로 시작
2. **문제 해결 공유**: "동일 환경에서의 문제"
3. **코드 호환성**: 같은 버전으로 실행
4. **자원 관리**: 일관된 정책

**표준화 도구**:
- **공유 Docker 이미지**: 모두 같은 환경
- **공유 conda 환경 파일**: 모두 같은 라이브러리
- **공유 문서**: 설치 가이드
- **공유 설정**: 연구실의 편의 설정

**역할**: 연구실에 이것이 없다면 시작할 수 있다. 후배에게 귀중한 기여.

**실천 가이드 — 첫 2주**.

오늘 AI 학습을 시작한다면 첫 2주 계획.

**Day 1**:
- Google 계정 준비
- Colab 첫 노트북 생성
- `import torch; torch.cuda.is_available()` 확인
- 첫 튜토리얼 (PyTorch의 Quickstart) 실행

**Day 2-3**:
- 튜토리얼 완주
- Colab의 기본 기능 학습 (저장, 공유, 드라이브 연동)
- GPU 할당 확인

**Day 4-7**:
- 간단한 프로젝트 (MNIST 분류)
- 결과를 Google Drive에 저장
- 코드를 GitHub에 업로드

**Day 8-10**:
- 학교 서버 접근 요청 (있으면)
- 또는 로컬 Python/Conda 설치
- 기본 환경 구축

**Day 11-14**:
- 로컬 vs Colab 비교
- 주 환경 결정
- 백업 전략 수립

**2주 후**: 안정적인 환경과 기본 워크플로.

**더 나아가 — 고급 환경 최적화**.

환경이 안정되면 최적화를 고려.

**최적화 1: SSH 키 관리**.
여러 서버를 쉽게 접근.

**최적화 2: Dotfiles**.
설정 (`.bashrc`, `.vimrc`)을 Git으로 관리.

**최적화 3: Tmux 또는 Screen**.
긴 작업이 SSH 연결 끊김에 영향 안 받게.

**최적화 4: 원격 개발**.
VS Code Remote로 서버 코드 편집.

**최적화 5: 자동 백업**.
Cron으로 정기 백업.

**최적화 6: 모니터링**.
GPU 사용률, 디스크 공간 자동 모니터링.

이것들은 박사 1-2년차에는 overkill. 박사 3-4년차에 점진적으로 도입.

이 12가지를 완료하면 환경 기반이 단단해진다.

**장기 관점 — 환경과 박사 경력**.

환경 선택이 박사 경력에 미치는 장기 영향.

**박사 과정**:
- 환경이 안정적이면 연구에 집중
- 환경 문제로 시간 낭비 최소화

**박사 후**:
- 같은 도구와 워크플로 이전
- 새 환경 적응 시간 감소
- 전문성의 확장

**산업체 진출**:
- 산업체 환경 (클라우드, 컨테이너)과 유사
- 전이 쉬움

**창업/독립**:
- 저비용으로 빠른 시작
- 환경 구축 경험이 자산

**박사 과정에서 배운 환경 관리 기술이** 평생 쓰인다.

> AI 환경 구축은 학습의 전제 조건이 아니라 학습의 첫 단계다. 환경에 시간을 투자하면 나중의 수백 시간이 절약된다. 너무 많이 투자하면 학습이 늦어지고, 너무 적게 투자하면 반복적인 문제로 고통받는다. 적절한 균형을 찾는 것이 중요하다. 박사 초기에 Colab으로 시작해, 필요에 따라 학교 서버와 로컬로 확장하라. 환경은 고정이 아니라 박사 과정 동안 진화한다. 환경을 의식적으로 관리하는 것이 생산적 박사 연구의 조용한 기반이다.
