# 코드를 쓸 줄 아는 연구자가 살아남는다

대학원에 입학하면 어느 순간 깨닫게 된다. 연구의 절반은 코딩이라는 사실을. 수식을 유도하는 것도 중요하지만, 그 수식을 실제로 돌려보지 않으면 아무것도 증명할 수 없다. 시뮬레이션을 돌리든, 데이터를 분석하든, 그래프를 그리든, 논문의 결과 섹션을 채우는 것은 결국 코드다.

"나는 실험을 하니까 프로그래밍은 필요 없다"고 생각할 수도 있다. 하지만 실험 데이터를 엑셀에서 수작업으로 정리하는 것과, Python 스크립트 하나로 자동 처리하는 것은 효율에서 10배 이상 차이가 난다. 실험을 100번 반복해야 할 때, 매번 엑셀을 여는 사람과 코드 한 줄로 결과를 뽑는 사람의 생산성 격차는 시간이 갈수록 벌어진다.

프로그래밍은 연구자의 기본 언어다. 영어가 국제 학술 커뮤니케이션의 언어라면, 프로그래밍은 컴퓨터와 소통하는 언어다. 이 장에서는 공학 대학원생이 프로그래밍을 시작하고, 연구에 활용하고, 코드 품질을 유지하는 방법을 다룬다. 이미 코딩을 잘하는 사람에게는 불필요한 내용일 수 있지만, 대부분의 공학 대학원생은 학부에서 C언어 한 과목을 들은 정도에서 시작한다. 그 수준에서 출발해도 충분하다.

## 왜 프로그래밍을 배워야 하는가

### 수식은 코드로 검증된다

공학 연구의 핵심은 수학적 모델링이다. 지배 방정식을 세우고, 경계 조건을 설정하고, 해를 구한다. 학부에서는 이 과정을 손으로 풀었다. 간단한 보의 처짐, 단순한 회로의 전압, 이상 기체의 상태 변화. 하지만 대학원에서 다루는 문제는 해석해(analytical solution, 수학적 공식만으로 정확한 답을 구할 수 있는 풀이)가 존재하지 않는 경우가 대부분이다. 비선형 편미분방정식, 복잡한 경계 조건, 다물리 연성 문제(열, 유체, 구조 등 여러 물리 현상이 서로 영향을 주고받는 문제)는 손으로 풀 수 없다. 컴퓨터가 풀어야 한다. 그리고 컴퓨터에게 "이 방정식을 풀어라"고 말하는 방법이 바로 프로그래밍이다.

수치해석 교과서에 나오는 Newton-Raphson 방법(비선형 방정식의 근을 반복 계산으로 찾아가는 대표적인 수치해석 기법)을 배웠다고 치자. 교과서에서는 2x3 정도의 문제를 손으로 풀어본다. 하지만 실제 연구에서는 수십만 개의 자유도를 가진 시스템에 이 방법을 적용해야 한다. 그 과정을 구현하는 것이 코딩이다.

### 데이터 처리와 시각화

실험을 하든, 시뮬레이션을 돌리든, 결과는 데이터로 나온다. CSV 파일, 텍스트 파일, 이미지 파일, 때로는 수 GB짜리 바이너리 파일. 이 데이터를 처리하고, 의미 있는 패턴을 찾아내고, 논문에 실을 수 있는 수준의 그래프를 만드는 것이 데이터 처리와 시각화다.

엑셀로도 할 수 있다. 데이터가 100행 이하이고, 그래프 하나만 그리면 되고, 한 번만 하면 된다면. 하지만 현실은 그렇지 않다. 실험을 50번 반복하고, 각 실험마다 10개의 변수를 기록하고, 이걸 조건별로 비교해야 한다면, 엑셀은 지옥이 된다. Python의 pandas와 matplotlib을 쓰면, 50개 실험 결과를 읽어서 평균/표준편차를 계산하고 그래프를 그리는 데 20줄이면 충분하다. 그리고 내일 실험 10개를 더 추가해도, 코드를 다시 돌리기만 하면 된다.

### 연구 생산성의 레버리지

프로그래밍을 할 줄 아는 연구자와 못하는 연구자의 차이는 시간이 갈수록 복리처럼 불어난다. 첫 달에는 별 차이가 없다. 코드를 배우는 시간이 오히려 투자 비용이다. 하지만 6개월 후, 코딩을 할 줄 아는 사람은 이전 실험 코드를 재활용하고, 파라미터만 바꿔서 새로운 실험을 하고, 결과를 자동으로 정리한다. 코딩을 못하는 사람은 매번 처음부터 수작업을 반복한다.

논문 리비전이 왔을 때 차이가 극명해진다. "Figure 3의 x축 범위를 0-100에서 0-150으로 바꾸고, 새로운 조건 3개를 추가해서 다시 그려 주세요." 코드가 있으면 파라미터 몇 개 수정하고 실행하면 끝이다. 코드가 없으면? 처음부터 다시 해야 한다. 그 차이가 리비전 기간 2주와 2개월의 차이로 이어진다.

### Python vs MATLAB: 신입생의 영원한 질문

공학 대학원 신입생이 가장 많이 하는 질문이 "Python을 배워야 하나요, MATLAB을 배워야 하나요?"다. 답은 상황에 따라 다르지만, 몇 가지 기준을 제시할 수 있다.

**연구실의 기존 코드가 MATLAB이면 MATLAB부터 시작한다.** 연구실에서 선배들이 쌓아온 코드 자산이 MATLAB이라면, 혼자 Python으로 전환하겠다고 고집부리는 것은 비효율적이다. 선배 코드를 읽고, 수정하고, 확장하는 것이 연구의 출발점이다. MATLAB을 먼저 익히고, 필요할 때 Python을 추가로 배우는 것이 현명하다.

**머신러닝/딥러닝을 할 예정이라면 Python이 거의 유일한 선택이다.** TensorFlow, PyTorch, Hugging Face 등 ML/AI 생태계는 사실상 Python 독점이다. MATLAB에도 Deep Learning Toolbox가 있지만, 최신 모델, 사전학습 가중치, 커뮤니티 코드의 99%는 Python이다. ML/AI를 연구 방법론으로 쓸 계획이라면, Python을 주력으로 삼아야 한다.

**졸업 후 산업계를 고려한다면 Python이 유리하다.** MATLAB은 학교 라이선스로 무료로 쓸 수 있지만, 졸업 후에는 연간 수백만 원의 라이선스 비용이 든다. 반면 Python은 영원히 무료다. 산업계에서도 Python의 수요가 MATLAB을 크게 앞서고 있다.

**"둘 다 배워라"가 가장 솔직한 답이다.** 현실적으로 대학원 2-5년 동안 한 언어만 쓰는 경우는 드물다. 기존 코드가 MATLAB이어서 MATLAB로 시작하더라도, 어느 순간 Python이 필요해지는 때가 온다. 두 언어의 기본 문법을 익히는 데는 각각 2-3주면 충분하다. 하나를 깊이 파고, 다른 하나는 기본을 알아두는 전략이 가장 실용적이다.

## 개발 환경 세팅

프로그래밍을 시작하려면 먼저 개발 환경을 세팅해야 한다. 처음 해보는 사람에게는 이 단계가 의외로 큰 벽이다. 코드 한 줄 안 쳤는데, 설치에서 막히면 의욕이 꺾인다. 최대한 간결하게 핵심만 정리한다.

### IDE 선택

### 터미널 기초

터미널(Terminal, 명령 프롬프트)은 컴퓨터를 텍스트 명령어로 조작하는 인터페이스다. GUI에 익숙한 사람에게는 불편해 보이지만, 연구 환경에서 터미널은 필수다. 서버에서 코드를 돌리거나, 패키지를 설치하거나, Git을 사용할 때 터미널을 쓴다.

기본 명령어 몇 가지만 알아도 충분히 시작할 수 있다. `cd`(디렉토리 이동), `ls`(파일 목록 보기), `mkdir`(폴더 만들기), `cp`(파일 복사), `mv`(파일 이동), `rm`(파일 삭제). 이 여섯 가지면 기본적인 파일 관리가 가능하다. 외울 필요 없다. 처음에는 필요할 때마다 검색하면 되고, 반복하다 보면 자연스럽게 손에 익는다.

Mac에서는 Terminal 앱을, Windows에서는 PowerShell이나 Windows Terminal을 쓰면 된다. VS Code에 내장된 터미널을 사용하면 에디터에서 바로 명령어를 실행할 수 있어서 편리하다.

### 가상환경

가상환경은 프로젝트별로 독립된 패키지 환경을 만드는 것이다. 왜 필요한가? 프로젝트 A에서는 numpy 1.21이 필요하고, 프로젝트 B에서는 numpy 1.24가 필요할 수 있다. 하나의 환경에 두 버전을 동시에 설치할 수 없으므로, 프로젝트별로 환경을 분리해야 한다.

**conda**를 추천한다. Anaconda 또는 Miniconda를 설치하면 conda 명령어를 사용할 수 있다. Miniconda가 가볍고 빠르다. 새 환경을 만들 때는 `conda create -n myproject python=3.11`, 활성화할 때는 `conda activate myproject`, 패키지 설치할 때는 `conda install numpy` 또는 `pip install numpy`를 쓴다.

처음에는 번거로워 보이지만, 가상환경 없이 작업하다가 패키지 충돌로 하루를 날리는 경험을 한 번 하면, 그 이후로는 무조건 가상환경부터 만들게 된다. 미래의 나를 위해, 프로젝트를 시작할 때 가상환경을 만드는 습관을 들여라.

## 코딩 습관: 미래의 나를 위한 투자

연구 코드의 가장 큰 문제는 "돌아가기만 하면 된다"는 마인드셋이다. 당장은 돌아가니까 괜찮다. 하지만 3개월 후에 그 코드를 다시 열어야 할 때, 지옥이 시작된다. 변수 이름이 `a`, `b`, `c`이고, 주석은 없고, 함수 하나가 500줄이며, 어디서부터 어디까지가 하나의 기능인지 알 수 없다. 과거의 내가 현재의 나에게 남긴 저주다.

좋은 코딩 습관은 코드를 "잘" 짜는 것이 아니다. 미래의 내가, 또는 다른 사람이 읽었을 때 이해할 수 있도록 짜는 것이다. 몇 가지 핵심 습관을 정리한다.

### 변수명과 함수명

변수명은 그 변수가 무엇인지를 설명해야 한다. `x`보다 `displacement`, `a`보다 `acceleration`, `n`보다 `num_elements`가 낫다. 물론 수학적 관례가 있는 경우(물리량 표기 등)에는 짧은 이름이 자연스러울 수 있다. 강성 행렬을 `K`, 변위 벡터를 `u`로 쓰는 것은 해당 분야 사람이면 모두 이해한다. 하지만 그 외의 변수들, 예를 들어 반복 횟수, 파일 경로, 설정값 등은 의미가 드러나는 이름을 써야 한다.

함수명도 마찬가지다. `func1`, `process`, `do_stuff` 같은 이름은 아무 정보도 전달하지 않는다. `calculate_stress`, `load_experimental_data`, `plot_convergence` 같이 동사+목적어 형태로 명확하게 쓰는 것이 좋다.

### 주석

주석은 "왜(Why)"를 설명하는 것이다. "무엇을(What)"은 코드 자체가 설명해야 한다. `x = x + 1 # x에 1을 더한다` 같은 주석은 무의미하다. 대신 `x = x + 1 # 경계 조건에서의 오프셋 보정`이 유용한 주석이다.

특히 연구 코드에서 중요한 주석은 다음과 같다. 수식의 출처(어떤 논문의 몇 번 식인지), 매직 넘버의 의미(0.4가 어디서 나온 상수인지), 특이한 처리를 한 이유(왜 여기서 데이터를 잘라내는지). 이런 정보가 없으면, 코드를 읽는 사람은 그 숫자가 왜 그 값인지, 그 처리가 왜 필요한지를 역추적해야 한다. 그 사람이 3개월 후의 나 자신이라면, 특히 고통스럽다.

### 함수 분리

하나의 함수는 하나의 일을 해야 한다. 데이터를 읽는 코드, 전처리하는 코드, 계산하는 코드, 결과를 그리는 코드가 하나의 거대한 스크립트에 순서대로 나열되어 있으면 스파게티 코드다. 각각을 함수로 분리하면 구조가 명확해진다.

```python

data = data[data["value"] > 0]
```

### README와 requirements.txt

프로젝트 폴더에 README 파일을 만들어라. 거창할 필요 없다. 이 프로젝트가 무엇인지(한 줄), 어떻게 실행하는지(명령어), 필요한 패키지가 무엇인지, 데이터 파일은 어디 있는지. 이 네 가지만 적어두면 된다.

`requirements.txt`는 이 프로젝트에 필요한 Python 패키지 목록이다. `pip freeze > requirements.txt`로 자동 생성할 수 있다. 다른 사람(또는 미래의 내)이 `pip install -r requirements.txt` 한 줄로 동일한 환경을 구축할 수 있게 해준다. conda를 쓴다면 `environment.yml`도 같은 역할을 한다.

## 디버깅: 에러와 친해지는 법

코드를 쓰면 에러가 난다. 이것은 실력의 문제가 아니라 코딩의 본질이다. 경력 20년의 소프트웨어 엔지니어도 매일 에러를 만난다. 차이는 에러를 만났을 때 대처하는 속도와 체계다.

### 에러 메시지를 읽어라

가장 기본적이지만 가장 많은 사람이 건너뛰는 단계다. 에러 메시지는 "무엇이 잘못되었는지"를 알려준다. Python의 에러 메시지는 특히 친절한 편이다. `FileNotFoundError: [Errno 2] No such file or directory: 'data.csv'`라고 나오면, `data.csv` 파일이 없다는 뜻이다. 파일 경로를 확인하면 된다.

에러 메시지의 마지막 줄이 핵심이다. `Traceback`(역추적)은 에러가 발생한 경로를 보여주는데, 아래에서 위로 읽는다. 마지막 줄에 에러 종류와 설명이 나오고, 그 위에 에러가 발생한 코드 줄이 표시된다.

### 에러 유형별 대처

에러에는 패턴이 있다. 몇 가지 자주 만나는 유형을 알아두면 대처가 빨라진다.

**SyntaxError**: 문법 오류. 괄호를 닫지 않았거나, 콜론을 빼먹었거나, 들여쓰기가 잘못되었다. 에러 메시지가 가리키는 줄과 그 바로 윗줄을 확인한다.

**NameError**: 정의되지 않은 변수를 사용했다. 변수명 오타인 경우가 대부분이다. `displacement`를 `displacment`으로 쓴 것 같은.

**IndexError**: 리스트나 배열의 범위를 벗어났다. 길이가 10인 배열의 10번째 인덱스(0부터 시작하므로 9가 마지막)에 접근하려 했다.

**TypeError**: 잘못된 타입의 데이터를 연산에 사용했다. 문자열과 숫자를 더하려 했다거나, 함수에 잘못된 인자를 넘겼다.

**ValueError**: 값이 올바르지 않다. 행렬 크기가 맞지 않아서 곱셈이 안 된다거나, 음수에 로그를 취하려 했다.

### print 디버깅

가장 원시적이지만 가장 확실한 방법이다. 의심되는 지점에 `print()`를 넣어서 변수의 값, 데이터의 형태, 코드의 실행 흐름을 확인한다. 디버깅이 끝나면 print 문을 지워야 한다는 것만 잊지 않으면 된다.

```python
# 디버깅 예시
print(f"data shape: {data.shape}")      # 데이터 크기 확인
print(f"max value: {data.max()}")       # 최대값 확인
print(f"type: {type(result)}")          # 타입 확인
print("여기까지 실행됨")                  # 실행 흐름 확인
```

### breakpoint 디버깅

print 디버깅의 업그레이드 버전이다. 코드의 특정 지점에서 실행을 멈추고, 그 시점의 모든 변수를 탐색할 수 있다. Python에서는 `breakpoint()`를 코드에 넣으면 해당 지점에서 디버거가 작동한다. VS Code의 디버깅 기능을 활용하면 GUI에서 편리하게 breakpoint를 설정하고, 변수를 실시간으로 모니터링할 수 있다.

처음에는 print 디버깅만으로도 충분하지만, 코드가 복잡해지면 breakpoint 디버깅이 훨씬 효율적이다. 특히 반복문 안에서 특정 조건일 때만 멈추고 싶다면, 조건부 breakpoint를 활용한다.

## 연구 코드 vs 프로덕션 코드

연구 코드와 산업계의 프로덕션 코드는 근본적으로 다른 목적을 가진다. 프로덕션 코드(production code, 실제 서비스 환경에서 사용자에게 제공되는 코드)는 수백만 사용자가 쓸 서비스를 구동하는 코드다. 안정성, 보안, 확장성이 최우선이다. 연구 코드는 가설을 검증하고, 실험 결과를 생산하고, 논문을 쓰기 위한 코드다. 목적이 다르므로 기준도 달라야 한다.

### 완벽할 필요는 없다

연구 코드에 대해 가장 중요한 조언은 이것이다. 완벽한 코드를 쓰려고 하지 마라. 소프트웨어 공학의 모든 원칙을 적용하려고 하면, 코드를 쓰는 데 연구 시간의 80%를 쓰게 된다. 디자인 패턴(자주 반복되는 설계 문제의 정형화된 해결 방식), 유닛 테스트 100% 커버리지(코드의 모든 기능을 자동으로 검증하는 테스트의 완전한 적용), CI/CD 파이프라인(코드 변경 시 자동으로 테스트하고 배포하는 자동화 시스템) 같은 것들은 프로덕션 코드에는 필수지만, 한 번 돌리고 결과를 확인할 연구 코드에는 과하다.

하지만 "완벽할 필요 없다"가 "아무렇게나 짜도 된다"는 뜻은 아니다. 최소한의 기준이 있다.

### 최소한의 기준: 재현 가능성

연구 코드의 핵심 기준은 **재현 가능성(Reproducibility)**이다. 같은 코드를 같은 데이터에 같은 환경에서 돌렸을 때, 같은 결과가 나와야 한다. 당연한 것 같지만, 실제로는 놀랍도록 어렵다.

재현 가능성을 확보하기 위한 최소한의 체크리스트가 있다.

<div class="check-list">
  <label><input type="checkbox"> 랜덤 시드를 고정했는가? (np.random.seed(42) 또는 torch.manual_seed(42))</label>
  <label><input type="checkbox"> 패키지 버전을 기록했는가? (requirements.txt 또는 environment.yml)</label>
  <label><input type="checkbox"> 데이터 전처리 과정이 코드에 포함되어 있는가? (수작업 전처리 금지)</label>
  <label><input type="checkbox"> 결과 파일의 이름에 실험 조건이 포함되어 있는가?</label>
  <label><input type="checkbox"> 어떤 코드 버전으로 어떤 결과를 생성했는지 기록했는가?</label>
</div>

### 실험 결과 저장

시뮬레이션을 돌릴 때마다 결과를 체계적으로 저장해야 한다. "나중에 정리하지" 하고 미루면, 3일 후에 어떤 결과가 어떤 조건으로 생성된 것인지 알 수 없게 된다.

가장 간단한 방법은 결과 폴더에 타임스탬프와 실험 조건을 포함하는 것이다. 예를 들어 `results/2025-03-15_lr0.001_epoch100/` 같은 식이다. 각 폴더에 실험 설정(하이퍼파라미터, 데이터 경로 등)을 JSON이나 YAML 파일로 저장하면, 나중에 어떤 조건으로 돌렸는지를 코드 없이도 확인할 수 있다.

더 체계적인 방법으로는 MLflow, Weights & Biases(W&B) 같은 실험 추적 도구가 있다. 이런 도구들은 하이퍼파라미터, 메트릭, 아티팩트를 자동으로 기록하고, 실험 간 비교를 쉽게 해준다. 머신러닝 연구라면 적극 권장한다.

### 랜덤 시드의 중요성

머신러닝은 물론이고, 몬테카를로 시뮬레이션(난수를 반복적으로 생성하여 확률적으로 결과를 추정하는 계산 기법), 최적화 알고리즘 등 랜덤성이 포함된 모든 연구에서 랜덤 시드 고정은 필수다. 시드를 고정하지 않으면, 같은 코드를 돌려도 매번 다른 결과가 나온다. "어제는 정확도가 95%였는데 오늘은 87%네?" 이게 코드 문제인지, 데이터 문제인지, 단순히 랜덤 시드 때문인지 구분할 수 없게 된다.

Python에서 랜덤 시드를 고정하려면 관련된 모든 라이브러리에서 시드를 설정해야 한다. `random.seed(42)`, `np.random.seed(42)`, `torch.manual_seed(42)`, 그리고 GPU를 쓴다면 `torch.cuda.manual_seed_all(42)`까지. 42라는 숫자에 특별한 의미는 없다. 관례적으로 많이 쓰일 뿐이다. 어떤 숫자든 상관없지만, 일관되게 같은 시드를 쓰는 것이 중요하다.

## 연구에서 자주 쓰는 Python 패턴

마지막으로, 공학 연구에서 자주 반복되는 Python 코드 패턴 몇 가지를 소개한다. 이 코드들을 통째로 외울 필요는 없다. 이런 것이 가능하다는 것만 알아두고, 필요할 때 찾아서 쓰면 된다.

**여러 파일을 한 번에 읽기**: 실험 데이터가 `exp_01.csv`, `exp_02.csv`, ... 으로 저장되어 있을 때, 반복문으로 한 번에 읽어 합칠 수 있다. `glob` 모듈이나 `pathlib`을 활용한다.

**파라미터 스위프**: 시뮬레이션의 입력 변수를 바꿔가며 여러 번 실행하는 것. 이중 반복문이나 `itertools.product`를 사용한다. 결과를 데이터프레임으로 정리하면 분석이 편해진다.

**결과 자동 플롯**: 실험 결과를 자동으로 그래프로 만드는 스크립트. 한 번 만들어 두면, 새 데이터가 추가될 때마다 스크립트만 돌리면 된다. 논문 리비전에서 특히 위력을 발휘한다.

**설정 파일 분리**: 코드 안에 하드코딩된 파라미터를 JSON이나 YAML 파일로 분리하는 것. 코드를 수정하지 않고 설정 파일만 바꿔서 다른 실험을 실행할 수 있다.

프로그래밍은 결국 도구다. 연구를 위한 도구. 도구 자체에 매몰되지 말고, 도구를 써서 무엇을 할 수 있는지에 집중해야 한다. 완벽한 코드를 쓰는 데 3개월을 쓰는 것보다, 돌아가는 코드를 빨리 만들고 연구 결과를 내는 것이 대학원생에게는 더 중요하다. 단, 최소한의 기록과 재현 가능성만은 지켜야 한다. 논문에 실린 결과를 재현할 수 없는 코드는, 아무리 좋은 결과를 냈더라도 의미가 없다.

공학 연구자에게 프로그래밍이란, 펜과 종이에서 벗어나 자신의 아이디어를 실제로 구현하고 검증하는 힘이다. 처음에는 어색하고 답답하겠지만, 한 줄씩 쓰다 보면 어느 순간 코드가 손에 익고, 그때부터 연구의 속도가 완전히 달라진다. 시작이 반이다. 오늘 Miniconda를 설치하고, VS Code를 열고, `print("Hello, Research")`를 실행해 보자.

## 코드 품질 관리

"돌아가기만 하면 되지"라는 말은 연구 코드에서 가장 위험한 생각이다. 당장은 돌아가더라도, 3개월 후 리비전이 왔을 때 자기가 쓴 코드를 자기가 이해하지 못하는 상황이 온다. 코드 품질 관리는 소프트웨어 회사만의 이야기가 아니다. 연구 코드에도 최소한의 품질 기준이 필요하다. 이 섹션에서는 린터와 포매터, 테스트, 코드 리뷰라는 세 가지 축으로 연구 코드의 품질을 높이는 방법을 다룬다.

### 린터와 포매터

린터(Linter)는 코드의 잠재적 오류와 스타일 문제를 자동으로 찾아주는 도구다. 포매터(Formatter)는 코드의 형식을 자동으로 정리해주는 도구다. 둘 다 "실행하기 전에" 문제를 잡아준다.

**왜 연구 코드에도 린터가 필요한가.** 연구 코드는 빠르게 작성되고, 여러 번 수정되고, 가끔 다른 사람에게 넘겨진다. 이 과정에서 변수명 오타, 사용하지 않는 import, 정의되지 않은 변수 참조 같은 문제가 쉽게 생긴다. 이런 문제는 코드를 실행해야 발견되고, 실행해도 즉시 에러가 나지 않고 조용히 잘못된 결과를 만들어낼 수 있다. 린터는 이런 문제를 코드를 실행하지 않고도 잡아준다.

**Python 린터 추천.** `flake8`은 가볍고 빠르다. PEP 8 스타일 위반, 사용하지 않는 변수, 복잡도가 높은 함수 등을 잡아준다. `pylint`는 더 엄격하고 상세하다. 코드 품질 점수까지 매겨준다. 처음에는 flake8으로 시작하고, 익숙해지면 pylint로 넘어가는 것을 추천한다. 최근에는 `ruff`라는 도구가 flake8과 pylint의 기능을 통합하면서도 훨씬 빠르게 동작하여 인기를 얻고 있다.

**Python 포매터 추천.** `black`은 "opinionated formatter"로, 설정할 것이 거의 없다. 실행하면 코드를 일관된 스타일로 자동 정리한다. 들여쓰기, 줄 바꿈, 따옴표 스타일 등을 통일해준다. `isort`는 import 문을 알파벳순, 카테고리순으로 자동 정렬한다. 둘을 함께 쓰면 코드 스타일에 대한 고민을 완전히 없앨 수 있다.

**타입 힌트(Type Hints) 기초.** Python은 동적 타입 언어라서 변수에 어떤 타입이 들어가든 에러가 나지 않는다. 하지만 타입 힌트를 추가하면 코드의 의도가 명확해지고, `mypy` 같은 도구로 타입 불일치를 사전에 잡을 수 있다. 예를 들어 `def calculate_stress(force: float, area: float) -> float:`처럼 함수의 입출력 타입을 표시하면, 이 함수가 무엇을 받아서 무엇을 반환하는지 코드 자체가 문서가 된다.

### 연구 코드에도 테스트가 필요하다

"내 코드는 내가 직접 확인했으니 괜찮다"고 생각할 수 있다. 하지만 사람의 눈은 놀라울 정도로 자기 코드의 버그를 못 본다. 특히 데이터 전처리, 모델 입출력, 평가 지표 계산 같은 부분에서 미묘한 버그가 숨어 있으면, 논문의 결과 전체가 흔들릴 수 있다. "it works on my machine"은 연구에서는 충분하지 않다. 코드가 정확한지 체계적으로 검증하는 방법이 테스트다.

**pytest 기초.** Python의 대표적인 테스트 프레임워크인 `pytest`는 사용이 간단하다. 테스트 파일을 만들고(`test_*.py`), 함수 이름을 `test_`로 시작하고, `assert`로 기대값을 확인하면 된다. 터미널에서 `pytest`를 실행하면 모든 테스트가 자동으로 돌아간다.

**연구 코드에서 무엇을 테스트할 것인가.** 모든 코드를 테스트할 필요는 없다. 하지만 다음 네 가지는 반드시 테스트해야 한다.

첫째, 데이터 로딩과 전처리다. 입력 데이터의 shape이 기대와 같은지, 결측치 처리가 올바른지, 정규화 후 값의 범위가 올바른지 등을 확인한다. 데이터 전처리 버그는 가장 흔하고, 발견하기 가장 어렵다.

둘째, 모델의 입출력 shape이다. 모델에 [batch_size, input_dim] 데이터를 넣었을 때, 출력이 [batch_size, output_dim]인지 확인한다. shape 불일치는 런타임 에러로 이어지거나, 더 나쁘게는 에러 없이 잘못된 결과를 만들어낸다.

셋째, 알려진 결과와의 비교다. 해석해(analytical solution)가 있는 간단한 케이스에서 코드의 출력이 해석해와 일치하는지 확인한다. 예를 들어, 외팔보의 끝단 처짐을 계산하는 FEM 코드가 있다면, 교과서의 공식과 비교할 수 있다.

넷째, 평가 지표 계산이다. 정확도, RMSE, R-squared 등의 평가 지표를 수동으로 계산한 값과 코드의 출력이 일치하는지 확인한다. 이 부분에서 실수하면 논문의 핵심 수치가 틀릴 수 있다.

**테스트의 현실적 수준.** 100% 코드 커버리지를 추구할 필요는 없다. 연구 코드는 빠르게 변하고, 탐색적 성격이 강하기 때문이다. 핵심 함수 10개에 대해 기본적인 테스트 20개를 작성하는 것만으로도 큰 차이를 만든다. "테스트를 작성하는 시간이 아깝다"고 느낄 수 있지만, 논문 제출 후 결과가 재현되지 않아 밤새 디버깅하는 시간에 비하면 훨씬 작은 투자다.

### 코드 리뷰 문화

코드 리뷰란, 자기가 작성한 코드를 다른 사람이 읽고 피드백을 주는 과정이다. 소프트웨어 업계에서는 당연한 관행이지만, 대학원 연구실에서는 흔하지 않다. 하지만 연구 코드야말로 리뷰가 필요한 코드다. 왜냐하면 그 코드의 결과가 논문에 실리기 때문이다.

**코드 리뷰의 이점.** 첫째, 버그를 잡는다. 자기 코드는 자기 눈에 익어서 실수를 놓치기 쉽다. 다른 사람의 눈은 자기가 보지 못한 것을 본다. 둘째, 지식이 공유된다. 선배의 코드를 리뷰하면서 새로운 기법을 배우고, 후배의 코드를 리뷰하면서 기초를 다시 점검할 수 있다. 셋째, 코드의 가독성이 올라간다. "다른 사람이 읽을 것"이라는 의식이 있으면, 변수명도 신경 쓰고, 주석도 달게 된다.

**연구실에서 코드 리뷰를 시작하는 법.** 공식적인 프로세스가 없어도 시작할 수 있다. 가장 간단한 방법은 동기나 선배에게 "이 코드 한번 봐줄 수 있어?"라고 부탁하는 것이다. 전체 코드가 아니라, 핵심 함수 2-3개만 봐달라고 하면 부담이 적다. "이 함수가 하려는 것은 X인데, 제가 제대로 구현한 건지 모르겠어요"라고 구체적으로 요청하면 리뷰어도 초점을 맞추기 쉽다.

ch16 '버전 관리'에서 다루는 GitHub을 사용한다면, Pull Request를 통한 코드 리뷰가 자연스럽다. 코드 변경 사항을 PR로 올리고, 리뷰어가 라인별로 코멘트를 남기는 방식이다. 기록이 남으므로 나중에 "왜 이렇게 바꿨지?"를 추적할 수 있다.

**코드 리뷰 시 확인할 것.** 연구 코드 리뷰에서 특히 중요한 체크 포인트가 있다. (1) 물리적/수학적 의미가 맞는가? 수식이 코드로 정확하게 옮겨졌는가? (2) 단위(unit)가 일관성 있는가? SI 단위와 다른 단위가 섞여 있지 않은가? (3) 하드코딩된 값이 없는가? 매직 넘버(설명 없이 코드에 직접 박혀 있는 숫자)가 있다면 변수나 상수로 바꿔야 한다. (4) 경계 조건은 올바른가? 입력이 0이거나 매우 큰 값일 때도 코드가 정상 작동하는가?

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>코드 리뷰는 비판이 아니라 협력이다</strong></p>
    <p>코드 리뷰를 받을 때 방어적이 되기 쉽다. "내 코드에 뭔 문제가 있다는 거야?"라는 감정이 올라올 수 있다. 하지만 리뷰의 목적은 코드를 더 좋게 만드는 것이지, 사람을 평가하는 것이 아니다. 리뷰를 줄 때도 "이 코드가 나쁘다"가 아니라 "이 부분을 이렇게 바꾸면 더 읽기 쉬울 것 같다"는 식으로 건설적으로 표현해야 한다.</p>
  </div>
</div>

## 연구 코드의 디버깅 전략

박사 과정에서 코드를 작성하는 시간보다 디버깅하는 시간이 더 많을 수 있다. 디버깅 능력이 연구 진도의 큰 부분을 결정한다. ch20에서 ML 모델의 디버깅을 다뤘다면, 이 섹션은 일반 연구 코드의 디버깅을 다룬다.

<div class="highlight-box info">
  <span class="highlight-box-icon">ℹ️</span>
  <div class="highlight-box-content">
    <p><strong>디버깅의 본질</strong></p>
    <p>디버깅은 무언가에 대해 잘못 알고 있는 부분을 찾는 과정이다. 코드가 잘 작동할 것이라고 믿었지만 실제로는 다르게 작동한다. 디버깅의 목표는 가정과 실제 동작 사이의 차이를 찾는 것이다.</p>
  </div>
</div>

**디버깅의 5가지 도구.**

**도구 1: Print 디버깅.**
가장 단순하고 흔한 방법. 변수 값, 함수 진입점, 흐름 순서를 print로 출력한다.

```python
def my_function(x, y):
    print(f"DEBUG: x={x}, y={y}")  # 입력 확인
    result = x * y + 1
    print(f"DEBUG: result={result}")  # 출력 확인
    return result
```

장점: 즉시 사용 가능, 학습 곡선 없음.
단점: 디버깅 후 print를 모두 제거해야 한다.

**팁**: print 대신 logging 모듈을 사용하면 디버깅 후 로그 레벨만 변경하면 된다 (DEBUG → INFO).

```python
import logging
logging.basicConfig(level=logging.DEBUG)
logger = logging.getLogger(__name__)

def my_function(x, y):
    logger.debug(f"x={x}, y={y}")
    result = x * y + 1
    logger.debug(f"result={result}")
    return result
```

**도구 2: 인터랙티브 디버거 (pdb / ipdb).**
코드의 특정 지점에서 실행을 멈추고 변수 상태를 검사할 수 있다. Python의 `pdb` 또는 더 사용하기 쉬운 `ipdb`를 쓴다.

```python
import ipdb

def my_function(x, y):
    result = x * y + 1
    ipdb.set_trace()  # 여기서 실행이 멈춘다
    return result
```

`ipdb.set_trace()`에서 변수를 검사하고, 한 줄씩 실행하고, 함수에 진입할 수 있다.

**기본 명령어**:
- `n` (next): 다음 줄로
- `s` (step): 함수에 진입
- `c` (continue): 다음 breakpoint까지 계속
- `p variable` (print): 변수 출력
- `q` (quit): 디버거 종료

장점: print보다 강력, 모든 변수를 즉시 검사.
단점: 약간의 학습 곡선.

**도구 3: IDE 디버거.**
VS Code, PyCharm 같은 IDE는 시각적 디버거를 제공한다. 라인 옆에 빨간 점(breakpoint)을 클릭으로 추가하고 디버거를 시작한다.

**VS Code의 Python 디버거 설정**:
1. Python 확장 설치
2. 코드 라인 옆에 breakpoint 클릭
3. F5로 디버거 시작
4. 변수를 사이드 패널에서 검사

장점: 시각적이고 사용하기 쉽다.
단점: IDE에 의존.

**도구 4: 로깅 (Logging).**
큰 프로젝트나 장시간 실행되는 코드(시뮬레이션, 학습)에서는 로깅이 디버깅의 핵심이다. print는 콘솔이 사라지면 정보가 사라지지만, 로그는 파일에 저장된다.

```python
import logging

logging.basicConfig(
    filename='experiment.log',
    level=logging.INFO,
    format='%(asctime)s - %(levelname)s - %(message)s'
)

logger = logging.getLogger(__name__)

logger.info("실험 시작")
logger.info(f"하이퍼파라미터: lr={lr}, batch={batch}")
# ... 실험 진행 ...
logger.warning("loss가 발산하기 시작함")
logger.error("학습 실패: out of memory")
```

실험 후 로그 파일을 검토하면 무엇이 잘못되었는지 알 수 있다.

**도구 5: Git Bisect — 버그가 언제 들어왔는가.**
어제까지 잘 작동하던 코드가 오늘은 안 된다. 어떤 변경이 버그를 도입했는지 모를 때, `git bisect`가 이진 탐색으로 버그 도입 시점을 찾아준다.

```bash
git bisect start
git bisect bad                    # 현재 commit이 문제 있음
git bisect good a3f5b2c           # 이 commit은 정상이었음
# Git이 중간 commit으로 자동 이동
# 코드를 테스트하고:
git bisect good                   # 또는 bad
# 반복하면 버그 도입 commit이 발견됨
git bisect reset                  # 종료
```

1000개의 commit 중 버그 commit을 10단계 안에 찾을 수 있다. ch16의 Git 고급 기법 참조.

**디버깅의 5가지 원칙.**

1. **가정을 의심하라**: 디버깅의 가장 큰 함정은 "이 부분은 분명히 맞다"는 가정. 가장 확신하는 부분이 가장 의심해야 할 부분이다.
2. **단순한 사례부터**: 큰 입력으로 안 되면 작은 입력으로 시도한다. 작은 입력에서 버그가 더 명확히 보인다.
3. **한 번에 하나만 변경**: 여러 부분을 동시에 바꾸면 무엇이 효과 있었는지 모른다. 한 번에 하나씩 변경하고 결과를 확인한다.
4. **코드를 다른 사람에게 설명**: 코드를 동료에게 설명하다 보면 종종 직접 버그를 발견한다. 이것을 "Rubber Duck Debugging"이라고 한다(고무 오리에게 설명해도 작동한다).
5. **휴식 후 다시**: 1-2시간 막혔다면 잠시 떠난다. 산책 후 다시 보면 새 시각으로 보인다.

**에러 메시지 읽기.**

디버깅의 첫 단계는 에러 메시지를 정확히 읽는 것이다. 박사 학생의 흔한 실수는 에러 메시지를 무시하고 추측하는 것이다.

Python의 traceback 읽는 법:

```
Traceback (most recent call last):
  File "main.py", line 15, in <module>
    result = analyze(data)
  File "analyzer.py", line 42, in analyze
    return data.mean() / data.std()
ZeroDivisionError: float division by zero
```

traceback의 마지막 줄이 가장 중요하다. `ZeroDivisionError: float division by zero` — 0으로 나누고 있다. 위로 올라가면 어디서 0으로 나누는지 보인다(`analyzer.py`의 42번째 줄).

traceback을 처음부터 끝까지 읽는다. 에러를 검색(Google, Stack Overflow, AI 도구)하기 전에 먼저 에러 메시지를 정확히 이해한다.

디버깅 능력은 박사 학위의 핵심 기술 중 하나다. 박사 졸업 시점에 디버깅의 마스터가 되어 있다면, 산업체든 학계든 어디서나 가치 있는 사람이 된다.

## 연구 코드의 성능 프로파일링 — 느린 코드를 빠르게

연구 코드가 작동은 하지만 너무 느려서 실험을 돌릴 수 없다. 시뮬레이션이 3일 걸리고, 파라미터 스터디를 시도하면 한 달이 넘는다. 이런 상황에서 많은 학생이 "더 빠른 컴퓨터가 필요하다"고 생각하지만, 대부분의 경우 **코드 자체에 10-100배 속도 향상의 여지**가 있다. 필요한 것은 더 빠른 하드웨어가 아니라 **프로파일링(profiling)** 기술이다. 어디가 느린지 측정하고, 그 부분만 최적화하는 것이다.

<div class="highlight-box highlight-important">

**"빠를 것 같은 부분"은 틀리기 쉽다.** "이 부분이 느릴 것 같다"고 추측한 것이 실제로는 전체 실행 시간의 5%만 차지하고, "빠를 것 같다"고 생각한 부분이 80%를 차지하는 경우가 매우 흔하다. Donald Knuth의 유명한 격언 "premature optimization is the root of all evil"이 이것을 경계한다. 추측 대신 측정한다.

</div>

**프로파일링의 3단계.**

**1단계: 전체 실행 시간 측정.**
먼저 코드가 실제로 얼마나 걸리는지 기록한다. 간단한 타이머로 충분하다.

```python
import time
start = time.time()
main()
print(f"Total: {time.time() - start:.2f}s")
```

이것이 기준선(baseline)이다. 최적화 후 이 수치와 비교하여 효과를 측정한다.

**2단계: 함수 수준 프로파일링.**
어느 함수가 시간을 많이 쓰는지 파악한다. Python은 `cProfile`이 표준이다.

```python
import cProfile
import pstats

profiler = cProfile.Profile()
profiler.enable()
main()
profiler.disable()

stats = pstats.Stats(profiler)
stats.sort_stats('cumulative')
stats.print_stats(20)  # 상위 20개 함수
```

출력 예시:
```
ncalls  tottime  percall  cumtime  percall filename:lineno
  1000    5.234    0.005   12.456    0.012 analyze.py:45(process_batch)
  1000    3.123    0.003    3.123    0.003 numpy.py:123(dot)
 10000    0.123    0.000    4.567    0.000 utils.py:12(normalize)
```

- **tottime**: 이 함수 자체에서 쓴 시간 (호출한 하위 함수 제외).
- **cumtime**: 이 함수와 그 하위 호출의 누적 시간.
- **ncalls**: 호출 횟수.

가장 먼저 봐야 할 것은 **cumtime이 큰 함수**다. 그 함수가 코드의 병목(bottleneck)이다.

**3단계: 라인 수준 프로파일링.**
병목 함수 안에서도 어느 **줄**이 느린지 알고 싶다. `line_profiler`를 쓴다.

```bash
pip install line_profiler
```

```python
@profile  # 함수에 데코레이터 추가
def process_batch(data):
    result = np.zeros(len(data))
    for i in range(len(data)):
        result[i] = expensive_computation(data[i])
    return result
```

```bash
kernprof -l -v my_script.py
```

출력이 각 줄의 시간을 보여준다.

```
Line #   Hits  Time    Per Hit  % Time  Line Contents
    15   1000    45      0.05    0.5   result = np.zeros(len(data))
    16   1000    23      0.02    0.3   for i in range(len(data)):
    17 100000  8234      0.08   95.0   result[i] = expensive_computation(data[i])
    18   1000    12      0.01    0.1   return result
```

17번 줄이 95%를 차지한다. 최적화의 대상이 명확해진다.

**메모리 프로파일링.**

시간뿐 아니라 메모리 사용도 프로파일링할 수 있다. `memory_profiler`를 쓴다.

```bash
pip install memory_profiler
```

```python
@profile
def my_function():
    a = [1] * 1000000
    b = [2] * 2000000
    del b
    return a
```

```bash
python -m memory_profiler my_script.py
```

메모리 부족(Out of Memory) 문제가 있을 때 특히 유용하다.

**가장 흔한 성능 문제 5가지와 해결.**

**1. Python 루프 대신 NumPy 벡터화.**
Python의 for 루프는 느리다. 같은 연산을 NumPy로 벡터화하면 10-100배 빨라진다.

```python
# 느림 (Python 루프)
result = np.zeros(len(data))
for i in range(len(data)):
    result[i] = data[i] ** 2 + 3 * data[i] + 1

# 빠름 (NumPy 벡터화)
result = data ** 2 + 3 * data + 1
```

**2. 반복 계산 캐싱.**
같은 값을 여러 번 계산하지 말고 한 번 계산하여 저장한다.

```python
# 느림: 같은 값을 매번 다시 계산
for x in data:
    process(x, get_threshold())  # get_threshold()가 매번 호출됨

# 빠름: 한 번만 계산
threshold = get_threshold()
for x in data:
    process(x, threshold)
```

Python의 `@functools.lru_cache` 데코레이터로 함수 결과를 자동 캐싱할 수도 있다.

**3. 잘못된 자료구조.**
리스트에서 멤버십 검사(`x in list`)는 O(n)이지만, 집합(set)에서는 O(1)이다.

```python
# 느림: O(n) per check
visited = []
for item in items:
    if item not in visited:  # O(n)
        visited.append(item)
        process(item)

# 빠름: O(1) per check
visited = set()
for item in items:
    if item not in visited:  # O(1)
        visited.add(item)
        process(item)
```

**4. 불필요한 복사.**
NumPy 배열이나 큰 자료구조를 복사하는 것은 비싸다. 필요하지 않으면 피한다.

```python
# 느림: 전체 배열 복사
result = big_array.copy()
result[0] = 1

# 빠름: 뷰 사용 (가능한 경우)
big_array[0] = 1
```

**5. I/O 병목.**
파일 읽기/쓰기가 계산보다 느릴 때가 있다. 한 번에 많은 데이터를 읽는 것이 작게 여러 번 읽는 것보다 빠르다.

```python
# 느림: 매 루프마다 디스크 접근
for i in range(1000):
    data = np.load(f"file_{i}.npy")
    process(data)

# 빠름: 한 번에 많은 데이터를 로드
all_data = [np.load(f"file_{i}.npy") for i in range(1000)]
for data in all_data:
    process(data)
```

**병렬화 (Parallelization).**

단일 코드가 충분히 최적화된 후에도 느리다면 병렬화를 고려한다. Python은 GIL(Global Interpreter Lock) 때문에 스레드 병렬화가 제한적이다. 프로세스 병렬화가 일반적이다.

```python
from multiprocessing import Pool

def process_one(item):
    return expensive_computation(item)

if __name__ == '__main__':
    with Pool(8) as pool:  # 8개 프로세스
        results = pool.map(process_one, items)
```

8개 코어로 돌리면 이론적으로 8배 빨라진다. 실제로는 프로세스 시작 비용과 통신 오버헤드 때문에 5-6배 정도가 일반적이다.

NumPy 내부 연산은 이미 BLAS 라이브러리로 병렬화되어 있으므로, 수치 계산 코드에서는 명시적 병렬화가 필요 없을 수 있다. `np.dot()`, `np.linalg.solve()` 같은 함수는 내부적으로 여러 코어를 사용한다.

**GPU 가속의 판단.**

계산이 매우 많고 벡터화 가능한 경우 GPU로 옮기면 10-50배 빨라질 수 있다. 다만 GPU로 옮기는 데 드는 노력과 학습 곡선을 고려한다.

- **CuPy**: NumPy와 거의 같은 API로 GPU 연산. 기존 NumPy 코드를 최소한의 변경으로 GPU로 옮길 수 있다.
- **PyTorch / JAX**: 자동 미분과 GPU 가속을 모두 제공. 과학 계산에 점점 많이 쓰인다.
- **Numba**: Python 코드를 JIT(Just-In-Time) 컴파일하여 CPU 또는 GPU에서 가속. `@njit` 데코레이터 하나로 10배 향상이 가능.

**"최적화하지 말아야 할 코드".**

모든 코드를 최적화할 필요는 없다. 다음 기준을 따른다.

1. **일회성 스크립트**: 한 번만 돌릴 코드는 최적화할 필요가 없다. 1시간 돌려서 끝나면 그걸로 충분.
2. **디버깅 중인 코드**: 아직 정확성이 확립되지 않은 코드를 최적화하면, 최적화가 버그를 숨긴다. 먼저 정확성을 확보하고 그 다음 성능.
3. **가독성 희생이 큰 최적화**: 10% 향상을 위해 코드를 읽기 어렵게 만드는 것은 나쁜 거래다. 미래의 본인(또는 후배)이 이해할 수 없으면 유지보수가 불가능해진다.

최적화는 코드의 1-10%에 집중한다. 나머지는 읽기 쉽게 유지한다.

**프로파일링과 최적화의 반복 주기.**

최적화는 한 번에 끝나지 않는다. 다음 주기를 반복한다.

1. **측정**: 현재 속도 기록.
2. **프로파일링**: 가장 느린 부분 식별.
3. **가설**: 왜 느린지 이유 추정.
4. **수정**: 개선 시도.
5. **재측정**: 실제로 빨라졌는지 확인.
6. **반복**: 다른 병목으로 이동.

각 주기마다 "몇 %나 빨라졌는가"를 기록한다. 때로는 수정이 오히려 느리게 만들거나 미미한 효과만 있다. 그것도 데이터다.

> 프로파일링은 박사 과정에서 많이 쓰이지 않지만, 알면 강력한 도구다. 시뮬레이션이나 분석이 느려서 진도가 안 나간다면, 하드웨어 탓을 하기 전에 1-2시간의 프로파일링을 해본다. 대부분의 경우 코드의 한두 부분만 수정해도 전체가 10배 빨라진다. 이 시간 절약이 박사 과정의 나머지 시간을 훨씬 여유 있게 만든다.

### 박사 단계별 활용 전략

**박사 1-2년차 (기초 학습기)**:
- **최소 의존**: AI를 50% 이상 쓰지 말 것. 본인이 직접 문제를 풀어보는 것이 학습에 중요.
- **학습 용도**: AI에게 "이 함수가 왜 이렇게 작동하는지 설명해줘"로 사용. 대안을 제안받아 검토.
- **주의**: 과제·연습 코드를 AI에게 풀게 하면 본인의 프로그래밍 근육이 자라지 않음.

**박사 3-4년차 (생산성 추구기)**:
- **적극 활용**: AI를 연구 속도 배가의 도구로. 보일러플레이트·표준 알고리즘·데이터 정리는 AI에 위임.
- **본인의 핵심 기여**: 새 알고리즘·핵심 실험 설계는 본인이 주도. AI는 구현 조수.
- **페어 프로그래밍**: Cursor Composer 또는 Claude Code agent로 복잡한 리팩토링을 빠르게.

**박사 5-7년차 (완성 단계)**:
- **AI와 공동 저자**: 박사 논문의 코드에서 AI 기여를 명시 (ch40의 IP·ch54의 분야 기준 참조).
- **검증과 테스트**: AI 생성 코드의 검증 능력이 핵심. 본인이 AI의 실수를 잡아낼 수 있어야 함.
- **후배 지도**: AI 페어 프로그래밍을 후배에게 가르침 (ch41 참조).

### AI 생성 코드 검증의 다섯 가지 기술

AI 페어 프로그래밍의 가장 큰 위험은 **잘못된 코드를 의심 없이 수용**하는 것. 검증 기술:

**기술 1: 이해하지 못한 코드는 쓰지 않기**.
AI가 제안한 코드의 모든 줄을 이해하지 못하면 쓰지 않음. "왜 이렇게 작동하는지" 설명할 수 없는 코드는 본인의 책임을 지지 못함.

**기술 2: 테스트 우선**.
AI가 함수를 생성하면 먼저 단위 테스트를 쓰고 (또는 AI에게 작성 요청) 실행. 경계 조건·예외 상황·큰 입력에 대한 테스트. Pass하지 않으면 코드 수정.

**기술 3: 핵심 알고리즘 손으로 추적**.
AI가 작성한 핵심 알고리즘을 종이·화이트보드에 손으로 추적. 입력 예시로 각 단계를 따라가 본인이 직접 결과를 계산. AI 답과 일치하는지 확인.

**기술 4: Edge case 탐색**.
AI가 보통 edge case (빈 리스트·null·음수·0·매우 큰 수)에 약함. 본인이 의도적으로 극단 입력을 시도.

**기술 5: 코드 리뷰의 일상화**.
매일 작성한 AI 생성 코드의 5-10분 리뷰 습관. 어제의 본인의 코드를 오늘의 본인이 검토하듯이.

### 학습 손실의 방지

AI 페어 프로그래밍의 가장 큰 함정은 **프로그래밍 능력의 암묵적 약화**. 매번 AI가 답하면 본인이 직접 푸는 연습이 줄어든다. 방어 기술:

**방어 1: AI-Free 시간 주 3-5시간**.
주 몇 시간은 AI 완전 차단. 본인만의 사고로 문제 해결. 어려움에 부딪혀도 AI를 켜지 않음.

**방어 2: 손으로 알고리즘 쓰기**.
새 알고리즘을 처음 배울 때 종이에 손으로 구현. AI가 도와주기 전에 본인의 이해가 먼저.

**방어 3: 코드 리뷰에서 AI 사용 명시**.
랩메이트와 코드 리뷰 시 "이 부분은 AI가 생성, 본인이 검증함" / "이 부분은 본인이 직접 작성"을 구분. 본인의 기여를 명확히.

**방어 4: 알고리즘 면접 연습**.
LeetCode·HackerRank 문제를 AI 없이 정기적으로 풀기. 기본 알고리즘 근육 유지. 박사 후 산업체 지원 시에도 유리.

**방어 5: 주석을 본인의 언어로**.
AI가 생성한 코드의 주석을 본인의 언어로 다시 쓰기. 본인의 이해를 명확히 문서화.

### 한국 박사의 AI 도구 접근성

- **학교 계정**: 많은 한국 대학이 GitHub 학생 프로그램을 안내하지만 활용 저조. 계정 확인·설정.
- **랩 라이선스**: 일부 대학·랩이 Cursor·GitHub Copilot의 라이선스를 단체 구매. 학과 사무실 확인.
- **한국어 지원**: 한국어 프롬프트·주석도 잘 처리되지만, 영어가 여전히 더 정확. 한국어로 먼저 쓰고 AI 요청 시 영어로 전환 연습.
- **결제 장벽**: 해외 결제 가능 카드 또는 PayPal 준비. 학교 연구비로 결제 가능한 경우도 있음.

### AI 페어 프로그래밍의 미래 전망

2024-2025에 "에이전트 모드" (AI가 여러 파일을 자동으로 편집·실행)가 빠르게 성장. Cursor Composer, Claude Code agent, GitHub Copilot Workspace. 박사 과정의 코딩이 "타이핑"에서 "지시와 검증"으로 점진적 전환 중.

박사 5-7년 동안 도구가 2-3번 크게 바뀔 것. 특정 도구에 갇히지 않고 **핵심 능력(문제 정의·알고리즘 이해·검증)**을 키우는 것이 가장 중요.

### 마지막 — AI 페어 프로그래밍은 박사 프로그래밍의 기본이 되었다

2025년 박사 과정에서 AI 페어 프로그래밍 도구를 쓰지 않는 것은 선택이 아니라 이미 생산성의 큰 손실이다. 그러나 무분별한 의존은 박사 학위의 핵심인 **깊은 이해와 독립적 사고**를 해친다. 단계별 전략(1-2년차 최소 의존 → 3-4년차 적극 활용 → 5-7년차 검증 중심)과 다섯 가지 검증 기술, 다섯 가지 학습 손실 방지 기술을 의식적으로 쓰면 AI가 박사 과정의 조수가 되고 대체자가 되지 않는다. 박사 과정의 진짜 산출물은 코드가 아니라 본인의 사고 능력이다.

> AI 페어 프로그래밍은 2025년 박사 과정의 기본이다. GitHub Copilot·Cursor·Claude Code의 3가지 모드(제안·대화·에이전트)를 단계별로 활용하되, 다섯 가지 검증 기술과 다섯 가지 학습 손실 방지 기술로 본인의 프로그래밍 근육을 지킨다. 박사 과정의 진짜 산출물은 코드가 아니라 본인의 사고 능력이며, AI는 조수일 뿐 대체자가 아니다.

## 박사 5년의 코드 진화 — 장기 코드 베이스의 관리 전략

박사 과정 전체에 걸쳐 본인이 쓰게 될 코드의 총량은 상상 이상이다. 실험 스크립트 수백 개, 전처리 파이프라인 수십 개, 분석 노트북 수백 개, 논문 그림 생성 코드 수십 개, 그리고 그것들을 뒷받침하는 개인 라이브러리. 2-3년이 지나면 **코드 베이스가 본인의 기억보다 크다**. 3년 전 본인이 쓴 코드를 본인이 이해 못하는 상황이 일상이 된다. 이 섹션은 박사 5년간 누적되는 코드 베이스를 관리하고 진화시키는 실전 전략을 다룬다. AI 도구 활용은 위 섹션에서 다뤘고, 여기서는 **박사 전체 기간의 코드 생애주기** 관점이다.

**박사 기간 코드 베이스의 3단계 진화.**

단계 1: **탐색기 (박사 1-2년차)**. 본인이 무슨 연구를 할지 모르는 시기. 여러 실험, 여러 데이터셋, 여러 방법을 시도. 코드는 대부분 일회성 노트북, 짧은 스크립트. 이 시기 코드의 특징: 구조화가 약하고, 하드코딩된 경로, 매직 넘버, 복붙이 많다. **이 단계에서 과도한 구조화는 시간 낭비다**. 일회성 탐색은 일회성 코드로 족하다.

단계 2: **정착기 (박사 2-3년차)**. 연구 방향이 잡히고 반복 실험이 시작. 비슷한 전처리, 비슷한 평가, 비슷한 시각화를 반복한다. 이 시기의 핵심 과제는 **"반복되는 것을 모듈화"**다. `utils.py`, `data_loader.py`, `eval_metrics.py` 같은 개인 라이브러리가 형성되기 시작. 노트북은 탐색, 스크립트는 재현, 모듈은 공통 함수로 역할 분리.

단계 3: **확장기 (박사 4-5년차)**. 여러 논문을 병행하고, 후배와 협업하고, 연구실 내 다른 프로젝트에 본인 코드를 빌려주는 시기. 이 시기 코드는 **본인만 쓰는 것이 아니다**. 문서화, 테스트, 버전 명시가 실질적으로 필요해진다. 이 단계에서 코드를 GitHub 리포지토리로 독립시키거나 pip 설치 가능한 내부 패키지로 만드는 것이 현실적 선택.

많은 박사가 1단계에서 2단계로 넘어가지 못하고 5년 내내 탐색기 스타일로 코드를 쓴다. 결과: 5년차에 코드 베이스가 무질서의 극한에 도달하고, 졸업 논문의 재현이 불가능해진다. 반대로 1단계에서 바로 3단계 수준의 구조를 강요하면 탐색 속도가 느려진다. **단계에 맞는 구조화 수준**이 핵심이다.

**연구 코드 재사용의 3계층.**

박사 기간의 모든 코드는 3가지 재사용 수준으로 분류된다. 본인 코드를 쓸 때 **이 줄은 어느 계층인가**를 의식하는 것이 장기 관리의 출발점이다:

계층 1, **일회성 코드 (Throwaway)**: 특정 실험, 특정 분석, 특정 그림을 위한 코드. 한 번 쓰고 버린다. 박사 코드의 60-70%가 이 계층. 이 계층은 노트북으로 쓰고, 결과를 논문에 반영한 후 `/archive` 폴더로 이동하거나 그대로 둔다. **재사용 의도가 없는 코드를 억지로 재사용 가능하게 만들려고 하지 마라.**

계층 2, **프로젝트 코드 (Project-scoped)**: 하나의 논문/프로젝트에 걸쳐 반복 사용. 예: 해당 논문의 전처리 파이프라인, 모델 학습 스크립트, 평가 루틴. 박사 코드의 20-30%. 이 계층은 명확한 폴더 구조, README, requirements를 가진다. 하지만 다른 프로젝트로 가져갈 필요는 없다.

계층 3, **재사용 가능 라이브러리 (Reusable)**: 여러 프로젝트에서 재사용되는 공통 코드. 본인이 박사 전체에서 반복적으로 쓰는 함수들 (`load_dataset`, `plot_metric_curve`, `compute_confidence_interval` 등). 박사 코드의 5-10%. 이 계층은 별도 리포지토리, 문서화, 테스트, pip install 가능한 패키지로 진화. 졸업 후에도 쓸 수 있는 본인의 "연구 자산".

박사 초년생의 실수: **계층 1과 계층 3을 혼동**. 일회성 분석 코드에 과도한 구조를 부여해서 시간을 낭비하거나, 반대로 재사용되는 공통 함수가 여러 프로젝트에 복붙되어 있어서 버그 수정이 악몽이 된다. 3계층을 의식적으로 구분하는 것이 박사 코드 관리의 기초.

**Dead Code의 관리 — 버려야 할 코드를 결정하는 기준.**

박사 3-4년차가 되면 코드 베이스에 "2년 전에 잠깐 썼다가 버려진 코드" 수백 파일이 쌓인다. Dead code. 이것을 어떻게 관리할 것인가:

- **완전 삭제 금지 원칙**: Git 기록은 남기되, 리포지토리 내에서 dead code를 `/archive/` 또는 `/deprecated/` 폴더로 이동. 완전 삭제는 금지 (나중에 필요할 수 있음).
- **Archive의 주기적 정리**: 6개월에 1번, `/archive/` 폴더를 검토. 1년 넘게 참조되지 않은 것은 별도 git branch로 옮기고 main에서 제거.
- **분기별 git log 리뷰**: 지난 3개월 커밋 로그를 훑으며 "어느 코드가 살아있고, 어느 코드가 죽었는지" 확인. 죽은 코드의 이동/정리.
- **논문 수록 코드 고정**: 논문에 수록한 결과를 낸 코드는 해당 논문의 git tag로 고정(`v1.0-paper-NeurIPS2024`). 그 시점 상태를 영구 보존.

Dead code를 방치하면 박사 5년차에 리포지토리 크기의 70%가 쓸모없는 코드가 된다. 새로 들어온 후배나 본인조차 "이 파일이 살아있는 건가 죽은 건가"를 판단 못해 모든 파일을 조심스럽게 다루게 된다. 결과: 모두가 불필요한 인지 부하를 진다.

**브랜치 전략 — 박사 연구의 Git 활용.**

소프트웨어 회사의 Git Flow는 박사 연구에 맞지 않는다. 박사는 다음과 같은 단순화된 전략이 실용적:

- **main**: 현재 "정식" 상태의 연구 코드. 논문 제출, 발표 준비 등에서 참조하는 기준.
- **exp/[실험명]**: 실험적 시도. 예: `exp/loss-reweighting`, `exp/new-dataset`. 실패하면 삭제, 성공하면 main에 병합.
- **paper/[논문명]**: 특정 논문 제출 상태의 스냅샷. 예: `paper/NeurIPS2024`. 심사 중 수정이 일어나도 원본은 고정.
- **archive**: 오래된 코드의 보관소. 실제로는 `git branch archive-2023`처럼 연도별로.

많은 박사가 main에서만 작업한다. 결과: 2년 전 논문의 재현을 하려는데 main이 크게 변해서 당시 코드를 찾을 수 없다. `paper/` 브랜치 하나만 만들어도 이 문제가 해결된다. **논문 제출 당일 `git tag`를 만드는 습관**이 박사 5년의 재현성을 구원한다.

**박사 졸업 직전의 코드 정리 — 6개월 계획.**

박사 5년차 후반, 졸업이 임박하면 코드 정리에 6개월을 투자하는 것이 장기적으로 본인에게 이득이다. 이 정리의 산출물은 졸업 후에도 수년간 본인의 연구 자산이 된다:

- **6개월 전**: 박사 기간 전체의 코드 베이스 현황 스냅샷. 어느 프로젝트가 살아있고 어느 것이 죽었는지 목록화.
- **5개월 전**: 재사용 가능 라이브러리(계층 3)를 별도 리포지토리로 분리. 명확한 README, pip 설치 가능하게 정리.
- **3개월 전**: 논문별 재현 환경 고정. 각 논문의 `paper/` 브랜치에서 Docker 이미지 또는 conda env 파일 생성.
- **2개월 전**: 공개 가능한 코드를 GitHub public으로 전환. Zenodo에 DOI 발급 신청.
- **1개월 전**: 연구실에 남길 코드를 후배에게 인수인계. 인수인계 문서 작성.
- **졸업 직후**: 개인 리포지토리로 이전 (연구실 GitHub → 본인 GitHub). 라이선스 정리.

이 정리를 생략하면 박사 졸업 후 2-3년 동안 "3년 전 내가 쓴 논문 코드를 재현할 수 없어서 논문 수정 요청을 거절해야 하는" 상황을 겪는다. 6개월의 정리가 10년의 재현성을 구원한다.

**오래된 코드를 이해하는 실전 — AI 시대의 코드 고고학.**

박사 3-4년차가 되면 "2년 전 본인이 쓴 코드를 본인이 이해 못하는" 상황이 일상이다. 2024년 이후 AI 도구가 이 문제를 획기적으로 바꾼다:

- **Claude Code / Cursor에 폴더 통째로 맥락 주기**: 오래된 프로젝트 폴더를 열고 "이 코드의 전체 구조와 핵심 플로우를 설명해"라고 요청. 5년 된 코드도 30분이면 구조 파악 가능.
- **함수 단위 의도 역추적**: 특정 함수가 무엇을 하는지 불분명할 때, AI에게 "이 함수의 역할과 호출 맥락을 추론해". 단, AI의 추론은 **가설**이므로 핵심 부분은 본인이 직접 검증.
- **주석 누락 보완**: 옛 코드에 주석이 없을 때, AI가 주석 초안을 생성. 본인이 검토/수정 후 커밋. 미래의 본인에게 주는 선물.
- **테스트 생성**: 오래된 함수가 여전히 정확히 작동하는지 확신이 없을 때, AI에게 테스트 케이스 생성 요청. 실제 테스트 실행으로 검증.

이 AI-assisted 코드 고고학은 2024년 이후 박사 후기의 코드 관리 비용을 크게 낮췄다. 하지만 **AI의 설명이 원저자의 의도와 다를 수 있다**. 결정적 부분은 반드시 본인의 옛 노트, 커밋 메시지, 당시 논문 초고로 교차 검증.

**마지막 — 박사 5년은 코드를 쌓는 게 아니라 진화시키는 시간이다.**

박사 과정을 시작할 때 대부분 생각하는 것은 "논문을 쓰는 것". 코드는 그 도구로 간주된다. 하지만 박사 5년을 지나면 알게 된다: **본인의 코드 베이스는 본인의 연구 정체성의 일부**다. 그것이 잘 관리되면 박사 후 5-10년간 본인 연구의 기반이 된다. 관리되지 않으면 졸업과 함께 죽은 자산이 된다. 3단계 진화를 의식하고, 3계층 재사용을 구분하고, dead code를 정리하고, 브랜치를 활용하고, 졸업 직전 6개월을 정리에 투자하는 것 — 이것이 박사 5년 코드의 진짜 관리법이다. AI 도구는 이 관리 비용을 낮추지만, 본질적 설계 결정은 여전히 본인이 해야 한다.

> 박사 5년의 코드는 탐색·정착·확장의 3단계로 진화. 모든 코드는 일회성·프로젝트·재사용의 3계층으로 분류. Dead code는 archive 폴더로 이동하고 주기적 정리. 단순한 브랜치 전략(main·exp·paper·archive)과 논문 제출일 git tag 습관이 5년의 재현성을 구원. 졸업 6개월 전부터 코드 정리 계획을 실행. AI 도구가 코드 고고학의 비용을 낮추지만 본질적 설계는 본인 몫. 코드 베이스는 박사 연구 정체성의 일부이며 졸업 후 10년의 연구 기반이 된다.

## GPU 프로그래밍의 박사 실전 — CUDA·메모리·multi-GPU 관리

2020년 이후 박사 연구의 상당수가 GPU에 의존한다. AI·물리·화학·생물·의학·공학 등에서 GPU 코드가 필수. 하지만 많은 박사가 "GPU는 PyTorch가 알아서"라고 생각하고 깊이 들어가지 않는다. 결과: GPU 메모리 부족(OOM)에 당황, multi-GPU 실패, 성능 병목 진단 불가. 이 섹션은 박사가 GPU 프로그래밍을 체계적으로 다루는 실전을 다룬다. ch15의 다른 섹션이 **일반 프로그래밍**이었다면, 이 섹션은 **GPU 특화**다.

**GPU 프로그래밍의 3가지 계층.**

박사의 GPU 사용은 3가지 계층으로 구분:

**계층 1, 프레임워크 수준 (PyTorch/JAX)**: `model.cuda()`, `tensor.to("cuda")`. 대부분 박사의 90%의 작업. 블랙박스적.

**계층 2, 저수준 연산 (CUDA 내장 함수)**: `torch.cuda.synchronize()`, custom CUDA kernels via `torch.compile`. 성능 최적화 필요 시.

**계층 3, 네이티브 CUDA (C++/CUDA)**: 완전 커스텀 커널. 박사 후 5%만 필요. 극한 성능·특수 연산.

**박사의 권장**: 계층 1이 기본, 필요 시 계층 2까지. 계층 3은 대부분 박사에게 불필요.

**GPU 메모리 관리 — 박사의 핵심 기술.**

OOM (Out of Memory) 에러는 박사의 일상. 해결 기법:

**기법 1, 배치 크기 조정**: 가장 기본. GPU 메모리가 부족하면 batch_size 축소. 학습이 느리지만 가능.

**기법 2, Gradient Accumulation**: 작은 배치를 여러 번, 그라디언트 누적. 큰 배치 효과 + 적은 메모리.

**기법 3, Mixed Precision (FP16/BF16)**: float32 → float16. 메모리 반, 성능 비슷. `torch.cuda.amp`, `accelerate`.

**기법 4, Gradient Checkpointing**: 중간 activation을 저장 안 하고 재계산. 메모리 감소 + 계산 증가. PyTorch의 `checkpoint`.

**기법 5, 모델 Offloading**: CPU/디스크로 모델 일부 이동. 속도 느리지만 큰 모델 실행. DeepSpeed ZeRO, FSDP.

**기법 6, Quantization**: 모델 가중치를 FP32 → INT8/INT4. 추론 시 주로. bitsandbytes, GPTQ.

이 기법들을 **조합**하는 것이 박사의 실전.

**GPU 메모리 프로파일링 도구.**

문제를 측정해야 해결 가능:

- **`nvidia-smi`**: GPU 사용량·메모리 실시간. 가장 기본.
- **`torch.cuda.memory_allocated()`**: PyTorch 할당 메모리.
- **`torch.cuda.max_memory_allocated()`**: 최대 사용량.
- **PyTorch Profiler**: 상세한 메모리·시간 분석. `torch.profiler`.
- **NVIDIA Nsight Systems**: 전문가 수준. 커널별 분석.
- **`py-spy`**: Python 레벨 프로파일링.
- **Weights & Biases**: 실험 모니터링 + GPU 메트릭.

박사 2-3년차에 이 도구들을 능숙하게.

**Multi-GPU 전략 — 4가지 병렬화.**

단일 GPU로 부족할 때:

**전략 1, Data Parallelism (DP/DDP)**: 같은 모델을 여러 GPU에 복제, 데이터 분할. 가장 흔함. `torch.nn.DataParallel` (구식), `DistributedDataParallel` (표준).

**전략 2, Model Parallelism**: 모델을 여러 GPU에 분할. 매우 큰 모델. 구현 복잡. Megatron-LM, DeepSpeed.

**전략 3, Pipeline Parallelism**: 모델의 layer를 순차 GPU에 배치. 파이프라인. GPipe, PipeDream.

**전략 4, Tensor Parallelism**: 단일 layer의 연산을 여러 GPU에 분할. 큰 LLM에 필수. Megatron-LM.

**박사의 현실**: 대부분 DDP면 충분. 모델 parallelism은 연구실의 인프라·코드에 의존.

**DistributedDataParallel (DDP)의 실전.**

단일 → multi-GPU 전환의 가장 흔한 경로:

```python
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

# 초기화
dist.init_process_group(backend="nccl")

# 모델을 DDP로 wrap
model = model.to(local_rank)
model = DDP(model, device_ids=[local_rank])

# DataLoader의 sampler 교체
from torch.utils.data.distributed import DistributedSampler
sampler = DistributedSampler(dataset)
```

**주의**:
- `torchrun` 또는 `accelerate launch`로 실행.
- Random seed 일관성.
- Checkpoint 저장은 rank 0만.
- `ddp_sync_batchnorm` 고려.

**박사의 DDP 학습 곡선**: 처음에는 복잡하지만 1주일 투자로 평생의 기술.

**GPU 디버깅의 독특한 어려움.**

GPU 코드는 CPU보다 디버깅 어려움:

- **비동기 실행**: GPU 커널이 비동기. 에러가 나중 시점에 나타남.
- **CUDA OOM의 추적**: 어느 텐서가 메모리 과점하는지 찾기.
- **NaN/Inf**: Mixed precision에서 자주. `torch.autograd.detect_anomaly()`.
- **Deadlock**: DDP의 rank 간 동기화 문제.
- **Race condition**: 비결정적 결과.

**디버깅 전략**:
- `CUDA_LAUNCH_BLOCKING=1`: 동기 실행 강제. 에러 위치 정확.
- `torch.cuda.synchronize()`: 수동 동기화.
- Smaller batch size로 먼저 테스트.
- Single GPU로 먼저 디버깅 후 DDP.

한국 박사의 GPU 접근:

## Docker·Container — 박사 연구 환경의 재현성

박사 논문의 결과는 **특정 환경**에서만 재현된다. Python 3.9, PyTorch 2.0, CUDA 11.8, Ubuntu 20.04. 다른 버전에서는 오류 또는 다른 결과. 6개월 후 본인이 돌아왔을 때, 또는 다른 사람이 재현할 때, 이 환경이 **재현 불가**. Docker와 컨테이너 기술이 이 문제의 표준 해법. 이 섹션은 박사가 Docker를 연구 재현성에 활용하는 실전을 다룬다. ch15의 다른 섹션(AI 페어 프로그래밍·GPU·5년 코드 진화)이 **코딩**이었다면, 이 섹션은 **환경**이다.

**Docker가 박사에게 주는 5가지 가치.**

**가치 1, 환경 재현성**: Dockerfile 하나로 어디서든 동일 환경.

**가치 2, 의존성 격리**: 프로젝트 A와 B의 Python 버전 충돌 없음.

**가치 3, 공동 저자와의 환경 공유**: "저는 이 컨테이너에서 돌아가요" 말로 끝.

**가치 4, 논문 재현의 표준**: 2020년대 많은 논문이 Docker 이미지 공개.

**가치 5, 학위논문 10년 후 재현**: 박사 졸업 10년 후에도 실험 재현 가능.

이 5가지가 Docker를 **박사 연구 재현성의 핵심 도구**로 만듦.

**Docker의 3가지 기본 개념.**

**개념 1, 이미지 (Image)**:
- 소프트웨어의 "설계도".
- Dockerfile로 정의.
- 읽기 전용.
- 예: `pytorch/pytorch:2.0.0-cuda11.8-cudnn8-runtime`.

**개념 2, 컨테이너 (Container)**:
- 이미지의 "실행 인스턴스".
- 격리된 프로세스.
- 삭제·재시작 가능.

**개념 3, 레지스트리 (Registry)**:
- 이미지 저장소.
- Docker Hub, NVIDIA NGC, GitHub Container Registry.

**박사의 Dockerfile 템플릿.**

ML 박사의 전형적 Dockerfile:

```dockerfile
# Base image: CUDA 지원 PyTorch
FROM pytorch/pytorch:2.1.0-cuda12.1-cudnn8-runtime

# 환경 설정
ENV PYTHONUNBUFFERED=1
ENV DEBIAN_FRONTEND=noninteractive

# 시스템 패키지
RUN apt-get update && apt-get install -y \
    git curl wget \
    vim htop tmux \
    libgl1-mesa-glx libglib2.0-0 \
    && rm -rf /var/lib/apt/lists/*

# Python 패키지
COPY requirements.txt /tmp/
RUN pip install --no-cache-dir -r /tmp/requirements.txt

# 작업 디렉토리
WORKDIR /workspace

# 기본 명령어
CMD ["python"]
```

**설명**:
- `FROM`: base image.
- `RUN`: 빌드 시 실행 명령.
- `COPY`: 호스트 파일을 이미지로.
- `WORKDIR`: 작업 디렉토리.
- `CMD`: 컨테이너 시작 시 기본 명령.

**Docker 빌드·실행의 기본 명령.**

**빌드**:
```bash
docker build -t my-research:v1 .
```

**실행 (CPU)**:
```bash
docker run -it --rm \
    -v $(pwd):/workspace \
    my-research:v1 bash
```

**실행 (GPU)**:
```bash
docker run -it --rm \
    --gpus all \
    -v $(pwd):/workspace \
    my-research:v1 bash
```

**백그라운드 실행**:
```bash
docker run -d --name my-exp \
    --gpus all \
    -v $(pwd):/workspace \
    my-research:v1 python train.py
```

**Docker Compose — 복잡한 환경.**

여러 서비스의 동시 실행:

```yaml
# docker-compose.yml
version: '3.8'
services:
  jupyter:
    image: my-research:v1
    ports:
      - "8888:8888"
    volumes:
      - ./:/workspace
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]

  tensorboard:
    image: my-research:v1
    command: tensorboard --logdir=/workspace/logs --host=0.0.0.0
    ports:
      - "6006:6006"
    volumes:
      - ./:/workspace
```

**실행**:
```bash
docker-compose up -d
```

**박사의 전형적 사용**: Jupyter + Tensorboard + 훈련 스크립트 동시.

**박사의 Docker 사용 5가지 시나리오.**

**시나리오 1, 개인 연구 환경**:
- 본인 노트북/워크스테이션에 Docker 설치.
- 프로젝트별 컨테이너.
- 환경 충돌 방지.

**시나리오 2, 연구실 서버 공유**:
- 여러 박사가 같은 서버 사용.
- 각자 컨테이너로 격리.
- 관리자 권한 없이 Python 버전 선택.

**시나리오 3, 클라우드 실험**:
- AWS EC2, GCP에 컨테이너 배포.
- 로컬 개발 → 클라우드 실행의 원활한 전환.

**시나리오 4, 논문 재현성 패키지**:
- 논문 공개 시 Dockerfile + 코드.

**시나리오 5, 학위논문 동결 상태**:
- 학위논문 심사 시 Docker 이미지 저장.
- 10년 후에도 재현 가능.

**Docker의 5가지 흔한 실수.**

**실수 1, 이미지의 무한 비대**: 불필요한 패키지 설치. 수 GB 이미지. 해결: multi-stage build, `--no-cache-dir`.

**실수 2, Hardcoded 비밀**: API 키·비밀번호를 Dockerfile에. 해결: 환경 변수, Docker secrets.

**실수 3, Cache 활용 부족**: `pip install`을 매번 다시. 해결: 의존성을 별도 layer.

**실수 4, root 사용자**: 컨테이너 내부 root. 보안 위험. 해결: `USER` 지시어로 비root 사용자.

**실수 5, Latest 태그 의존**: `FROM ubuntu:latest`의 불확실성. 해결: 버전 명시 `ubuntu:22.04`.

**Docker 대안 — 2024+ 도구.**

**Podman**: Docker의 오픈소스 대안. 데몬 없음, root 불필요.

**Singularity/Apptainer**: HPC 환경의 표준. 연구실 GPU 클러스터에 흔함.

**Nix**: 함수형 패키지 관리. 극한의 재현성.

**conda/mamba**: Python 중심 환경. 가볍지만 Docker만큼 격리 없음.

**uv**: 2024년 등장. 빠른 Python 환경.
